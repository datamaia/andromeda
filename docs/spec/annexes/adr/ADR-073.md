### ADR-073 — Third-party tool delivery: plugins and MCP servers only, no in-process loading

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Tool Runtime, Extension SDK, Plugin Runtime, MCP Runtime, Sandbox Engine
- Related requirements: FR-TOOL-001, FR-SDK-001; FR-PLUG-001 and FR-MCP-001 (Volume 6 chapters 05/08)

#### Context

The Tool entity's `origin` enum is closed: `builtin`, `plugin`, `mcp` (Volume 2, chapter 04).
ADR-009 fixes subprocess JSON-RPC plugins after rejecting Go's in-process `plugin` package,
and ADR-010 fixes the official MCP SDK. The Extension SDK ships a Go tool-authoring kit
(Volume 3, chapter 06). What remains open is whether the SDK's in-process `ToolPort`
authoring surface is itself a third-party delivery channel — that is, whether Andromeda ever
loads third-party tool code into its own process.

#### Problem

Decide the sanctioned delivery channels for third-party tools, and the role of the SDK's
in-process authoring surface, such that the process-boundary security model of ADR-009/ADR-021
cannot be bypassed by an "easier" loading path.

#### Forces and constraints

- Fault and permission isolation for third-party code requires a process boundary the Sandbox
  Engine can police (ADR-021); in-process code has ambient authority by construction.
- Go has no viable dynamic in-process loading (ADR-009 rejected `plugin` outright).
- Contributors adding *built-in* tools to the monorepo need a first-class in-process kit.
- One tool contract for all origins (Principle 4) must survive whatever the channel is.

#### Alternatives considered

1. **SDK-loaded in-process third-party tools** (recompile Andromeda with extra tools, or a
   sideloading convention) — rejected for distribution: custom binaries fork the update and
   signing chain (Volume 14) and place unaudited code inside the permission boundary.
2. **Plugins only, no MCP tool bridging** — rejected: MCP is the ecosystem's largest tool
   source and first-class support is mandated (PRD-007; MCP client keystone).
3. **Scripting runtime embedded in Andromeda** (Lua/JS tools in-process) — rejected for this
   horizon: a second execution semantics inside the permission boundary, with its own sandbox
   problem; WASM remains the studied v2 candidate under ADR-009's review conditions.

#### Decision

Third-party tools reach a running Andromeda through exactly two channels: **plugins** over the
Andromeda Runtime Protocol (origin `plugin`, ADR-009) and **MCP servers** (origin `mcp`,
ADR-010). Both execute outside Andromeda's process and register tools that the Tool Runtime
mediates identically to built-ins. The Extension SDK's in-process tool kit is the authoring
surface for `builtin` tools (in-tree contributions compiled into the product) and for the
tool-serving side of plugins; it is not a runtime loading mechanism. Andromeda MUST NOT load
third-party executable code into its own process in any committed phase; WASM-based
in-process isolation remains a v2 candidate, PENDING VALIDATION per ADR-009.

#### Rationale

The decision makes the security architecture and the extension story the same story: the only
ways in are the ways the Sandbox Engine and Permission Manager already govern. Contributors
keep a zero-boilerplate in-process kit — but its output ships through review into the product
binary, which is precisely the trust distinction between `builtin` and the other origins that
INV-TOOL-03 makes visible.

#### Positive consequences

- One enforcement story: every third-party tool call crosses a supervised process boundary.
- The `origin` enum stays closed and truthful; trust classification maps to real boundaries.
- SDK effort concentrates on one kit reused by built-ins and plugin authors (SM-02, SM-03).

#### Negative consequences

- In-process performance is unavailable to third parties; chatty third-party tools pay
  process-boundary latency (accepted in ADR-009; coarse-grained API design mitigates).
- Users wanting a private tool must run a plugin process or MCP server rather than dropping a
  file in a directory.

#### Risks

- Pressure for a "quick local tool" path could produce unofficial sideloading forks;
  mitigated by keeping plugin scaffolding cost low (SM-03 ≤ 8 person-hours) and documenting
  the single-file-plugin pattern.
- The two channels drift in capability; mitigated by the shared tool contract conformance
  suite that both bridges must pass (FR-TOOL-001).

#### Reversal plan

Adding a third channel (e.g., WASM in v2) is additive: a new `origin` value via the Volume 2
change procedure, a new host component, and the same Tool Runtime mediation. No existing
contract changes; this ADR would be superseded, not unwound.

#### Review conditions

- The ADR-009 WASM spike validates a workable host-function surface (v2 gate).
- Field evidence that plugin scaffolding cost is blocking a significant class of tool authors.
