### ADR-023 — Concurrency model: goroutines with context cancellation, errgroup structure, supervised tasks, bounded pools

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Task Scheduler, Runtime, Agent Engine, Execution Engine, Event Bus, Tool Runtime
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation.

#### Context

Andromeda's Runtime is inherently concurrent: an interactive TUI (ADR-006) stays responsive
while the Agent Engine streams provider responses, the Execution Engine runs plan tasks, tools
and plugins (ADR-009) execute as subprocesses, the Indexing Engine updates indexes, and the
Event Bus (ADR-012) fans out events. Cancellation is first-class product behavior: users
interrupt runs, timeouts fire (exit code 8 per ADR-016), and permission denials abort in-flight
work. The glossary names a Task Scheduler that "schedules and supervises concurrent tasks with
cancellation and backpressure"; this ADR fixes the model behind it.

#### Problem

Which concurrency model and conventions govern all concurrent work in Andromeda, such that
cancellation reliably reaches every in-flight operation, goroutine leaks and orphaned work are
structurally prevented, and load on slow consumers degrades gracefully instead of exhausting
memory?

#### Forces and constraints

- Go (ADR-001) provides goroutines, channels, and `context.Context` as native primitives; any
  model fights or embraces them.
- Cancellation MUST propagate end-to-end: user interrupt → run → tasks → tool subprocesses →
  provider HTTP requests (ADR-019), with prompt resource release.
- Unsupervised goroutines are how leaks, deadlocks-after-shutdown, and lost panics happen; a
  large agent runtime cannot audit ad-hoc spawns.
- Producers (event emitters, providers streaming tokens) can outpace consumers (TUI rendering,
  disk writes); unbounded buffering is a memory-exhaustion path.
- Contributor accessibility: conventions must be recognizable idiomatic Go, checkable in review
  and lint (ADR-018).

#### Alternatives considered

1. **Actor frameworks** (actor-model libraries with mailboxes and supervision trees) —
   rejected: they impose a foreign programming model over Go's native primitives, add a
   framework dependency at the heart of the Runtime, complicate debugging (stack traces cross
   mailbox boundaries), and their supervision benefits are achievable with errgroup plus the
   Task Scheduler at far lower cost.
2. **Callback-based async** (completion callbacks, promise-like chaining) — rejected: inverts
   control flow against the grain of Go, makes cancellation and error propagation manual at
   every hop, and produces the exact leak and orphan classes this ADR exists to prevent.
3. **Unstructured goroutines with per-case discipline** — rejected: relies on every author
   remembering rules the structure should enforce; unauditable at this codebase's scale.

#### Decision

- Concurrency uses **goroutines with `context.Context` cancellation propagated through every
  engine**: every blocking or long-running function in the Runtime accepts a context, and every
  I/O path (provider HTTP, subprocesses, database, Event Bus delivery) honors it.
- **`errgroup` is the structured-concurrency primitive**: related goroutines run in groups
  whose lifetime is bounded by a parent context, with first-error propagation and joined
  shutdown.
- **No naked goroutines**: outside the Task Scheduler's own internals, components MUST NOT
  spawn free-running goroutines; all concurrent work is submitted to and supervised by the
  Task Scheduler, which owns panic capture, lifecycle observability, and cancellation wiring.
- The **Event Bus is channel-based** (ADR-012) with defined delivery semantics; consumers
  receive through bounded subscriptions.
- **Worker pools are bounded with backpressure**: every pool declares its size and queue bound;
  when full, producers block or shed per a declared policy — unbounded queues are prohibited.

#### Rationale

This is Go's native model applied with structure: contexts and errgroup are standard,
well-understood tools, so the conventions read as ordinary idiomatic code rather than a
framework. Centralizing spawning in the Task Scheduler turns "did we leak a goroutine?" from a
whole-codebase question into a component invariant, and gives one place to attach the
observability (ADR-011) and cancellation guarantees the specification makes user-visible.
Bounded-with-backpressure is the only defensible default for a long-running interactive
process where producers include a token-streaming network peer.

#### Positive consequences

- Interrupts and timeouts behave uniformly everywhere; exit code 8 semantics are implementable
  honestly.
- Goroutine lifecycles are observable and leak-checkable in tests (Volume 13 can assert
  zero-leak on shutdown).
- Backpressure converts overload into visible latency instead of invisible memory growth.
- Panics in concurrent work are captured and reported through one path with run correlation.

#### Negative consequences

- Threading contexts through every API adds ceremony to signatures.
- The Task Scheduler becomes core infrastructure whose own correctness is critical and needs
  rigorous testing.
- Bounded queues require choosing sizes and shedding policies per pool — tuning work that
  unbounded designs defer (until they fail).

#### Risks

- Context misuse (storing values instead of cancellation, dropping contexts at boundaries);
  mitigation: lint rules (ADR-018) and review checklist.
- Backpressure deadlocks if two bounded stages block on each other cyclically; mitigation:
  the Event Bus and Task Scheduler define an acyclic flow topology, and shedding policies break
  cycles.
- "No naked goroutines" erosion over time; mitigation: lint-detectable pattern plus periodic
  audit in quality review.

#### Reversal plan

The conventions are enforced at review/lint level and localized in the Task Scheduler
implementation. Relaxing or replacing the model (for example, adopting a future language-level
structured concurrency feature) proceeds component-by-component behind the Task Scheduler's
submission interface without breaking the engines that submit work.

#### Review conditions

- If the Go toolchain ships first-class structured concurrency or scheduler observability that
  subsumes errgroup patterns, re-evaluate the primitives.
- If profiling (Volume 12) shows scheduler supervision overhead exceeding its latency budget,
  revisit the no-naked-goroutines mechanics (not the principle).
- Review pool bounds and shedding policies against real workload data at each phase transition.
