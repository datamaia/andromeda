### ADR-138 — Logging sinks: per-process JSONL files with built-in size/age rotation; no third-party rotation dependency

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Logging, Platform Abstraction Layer (PAL)
- Related requirements: FR-OBS-002, FR-OBS-003; ADR-011, ADR-022

#### Context

ADR-011 fixed `log/slog` with a JSON handler as the logging API and prohibited third-party
logging frameworks; ADR-022 fixed directory resolution. Still open: the concrete sink
layout (how many files, named how, shared how across concurrent processes) and how rotation
and retention are implemented, given that several `andromeda` processes (multiple
workspaces, headless instances) may log concurrently on one machine.

#### Problem

Choose a file layout and rotation mechanism that needs no cross-process coordination on the
hot path, keeps the footprint bounded and configurable, and adds no dependency beyond the
standard library.

#### Forces and constraints

- SQLite-style single-writer contention is unacceptable on the logging hot path; ADR-011
  budgets logging as human-scale but concurrent processes must not serialize on a lock.
- Retention must be deterministic and verifiable (RISK-OBS-001).
- ADR-011 rule 5 prohibits third-party logging frameworks; pulling a rotation library in
  through the back door would violate its spirit.
- Log files are diagnostics, not the durable record (events/traces/audit are) — losing a
  rotated file to retention is acceptable by design.

#### Alternatives considered

1. **Single shared log file with file locking** — cross-process lock contention on every
   append, interleaving risk on crash, and PAL File Locking on the hottest path. Rejected.
2. **Logging into SQLite (a `logs` table)** — queryable, but puts high-frequency appends
   into the workspace database's write path, competes with the run record stream, and makes
   logs unreadable without tooling. Rejected: logs must be inspectable with standard text
   tools.
3. **Third-party rotation library (e.g., a lumberjack-style writer)** — mature, but adds a
   dependency for ~200 lines of well-understood logic and contradicts the ADR-011 posture.
   Rejected.
4. **OS-delegated rotation (logrotate, launchd)** — not portable across the platform
   matrix, invisible to Andromeda's own retention accounting, unavailable on user machines
   by default. Rejected.

#### Decision

1. One JSONL file **per process instance**, named `andromeda-<UTC start instant>-<pid>.jsonl`,
   under the PAL-resolved log directory (`<state_dir>/andromeda/logs/`, ADR-022).
2. **Built-in size rotation** (`logging.file.max_size_mb`, default 32) within the process;
   records are never split across files.
3. **Startup/rotation retention sweep**: delete oldest-first beyond
   `logging.file.max_files` (10) or `logging.file.max_age_days` (30), excluding live
   processes' active files.
4. **Degradation**: sink failure falls back to stderr with drop counters (E-OBS-007),
   never failing the logged operation.
5. Implementation uses the standard library only.

#### Rationale

Per-process files eliminate coordination by construction — the only cross-process touch
point is the sweep, which is idempotent, liveness-aware, and off the hot path. Built-in
rotation keeps the dependency graph flat where the logic is small and testable, and makes
retention Andromeda-observable (events, counters, doctor output) instead of delegated and
invisible.

#### Positive consequences

- Zero hot-path locking across processes; trivially correct concurrent logging.
- Footprint bound is arithmetic: files × size × age window.
- Logs remain plain JSONL, inspectable with standard text tooling offline.

#### Negative consequences

- Diagnosing a multi-process incident means reading several files (correlation IDs and
  timestamps make merging mechanical).
- Short-lived processes create many small files within the retention window.
- Rotation logic is ours to test (property and fault-injection tests per Volume 13).

#### Risks

- Sweep racing a starting process whose file looks stale: mitigated by PID liveness checks
  and a minimum-age guard before deletion eligibility.
- Clock skew corrupting age-based retention: future-dated files are treated as age zero and
  logged at `WARN`.

#### Reversal plan

The sink is one writer behind the slog handler chain; swapping to a shared-file, SQLite, or
library-based sink changes one component without touching call sites, record schema, or
redaction. Retention configuration keys are sink-agnostic.

#### Review conditions

- Volume 12 soak tests show file-count or sweep costs mattering at realistic session
  volumes.
- A future daemon/multi-process architecture changes the per-process assumption.
- slog gains a stdlib rotation-capable handler upstream — adopt via amendment if it
  simplifies rule 5.
