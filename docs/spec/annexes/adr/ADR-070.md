### ADR-070 — Tool identity: dotted names, reserved namespaces, and collision rejection

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Tool Runtime, Extension SDK, Plugin Runtime, MCP Runtime, CLI, TUI
- Related requirements: FR-TOOL-001, FR-TOOL-003; INV-TOOL-01 (Volume 2)

#### Context

Every tool — built-in, plugin-provided, or MCP-provided — is addressed by `name` by agents,
users, configuration selectors, and Agent Profile tool policies (Volume 2, chapter 04). The
registry's natural key is `(name, version, origin, origin_ref)`, and INV-TOOL-01 forbids
storing ambiguous rows, but leaves collision *handling* to Volume 6. Models see tool names in
their tool declarations, so names are also a prompt-injection and typosquatting surface
(Volume 9 threat model).

#### Problem

Define one identity grammar for tool names and one deterministic rule for what happens when
two origins offer the same name, such that resolution is unambiguous, built-in names cannot be
impersonated, and users can predict which tool runs.

#### Forces and constraints

- Agents resolve by bare `name`; records pin the exact registered row (Volume 2).
- Third-party tools arrive from sources Andromeda does not control (plugins, MCP servers);
  their declared names are untrusted input.
- A model-facing name set that changes shape per installation harms prompt portability.
- Silent shadowing of a trusted tool by a look-alike is a supply-chain attack primitive.

#### Alternatives considered

1. **Flat names, last-registration-wins** — rejected: silent shadowing; a malicious MCP server
   could replace `fs.read` semantics without any visible change.
2. **Origin-prefixed names everywhere** (e.g., `mcp:server/tool`) — rejected as the primary
   form: verbose in model prompts, breaks prompt portability across installations, and pushes
   trust bookkeeping into string parsing.
3. **Automatic suffixing on collision** (`name-2`) — rejected: nondeterministic identity across
   installations and restarts; records would pin names users never chose.

#### Decision

Tool names use the grammar `namespace.action`: two or more lowercase segments matching
`^[a-z][a-z0-9_]*(\.[a-z][a-z0-9_]*)+$`, at most 64 characters total. The first segment is the
namespace. The namespaces of the built-in catalog (`fs`, `git`, `terminal`, `process`, `http`,
`browser`, `docker`, `kubernetes`, `sqlite`, `github`, `gitlab`, `jira`, `notion`, `slack`,
`linear`) plus `andromeda` are **reserved**: only `builtin` origin may register into them.
Across origins, a registration whose name equals an already-registered enabled tool of a
*different* origin (or different `origin_ref`) is **rejected**, not shadowed; the rejection is
recorded and surfaced (`tool.registration.rejected`, E-TOOL-002). Within one origin, multiple
versions coexist and resolution picks the highest enabled version. Users MAY declare explicit
aliases in `[tools]` configuration to expose a rejected tool under a different name; aliasing
is the only collision escape hatch and is always user-initiated.

#### Rationale

Rejection is the only deterministic, auditable outcome that keeps INV-TOOL-01 true without
inventing identity on the author's behalf. Reserved namespaces make the built-in name set a
stable, non-impersonable vocabulary that prompts and documentation can rely on, and the
user-declared alias keeps legitimate coexistence possible with explicit consent instead of
automatic magic.

#### Positive consequences

- Resolution is a pure function of registry contents; no install-order dependence for enabled
  tools.
- Built-in names cannot be claimed or replaced by any third-party source.
- Collisions become visible registration events users can act on, instead of silent behavior
  changes.

#### Negative consequences

- Two independently useful third-party tools with the same natural name cannot both be enabled
  without a user aliasing step.
- Reserved namespaces constrain plugin authors who would prefer to extend `fs.*` et al.
  directly.

#### Risks

- Alias configuration adds a second name for one tool; mitigated by records always pinning the
  registered canonical name and the alias being visible in registry listings.
- Typosquatting of *unreserved* namespaces remains possible; mitigated by origin and trust
  visibility (INV-TOOL-03) and Volume 9 trust policy.

#### Reversal plan

The grammar and reservation list are registry validation rules, not persisted structure.
Loosening them (e.g., permitting namespaced third-party extensions under a vetting policy) is
an additive change gated by a superseding ADR; tightening after release would require a
deprecation window per ADR-015 because names are a public contract surface (SM-20).

#### Review conditions

- A future marketplace (Volume 6 chapter 09) introduces publisher identities that could carry
  namespace ownership.
- Recurring real-world collisions show aliasing is too much friction.
- The MCP ecosystem standardizes server-side tool namespacing that Andromeda should map onto.
