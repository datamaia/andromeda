### ADR-020 — Embeddings stored in SQLite with exact in-process cosine similarity for MVP; ANN acceleration deferred

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Indexing Engine, Persistence Layer, Memory Manager, Context Manager
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation.

#### Context

Andromeda's semantic indexing and retrieval (Volume 7) turns workspace content and memory
records into embeddings and answers similarity queries for the Context Manager and Memory
Manager. ADR-007 fixes the Persistence Layer on `modernc.org/sqlite`, a cgo-free pure-Go SQLite
driver, precisely to keep the single-binary, cross-compilable, local-first distribution story
intact. The corpus in scope is a developer workspace — source files, documentation, and memory
records — not a web-scale document store.

#### Problem

Where do embedding vectors live and how are nearest-neighbor queries answered, given that the
Persistence Layer is already SQLite, the pure-Go constraint excludes CGO extensions by default,
and workspace-scale corpora are orders of magnitude smaller than the workloads approximate
nearest neighbor (ANN) indexes were built for?

#### Forces and constraints

- Local-first: retrieval MUST work offline with no external service.
- ADR-007's pure-Go, cgo-free constraint governs everything loaded into the SQLite process
  space; native extensions conflict with it.
- One storage engine for state (sessions, memory, indexes, audit) keeps backup, migration, and
  integrity handling (exit code 9 semantics per ADR-016) in one place.
- Exactness matters at small scale: approximate indexes trade recall for speed — a trade that
  only pays once corpora are large.
- Scale assumption, stated explicitly: the design target is **at most 100,000 indexed chunks
  per workspace**. This is a recorded technical assumption, not a verified fact about all users.

#### Alternatives considered

1. **Dedicated external vector database** (server process or embedded native library) —
   rejected: adds an operational dependency or a native-code dependency, breaking local-first
   simplicity and the single-binary distribution for a scale of data that does not need it.
2. **`sqlite-vec` (or similar SQLite ANN extension) from day one** — deferred, not adopted:
   loading it may require CGO or native extension loading, which conflicts with ADR-007's
   pure-Go driver constraint. Its adoption is **PENDING VALIDATION** (see Risks and Review
   conditions).
3. **In-process ANN library (pure Go) now** — rejected for MVP: index build/maintenance
   complexity and approximate recall are costs paid immediately for a benefit that only
   materializes beyond the stated scale target.

#### Decision

- Embedding vectors are stored **in SQLite via the Persistence Layer (ADR-007)**, as vector
  blobs in normal tables alongside chunk metadata, covered by the same migration and integrity
  machinery as all other state.
- Similarity queries for MVP use **exact cosine similarity computed in-process**: candidate
  vectors are read from SQLite and scored in Go; results are exact, not approximate.
- Brute-force scoring is the accepted MVP algorithm under the stated assumption of at most
  100,000 chunks per workspace; the Indexing Engine records corpus size so the assumption is
  observable.
- ANN acceleration (for example `sqlite-vec`) is explicitly deferred and PENDING VALIDATION.

#### Rationale

At workspace scale, exact brute-force scoring is simple, correct, and has zero recall loss;
100,000 vector comparisons per query is an in-memory workload a Go loop handles without an
index structure. Every alternative buys speed Andromeda does not yet need at the price of
things it definitely needs: pure-Go builds (ADR-007), one storage engine, and no operational
dependencies. Stating the 100k-chunk assumption in the decision makes the boundary of validity
explicit and testable rather than implicit.

#### Positive consequences

- No new storage or native dependencies; backups, migrations, and integrity checks remain
  unified in the Persistence Layer.
- Exact results: retrieval quality issues can never be caused by ANN recall loss in MVP.
- Cross-compilation and single-binary distribution (ADR-013) are unaffected.
- The Context Manager and Memory Manager consume one retrieval interface that can later be
  re-backed without contract change.

#### Negative consequences

- Query latency grows linearly with corpus size; beyond the stated target it will degrade
  noticeably.
- Vector blobs enlarge the SQLite database file; retention and re-embedding policies (Volume 7)
  must manage growth.
- Reading candidate vectors per query costs I/O that an ANN index with quantization would avoid.

#### Risks

- The ≤100k-chunk assumption may not hold for monorepo users; mitigation: corpus-size metrics
  plus the review condition below.
- **PENDING VALIDATION**: `sqlite-vec` (or any SQLite ANN extension) may require CGO or native
  extension loading, conflicting with ADR-007's pure-Go constraint; whether a compliant loading
  path exists must be validated before any adoption.
- Embedding dimensionality growth (larger models) multiplies both storage and scan cost;
  mitigation: dimension is recorded per index and bounded by Volume 7 configuration.

#### Reversal plan

The retrieval interface (query in, ranked chunk references out) hides the scoring
implementation. Introducing ANN — whether a pure-Go index, a validated `sqlite-vec` path, or a
sidecar — replaces the scorer behind that interface plus a one-time index build; stored vectors
remain the source of truth, so no data migration is destructive and rollback is deleting the
derived index.

#### Review conditions

- When any real workspace exceeds 100,000 indexed chunks, or p95 retrieval latency exceeds the
  Volume 12 budget at smaller sizes: open the ANN evaluation.
- Resolve the PENDING VALIDATION on `sqlite-vec` CGO requirements before any ANN adoption
  decision; if CGO is required, evaluate pure-Go ANN alternatives instead.
- Re-examine storage growth policy when Volume 7 fixes retention rules for embeddings.
