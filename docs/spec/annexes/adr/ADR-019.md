### ADR-019 — Provider adapters use stdlib net/http against documented public APIs; official SDKs adoptable per adapter

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Provider Layer, Authentication Layer, Core Domain
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation.

#### Context

Andromeda is vendor-agnostic by principle: the glossary defines this as "no provider-specific
logic outside that provider's adapter". The Provider Layer defines a common provider contract
(Volume 5) with per-provider adapters for cloud services (OpenAI, Anthropic, OpenAI-compatible
endpoints, and others) and local servers (Ollama). Adapters speak HTTP to documented public
APIs, handle streaming, and integrate with the Authentication Layer. Facts web-verified
2026-07-11: `github.com/openai/openai-go` is the official OpenAI SDK, GA, module major `/v3`,
v3.41.1 (2026-07-08), Apache-2.0, Go 1.22+ — three majors in roughly two years;
`github.com/anthropics/anthropic-sdk-go` is the official Anthropic SDK, GA, v1.57.0
(2026-07-10), MIT, Go 1.24+; Ollama exposes a documented REST API on `localhost:11434`
(`/api/chat`, `/api/generate`, `/api/embed`), and importing its official Go client
`github.com/ollama/ollama/api` (v0.31.2, MIT) pulls in the entire ollama module.

#### Problem

Do provider adapters talk to provider HTTP APIs directly with the standard library, or through
each vendor's official SDK — balancing dependency weight, churn exposure, and streaming
control, without leaking vendor types into the core?

#### Forces and constraints

- Vendor-agnostic architecture: provider SDK types MUST NOT appear outside the owning adapter;
  the Provider Layer contract is the only surface the Runtime sees.
- Only official, documented public APIs are used (same legality constraint as ADR-014); an SDK
  is a convenience over such an API, never a source of undocumented capability.
- SDK churn is real: openai-go shipped 3 majors in about 2 years — every major is forced work.
- Local-first: the Ollama adapter is on the no-network core path and its dependency weight
  matters for binary size and build hygiene.
- Streaming, timeout, retry, and observability behavior must follow Volume 5 and ADR-011
  conventions uniformly across adapters; heterogeneous SDK behaviors work against that.

#### Alternatives considered

1. **SDK-first for all adapters** — rejected as the baseline: it imports each vendor's HTTP
   stack, retry logic, and type surface wholesale, multiplies major-version migration work
   (three openai-go majors in two years, verified), and hinders cross-adapter uniformity
   because each SDK owns retries, tracing, and errors differently.
2. **A third-party multi-provider abstraction library** — rejected: it would place the pivotal
   vendor-agnostic boundary of the product in an external project's hands, contradicting the
   Provider Layer being a first-class, specified contract of Andromeda itself.
3. **`github.com/ollama/ollama/api` for the Ollama adapter** — rejected for dependency weight:
   importing the client pulls the entire ollama module into the build (verified); a thin client
   against three documented endpoints does not justify that.

#### Decision

- The **baseline for every provider adapter is stdlib `net/http` against the provider's
  documented public API**, including streaming response handling, under the Provider Layer
  contract with uniform retry, timeout, tracing, and redaction behavior.
- **Official Go SDKs MAY be adopted per adapter** after an evaluation at that adapter's
  implementation time, when the SDK demonstrably reduces maintenance (for example, complex auth
  flows or streaming edge cases). Candidates verified today: openai-go v3 (pin the major —
  churn risk noted above) and anthropic-sdk-go v1.57.0. **Each per-adapter SDK adoption is
  PENDING VALIDATION at that adapter's implementation.**
- The **Ollama adapter uses a thin hand-rolled HTTP client** against the documented REST API on
  `localhost:11434` (`/api/chat`, `/api/generate`, `/api/embed`).
- SDK types, when adopted, are confined to the adapter package; the adapter translates to
  Provider Layer contract types at its boundary.

#### Rationale

net/http as the baseline keeps every adapter uniform where Andromeda needs uniformity
(observability, retries, cancellation per ADR-023) and keeps dependency weight proportional.
Documented public APIs are the contract that matters; official SDKs wrap those same APIs, so
adopting one is an implementation optimization judged adapter-by-adapter with real evidence —
hence per-adapter PENDING VALIDATION rather than a blanket rule in either direction.

#### Positive consequences

- Uniform streaming, retry, timeout, and telemetry behavior across all adapters.
- Dependency surface and binary size stay proportional; the local-first Ollama path stays lean.
- SDK major-version churn affects at most one adapter at a time, behind a stable contract.
- The Provider Layer contract remains the single, specifiable source of truth.

#### Negative consequences

- Andromeda maintains request/response types and streaming decoders per adapter that an SDK
  would provide for free.
- Provider API changes must be tracked from provider documentation and changelogs directly.
- Two implementation styles (raw HTTP vs SDK-backed) may coexist across adapters, requiring
  clear adapter-authoring guidance in Volume 5.

#### Risks

- Hand-rolled clients can drift from provider behavior changes; mitigation: contract tests per
  adapter with recorded fixtures, run in CI.
- PENDING VALIDATION per-adapter SDK decisions could stall or be made inconsistently;
  mitigation: Volume 5 defines the evaluation checklist (maintenance cost, streaming fidelity,
  dependency weight, license) and each outcome is recorded in the decision register.
- openai-go's major cadence could strand a pinned adapter; mitigation: major pins with scheduled
  review.

#### Reversal plan

Adapters are leaf packages behind the Provider Layer contract. Swapping an adapter between raw
HTTP and an official SDK changes one package and its tests; no Runtime, Core Domain, or other
adapter changes. A full reversal to SDK-first would proceed adapter-by-adapter under the same
contract tests.

#### Review conditions

- At each adapter's implementation: resolve its PENDING VALIDATION SDK evaluation and record the
  outcome.
- Re-evaluate when openai-go ships a new major or anthropic-sdk-go changes majors.
- If Ollama's documented REST API changes incompatibly or its client module becomes importable
  without whole-module weight, revisit the thin-client choice.
