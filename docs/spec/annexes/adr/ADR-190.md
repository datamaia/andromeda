### ADR-190 — Distribution artifact set, naming scheme, and package formats

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Updater, Package Manager, CLI
- Related requirements: FR-REL-001, FR-REL-002, FR-REL-003, FR-REL-004

#### Context

ADR-013 fixes the release tooling (goreleaser in GitHub Actions, cosign keyless signatures,
syft SBOMs, SLSA provenance, a Homebrew tap) and ADR-001/ADR-007 guarantee CGO-free
cross-compiled binaries. Volume 1 fixes the platform tiers: macOS arm64 and Linux
x86_64/arm64 are Tier 1 at MVP; macOS Intel is Tier 2 "MVP when viable" and PENDING
VALIDATION. What remains undecided is the concrete artifact set a release publishes, how
artifacts are named so scripts and the shell installer can address them without lookup
tables, which Linux package formats ship, and whether a macOS universal binary is produced.

#### Problem

Fix the exact, stable artifact inventory and naming grammar for every release, such that the
Updater, the shell installer, mirrors, and humans can locate and verify artifacts
mechanically, and decide the universal-macOS and Linux-package questions explicitly.

#### Forces and constraints

- Artifact names are a public contract: the Updater, install scripts, mirrors, and CI
  verification jobs parse them; renames break every consumer at once.
- `uname -s` / `uname -m` are the tokens a portable shell installer has natively; arch
  aliases (`amd64` vs `x86_64`) must be decided once.
- A universal macOS binary doubles artifact size and requires an Intel build — whose Tier 2
  viability is itself PENDING VALIDATION (Volume 1 platform scope).
- goreleaser produces deb/rpm/apk via nfpm at near-zero marginal cost (ADR-013), but
  *hosting* apt/dnf/apk repositories is recurring infrastructure the project has not
  committed to.
- Air-gapped installation needs the artifact set to be self-describing: checksums,
  signatures, and SBOMs must be downloadable files, not service lookups.

#### Alternatives considered

1. **goreleaser default naming (`amd64`, `Darwin` capitalization variants)** — rejected:
   defaults drift across goreleaser majors and force the installer to maintain alias maps;
   fixing the grammar in configuration costs nothing and freezes the contract.
2. **Universal macOS binary as the only macOS artifact** — rejected: it makes every macOS
   download pay for an Intel slice whose build viability is unvalidated, and Homebrew serves
   per-arch bottles anyway.
3. **No Linux packages (archives only)** — rejected: deb/rpm/apk are produced by the
   existing pipeline at negligible cost and materially widen reachable installations.
4. **Hosted apt/dnf/apk repositories at MVP** — rejected for now: recurring signing and
   hosting infrastructure without a validated need; package *files* attached to releases
   deliver most of the value. Hosted repositories are classified Future.

#### Decision

1. Every release publishes, for each platform in the matrix: a binary archive
   `andromeda_<version>_<os>_<arch>.tar.gz` with `<os>` ∈ {`darwin`, `linux`} and `<arch>` ∈
   {`arm64`, `x86_64`} (uname-style tokens, lowercase), containing the `andromeda` binary,
   `LICENSE`, and third-party notices.
2. The MVP matrix is: `darwin_arm64`, `linux_x86_64`, `linux_arm64` (Tier 1).
   `darwin_x86_64` and `andromeda_<version>_darwin_universal.tar.gz` (lipo universal binary)
   are added if and when macOS Intel Tier 2 builds are validated — PENDING VALIDATION,
   tracked in the Volume 14 register.
3. Linux package files `andromeda_<version>_linux_<arch>.deb` / `.rpm` / `.apk` are produced
   via goreleaser's nfpm integration and attached to the release. Hosted package
   repositories are Future.
4. Integrity metadata per release: `andromeda_<version>_checksums.txt` (SHA-256, one line
   per artifact); cosign keyless signature of the checksums file
   (`andromeda_<version>_checksums.txt.sig` + `.pem`); one SBOM per archive
   (`<artifact>.sbom.spdx.json`, SPDX JSON via syft); SLSA provenance attestation in in-toto
   format covering all artifacts. Verifying any artifact means: verify the checksums-file
   signature, then verify the artifact's SHA-256 against it.
5. `<version>` in file names carries no `v` prefix; the release tag carries the prefix per
   Volume 11 tag conventions. The naming grammar is a public contract under ADR-015: changes
   are breaking changes.

#### Rationale

One grammar, chosen once, makes every consumer — Updater, installer, mirror builder, human
with `sha256sum` — address artifacts by construction instead of by lookup. Signing the
checksums file (instead of every artifact individually) is the smallest verifiable unit that
still covers the full set transitively, and matches the tooling's native flow. Deferring the
universal binary keeps the unvalidated Intel dependency out of the Tier 1 path while leaving
a named slot for it.

#### Positive consequences

- The shell installer is a portable script with zero platform tables: `uname` output maps
  directly to an artifact name.
- Air-gapped verification needs only files published with the release.
- Linux package users get native install/uninstall semantics at no pipeline cost.
- The artifact inventory is auditable: a release is complete if and only if the enumerated
  set exists.

#### Negative consequences

- Freezing the grammar means even cosmetic renames are major-version events.
- Two macOS artifacts (arm64 now, universal later) create a transition question when the
  universal artifact lands.
- Package files without hosted repositories do not auto-update through apt/dnf; those
  installs update through the package file or the Updater's deference rules (FR-REL-009).

#### Risks

- goreleaser major upgrades changing template semantics could silently alter names;
  mitigated by a CI release-audit check comparing the artifact inventory against this ADR's
  grammar (FR-REL-001 acceptance criteria).
- The PENDING VALIDATION Intel/universal decision could linger; the Volume 14 register entry
  binds it to the Volume 1 platform-scope question so both resolve together.

#### Reversal plan

The grammar is configuration, not code: goreleaser name templates plus the installer's
mapping function. Changing it is a coordinated major release: publish both names for one
transition release, update installer and Updater matchers, then drop the old names. SBOM or
signature format changes follow the same dual-publish transition.

#### Review conditions

- macOS Intel Tier 2 validation resolves (either way) — activate or delete the
  `darwin_x86_64`/universal rows and resolve the register entry.
- Windows phase entry (v2 candidate): extend the grammar (`windows_x86_64`, archive format,
  Authenticode) by a superseding ADR per ADR-013's review conditions.
- Hosted package repositories are reclassified from Future by the change procedure.
- goreleaser v3 or a signing-format change per ADR-013's review conditions.
