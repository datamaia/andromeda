### ADR-004 — Branching Model: Trunk-Based Development

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: None (development-process decision; applies to the `andromeda` repository, automated per Volume 11)
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation

#### Context

Andromeda is developed openly in the `andromeda` monorepo (ADR-003) by human engineers and AI implementation agents. The traceability chain (Volume 11) binds every change to an issue, a branch, a PR, required checks, and a release. Releases are automated from tags (ADR-013). The branching model determines integration frequency, conflict cost, release mechanics, and how much process contributors must learn. The project brief explicitly forbids adopting Git Flow by default; the model MUST be argued.

#### Problem

Choose the branching and integration model for the repository: how long branches live, where integration happens, how releases are cut and stabilized, and what is required to merge.

#### Forces and constraints

- Contributions arrive as many small, spec-traceable changes; AI agents in particular produce short-lived, narrowly scoped branches and benefit from immediate integration feedback.
- `main` MUST always build, pass all required checks, and be releasable (a prerequisite for tag-driven release automation).
- Pre-1.0, Andromeda ships a single supported line — there is no long-term parallel maintenance of multiple majors — but release candidates need a stabilization surface, and patch releases need a backport path.
- Long-lived divergence multiplies merge conflicts in a monorepo where changes cross component boundaries.
- Direct pushes to `main` MUST be impossible; every change flows through a PR with required checks (branch protection).

#### Alternatives considered

| Property | Trunk-based (selected) | GitHub Flow | Git Flow |
|---|---|---|---|
| Long-lived branches | `main` + temporary `release/*` | `main` only | `main`, `develop`, `release/*`, `hotfix/*` |
| Integration point | `main`, continuously | `main`, continuously | `develop`, deferred to release |
| Branch lifetime | Short-lived (days) | Short-lived | Feature branches often long-lived |
| Release mechanism | Tag on `main`; `release/*` only for stabilization/backports | Tag on `main`; no stabilization surface | Merge `release/*` into `main` + back-merge into `develop` |
| Ceremony per change | One PR | One PR | PR + dual-branch bookkeeping |
| Fit for tag-driven automation | Direct | Direct | Indirect (two integration branches) |

- **Git Flow — evaluated and rejected.** Git Flow's value proposition is parallel maintenance of scheduled, boxed releases across a persistent `develop`/`main` split. Andromeda has the opposite profile: continuous integration of small changes into a single supported line. The permanent `develop` branch creates a second source of truth, delays integration (conflicts surface at release time instead of merge time), doubles back-merge bookkeeping, and confuses the traceability chain, which assumes one integration target. The brief's prohibition on defaulting to Git Flow is therefore confirmed on the merits, not just on mandate.
- **GitHub Flow — viable but incomplete.** It is trunk-based development minus release branches. It offers no surface for stabilizing a release candidate while `main` keeps moving, and no branch to receive backported fixes for a published minor while the next one is in flight. Since Beta onward requires exactly that (release hardening, patch releases), GitHub Flow alone is insufficient.

#### Decision

The repository MUST use trunk-based development:

1. `main` is the only permanent branch and MUST be protected: PRs required, required checks green, no direct pushes, linear history.
2. Work happens on short-lived branches named per the Volume 11 convention and SHOULD merge within 3 working days of creation; branches exceeding 10 working days MUST be split or closed.
3. Incomplete features land behind phase gates or feature flags rather than living on long branches.
4. `release/vX.Y` branches are cut from `main` only to stabilize a release candidate or to receive cherry-picked fixes for a published release; they MUST NOT receive feature work and are deleted or frozen when the line is out of support.
5. Releases are tags (on `main` or on a `release/*` branch), which trigger the ADR-013 pipeline.

#### Rationale

Trunk-based development minimizes integration latency — the dominant cost in a monorepo with cross-cutting changes — while the narrowly scoped `release/*` mechanism supplies the one thing GitHub Flow lacks. It keeps process ceremony at one PR per change, which matches how AI agents contribute and keeps the Volume 11 automation simple: one integration target, one branch-naming rule, one tag-to-release mapping.

#### Positive consequences

- Conflicts surface at merge time, small and fresh, instead of at release time.
- `main` is always releasable, enabling on-demand releases and simple revert-first incident response.
- Minimal model to teach contributors; branch protection encodes most of it mechanically.
- Stabilization and backports have a defined home without a permanent second branch.

#### Negative consequences

- Requires investment in feature flags/phase gating for work that spans multiple PRs.
- A red `main` blocks everyone; check reliability becomes critical infrastructure.
- Cherry-pick-based backporting is manual and can miss fixes without release-management discipline.

#### Risks

- **Flaky required checks** would throttle all integration. Mitigation: Volume 13 quality gates include a flakiness budget; persistently flaky tests are quarantined by policy rather than ignored.
- **Flag debt** from feature gating accumulating in the codebase. Mitigation: each flag is registered with an owner and a removal condition; stale flags are release-blocking findings in the Volume 13 audit.

#### Reversal plan

All candidate models share the PR-into-protected-branch mechanic, so migration is configuration, not history surgery. If sustained parallel maintenance of multiple major versions ever becomes necessary (e.g., an LTS line), the model is extended with long-lived `release/vX` maintenance branches — a superset of the current rules — via a superseding ADR. Reverting to GitHub Flow (dropping release branches) requires only deleting rule 4.

#### Review conditions

- The project commits to maintaining more than one major version simultaneously.
- Median time-to-merge or conflict rate degrades past the Volume 13 process targets for two consecutive months.
- Release stabilization repeatedly requires more than one active `release/*` branch, indicating the cadence has outgrown the model.
