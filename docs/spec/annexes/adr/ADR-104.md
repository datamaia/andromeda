### ADR-104 — Extension-contributed commands mount under the reserved `x` namespace

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: CLI, Plugin Runtime, Package Manager
- Related requirements: FR-CLI-004, FR-CLI-001, PRD-007; Principle 6 (commands are a named extension surface)

#### Context

Principle 6 and PRD-007 name *commands* among Andromeda's extension surfaces, and Volume 3
(chapter 06) records plugin/package-contributed commands as the CLI's extensibility path.
Plugins run as subprocesses speaking the Andromeda Runtime Protocol (ADR-009); a plugin
that declares a command surface needs a place in the `andromeda` command tree. Meanwhile
ADR-100 closes the top-level namespace so the built-in grammar can grow across releases
without colliding with anything third parties shipped in the interim. Tools in the
ecosystem solve this differently: git merges external `git-foo` binaries into its top-level
namespace; kubectl mounts plugins by prefix resolution; gh scopes extensions as
`gh extension-name` at top level with registration.

#### Problem

Where do third-party commands appear in the command tree, how are collisions prevented —
among extensions and against future built-ins — and which conventions (output, exit codes,
permissions) bind extension commands so the surface stays coherent?

#### Forces and constraints

- The built-in tree must keep growing: any namespace extensions share with built-ins turns
  every future built-in command into a potential breaking collision.
- Two extensions must be co-installable even when both pick the same command name.
- Extension commands must not be able to impersonate built-ins (`andromeda auth …` from a
  plugin is a phishing vector for credentials).
- Discovery must be offline and deterministic: the tree is assembled from local
  registrations (Package Manager rows), never from network lookup at parse time.
- Extension commands are still Andromeda surface: FR-CLI-005/006 conventions (global flags,
  `--json`, exit codes) and full permission mediation apply (PRD-005).

#### Alternatives considered

1. **Reserved `x` group: `andromeda x <extension> <command> [...]` (selected).**
2. **Top-level merging (git-style)** — rejected: collides with future built-ins, enables
   built-in shadowing/impersonation, and makes `--help` a mixture of trusted and
   third-party content with no visual boundary.
3. **Per-extension top level with registration-time collision arbitration (gh-style)** —
   rejected: first-installed-wins arbitration creates install-order-dependent CLIs; two
   machines with the same extensions installed in different order would have different
   grammars, breaking script portability.
4. **No CLI extension surface (defer to plugin tools only)** — rejected: contradicts
   Principle 6's explicit listing of commands as an extension surface and PRD-007.

#### Decision

1. Extension-contributed commands are reachable exclusively as
   **`andromeda x <extension-name> <command> [args…]`**, where `<extension-name>` is the
   installed extension's registered name (unique per the Package Manager's registry,
   Volume 6). The `x` group is reserved by FR-CLI-001; nothing else mounts there.
2. Command declarations arrive through the extension's manifest (Volume 6 owns the format):
   command path under the extension, help text, argument/flag declarations, and required
   permissions. The CLI assembles the `x` subtree from local Extension registry rows at
   startup; unavailable plugins list with a diagnostic and fail with E-CLI-004 semantics
   when invoked.
3. Extension commands inherit the CLI conventions of chapter 02 mechanically: global flags
   are parsed by Andromeda (not forwarded), `--json` output uses the FR-CLI-006 envelope
   with the extension's payload under `data`, exit codes map through the ADR-016 scheme,
   and every side effect passes the Permission Manager under the extension's declared and
   granted permissions — identical to tool citizenship (Principle 4).
4. Help output labels every `x` command with its extension name, origin, and trust level
   (the Volume 6 vocabulary), so third-party surface is always visually distinguishable
   from built-in surface.

#### Rationale

A single reserved mount point makes the collision policy structural instead of procedural:
extensions cannot collide with built-ins because they never share a namespace, and two
extensions cannot collide with each other because their registered names — already unique —
are part of the path. The cost (three tokens before the extension's own command) buys
script portability across machines, an unambiguous trust boundary in help and completion,
and a top-level namespace the product can keep evolving under SM-20 without auditing the
extension ecosystem first.

#### Positive consequences

- Zero collision classes by construction; no arbitration machinery to specify or test.
- Trust boundary is visible in the invocation itself (`x` reads as "external").
- Built-in grammar evolution never breaks installed extensions' invocations.
- Completion and help for extension commands come from registered manifests offline.

#### Negative consequences

- Longer invocations than git/gh-style mounting; frequently used extension commands pay
  three prefix tokens (shell aliases are the user-side remedy).
- Extension authors cannot offer top-level ergonomics even for flagship integrations; a
  future first-party promotion path would require a superseding ADR.

#### Risks

- The `x` group becomes a dumping ground of low-quality commands harming perceived product
  quality; mitigation: trust/origin labeling plus Volume 6 packaging policy; the boundary
  itself communicates provenance.
- Extensions requesting argv passthrough of global flags (conflicting with rule 3);
  mitigation: manifest schema has no mechanism to claim global flag names — structurally
  impossible rather than reviewed-away.

#### Reversal plan

The `x` subtree is assembled by one registrar component reading Extension rows. Moving to a
different mounting policy (e.g., opt-in top-level promotion for verified extensions) adds a
second registrar path and an ADR; existing `x` paths would remain valid as DEPRECATED
aliases through a deprecation window, so no invocation breaks retroactively.

#### Review conditions

- A first-party-quality extension ecosystem emerges and three-token prefixes are a
  demonstrated adoption barrier (measured in extension usage telemetry under consent).
- Volume 6's manifest evolution adds command capabilities (e.g., interactive TUI panels)
  that need mounting semantics this ADR does not cover.
- The marketplace phase (post-v1) introduces verified publishers — revisit promotion.
