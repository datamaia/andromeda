### ADR-072 — Automatic tool retries bound to declared idempotency

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Tool Runtime, Execution Engine, Agent Engine
- Related requirements: FR-TOOL-001, FR-TOOL-008; INV-TINV-06 (Volume 2)

#### Context

Tool invocations fail for transient reasons (timeouts against a busy origin, connection resets
on network tools) and for permanent ones (invalid input, denied permission). The Tool
Invocation entity supports retry chains (`retry_of_id`, acyclic, within one run — INV-TINV-06),
and the ADR-016 error envelope carries per-error retryability. What is undecided is *who* may
retry *what* automatically: an automatic re-execution of a non-idempotent side-effecting tool
(a second `git.exec` commit, a second Slack message) is a correctness and safety defect, not a
resilience feature.

#### Problem

Fix the rule by which the Tool Runtime performs automatic retries, versus leaving the decision
to the agent loop or the user, without ever duplicating side effects.

#### Forces and constraints

- Safety precedence: duplicated side effects are worse than surfaced failures.
- SM-10 reliability counts structured errors as acceptable outcomes — retries are an
  optimization, not a requirement, for that metric.
- The permission context of an invocation is already granted; a retry must not silently widen
  or re-consume consent.
- Agents can reason about failures; workflows have their own retry policies (Volume 4).

#### Alternatives considered

1. **No automatic retries at all** — safe but wasteful: schema-valid, declared-idempotent
   read-only tools (e.g., `fs.read` hitting a transient lock) would round-trip through the
   model for a mechanical re-attempt, spending tokens and latency for nothing.
2. **Runtime retries everything retryable** — rejected: retryability of the *error* says
   nothing about the safety of re-running the *tool*; duplicated non-idempotent effects are
   unacceptable.
3. **Per-invocation agent-requested retries only** — workable but pushes a mechanical policy
   into every agent prompt; keeps no uniform audit shape.

#### Decision

The Tool Runtime performs an automatic retry of a failed or timed-out invocation only when
**all** of the following hold: the tool's declaration states `idempotent = true`; the
terminal error's envelope is retryable; the attempt count is below both the tool's declared
`max_auto_retries` and the `tools.max_auto_retries` configuration cap (default 2); and the
invocation was not `denied` or `cancelled`. Each retry is a **new** Tool Invocation row
chained via `retry_of_id`, reusing the still-valid permission grants of the original without
re-prompting; if any required grant has expired or been revoked, the retry re-enters
permission evaluation instead of inheriting. Non-idempotent tools are never retried
automatically; their failures return to the caller (agent, workflow, or user), which may
issue a new invocation deliberately. Backoff between automatic retries is exponential with
jitter, starting at 250 ms, doubling per attempt.

#### Rationale

Idempotency is the tool author's declaration of re-execution safety, and the error envelope is
the failure's declaration of transience; automatic retry is justified exactly at their
intersection and nowhere else. New rows per attempt keep the audit trail complete (every
attempt, its permissions, and its outcome are individually recorded), matching PRD-006 rather
than hiding attempts inside a counter.

#### Positive consequences

- Transient failures of read-only tools self-heal without model round-trips.
- Zero risk of runtime-duplicated side effects on non-idempotent tools.
- Retry chains are first-class records: SM-13 traceability covers every attempt.

#### Negative consequences

- Honest failures of non-idempotent tools always cost an agent turn to recover.
- Tools that are idempotent in fact but undeclared get no automatic resilience until their
  declaration is fixed.

#### Risks

- Authors may declare `idempotent = true` optimistically; mitigated by the conformance suite's
  double-execution test (FR-SDK-001) and trust policy consequences for violators (Volume 9).
- Retry storms against a degraded origin; mitigated by the attempt caps and backoff, and by
  origin availability gating (E-TOOL-010) failing fast when the origin is down.

#### Reversal plan

The rule is runtime policy over unchanged records. Disabling automatic retries entirely is a
configuration default change (`tools.max_auto_retries = 0`); replacing the rule with a
policy-engine-driven one would extend, not break, the declaration contract.

#### Review conditions

- SM-10 field data shows a material class of transient failures on non-idempotent tools that
  agents recover from poorly.
- Volume 4 workflow retry policies and this rule interact in ways that double-retry;
  consolidation must verify single ownership of each retry decision.
