### ADR-025 — Git engine: shell out to system git behind an encapsulated adapter, minimum version pinned

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Git Engine, Platform Abstraction Layer (PAL), Workspace Engine
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation.

#### Context

Git operations are central to Andromeda's workflows: status, diff, commit, branch, and worktree
operations back file-change tracking, patch review, and the GitHub-based development process
(Volume 11). The glossary defines the Git Engine as "encapsulated Git operations", and the
target environment is a developer machine where git is effectively universal. Repositories in
scope include large monorepos where fidelity (diffs, renames, attributes, hooks, sparse
checkout) and performance are both user-visible. Candidate strategies in Go: shell out to the
system `git` binary, the pure-Go `go-git` library, or CGO bindings to libgit2.

#### Problem

How does the Git Engine execute Git operations so that behavior matches what developers observe
from their own `git` invocations — including on large repositories, worktrees, and partial
clones — without native-code dependencies or reimplementing Git semantics, while keeping the
door open to in-process optimizations for cheap read-only queries?

#### Forces and constraints

- Fidelity: Andromeda's view of a repository MUST match `git`'s view; a diff or status that
  disagrees with the user's own git output destroys trust and corrupts the patch workflow.
- Features in active use — worktrees, sparse checkout, filters, hooks, `.gitattributes`,
  credential helpers — must behave exactly as the user configured them.
- ADR-007's cgo-free stance and single-binary distribution (ADR-013) weigh against native
  bindings.
- The target audience (developers) has git installed; but version behavior varies, so a floor
  must be pinned and checked.
- Subprocess execution cost is nonzero; hot read-only paths (for example, frequent status
  polling for the TUI) may later justify an in-process fast path.

#### Alternatives considered

1. **go-git for everything** — rejected: pure-Go and dependency-light, but carries fidelity and
   performance risks on large repositories, and has gaps in worktree and sparse-checkout
   support relative to system git. Users judge Andromeda's output against their own git binary,
   so any divergence is a defect Andromeda would own.
2. **libgit2 bindings** — rejected: a CGO dependency contradicting the cgo-free build posture
   (ADR-007 context) and complicating cross-compilation and release (ADR-013); it also tracks
   upstream git semantics with its own lag.
3. **go-git as default with shell-out for gaps** — rejected: the default would carry the
   fidelity risk; inverting it (shell-out default, go-git as validated optimization) keeps
   correctness as the baseline. That inversion is the decision below.

#### Decision

- The Git Engine **shells out to the system `git` binary** through an **encapsulated adapter
  inside the PAL boundary**: one component owns process invocation, environment control
  (ADR-021 filtering applies), parsing of machine-readable output formats, timeout and
  cancellation (ADR-023), and error mapping to the ADR-016 envelope.
- The **minimum supported git version is pinned at 2.40**; the Git Engine detects the installed
  version at startup and refuses operation below the floor with a defined configuration error.
  The pin carries a review condition below.
- **go-git MAY be used for selected read-only operations** (for example, ref inspection on hot
  paths); **each such operation is PENDING VALIDATION** individually — validated for output
  equivalence against system git before adoption, reverted if equivalence fails.
- No component outside the Git Engine invokes git or parses git output.

#### Rationale

System git is the semantic authority: shelling out delegates correctness for the entire feature
matrix (worktrees, sparse checkout, filters, hooks, helper-based auth) to the exact binary and
configuration the user already trusts, at zero reimplementation cost and with no native
dependencies. The adapter contains the classic costs of shelling out — parsing, versions,
environments — in one tested place. Pinning 2.40 gives a concrete floor for the machine-readable
output formats the adapter parses, instead of an undefined "recent git". The go-git escape
hatch is narrow and evidence-gated because its value is performance, not correctness.

#### Positive consequences

- Git behavior matches user expectations by construction, including user config, hooks, and
  credential helpers.
- No CGO, no vendored Git; release and cross-compilation posture (ADR-013) is unaffected.
- New git features become available as users upgrade git, without library waits.
- One adapter centralizes subprocess policy: env filtering, timeouts, cancellation, error
  mapping.

#### Negative consequences

- A runtime prerequisite exists: system git at or above 2.40; installation documentation and
  startup diagnostics must handle its absence.
- Output parsing couples to git's machine-readable formats; their evolution requires adapter
  maintenance.
- Per-operation subprocess overhead on hot paths until validated go-git fast paths land.

#### Risks

- Version drift: users below the 2.40 floor get hard failures; mitigation: precise startup
  diagnostic with remediation guidance, and a floor kept as low as the parsed formats allow.
- Parsing output where no stable format exists; mitigation: the adapter prefers porcelain and
  NUL-terminated formats and treats anything else as internal risk needing per-version tests.
- **PENDING VALIDATION**: each go-git read-only fast path must prove output equivalence with
  system git on representative repositories (including worktrees and sparse checkouts) before
  adoption; divergence at validation time blocks that path.

#### Reversal plan

The Git Engine's interface is operation-level (status, diff, commit, branch, worktree, …), not
transport-level. Reversing toward go-git or future bindings replaces adapter internals per
operation, gated by the same equivalence tests; reversing a validated go-git fast path is a
fallback to the subprocess implementation.

#### Review conditions

- Review the 2.40 minimum-version pin at each phase transition (Beta, v1) against git versions
  shipped by supported platforms and the formats the adapter parses.
- Resolve each go-git PENDING VALIDATION individually at its proposed adoption, recording
  equivalence evidence in the decision register.
- If a supported platform stops shipping git, re-evaluate the bundling question.
