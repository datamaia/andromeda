### ADR-010 — MCP Support: Official modelcontextprotocol/go-sdk

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: MCP Runtime, Tool Runtime
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation

#### Context

Model Context Protocol (MCP) support is a first-class integration surface for Andromeda: the MCP Runtime manages client connections to external MCP servers whose tools, resources, and prompts are surfaced through the Tool Runtime (Volume 6). MCP is an evolving standard with dated specification revisions; wire-level correctness, revision negotiation, and timely tracking of the spec are the core engineering burdens.

Web-verified facts (2026-07-11): `github.com/modelcontextprotocol/go-sdk` is the **official** Go SDK, hosted in the `modelcontextprotocol` organization and developed in collaboration with Google. It is **stable at v1.6.1 (2026-05-22)**, licensed Apache-2.0 (new code; MIT legacy), and supports specification revisions 2025-11-25, 2025-06-18, 2025-03-26, and 2024-11-05. Client-side OAuth support is still experimental. A release candidate of the next specification revision is expected around 2026-07-28.

#### Problem

Decide how Andromeda implements MCP: adopt the official Go SDK, adopt a community implementation, or implement the protocol in-house — and how to handle the moving parts (spec revisions, experimental OAuth).

#### Forces and constraints

- Vendor-agnosticism makes broad MCP interoperability a product objective; users will point Andromeda at arbitrary third-party servers.
- The protocol revises on dated revisions; Andromeda MUST negotiate correctly against servers of varying age (the SDK's four supported revisions cover this).
- ADR-009 deliberately mirrors the MCP stdio transport model for plugins; keeping one JSON-RPC mental model is a standing goal.
- The ADR-002 license policy requires permissive licensing — Apache-2.0 verified.
- Some MCP servers require OAuth-based authorization; the SDK's client OAuth is experimental, and credential handling is owned by ADR-014/Volume 9.

#### Alternatives considered

| Criterion | Official go-sdk (selected) | Community implementations | In-house implementation |
|---|---|---|---|
| Provenance | modelcontextprotocol org, Google collaboration (verified) | Third-party maintainers | Us |
| Stability | v1.6.1 stable (2026-05-22) | Not re-verified at decision time | n/a |
| Spec revisions supported | 2025-11-25, 2025-06-18, 2025-03-26, 2024-11-05 (verified) | Varies | Whatever we build and maintain |
| Tracking of future revisions | Upstream obligation of the official SDK | Best-effort | Entirely our burden |
| License | Apache-2.0 (verified) | Varies | n/a |
| Client OAuth | Experimental (verified) | Varies | Ours to design |

- **Community implementations** (Go MCP libraries predating the official SDK) were credible before it reached stability; none were re-verified for this decision because a stable official SDK removes their reason to be chosen — it tracks the spec as an upstream obligation, not volunteer goodwill.
- **In-house implementation** would give total control at the cost of permanently owning conformance with an actively evolving multi-revision protocol — a continuous tax with no differentiating value, and an interoperability risk against the long tail of third-party servers.

#### Decision

Andromeda MUST implement MCP client support using `github.com/modelcontextprotocol/go-sdk`, pinned within the stable v1 major line. Binding rules:

1. The MCP Runtime wraps the SDK behind Andromeda's own port interface (Volume 3); SDK types MUST NOT leak into the Tool Runtime contract or the Core Domain.
2. Revision negotiation is delegated to the SDK across its verified supported set (2025-11-25, 2025-06-18, 2025-03-26, 2024-11-05).
3. The protocol revision set Andromeda pins and certifies at implementation time is PENDING VALIDATION — it depends on the spec release cycle beginning with the 2026-07-28 RC and is tracked in the open questions register until resolved.
4. Because the SDK's client OAuth is experimental, Andromeda MUST treat OAuth-based MCP server authorization as PENDING VALIDATION and MUST NOT expose it as a stable MVP feature; non-OAuth servers (stdio; token/header HTTP auth via ADR-014 credential handling) are the supported MVP path.

#### Rationale

One option dominates: the official SDK is verified stable, permissively licensed, multi-revision capable, and maintained by the organization that owns the protocol, with Google. Choosing it turns spec-tracking from an internal liability into an upstream dependency update. The port-interface rule preserves reversibility and keeps the Tool Runtime ignorant of MCP specifics, consistent with how every external protocol is wrapped. The two PENDING VALIDATION items isolate exactly the genuinely unsettled parts (revision set; OAuth) instead of letting them destabilize the whole integration.

#### Positive consequences

- Immediate, conformant coverage of four spec revisions with upstream responsibility for future ones.
- License-homogeneous with the rest of the stack (Apache-2.0, ADR-002).
- Shared JSON-RPC/stdio mental model with ARP (ADR-009) and external IPC (ADR-012) reduces implementer and documentation surface.
- The port interface makes MCP conformance testable with recorded sessions independent of the SDK's internals.

#### Negative consequences

- Andromeda's MCP feature set is rate-limited by SDK release cadence; a spec feature the SDK lags on is unavailable without forking.
- The wrapper layer must be kept honest (no type leakage), a small permanent indirection cost.
- MVP excludes OAuth-gated MCP servers, narrowing the connectable set until validation completes.

#### Risks

- **Spec churn**: the 2026-07-28 RC may shift the supported-revision matrix shortly after implementation begins. Mitigation: rule 3's deferred pin and the matching review condition.
- **OAuth stabilizing slower than product need**: users demand OAuth-gated servers first. Mitigation: ADR-014's abstraction keeps credential flows outside the MCP Runtime, so an interim solution does not fork the SDK.
- PENDING VALIDATION items (also listed in Review conditions): pinned protocol revision set at implementation time; OAuth-based MCP server authorization support.

#### Reversal plan

The port interface is the reversal seam: replacing the SDK (community library or in-house implementation) re-implements one adapter package against Andromeda's MCP port, verified by the recorded-session conformance suite; no Tool Runtime, Core Domain, or configuration surface changes. The realistic partial reversal — a temporary SDK fork to pick up an urgent spec feature — is acceptable as a bridge; Apache-2.0 permits it.

#### Review conditions

- PENDING VALIDATION: after the 2026-07-28 specification RC lands and the SDK declares support, fix the pinned/certified protocol revision set and resolve the open-questions entry.
- PENDING VALIDATION: when the SDK graduates client OAuth from experimental, validate and specify OAuth-gated MCP server support (with ADR-014) or explicitly defer it by phase.
- The SDK fails to support a published spec revision within one quarter of its release.
- A v2 major of the SDK ships with breaking changes: evaluate migration versus staying on maintained v1.
