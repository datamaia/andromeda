### ADR-008 — TOML Parsing: pelletier/go-toml/v2

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Configuration Manager
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation

#### Context

Andromeda's configuration format is TOML by product identity: `andromeda.toml` at global and project scope, with schema, precedence, validation, and migration owned by Volume 10's Configuration Manager. Config parsing sits on the trust boundary — a config file is user-supplied input read at every startup — so parser correctness, strictness, and actionable error reporting directly shape both security posture and user experience (exit code 3, configuration error, depends on precise diagnostics).

Web-verified facts (2026-07-11): `github.com/pelletier/go-toml/v2` is MIT-licensed, active, v2.4.3 (2026-07-05), implements TOML v1.1.0, and must be imported via the `/v2` module path. It is stricter and faster than `BurntSushi/toml`. `BurntSushi/toml` v1.6.0 (2025-12) is an equally safe fallback with a slower release cadence.

#### Problem

Select the TOML parsing/serialization library the Configuration Manager builds on, with a documented fallback should the primary become unavailable.

#### Forces and constraints

- The parser MUST reject invalid TOML deterministically with position information (line/column) usable in user-facing configuration errors.
- The Configuration Manager needs both decoding (load) and encoding (config migration rewrites, `andromeda config` write paths).
- Strictness matters: silently ignored malformed input in a permissions- and credentials-adjacent file is a security-relevant defect class.
- TOML v1.1.0 support determines which syntax the Volume 10 schema may rely on.
- ADR-005 already excludes viper; this ADR selects the low-level parser the custom manager uses.
- Dependency policy (ADR-002): permissive license required — both candidates are MIT.

#### Alternatives considered

| Criterion | pelletier/go-toml/v2 (selected) | BurntSushi/toml | Hand-written parser |
|---|---|---|---|
| TOML version (verified) | v1.1.0 | v1.6.0 library release; spec coverage not re-verified beyond "equally safe" | Whatever we implement |
| Activity (verified) | v2.4.3, 2026-07-05 | v1.6.0, 2025-12; slower cadence | n/a |
| Strictness / speed (verified) | Stricter and faster of the two | Baseline | Unknown until built |
| Decode + encode | Both | Both | Both, at full cost |
| License (verified) | MIT | MIT | n/a |
| Risk profile | Must use `/v2` import path | Fewer releases between spec updates | Parser bugs on a trust boundary |

- **BurntSushi/toml** is a credible, safe choice — it is retained as the documented fallback precisely because it is adequate. It loses on the two properties that matter most here: strictness (pelletier v2 rejects more malformed input) and cadence (slower releases mean slower uptake of TOML spec revisions and fixes).
- **A hand-written parser** was dismissed: TOML has enough grammar surface (dates, multiline strings, table arrays) that a bespoke implementation would concentrate defect risk exactly where user-supplied input crosses the trust boundary, for zero differentiating value.

#### Decision

The Configuration Manager MUST use `github.com/pelletier/go-toml/v2` (imported via the `/v2` module path, pinned) for all TOML decoding and encoding. Decoding MUST run in strict mode: unknown keys and malformed constructs are surfaced as configuration errors (exit code 3) with file, line, and key context — unknown-key tolerance, where the Volume 10 migration rules require it, is implemented deliberately above the parser, not by loosening the parser.

Documented fallback: `github.com/BurntSushi/toml`. The parser is wrapped behind a small internal decode/encode interface owned by the Configuration Manager so that switching to the fallback is a one-package change.

#### Rationale

Both finalists are MIT-licensed and trustworthy; the selection follows the two discriminating verified facts — pelletier v2 is stricter and faster, and it tracks the TOML spec more closely (v1.1.0) on a faster release cadence. Strictness is the security-relevant property: on a trust-boundary input, rejecting ambiguity beats tolerating it. Speed is secondary (config files are small) but costs nothing. Keeping BurntSushi as a wrapped, pre-vetted fallback converts a potential upstream emergency into routine maintenance.

#### Positive consequences

- Strict, position-aware parse errors feed directly into the Volume 10 configuration-error UX and exit-code contract.
- TOML v1.1.0 syntax is available to the configuration schema design.
- The wrapper interface makes parser substitution (and parser-differential testing in ADR-017's property tests, e.g., round-trip encode/decode) cheap.

#### Negative consequences

- The `/v2` import-path requirement is a recurring contributor footgun (importing the v1 path builds but takes the wrong library); needs a CI import lint.
- Strict mode means configurations that older, laxer tools tolerated will be rejected — intentional, but it produces support friction that documentation must absorb.
- The wrapper interface is a small amount of indirection maintained solely for reversibility.

#### Risks

- **Upstream stall or breaking v3**: pelletier/go-toml has already crossed one major-version boundary; another would require migration. Mitigation: pinning plus the wrapper seam; fallback pre-selected.
- **Spec divergence**: TOML v1.1.0 features used in the schema might behave differently in the fallback parser. Mitigation: the Volume 10 schema restricts itself to constructs verified identical under both parsers; parser-differential tests enforce this.

#### Reversal plan

Switch the wrapper implementation to `BurntSushi/toml`, re-run the configuration test suite (including golden parse-error fixtures, which will need message updates since diagnostics differ), and confirm the schema's syntax subset parses identically — the parser-differential tests exist to keep this guaranteed continuously. No data or config-file migration is involved; TOML files are parser-independent.

#### Review conditions

- `pelletier/go-toml/v2` goes 12 months without a release, loses its maintainer, or announces a breaking v3.
- A TOML specification revision lands that the library does not track within two quarters.
- Parser-differential tests reveal a semantic divergence affecting any construct the Volume 10 schema uses.
