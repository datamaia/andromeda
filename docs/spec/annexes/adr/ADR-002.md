### ADR-002 — Project License: Apache-2.0

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Core Domain, Extension SDK, Package Manager
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation

#### Context

Andromeda is developed openly on GitHub and is intended for adoption by individual engineers, open-source projects, and companies that embed it in their engineering workflows. The outbound license governs who can adopt, redistribute, and extend Andromeda (including plugins, skills, and SDK-based tools), and it constrains which inbound dependencies are acceptable. The license decision is effectively permanent for a project that accepts external contributions, because relicensing later requires the agreement of every copyright holder.

#### Problem

Choose one outbound license for the `andromeda` repository (source, SDKs, documentation-adjacent code) that maximizes adoption — including by enterprises with formal open-source policies — while protecting contributors and users, and define the inbound dependency-license policy that follows from it.

#### Forces and constraints

- Enterprise adoption is a product objective; licenses with strong copyleft obligations are commonly disallowed or heavily reviewed by corporate open-source policies.
- The AI tooling ecosystem Andromeda integrates with is dominated by permissive licenses: verified examples include the MCP Go SDK (Apache-2.0), the OpenAI Go SDK (Apache-2.0), cosign and syft (Apache-2.0), the Charm stack and go-toml (MIT), and `modernc.org/sqlite` and `filippo.io/age` (BSD-3-Clause).
- Patent risk is real in the AI space; contributors and adopters benefit from an explicit patent grant.
- One verified dependency is not permissive: `pgregory.net/rapid` v1.3.0 (property-based testing, ADR-017) is MPL-2.0.
- Contribution provenance MUST be trackable (Volume 11 defines DCO sign-off in the traceability chain).

#### Alternatives considered

| License | Patent grant | Copyleft | Enterprise friction | Notes |
|---|---|---|---|---|
| **Apache-2.0** (selected) | Explicit grant + retaliation clause | None | Low | NOTICE mechanism; ecosystem norm for AI infrastructure |
| MIT | None (implied at best) | None | Low | Shortest text; no patent protection for users or contributors |
| MPL-2.0 | Yes | File-level | Medium | Modified files stay MPL; adds per-file compliance analysis for embedders |
| AGPL-3.0 | Yes | Strong, network-triggered | High | Would deter the embedding and enterprise use cases Andromeda targets |

- **MIT** was the closest runner-up: maximal simplicity and universal acceptance. It was rejected solely because it lacks an explicit patent license, which matters for a project expecting corporate contributions in a patent-active domain.
- **MPL-2.0** offers a middle ground but transfers ongoing file-level compliance work to every downstream embedder and to contributors moving code between files.
- **AGPL-3.0** maximizes reciprocity but directly conflicts with the adoption objective; organizations that embed developer tooling in internal services routinely prohibit AGPL dependencies.

#### Decision

The Andromeda repository MUST be licensed under **Apache-2.0**, applied repository-wide (core, CLI/TUI, `andromeda-sdk`, packaging, examples). The repository MUST carry `LICENSE` and `NOTICE` files at the root. Inbound contributions are accepted under the inbound=outbound model with DCO sign-off (Volume 11); no CLA is required.

Inbound dependency policy: runtime dependencies MUST use permissive licenses (Apache-2.0, MIT, BSD family, ISC or equivalent). Weak-copyleft licenses (MPL-2.0, EPL) MAY be accepted for **test-only or build-only** dependencies, recorded individually in the license policy. Strong-copyleft licenses (GPL, AGPL) MUST NOT enter the dependency graph of shipped artifacts. The known MPL-2.0 exception `pgregory.net/rapid` (test-only, consumed unmodified, file-level copyleft never triggered by use) is hereby recorded and MUST appear in the license policy file.

#### Rationale

Apache-2.0 combines permissive adoption terms with the two protections MIT lacks: an explicit patent grant from every contributor (§3) and a patent-retaliation clause that deters offensive litigation. It matches the licenses of Andromeda's most important verified dependencies and of the official MCP SDK, keeping the whole stack license-homogeneous and easing legal review for enterprise adopters. The NOTICE mechanism gives a standard place to credit bundled third-party work in release artifacts (ADR-013 embeds license and SBOM data in releases).

#### Positive consequences

- Frictionless adoption and embedding, including in closed-source and commercial contexts.
- Contributors and users receive an explicit patent license; retaliation clause discourages patent aggression.
- License homogeneity with the verified dependency set simplifies SBOM review (ADR-013).
- Inbound=outbound with DCO keeps the contribution process lightweight and auditable.

#### Negative consequences

- No reciprocity: third parties can ship proprietary forks or hosted derivatives without contributing back.
- Apache-2.0 is one-way incompatible with GPLv2-only projects, which cannot incorporate Andromeda code.
- The NOTICE file and per-dependency license inventory add release-process bookkeeping that MIT would not require.

#### Risks

- **License drift in the dependency graph:** a transitive dependency could introduce a copyleft license unnoticed. Mitigation: SBOM generation via syft on every release (ADR-013) plus a CI license-allowlist check; the allowlist is the license policy file named in the Decision.
- **Relicensing pressure:** if a sustainability model ever demanded a license change, accumulated external copyright would make it costly. Mitigation: none needed now; the DCO record at least preserves provenance.

#### Reversal plan

Moving to a *more* permissive license (MIT) requires consent of all copyright holders but is legally simpler because Apache-2.0 code can be relicensed only by its owners — in practice this means a formal consent process over the DCO-signed contributor list, feasible while the contributor set is small. Moving to a copyleft license is treated as a fork-level event and is not planned. If only the *inbound policy* proves too strict (e.g., a critical MPL-2.0 runtime library appears with no permissive alternative), the policy table in this ADR is amended by a superseding ADR without touching the outbound license.

#### Review conditions

- A critical runtime dependency becomes available only under a weak-copyleft license.
- Legal review during a future trademark or foundation-donation process (e.g., CNCF-style governance) requires alignment with a foundation's license policy.
- The `pgregory.net/rapid` dependency is promoted beyond test-only use, which would violate the recorded exception and force replacement or policy revision.
