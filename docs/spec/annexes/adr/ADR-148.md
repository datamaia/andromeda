### ADR-148 — Traceability enforcement: platform-native chain with required validators

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: None (development-process decision; applies to the `andromeda` repository)
- Related requirements: FR-GH-001, FR-GH-004; ADR-003, ADR-004, ADR-015

#### Context

Volume 0 chapter 03 mandates the traceability chain objective → requirement → epic → issue →
branch → commit → pull request → test → check → release → artifact, and assigns Volume 11
the GitHub-side automation that enforces it from Epic onward. ADR-015 fixes Conventional
Commits with a closed scope list and squash-merge discipline; ADR-004 fixes one integration
target (`main`); ADR-013 fixes tag-driven releases with provenance attestations binding
artifacts to source revisions. Contributions come from human engineers and AI agents in
comparable volume; enforcement must be mechanical, not reviewer memory.

#### Problem

Choose the concrete mechanism that makes every link of the chain machine-verifiable: where
each link is recorded, which grammar encodes it, and what enforces it — such that a chain
break blocks a merge rather than surfacing in an audit months later.

#### Forces and constraints

- Squash merge (ADR-015) means mainline commits correspond one-to-one to merged PRs; the PR
  is the natural join point of the chain.
- Requirement identifiers already have a lint-verifiable home (`docs/spec/`, checked by
  `scripts/spec_lint.py`); the chain should resolve against it, not against a second list.
- AI agents follow templates exactly when templates are machine-checkable; free-text
  conventions decay.
- The chain must survive platform export: links recorded in platform-native fields (issue
  forms, PR bodies, labels) are retrievable via official APIs and exportable.

#### Alternatives considered

1. **Requirement trailers on every commit** — rejected as the primary record. Under squash
   merge, per-commit trailers on branch commits are discarded at merge; requiring them on
   every working commit burdens contributors twice for a record that the PR already carries.
   (A trailer on the squash commit remains, generated from the PR — that part is kept.)
2. **External traceability database** (spreadsheet or service mapping IDs to PRs) —
   rejected. A second source of truth that drifts from the platform; violates the
   single-home discipline and adds an operational dependency.
3. **Manual release-time audit only** — rejected. Detects breaks after the fact, scales with
   auditor patience, and gives contributors no feedback at the moment they can fix things.
4. **Platform-native records + required validators** — selected, described below.

#### Decision

Each chain link has exactly one platform-native record and one mechanical validator:

1. **Objective → requirement:** the specification corpus (`docs/spec/`), validated by
   `scripts/spec_lint.py` in CI on every change touching `docs/spec/`.
2. **Requirement → epic → issue:** issues declare requirement IDs in a mandatory issue-form
   field; epics are tracking issues (label `type/architecture` or `type/release` with a task
   list). The traceability validator resolves every declared ID against the corpus.
3. **Issue → branch:** branch names follow `<type>/<issue-number>-<slug>` (grammar in
   FR-GH-003); validated on every PR.
4. **Branch/commit → PR:** PRs MUST link their issue with a closing keyword; PR titles
   follow Conventional Commits (they become the squash-commit subject). The squash commit
   body carries the generated trailer `Refs: #<issue>`; the commit-msg hook and CI check
   enforce format and the ADR-015 attribution prohibition.
5. **PR → test/check:** required checks (FR-GH-009/FR-GH-010) plus the traceability
   validator itself; merges are impossible with a red chain.
6. **Release → artifact:** the release pipeline (ADR-013) generates release notes grouped
   from merged PRs, resolving each to its issue and requirement IDs, and publishes the chain
   report as a release artifact; SLSA provenance binds artifacts to the tag's commit.
7. A **nightly mainline audit** re-walks the chain for every commit on `main` since the last
   audit and opens a `type/tech-debt` issue for any orphan (defense in depth against
   force-path anomalies and rule changes).

#### Rationale

Joining the chain at the PR exploits the squash-merge invariant: one mainline commit ↔ one
PR ↔ one issue ↔ declared requirements, each hop recorded where the platform already keeps
it and where official APIs can read it back. Validators run where contributors act (the PR),
so breaks are fixed by their author in minutes. Resolving requirement IDs against the spec
corpus reuses the linter's definition index — one source of truth for what exists.

#### Positive consequences

- The chain is enforceable with zero bespoke storage: forms, titles, labels, trailers,
  checks, release notes.
- Chain state is queryable for any commit via official APIs; the release-time report is a
  by-product, not a project.
- AI-agent contributions are traceable by construction — templates and validators do not
  care who typed.

#### Negative consequences

- Process rigidity: trivial changes still need an issue link (the taxonomy provides
  lightweight types; exemptions are enumerated in FR-GH-001, not ad hoc).
- Validator maintenance is real code with tests, owned like product code (`area/ci`).
- GitHub-specific field mechanics (issue forms, closing keywords) would need re-mapping on
  any platform migration (accepted; the chain data itself is exportable).

#### Risks

- Rule circumvention via admin merges; mitigation: branch protection applies to
  administrators; the nightly audit catches anomalies and files them visibly.
- Validator false positives blocking urgent fixes; mitigation: an explicitly labeled,
  audit-logged override label usable only by maintainers, itself flagged in the nightly
  audit report.

#### Reversal plan

Each validator is a separate required check; disabling one is a branch-protection
configuration change. Moving to another convention (for example, requirement IDs in PR
bodies instead of issue forms) changes the validator's resolution step only, because every
other link is independent. The chain definition itself lives in Volume 0 and changes only
through the change procedure.

#### Review conditions

- If validator-caused merge friction exceeds a maintainer-agreed nuisance threshold for two
  consecutive months, review the grammars (not the chain).
- When GitHub materially changes issue forms, Projects, or merge behavior, re-validate the
  mechanics.
- At each phase gate, review the nightly-audit orphan count; a non-zero steady state
  indicates a broken rule, not acceptable noise.
