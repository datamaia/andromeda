### ADR-074 — Built-in integration tools: official public APIs only, phased beyond MVP

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Tool Runtime, Authentication Layer, Secret Store, Sandbox Engine
- Related requirements: FR-TOOL-007; FR-AUTH-001 (Volume 5); Volume 1 provided constraint (official mechanisms only)

#### Context

The mandated built-in catalog includes tools for external systems and services: Docker,
Kubernetes, GitHub, GitLab, Jira, Notion, Slack, and Linear, plus a browser tool (Volume 6,
chapter 03). The product-wide provided constraint categorically excludes unofficial
integrations — reverse engineering, captured cookies, private APIs, web-UI automation against
a service's terms (Volume 1, chapter 05). The MVP minimum (Volume 1) includes none of these
integration tools. Each service's concrete API surface, authentication modes, versioning, and
rate limits are external facts this specification cannot assert without verification.

#### Problem

Fix the integration policy and delivery phasing for built-in tools that talk to external
services, without inventing endpoint-level facts and without letting integration breadth
compete with MVP focus.

#### Forces and constraints

- Legality and official-mechanisms-only rank second in the requirement precedence order,
  above all functional requirements.
- The assumptions policy forbids asserting endpoints, rate limits, or auth mechanics that have
  not been verified against official documentation.
- Every network-touching tool binds to the permission model (`network`,
  `external_service_access`) and to credential handling via Secret Store references only.
- Each integration is ongoing maintenance surface; APIs drift.

#### Alternatives considered

1. **Ship all integrations at MVP** — rejected: none is in the MVP minimum; each adds a
   verification, credential, and maintenance burden that competes with the MVP exit criteria.
2. **No built-in integrations; leave them to MCP servers** — genuinely attractive (the MCP
   ecosystem covers several of these services) but rejected as a blanket rule: the mandate
   names them as built-in catalog entries, and first-party tools carry `builtin` trust,
   uniform schemas, and product-grade conformance tests that third-party servers cannot be
   held to.
3. **Screen-scraping or web automation fallbacks where APIs are limited** — categorically
   rejected by the provided constraint.

#### Decision

Built-in integration tools interact with external services exclusively through their
**official, documented public APIs**, authenticated through the Authentication Layer with
credentials held as Secret Store references. Concrete per-service facts — API versions,
authentication modes, scopes, rate limits — are recorded as PENDING VALIDATION until verified
against official documentation when each tool's implementation phase begins; the tool schemas
in chapter 03 are contract sketches, not endpoint assertions. Delivery is phased: container
and data tools (`docker.control`, `sqlite.query`, `http.request`, `process.control`) and
`github.request` land in Beta; `kubernetes.control`, `gitlab.request`, `jira.request`,
`slack.request`, and `browser.control` land in v1; `notion.request` and `linear.request` land
in v2. The browser tool additionally requires selection of an official automation protocol
(W3C WebDriver or an equivalent documented standard), PENDING VALIDATION. Phasing changes go
through the Volume 0 change procedure.

#### Rationale

The phasing tracks value density and verification cost: developer-loop tools (HTTP, Docker,
SQLite, GitHub) serve the core engineering jobs first, while organization tools (Jira, Slack,
Notion, Linear) arrive once the platform surfaces they depend on (auth profiles, trust policy,
rate-limit handling in the field) have hardened. Holding endpoint facts as PENDING VALIDATION
keeps the specification honest under the assumptions policy while still fixing names, schema
shapes, permissions, and phases now.

#### Positive consequences

- The catalog commitment is kept without inflating MVP scope.
- Every integration inherits one credential, permission, and observability pattern.
- No integration can ever be built on a mechanism a service's terms prohibit.

#### Negative consequences

- Users wanting Jira/Slack/Notion/Linear before v1/v2 must bridge via MCP servers, with
  third-party trust semantics.
- PENDING VALIDATION items defer parts of the contract (operation lists per service) to
  implementation time.

#### Risks

- An official API may prove insufficient for a planned operation set; mitigation: scope the
  tool to what the API offers and record the gap — never fall back to unofficial paths.
- API drift breaks shipped tools; mitigation: conformance tests against recorded fixtures,
  RISK-TOOL-003 monitoring, and SemVer-governed tool versions for breaking schema changes.

#### Reversal plan

Dropping a planned integration tool is a phasing change (change procedure + register update);
its namespace stays reserved. Replacing a built-in integration with a first-party-maintained
MCP server would keep the tool name and schema surface via the MCP bridge and retire the
built-in, under the SM-20 deprecation regime.

#### Review conditions

- Per-service PENDING VALIDATION resolution at each tool's phase start.
- The MCP ecosystem produces official first-party servers for these services, changing the
  build-vs-bridge economics.
- Field demand reorders the phasing (change procedure with recorded justification).
