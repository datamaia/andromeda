### ADR-103 — CLI color and terminal-capability policy

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: CLI
- Related requirements: FR-UX-002, FR-CLI-005, FR-CLI-011; ADR-026 token vocabulary

#### Context

The CLI writes human-format output to terminals of very different capability, to pipes, and
to CI log collectors. ADR-026 fixes the brand token vocabulary and assigns Volume 8 the
token-to-ANSI mapping with degradation tiers (truecolor → 256 → 16 → no-color); chapter 08
of this volume (TUI theming) specifies those tiers in full. The CLI needs a narrower
decision first: *whether* to emit ANSI styling at all for a given invocation, honoring both
Andromeda-specific controls and the ecosystem conventions users already deploy (`NO_COLOR`,
`TERM=dumb`). Mis-deciding in either direction is a real defect: ANSI escapes in CI logs
and redirected files are noise that breaks grep-based tooling; stripped color on capable
terminals degrades the PRD-008 experience.

#### Problem

Define a deterministic enablement rule for styled CLI output — which signals are consulted,
in what order, with what default — and bound what the CLI styles so the rule stays simple.

#### Forces and constraints

- `NO_COLOR` (the informal cross-tool convention: any non-empty value disables color) and
  `TERM=dumb` are ecosystem signals Andromeda does not own but must respect; their status
  as conventions rather than standards is recorded as a register assumption.
- Explicit user intent (`--color=always`) must be able to force styling into a pipe —
  capturing styled output is a legitimate workflow (e.g., piping to a pager that renders
  ANSI).
- The decision must be identical for every command, or scripts get per-command surprises.
- stdout and stderr can differ (stdout piped, stderr on the TTY); progress and diagnostics
  on stderr should stay styled when only stdout is captured.
- JSON mode output must never contain ANSI styling regardless of the color decision.

#### Alternatives considered

1. **Ordered per-stream resolution with `auto` default (selected).**
2. **Color only when explicitly requested** — rejected: it makes the default experience at
   a TTY monochrome, failing PRD-008's interaction-quality bar for the common case.
3. **Always color, let users pipe through strippers** — rejected: pollutes CI logs and
   files by default; violates the ecosystem contract of `NO_COLOR`.
4. **Full terminfo-driven capability negotiation in the CLI** — rejected for the CLI
   surface: tier mapping beyond on/off matters for the TUI's dense rendering and is
   specified there (chapter 08); the CLI's styling is sparse enough that the 16-color-safe
   subset of the ADR-026 mapping suffices, keeping the CLI dependency-light.

#### Decision

1. Styling is resolved **per output stream** (stdout and stderr independently), first match
   wins:
   1. `--color=never` → off; `--color=always` → on.
   2. `cli.color` configuration key set to `never`/`always` → off/on.
   3. `NO_COLOR` set to any non-empty value → off.
   4. `ANDROMEDA_NO_COLOR` set to a truthy value → off (scoped variant for wrappers that
      must not disturb other tools).
   5. Stream is not a terminal, or `TERM` is `dumb` or unset → off.
   6. Otherwise (`auto`) → on.
2. `--json` output is never styled, independent of this resolution.
3. When styling is on, the CLI uses the ADR-026 token mapping's 16-color-safe subset as
   specified in chapter 08 of this volume; the CLI performs no tier probing beyond the
   on/off decision plus the `TERM`-based downgrade rule chapter 08 defines.
4. CI mode (FR-CLI-009) sets the effective default to off at step 5 regardless of
   pseudo-TTY allocation, still overridable by steps 1–2.

#### Rationale

The order encodes precedence of intent: explicit invocation beats persistent preference,
which beats ecosystem-wide signals, which beat environment heuristics. Honoring `NO_COLOR`
below Andromeda's own controls (rather than above) follows the convention's published
intent — a user-level default that individual invocations may override deliberately.
Per-stream resolution matches how developers actually compose pipelines. Bounding the CLI
to the 16-color-safe subset removes an entire class of capability-probing complexity from
the non-TUI path while keeping brand consistency through the shared token mapping.

#### Positive consequences

- Zero ANSI noise in captured output by default; zero configuration needed for the common
  cases at either extreme.
- One documented decision table serves every command; `doctor` prints the resolved outcome
  and the deciding rule for support.
- Wrappers and scripts get a scoped disable (`ANDROMEDA_NO_COLOR`) that does not affect
  other tools.

#### Negative consequences

- Six-step resolution is more to document than "color iff TTY"; the decision table must be
  kept testable (golden tests enumerate it).
- CLI output forgoes truecolor fidelity even on capable terminals; brand fidelity above 16
  colors is a TUI property.

#### Risks

- Divergence between the CLI's on/off subset and the TUI's tier mapping producing visibly
  different brand colors across surfaces; mitigation: both consume the single token mapping
  table in chapter 08.
- The `NO_COLOR` convention's semantics drift in the ecosystem; mitigation: register
  assumption; the check is one line to adjust.

#### Reversal plan

The resolution is one pure function plus a styling writer. Adding tier awareness to the CLI
(should sparse styling grow) means consuming more of chapter 08's existing mapping table —
an additive change. Removing a consulted signal is a behavior break requiring a documented
deprecation in release notes; the decision table's position in FR-UX-002 puts it under
change control.

#### Review conditions

- CLI output grows dense enough (tables, diff rendering) that 16-color-safe styling
  measurably harms readability against chapter 08's tier capabilities.
- The ecosystem standardizes a successor to `NO_COLOR` / `CLICOLOR` conventions.
- Recurrent support reports of misresolved styling in real terminals (decision-table gap).
