### ADR-054 — Compensation-based workflow rollback with git restore points for specification-driven development

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team (Volume 4 authoring)
- Components: Workflow Engine, Git Engine, Permission Manager
- Related requirements: FR-WF-007; RISK-WF-002

#### Context

The brief requires workflows to define rollback. Workflow steps produce side effects through
arbitrary tools: file changes, commands, git mutations, and — with granted permissions —
external calls. No transactional undo exists across that surface (RISK-ARCH-004 records the
same truth for recovery). Volume 2 gives Patches a `reverted` status and the Git Engine
(ADR-025) provides repository-level restore mechanics.

#### Problem

What does "rollback" mean for a workflow whose steps have heterogeneous, partially
irreversible side effects, and how does the flagship `spec-driven-dev` workflow make its
main mutation surface — the working tree — reliably restorable?

#### Forces and constraints

- Filesystem snapshots are platform-divergent and cannot cover external effects anyway.
- Pretending universal undo exists is worse than admitting bounded undo (RISK-WF-002).
- Compensations are themselves side-effecting work and must be permission-mediated and
  attributed like everything else (PRD-006).
- Most SDD mutations are workspace-local and git-tracked — the one surface with a reliable
  restore primitive.

#### Alternatives considered

1. **Automatic filesystem snapshot/restore** (copy-on-write, overlay mounts) — rejected:
   platform-specific, silent about external effects, and capable of destroying user edits
   made outside the run.
2. **Inverse-operation inference** (engine derives undo from recorded File Changes) —
   rejected as the general mechanism: inference is wrong exactly in the ambiguous cases
   that matter (concurrent edits, command side effects); retained only as *input shown to
   humans* in resume/rollback reports.
3. **No rollback (forward-only)** — rejected: the brief mandates rollback, and the
   git-tracked common case is genuinely restorable.

#### Decision

Rollback is **compensation-based**: a step MAY declare a `compensation` (agent goal or tool
action); rollback executes declared compensations of completed side-effecting steps in
reverse completion order, permission-mediated, halting honestly at the first failure
(E-WF-011) with a report of compensated versus remaining steps. Rollback triggers only via
declared `rollback_on` outcomes or explicit user request — never implicitly. The
`spec-driven-dev` definition records a **git restore point** (branch reference plus commit
created via the Git Engine under `git_mutation`) before its first side-effecting stage; its
implementation-stage compensation restores that point, and compensations never force-mutate
published remotes.

#### Rationale

Compensation makes reversibility a *declared, reviewable property* of each step — exactly
the posture the permission model takes toward side effects. Git restore points give the
dominant SDD case a precise, portable, user-familiar undo anchor without inventing snapshot
machinery.

#### Positive consequences

- Rollback semantics are inspectable in the definition before any run starts.
- Honest partial-rollback reports instead of silent best-effort guessing.
- SDD working-tree changes restore with standard git semantics users can verify themselves.

#### Negative consequences

- Steps without declared compensation are not rolled back — reversibility requires authoring
  effort.
- Reverse-order compensation is heuristic when steps' effects interleave on the same files.

#### Risks

- Compensations drift from what steps actually do; mitigation: rollback reports pair each
  compensation with the step's recorded effects, making drift visible in review.
- Restore-point branches accumulate; mitigation: restore points are cleaned at run
  completion per the definition's declared retention, and are ordinary branches users can
  delete.

#### Reversal plan

Compensation is the contract surface; adding stronger mechanisms later (e.g., OS-level
snapshot compensation implementations on platforms that support them) means shipping new
compensation *implementations*, not changing the model.

#### Review conditions

- Revisit if Beta rollback reports show > 25% of rollbacks halting on failed compensations.
- Revisit when Volume 9's Beta/v1 sandbox layers make per-step effect enumeration mechanical
  enough to auto-generate compensation proposals.
