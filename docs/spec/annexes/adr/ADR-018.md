### ADR-018 — Formatting and linting: gofmt plus golangci-lint with a curated, pinned linter set

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Core Domain, Runtime, CLI, TUI, Extension SDK
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation.

#### Context

Andromeda is an open-source Go codebase (ADR-001, ADR-002) developed on GitHub with trunk-based
development (ADR-004) and a monorepo layout (ADR-003). Contributions come from AI implementation
agents and human engineers of varying backgrounds. Style debates and inconsistent static
analysis are a known drag on open-source review throughput, and unpinned linters are a known
source of CI breakage: a linter release that adds checks turns previously green code red without
any code change.

#### Problem

Which formatting and static-analysis toolchain is authoritative for the repository, such that
formatting is never a review topic, analysis findings are consistent between local runs and CI,
and toolchain upgrades are deliberate events rather than ambient breakage?

#### Forces and constraints

- One canonical formatting output: two contributors formatting the same file MUST produce
  identical bytes.
- CI verdicts MUST be reproducible locally; tool version skew between the two is unacceptable.
- The linter set has to cover correctness-adjacent checks (unused results, shadowing,
  concurrency mistakes relevant to ADR-023) without drowning contributors in stylistic noise.
- AI implementation agents benefit disproportionately from mechanical, unambiguous rules —
  guidance that requires taste does not transfer.
- Tooling MUST remain standard `go` ecosystem tooling; bespoke build steps raise the
  contribution barrier.

#### Alternatives considered

1. **gofumpt as the mandatory formatter** — a stricter gofmt superset. Not adopted now: it adds
   rules beyond the universal Go baseline, and adopting it later is cheap (a one-time
   reformat), whereas adopting it prematurely adds friction for contributors whose editors
   default to gofmt. Recorded as a MAY-adopt-later candidate with an explicit review condition
   below.
2. **staticcheck alone** — a single high-quality analyzer. Rejected as the whole story: it
   covers one analyzer family, has no aggregation of other analyzers, no unified configuration
   and baseline handling, and would still need extra tooling for the checks it does not
   implement. staticcheck's checks remain available inside golangci-lint.
3. **No enforced linting (gofmt only)** — rejected: forfeits mechanical detection of defect
   classes the specification cares about (ignored errors, misuse of context, data-race-prone
   patterns), pushing that load onto human review.

#### Decision

- **gofmt** is the single authoritative formatter; CI fails on any file whose formatting
  differs from gofmt output.
- **golangci-lint** is the single lint entry point, running a **curated linter set** defined in
  the repository configuration file — deliberately selected analyzers rather than
  enable-all — so that findings are actionable and stable.
- The golangci-lint **version is pinned in CI** and in contributor tooling instructions;
  upgrades happen through pull requests that update the pin and fix or explicitly baseline any
  new findings in the same change.
- Lint configuration lives in the repository and is the only source of truth; per-file
  suppressions require a justifying comment.

#### Rationale

gofmt is the ecosystem-wide formatting baseline: zero configuration, universal editor support,
and no dialect drift. golangci-lint is the de-facto aggregation point for Go static analysis,
letting Andromeda compose staticcheck-class correctness analyzers with vet-style checks under
one configuration, one cache, and one CI step. Curating the set (instead of enabling
everything) keeps the signal-to-noise ratio compatible with review throughput, and pinning the
version makes every lint verdict a function of the repository state alone — which is what
trunk-based development (ADR-004) needs from its required checks.

#### Positive consequences

- Formatting disappears as a review topic; diffs contain only semantic changes.
- Local and CI lint results match exactly at a given commit.
- Defect classes with mechanical detectors are caught before human review.
- Toolchain upgrades are visible, reviewable events with their fallout contained in one PR.

#### Negative consequences

- The pinned version lags upstream; newly released analyzers arrive only on deliberate upgrade.
- Curating the linter set is ongoing maintenance work with judgment calls.
- Contributors must install a specific golangci-lint version locally to reproduce CI.

#### Risks

- Lint-set drift toward leniency (suppressions accumulating); mitigation: suppressions require
  justification comments and are counted in periodic quality review (Volume 13).
- A golangci-lint upstream compatibility break with a new Go toolchain release could block Go
  upgrades; mitigation: upgrade Go and the lint pin in the same change, validated in CI.
- Divergence between editor-integrated linting and the pinned CI version; mitigation: document
  the pin and provide a repository task that runs the pinned binary.

#### Reversal plan

Formatting and linting leave no trace in shipped artifacts. Replacing golangci-lint means
swapping the CI step and configuration file; replacing gofmt with a stricter formatter means a
one-time mechanical reformat commit. Either reversal is a single PR plus contributor
documentation updates.

#### Review conditions

- **gofumpt adoption**: revisit after the MVP phase; Andromeda MAY adopt gofumpt if the
  maintainers judge the reformat cost acceptable — this is the recorded trigger for that
  decision.
- Review the curated linter set at each phase transition (MVP, Beta, v1) against defect data
  from Volume 13 quality reporting.
- If golangci-lint ceases maintenance or changes license, re-evaluate the aggregation layer.
