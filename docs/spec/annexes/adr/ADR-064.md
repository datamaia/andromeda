### ADR-064 — Proactive single-flight token refresh with on-demand fallback

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Authentication Layer, Provider Layer, Task Scheduler
- Related requirements: FR-AUTH-010; Authentication Session machine (Volume 5 chapter 11)

#### Context

Renewable credentials (OAuth tokens, service-account exchanges) expire mid-session; agent
runs are long-lived (hours) and must not fail on predictable expiries (PRD-010). Volume 2
freezes the Authentication Session states including `refreshing`; AuthPort freezes `Refresh`
with the note that refreshes for one session serialize (single-flight). Many providers rotate
the refresh token on every use, so concurrent refresh attempts can invalidate each other.

#### Problem

When does refresh run, how do concurrent consumers coordinate, and what happens on failure —
so that long runs continue across expiries, refresh-token rotation is never raced, and
failures classify cleanly between "renew later" and "re-authenticate now"?

#### Forces and constraints

- Concurrent in-flight requests per provider are the norm (ADR-023 concurrency model).
- Refresh-token rotation-on-use makes racing refreshes actively destructive.
- Offline refresh attempts must fail fast without invalidating still-valid material
  (ADR-066).
- Clock skew makes local expiry evaluation approximate; provider signals are authoritative.

#### Alternatives considered

1. **On-demand only** (refresh after a 401-class rejection) — rejected as the sole strategy:
   every expiry costs a failed request plus retry latency mid-run, and streaming requests
   fail mid-stream.
2. **Background timer refresh for all sessions** — rejected: refreshes idle sessions nobody
   is using, generates unnecessary network activity (against the ADR-066 quiet-offline
   stance), and keeps wall-clock wakeups alive.
3. **Per-request refresh locks in adapters** — rejected: duplicates coordination logic per
   adapter; the Authentication Layer owns session state and must own its serialization.

#### Decision

Refresh triggers are exactly two: **proactive** — a session used within
`auth.refresh_lead_time_seconds` (default 300) of `expires_at` refreshes before the request
proceeds; and **on-demand** — a provider's documented expired-credential signal triggers
refresh and one request replay. Refreshes per session are **single-flight**: one exchange in
flight, all other consumers of that session block on its outcome. Transient failures retry
within `refreshing` (≤ 3 attempts, exponential backoff from 1 s, jittered, ≤ 30 s total);
with valid old material remaining, the session returns to `active` and retries at the next
window. Definitive failures classify to `expired` (renewable identity gone) or `failed`
(non-expiry error) per the chapter 11 machine.

#### Rationale

Use-triggered proactive refresh gets the benefit of never failing predictably without the
costs of background timers; single-flight is the only safe shape under rotation-on-use; the
transient/definitive split keeps offline windows from destroying otherwise-valid sessions.

#### Positive consequences

- Long runs cross token expiries without user-visible failures.
- Zero background network activity for idle sessions.
- Rotation-on-use providers are safe by construction.

#### Negative consequences

- The first request after a long idle period pays the refresh latency inline.
- Sessions with unknown `expires_at` get only the on-demand path.

#### Risks

- Providers whose expiry signals are nonstandard may mis-classify; mitigation: chapter 06
  normalization owns the classification and adapter contract tests pin it per provider.

#### Reversal plan

Trigger policy is internal to the Authentication Layer behind `AuthPort.Refresh`; moving to
timer-based or adapter-driven refresh changes no port, entity, or state name.

#### Review conditions

- Re-examine the default lead time when real token lifetimes from validated OAuth providers
  are known (V5B-OQ-2).
- Revisit if streaming-heavy workloads show mid-stream expiries that proactive refresh does
  not cover.
