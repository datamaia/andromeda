### ADR-006 — TUI Framework: Charm Bubble Tea v2 Stack

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: TUI
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation

#### Context

The TUI is Andromeda's primary interactive surface: streaming agent output, plan and task views, approval prompts, provider/session dashboards, and theming (Volume 8). It is a long-running, event-driven program that must render concurrent updates (tokens streaming from providers, events from the Event Bus per ADR-012) without blocking input. In the Go ecosystem (ADR-001) two families dominate: the Charm stack (Bubble Tea + Lip Gloss + Bubbles) with its Elm-style model-update-view architecture, and tview/tcell with an imperative widget model.

Web-verified facts (2026-07-11): the Charm v2 stack is active and shipped 2026-02-23 with breaking changes (Cursed renderer, declarative View API) and new vanity import paths — `charm.land/bubbletea/v2` v2.0.8, `charm.land/lipgloss/v2` v2.0.5, `charm.land/bubbles/v2` v2.1.1 (all released 2026-07), MIT-licensed. The companion test harness `teatest/v2` lives under `github.com/charmbracelet/x/exp/teatest/v2` (published 2026-07-05), experimental, pseudo-versioned.

#### Problem

Select the TUI framework and version line for the Andromeda TUI, including how the UI is tested, balancing architectural fit for event-driven streaming against ecosystem maturity and version-churn risk.

#### Forces and constraints

- The TUI renders high-frequency asynchronous updates; the rendering model MUST accommodate messages arriving from goroutines outside the UI loop.
- Volume 8 requires theming via design tokens; styling must be programmable, not hard-coded per widget.
- Volume 13 requires automated TUI testing; a framework without a testing story forces fragile PTY-scraping harnesses.
- A new project starting on Charm v1 would face a planned migration to v2 (released 2026-02-23) mid-development.
- Import-path correctness is a real defect class: v2 uses `charm.land/*` vanity paths, not the v1 `github.com/charmbracelet/*` paths.

#### Alternatives considered

| Criterion | Bubble Tea v2 stack (selected) | tview + tcell | Bubble Tea v1 |
|---|---|---|---|
| Architecture | Elm-style model/update/view; messages from any goroutine | Imperative widgets, callback-driven | Same as v2, older APIs |
| Fit for streaming/event UIs | Strong: async events are the native input | Possible; concurrent redraw discipline is manual | Strong |
| Styling/theming | Lip Gloss composable styles map onto design tokens | Per-widget attributes | Lip Gloss v1 |
| Ready-made components | Bubbles v2 (inputs, viewport, tables, spinners) | Rich widget set (forms, tables) | Bubbles v1 |
| Testing | teatest/v2 golden-frame harness | No dedicated harness | teatest v1 |
| Trajectory (verified) | Active current major, releases through 2026-07 | Not re-verified for this decision | Superseded major; new development targets v2 |

- **tview/tcell** excels at form- and table-centric apps with an imperative style, but Andromeda's TUI is dominated by streaming, incremental state — the exact shape the Elm architecture models naturally (every token, event, and keypress is a message folded into state). tview's maintenance trajectory was also not re-verified for this decision, whereas the Charm stack's was.
- **Bubble Tea v1** is the proven prior major, but adopting it in mid-2026 means building on a superseded line and scheduling a known breaking migration (renderer and View API changed in v2). For a greenfield TUI that trade has no upside.
- **Building directly on tcell** (no framework) was dismissed: it reimplements an event loop, layout, and styling for no differentiating benefit.

#### Decision

The TUI MUST be built on the Charm v2 stack: `charm.land/bubbletea/v2` (event loop and architecture), `charm.land/lipgloss/v2` (styling; the binding layer for Volume 8 design tokens), and `charm.land/bubbles/v2` (standard components). All three MUST be pinned to exact versions, and imports MUST use the `charm.land` vanity paths; v1 (`github.com/charmbracelet/*`) import paths MUST NOT appear in the tree. TUI tests use `github.com/charmbracelet/x/exp/teatest/v2` as a **test-only** dependency, pinned to an exact pseudo-version (it is experimental, under `/exp/`, with no stability promise); the broader testing stack is ADR-017.

#### Rationale

The Elm architecture is the decisive fit: streaming tokens, Event Bus notifications, and user input all reduce to messages processed by a single update function, giving deterministic state transitions that are directly assertable in tests. Lip Gloss provides the programmable styling layer the design-token system needs. Choosing v2 now — verified active, current, and five months into its release line — avoids a guaranteed v1→v2 migration and aligns with where upstream development and fixes land. teatest/v2 gives golden-frame regression testing that no alternative offers out of the box; its experimental status is contained by pinning and test-only scope.

#### Positive consequences

- Deterministic, message-driven UI state: unit-testable update logic plus golden-frame integration tests.
- Design tokens map cleanly onto Lip Gloss styles; theming stays declarative.
- Bubbles v2 supplies vetted components (viewport, text input, tables, spinners), reducing bespoke widget code.
- Single actively maintained stack from one upstream, MIT-licensed (verified), consistent with the ADR-002 policy.

#### Negative consequences

- v2 is a young major (shipped 2026-02-23); API refinements within the v2 line may force periodic pin bumps with code churn.
- The Elm architecture concentrates state in one model; large views need deliberate decomposition conventions to stay maintainable.
- teatest/v2 pseudo-version pinning is awkward tooling-wise and the harness may change under us between pins.

#### Risks

- **Upstream churn or stall** in the v2 line: mitigated by exact pinning, an internal package boundary that keeps Bubble Tea types out of the Runtime, and the review conditions below.
- **teatest/v2 instability** breaking the test suite on upgrade: mitigated by treating harness upgrades as isolated PRs with golden-file regeneration, never bundled with feature work.
- **Accidental v1/v2 import mixing** producing subtle duplicate-dependency defects: mitigated by a CI lint rule banning `github.com/charmbracelet/bubbletea` and sibling v1 paths.

#### Reversal plan

The TUI package is a consumer of Runtime events and resolved configuration; nothing outside it imports Charm types. Reversal to tview/tcell (or a later framework) rewrites the presentation layer against the same event contract and design tokens, guided by the Volume 8 wireframes and the existing golden frames as behavioral reference. The CLI (ADR-005) remains fully functional throughout, so a reversal can ship incrementally behind the TUI entry point.

#### Review conditions

- The Charm v2 stack goes unmaintained (no release for more than 12 months) or a v3 with breaking changes ships and receives all upstream fixes.
- teatest graduates out of `/exp/` (re-pin to tagged releases) or is abandoned (replace the harness).
- TUI performance targets in Volume 12 (frame latency under streaming load) are missed and profiling attributes the miss to the framework rather than to Andromeda's view code.
