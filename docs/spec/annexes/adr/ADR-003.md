### ADR-003 — Repository Strategy: Monorepo

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Core Domain, CLI, TUI, Extension SDK, Package Manager
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation

#### Context

Andromeda comprises the core engine, the CLI/TUI front ends, the public Extension SDK (`andromeda-sdk`), the specification corpus (`docs/spec/`), packaging and release configuration, and CI automation. Development is conducted on GitHub with a traceability chain (objective → requirement → issue → branch → PR → check → release) enforced by automation (Volume 11), and a substantial share of contributions is produced by AI implementation agents working from this specification.

#### Problem

Decide how the codebase is distributed across Git repositories: a single repository, one repository per deliverable, or a hybrid — such that atomic cross-cutting changes, SDK versioning, release engineering, CI cost, ownership, and security controls are all workable.

#### Forces and constraints

- The product identity fixes the main repository name: `andromeda`.
- Core, CLI/TUI, and SDK evolve in lockstep in the Core/MVP phases: a change to the Tool contract (Volume 6) typically touches core, SDK, tests, and docs in one logical change.
- The traceability chain is materially cheaper to enforce inside one repository (one issue tracker, one branch namespace, one set of required checks).
- AI-agent contributors work best when a single checkout contains the full context: specification, code, tests, and packaging.
- Go modules support multiple modules per repository, with subdirectory tags for independent versioning (e.g., `sdk/v0.3.0`).
- Release automation (ADR-013) and branch protection (ADR-004) multiply in cost per additional repository.
- Security posture: secrets, signing identities (cosign), and branch-protection rules are configured per repository.

#### Alternatives considered

| Criterion | Monorepo (selected) | Polyrepo | Hybrid (core + SDK repos) |
|---|---|---|---|
| Atomic cross-cutting changes | One PR | Coordinated multi-repo PRs | Coordinated for SDK-touching changes |
| SDK/core contract drift | Impossible (same commit) | Likely; needs contract test matrix | Likely at the seam |
| CI configuration | One pipeline, path filters | N pipelines | Two+ pipelines |
| Release engineering | One goreleaser config, one tag namespace (subdir tags for SDK) | Per-repo release trains | Split release trains |
| Traceability chain | Native | Cross-repo links, fragile | Partially fragile |
| Ownership granularity | CODEOWNERS paths | Repo-level | Mixed |
| Security controls (secrets, protections, scanning) | Configured once | Configured N times | Configured twice+ |
| Repo size / clone time growth | Grows over time | Small repos | Medium |
| Issue triage | Single tracker (label taxonomy needed) | Scoped trackers | Split trackers |

- **Polyrepo** optimizes for independent team ownership and small clones — neither is a current force: there is one team and one product. Its costs (contract drift between core and SDK, M×N CI/release/security configuration, broken atomicity) hit exactly the properties this project depends on.
- **Hybrid** (core monorepo, SDK in its own repository) is the conventional compromise when SDK consumers need a lean dependency surface. Go makes this unnecessary: a separate `go.mod` for `sdk/` inside the monorepo gives consumers an isolated module graph with subdirectory version tags, without a second repository.

#### Decision

Andromeda MUST use a single repository, `andromeda`, containing: core and runtime packages, the CLI/TUI, the Extension SDK as a separate Go module under `sdk/`, the specification corpus under `docs/spec/`, packaging/release configuration, and CI workflows. The SDK module MUST keep its dependency graph minimal and is versioned with subdirectory tags. Ownership boundaries are expressed with `CODEOWNERS` path rules; issue scoping uses the area labels from the taxonomy in Volume 0. Splitting any component into its own repository requires a superseding ADR.

#### Rationale

The dominant forces are atomicity, traceability, and low process overhead. In the Core/MVP phases nearly every meaningful change crosses the core/SDK/docs boundary; a monorepo makes those changes reviewable and revertible as single commits, which also keeps the requirement-to-release traceability chain enforceable with one set of GitHub automations. Go's multi-module support removes the classic technical argument for extracting SDKs. Security-relevant configuration (branch protection per ADR-004, release signing per ADR-013, secret scanning, Dependabot) is defined once and audited once.

#### Positive consequences

- Contract changes between Core Domain, Tool Runtime, and Extension SDK are atomic and bisectable.
- One CI pipeline with path-filtered jobs; one release pipeline; one security configuration surface.
- AI agents and new contributors get complete context from a single clone.
- The specification and the implementation it governs are versioned together.

#### Negative consequences

- CI duration grows with the tree; path filtering and test sharding become necessary maintenance work.
- A single issue tracker mixes concerns; discipline with area labels is required.
- Repository-level permissions cannot isolate subprojects; CODEOWNERS gates review, not read/write scope.
- Clone size grows monotonically (docs, golden test files); may eventually require partial-clone guidance for contributors.

#### Risks

- **CI cost creep** degrading PR feedback time as the tree grows. Mitigation: path-filtered workflows from day one; Volume 13 sets a measurable PR-check latency target that gates process changes.
- **SDK dependency bloat** if code is carelessly shared across the module boundary. Mitigation: the `sdk/` module has its own `go.mod`; CI fails if its dependency count grows beyond the allowlist in the license policy (ADR-002).

#### Reversal plan

Extraction is mechanical and history-preserving: `git filter-repo` (or equivalent) splits any path (most plausibly `sdk/`) into a new repository with full history; the Go module path either stays stable (vanity import) or bumps a major version. The monorepo keeps a read-only mirror of the extracted history. Because the SDK is already a separate Go module with subdirectory tags, consumers experience the extraction as, at most, one `go.mod` path change.

#### Review conditions

- PR-check latency exceeds the Volume 13 target for two consecutive months despite path filtering.
- A second, independently governed deliverable appears (e.g., a community plugin registry service) whose contributors need write access without core access.
- SDK consumers demonstrably need a release cadence decoupled from the core beyond what subdirectory tags provide.
