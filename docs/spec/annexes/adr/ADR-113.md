### ADR-113 — Clipboard integration: PAL-native first, OSC 52 fallback, bounded sizes, user-initiated only

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: TUI, Platform Abstraction Layer (PAL)
- Related requirements: FR-UX-075, RISK-UX-079

#### Context

Copying diffs, paths, commands, and error diagnostics out of the TUI is a core workflow.
The PAL Clipboard surface (Volume 3, chapter 07) abstracts the mechanisms: native clipboard
utilities locally (pbcopy/pbpaste, wl-clipboard/xclip) and the OSC 52 escape sequence as
the fallback over SSH, where no local clipboard exists. OSC 52 writes clipboard content by
emitting an escape sequence to the terminal — powerful, but it transmits the payload
through the terminal stream (visible to multiplexers, loggers, and the remote host path)
and emulators cap accepted sizes. The Volume 9 permission enum includes `clipboard`, and
secret redaction rules apply to everything Andromeda emits.

#### Problem

Define which mechanism serves a copy action, in what order, with what size and content
bounds, and who may trigger clipboard writes at all — so that the clipboard never becomes
a silent exfiltration channel and copies do not fail mysteriously over SSH.

#### Forces and constraints

- Precedence order (Volume 1): security of credentials outranks user experience — a copy
  path that could leak secrets by default is unacceptable.
- OSC 52 support and size limits vary by emulator and multiplexer configuration (tmux
  `set-clipboard`); these are external facts that cannot be assumed.
- Agents and tools must not read or write the user's clipboard without an explicit
  permission decision — the clipboard is user territory.
- A failed copy must be visible: silently doing nothing destroys trust in the affordance.

#### Alternatives considered

1. **Probed hybrid: native first, OSC 52 fallback, explicit failure (selected).**
2. **OSC 52 always** — rejected: routes every local copy through the terminal stream for
   no benefit, inheriting size caps and observability exposure locally.
3. **Native only** — rejected: no copy at all over plain SSH, one of the primary usage
   modes (Volume 3 constrained environments).
4. **Unrestricted agent clipboard access** — rejected outright: a prompt-injected agent
   could stage secret material onto the clipboard; violates the permission model's intent.

#### Decision

1. Copy actions resolve their mechanism per the `tui.clipboard` key (`auto` | `osc52` |
   `off`, default `auto`): under `auto`, the PAL-native mechanism when its probe succeeds,
   else OSC 52 when the session context permits it (remote/SSH or explicitly configured),
   else the action fails visibly with guidance (FR-UX-075).
2. Payloads are bounded by `tui.clipboard_max_bytes` (default 1 MiB); larger selections
   are refused with an offer to export to a file instead (`export` family, chapter 05).
3. Only **user-initiated** actions write the clipboard, and they carry implicit consent —
   no Approval prompt — but every write emits `tui.clipboard.copied` (content kind and
   byte count, never content). Agent- or tool-initiated clipboard access is mediated by
   the `clipboard` permission through the Permission Manager (Volume 9) like any other
   side effect; the TUI itself never performs it on an agent's behalf.
4. Copied content passes the same redaction rules as rendered content; error-diagnostic
   copies use the redacted envelope fields only.
5. Per-emulator OSC 52 availability and size limits are **PENDING VALIDATION**, verified
   by the Volume 13 compatibility suite per terminal (register entry, fragment C).

#### Rationale

Native-first keeps local copies off the terminal stream; OSC 52 fallback preserves the SSH
workflow; explicit failure with guidance beats silent no-ops. Splitting user-initiated
(implicit consent, audited) from agent-initiated (permission-mediated) matches how the rest
of the product treats side effects and closes the exfiltration path without adding friction
to a human pressing "copy".

#### Positive consequences

- Copy works in the three real environments — local, SSH, multiplexed — with one
  user-visible model and one audit event.
- The exfiltration channel through agents is closed by the same mediation machinery used
  everywhere else.
- Size bounds prevent both emulator truncation surprises and megabyte escape floods.

#### Negative consequences

- Two mechanisms to test per platform, plus multiplexer passthrough configuration to
  document.
- Users on emulators without OSC 52 and without native utilities get refusal-with-guidance
  rather than a copy — honest, but a worse experience.

#### Risks

- Sensitive content copied by the user lands in multiplexer buffers or terminal logs
  (RISK-UX-079). Mitigation: redaction before copy, size bounds, audit event, and the
  documentation note that OSC 52 traverses the terminal stream.
- tmux/screen passthrough misconfiguration makes fallback appear broken. Mitigation:
  `doctor` reports the probed clipboard path; the failure message names the multiplexer
  setting involved.

#### Reversal plan

The mechanism order and bounds live behind the PAL Clipboard surface and two configuration
keys; changing order, adding a mechanism (e.g., a future terminal protocol), or tightening
bounds are configuration-schema-level changes that do not touch ports or stored data.

#### Review conditions

- Compatibility-suite results (the PENDING VALIDATION above) contradict the default order
  or the 1 MiB bound.
- The terminal ecosystem standardizes a size-negotiated clipboard protocol superseding
  OSC 52.
