### ADR-021 — Layered sandboxing: process-level controls in MVP, OS-level isolation from Beta/v1

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Sandbox Engine, Tool Runtime, Terminal Engine, Permission Manager, Platform Abstraction Layer (PAL)
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation.

#### Context

Andromeda executes agent-chosen commands and tools on the user's machine. The Sandbox Engine
"applies isolation policies to tool/command execution" (glossary), under the Permission
Manager's decisions and the Volume 9 sandbox model. The product principle Safe by Default and
the precedence order (safety first, implementation convenience last) demand containment of side
effects, while local-first and single-binary distribution (ADR-013) argue against heavyweight
isolation dependencies. Platform targets are macOS and Linux first; the two differ
fundamentally in isolation primitives (macOS Seatbelt profiles via `sandbox-exec`; Linux
Landlock, namespaces, and helpers such as bubblewrap).

#### Problem

What containment does Andromeda actually provide for tool and command execution in each phase,
on each platform — without overstating guarantees the implementation cannot yet honor, and
without depending on infrastructure (containers, daemons) that contradicts local-first
operation?

#### Forces and constraints

- Safe by Default: side-effecting execution MUST pass the Permission Manager and run under the
  strictest containment available in the current phase.
- Honesty of guarantees: the specification MUST NOT claim OS-level isolation where only
  process-level controls exist; misrepresenting containment is a safety defect.
- Platform primitives differ per OS and per OS version; their availability and stability cannot
  be assumed without implementation-time validation.
- Local-first: no container runtime, daemon, or VM dependency for core operation.
- Secrets already in the environment (ADR-014 territory) MUST NOT leak into child processes by
  default.

#### Alternatives considered

1. **Containers-only sandboxing** (a container runtime as the isolation mechanism) — rejected:
   requires a container runtime installed and running — a heavy external dependency, frequently
   absent on developer laptops, with implied-VM semantics on macOS — and breaks the
   local-first, single-binary simplicity of the product.
2. **No sandbox — rely on the permission prompt alone** — rejected: violates Safe by Default;
   an approved command still deserves environment filtering, path policy, and resource limits,
   because approval is not a grant of unlimited side effects.
3. **OS-level isolation from day one (MVP)** — rejected: per-platform mechanism validation,
   profile authoring, and failure-mode handling are substantial; shipping MVP behind that work
   delays everything else, and shipping it half-validated would overstate guarantees.

#### Decision

Andromeda adopts a **layered sandbox model**:

- **MVP — process-level controls**, implemented with Go and portable OS primitives:
  environment and secret filtering (deny-by-default env passthrough), working-directory and
  path policy enforcement, command allowlists/denylists evaluated by the Permission Manager,
  and resource limits (CPU time, memory, output size, wall-clock timeouts) on child processes.
  The specification and UX MUST describe MVP containment as process-level controls and MUST
  NOT claim OS-level isolation in MVP.
- **Beta/v1 — OS-level isolation**, added per platform behind the Sandbox Engine:
  macOS Seatbelt profiles via `sandbox-exec`, and on Linux Landlock, namespaces, and/or
  bubblewrap. **Each mechanism is PENDING VALIDATION**: its availability, stability, and
  behavior on supported platform versions MUST be validated during implementation before any
  isolation claim is made for it.
- The Sandbox Engine exposes one policy interface; layers are additive, and the effective
  containment level is observable (logged and queryable) per execution.

#### Rationale

Layering matches reality: process-level controls are implementable now, portable, and already
eliminate the most common failure classes (secret leakage into child environments, writes
outside the workspace, runaway processes). OS-level isolation is strictly stronger but
platform-fragmented and version-sensitive — exactly the kind of external dependency Volume 0
requires marking PENDING VALIDATION rather than asserting. Phasing lets the MVP ship honest,
useful containment while the isolation work proceeds with per-platform validation.

#### Positive consequences

- MVP ships real, truthfully described containment with zero external dependencies.
- The one-interface design lets isolation strength grow per platform without changing the Tool
  Runtime or Terminal Engine contracts.
- Effective-containment observability lets users and audits see what protection applied to each
  execution.
- Denied-by-default environment passthrough directly protects Secret Store material (ADR-014).

#### Negative consequences

- MVP containment does not stop a malicious approved binary from, for example, opening network
  connections; users must understand the phase-dependent limits (documentation duty in
  Volumes 8 and 9).
- Two-layer implementation and testing across three mechanisms (Seatbelt, Landlock, namespaces/
  bubblewrap) is a large, platform-specific engineering surface.
- Behavioral differences between platforms in Beta/v1 complicate uniform documentation.

#### Risks

- **PENDING VALIDATION**: `sandbox-exec` on macOS — its profile language and tool availability
  must be validated per supported macOS version before reliance.
- **PENDING VALIDATION**: Linux Landlock (kernel-version dependent), namespaces (distribution
  and privilege dependent), and bubblewrap (external binary presence) must each be validated
  before reliance.
- Users may assume stronger isolation than the current phase provides; mitigation: the
  observable containment level and explicit UX wording.
- Resource-limit enforcement differs per OS; mitigation: PAL owns the per-OS implementation
  with conformance tests.

#### Reversal plan

The Sandbox Engine's policy interface isolates mechanism choices. Dropping or replacing an
OS-level mechanism (for example, if a platform deprecates `sandbox-exec`) removes one backend
without affecting the process-level layer, which remains the guaranteed floor. Containers could
later be added as an optional backend through the same interface without reversing this ADR.

#### Review conditions

- Before any Beta/v1 isolation claim per platform: resolve that platform's PENDING VALIDATION
  with documented mechanism tests.
- If a supported OS deprecates or removes a mechanism (for example `sandbox-exec`), re-evaluate
  that platform's isolation backend.
- Revisit the containers-as-optional-backend question if user demand for stronger isolation
  outgrows OS primitives.
