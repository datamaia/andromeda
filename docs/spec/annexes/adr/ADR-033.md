### ADR-033 — Dependency-rule enforcement: manifest-generated depguard plus an import-graph test

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: all components; CI; composition root
- Related requirements: FR-ARCH-001, FR-PORT-001, NFR-ARCH-001, NFR-PORT-004

#### Context

FR-ARCH-001 makes the Volume 3 chapter 01 dependency matrix normative, and FR-PORT-001
confines platform-conditional constructs to the PAL. Go enforces neither: any package can
import any visible package. ADR-018 already pins golangci-lint in CI, whose `depguard` linter
restricts imports by path rules; Go's tooling also makes the full package graph cheaply
computable in a test. Volume 1's risk register (scope/pressure dynamics) and RISK-ARCH-001
say erosion is the expected failure mode if enforcement is manual.

#### Problem

Choose the mechanism that makes the layer matrix and the PAL containment rule merge-blocking
from the first commit: what checks run, from what source of truth, and how exceptions are
handled.

#### Forces and constraints

- One source of truth: layer assignments must live in exactly one machine-readable place, or
  the checks will drift from the spec.
- Two rule kinds exist: import-direction rules (matrix) and construct rules (`runtime.GOOS`,
  `syscall`, naked `go` statements) — one tool rarely covers both well.
- The mechanism must run in seconds on every PR (Volume 13 CI budgets) and produce failures
  that name the offending edge or construct precisely.
- Exceptions (the sanctioned L2 `errgroup` import, test-file allowances) must be explicit
  data, not tribal knowledge.
- ADR-018 already owns the lint stack; this decision should extend it, not add a second lint
  system.

#### Alternatives considered

1. **Manifest-generated depguard rules + import-graph test + construct scanner (selected).**
2. **depguard alone, hand-maintained rules.** Covers import direction but the rule file
   becomes the de facto (unreviewed) architecture as it drifts from chapter 01; no coverage
   for construct rules or manifest completeness. Rejected as sole mechanism.
3. **A dedicated architecture-test library (go-arch-lint / arch-go class).** Purpose-built
   DSLs, but adds an external tool with its own rule language and maintenance risk while
   duplicating what depguard + a ~200-line test already do against the same manifest.
   Rejected; re-evaluable if rule complexity outgrows the simple approach.
4. **Review discipline + CODEOWNERS only.** Fails the premise: RISK-ARCH-001 exists because
   review-time vigilance decays; violations are cheapest to reject mechanically at PR time.
   Rejected.

#### Decision

Enforcement rests on one **layer manifest** (a machine-readable file in the repository
mapping every package path prefix to its layer, with an explicit exception list), from which
CI derives three required checks:

1. **depguard rules** (golangci-lint, ADR-018) generated from the manifest — fast,
   per-import-line failures during normal linting.
2. **An import-graph test**: a standard Go test that computes the production import graph,
   verifies every edge against the chapter 01 matrix, verifies manifest completeness (every
   package assigned exactly one layer), and verifies the `sdk/` no-`internal`-imports rule
   (ADR-031).
3. **A construct scanner** for the FR-PORT-001 prohibited constructs (`runtime.GOOS`, OS
   build tags, `syscall`/`x/sys` imports outside `internal/pal`) and the ADR-023 naked-`go`
   rule outside the Task Scheduler — implemented as lint rules where golangci-lint supports
   them (`forbidigo`/`gochecknoglobals`-class configuration) and as part of the graph test
   otherwise.

All three are required PR checks from the first commit (NFR-ARCH-001, NFR-PORT-004).
Exceptions enter only via manifest edits, which are change-procedure-reviewed because the
manifest mirrors this specification.

#### Rationale

Generating depguard from the manifest keeps the reviewed artifact singular while retaining
depguard's speed and precise diagnostics; the graph test catches what path-prefix rules
cannot (completeness, sdk isolation, matrix cells with conditions); the construct scanner
covers the rules that are not imports at all. Everything runs inside toolchains already
pinned by ADR-018 — no new lint ecosystem, no DSL to learn, failures in the same place
contributors already look.

#### Positive consequences

- Violations fail in seconds at PR time with file/line/edge named; erosion becomes expensive
  instead of silent.
- The manifest is a reviewable, diffable statement of the architecture — spec and enforcement
  cannot drift apart unnoticed.
- The graph test doubles as documentation: its fixture is the chapter 01 matrix, literally.

#### Negative consequences

- Three checks to maintain as the tree grows; manifest edits accompany every new package.
- Generated depguard config must be regenerated deterministically (a make/CI step) or it
  rots.
- Construct scanning has false-positive potential (strings resembling constructs); the
  scanner needs an annotation escape hatch, which must itself be audited.

#### Risks

- Exception-list growth hollowing out the rules; mitigation: exceptions carry a justification
  string and an owner, audited at each release (NFR-ARCH-001 audit).
- Check circumvention via force-merge; mitigation: FR-ARCH-001's scheduled full check catches
  post-hoc, and branch protection (ADR-004) makes bypass exceptional.
- Tool churn (depguard config format changes); mitigation: versions pinned per ADR-018; the
  graph test is toolchain-only and survives linter churn.

#### Reversal plan

Each check is independently removable or replaceable; the manifest is the durable artifact.
Migrating to a dedicated architecture-lint tool (alternative 3) would consume the same
manifest and retire the hand-rolled graph test. Abandoning mechanical enforcement entirely
would require superseding this ADR and amending FR-ARCH-001/NFR-ARCH-001 — deliberately
heavyweight.

#### Review conditions

- Rule complexity exceeds what manifest + depguard express cleanly (e.g., conditional edges
  per build profile) — evaluate alternative 3 against the same manifest.
- CI time for the three checks exceeds the Volume 13 static-check budget — optimize or shard
  before weakening rules.
- golangci-lint deprecates depguard — regenerate against its successor from the unchanged
  manifest.
