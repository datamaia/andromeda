### ADR-013 — Release Tooling: goreleaser with GitHub Actions, cosign, syft, and SLSA Provenance

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Updater, Package Manager
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation

#### Context

Releases are the trust boundary between the project and every user: Volume 14 requires versioned, reproducible, verifiable artifacts for macOS and Linux (amd64/arm64), with Windows later; the Updater verifies what it downloads; supply-chain integrity ranks under precedence items 1–3. Development and CI live on GitHub (Volume 11), releases are tag-driven (ADR-004), and builds are CGO-free cross-compiled Go binaries (ADR-001, ADR-007).

Web-verified facts (2026-07-11): goreleaser v2.17.0, MIT-licensed (no license change; "Pro" is a separate paid tier), natively integrates cosign signing and syft SBOM generation. cosign is at v3.1.1, Apache-2.0 — a v2→v3 major-version transition has occurred, so the major version MUST be pinned in CI. syft is at v1.46.0, Apache-2.0. All three are active.

#### Problem

Define the release pipeline: how binaries for the platform matrix are built, packaged, checksummed, signed, attested, published, and distributed to package managers — repeatably from CI, with no release engineer's laptop in the loop.

#### Forces and constraints

- One tag MUST produce the complete artifact set deterministically: binaries per OS/arch, archives, checksums, signatures, SBOMs, provenance, release notes.
- Users MUST be able to verify artifact integrity and origin (Updater does this mechanically; humans via documented commands).
- Signing MUST NOT depend on long-lived private keys stored in CI secrets if a keyless alternative exists (cosign's OIDC-based keyless flow against the GitHub Actions identity).
- Homebrew is the expected macOS/Linux install path for the target audience; a tap MUST be maintained from the same pipeline.
- macOS Gatekeeper treatment of downloaded binaries depends on Developer ID signing and notarization, which requires an Apple Developer account — an unresolved organizational decision.
- Versioning scheme is owned by ADR-015; this pipeline consumes tags, it does not define them.

#### Alternatives considered

| Criterion | goreleaser (selected) | Manual release scripts | ko | nfpm-only |
|---|---|---|---|---|
| Scope | Full pipeline: build matrix, archives, checksums, sign, SBOM, publish, brews | Whatever we write and maintain | Container images for Go apps | Linux packages (deb/rpm/apk) only |
| Fit for CLI binaries | Purpose-built | n/a | Wrong artifact type — Andromeda ships binaries, not images | Partial (no macOS, no archives/tap) |
| cosign/syft integration | Native (verified) | Hand-wired | Image signing focus | None |
| Homebrew tap publishing | Built-in | Hand-written | No | No |
| Maintenance burden | Config file in-repo | Highest; bespoke bash is where release bugs live | n/a | Medium, incomplete |
| License/status (verified) | MIT, v2.17.0, active | n/a | Not applicable to our artifact type | Subsumed: goreleaser uses nfpm internally for Linux packages |

- **Manual scripts** are the default failure mode of release engineering: every checksum, signing, and upload step hand-maintained, drifting, and untested until a release breaks. Rejected on maintenance burden and error surface.
- **ko** solves container-image publishing for Go services; Andromeda's deliverable is a host binary, so ko addresses the wrong artifact class. It MAY reappear if a containerized distribution channel is ever added (out of scope here).
- **nfpm-only** covers Linux package formats but nothing else in the matrix (no macOS archives, no tap, no signing orchestration); goreleaser already drives nfpm internally, making standalone use strictly inferior.

#### Decision

1. Releases MUST be produced by **goreleaser** (pinned v2 line) running in **GitHub Actions**, triggered exclusively by release tags per ADR-004/ADR-015; no artifacts are built or published from developer machines.
2. Every release MUST include: per-platform archives for macOS/Linux (amd64/arm64), a checksums file, **cosign** signatures — keyless (OIDC, GitHub Actions identity), with the cosign **v3 major pinned in CI** — and per-artifact SBOMs generated by **syft** (pinned v1 line).
3. The workflow MUST generate **SLSA provenance attestations** binding artifacts to the source revision and the CI workflow identity, published alongside the release.
4. A **Homebrew tap** (`homebrew-andromeda`) MUST be updated by the same pipeline; Linux deb/rpm/apk packages are produced via goreleaser's nfpm integration.
5. The Updater (Volume 14) MUST verify checksum and signature before applying any update; verification commands for humans are documented per release.
6. macOS Developer ID signing and notarization is PENDING VALIDATION: it requires an Apple Developer account decision (organizational, with cost and identity implications). Until resolved, macOS artifacts ship with cosign verification plus documented Gatekeeper guidance; the open-questions register tracks the account decision.

#### Rationale

goreleaser is the only candidate whose scope matches the whole problem — cross-compiled matrix builds, packaging, checksums, native cosign/syft orchestration, tap publishing — as verified, actively maintained, MIT-licensed configuration rather than bespoke scripting. Keyless cosign signing removes the worst secret-management liability (a long-lived signing key in CI) by rooting trust in the repository's OIDC identity, which is also what the SLSA provenance attests. SBOM-per-artifact operationalizes the ADR-002 license policy at the point of distribution. The single deliberately deferred item — Apple notarization — is isolated as PENDING VALIDATION so an organizational dependency cannot block the rest of the pipeline.

#### Positive consequences

- One-tag releases: reproducible, auditable, laptop-free; release bugs become config diffs, reviewable in PRs.
- End-to-end verifiability: checksum → signature → provenance → SBOM, mechanically checkable by the Updater and by downstream packagers.
- Homebrew and Linux package channels maintained at zero marginal effort per release.
- No signing keys to rotate, leak, or lose.

#### Negative consequences

- Deep coupling to GitHub Actions' OIDC identity: releases cannot move off GitHub CI without re-establishing the signing trust model.
- Keyless verification requires users/Updater to trust the Sigstore infrastructure chain, a heavier conceptual model than a static public key.
- Until the notarization question resolves, macOS users face Gatekeeper friction on direct downloads (Homebrew installs mitigate in practice).
- goreleaser config accretes; it must be treated as production code with review and staging (snapshot builds on PRs).

#### Risks

- **cosign v3→v4 or Sigstore infrastructure changes** breaking verification paths: mitigated by the pinned major, Updater verification tests in CI, and release-candidate dry runs.
- **Supply-chain compromise of the CI workflow** would sign malicious artifacts with valid identity: mitigated by branch protection (ADR-004), environment-protected release workflows, provenance transparency, and third-party rebuild verifiability from the SBOM.
- **PENDING VALIDATION — macOS Developer ID notarization** (Apple Developer account decision): unresolved Gatekeeper posture for direct downloads; tracked in the open-questions register and in Review conditions below.

#### Reversal plan

Artifact contracts, not the tool, are the stable surface: names, checksums, signature and SBOM formats are specified in Volume 14. Replacing goreleaser means reimplementing the same contract in CI steps (matrix `go build`, archive, cosign, syft, upload) — tedious but mechanical, verifiable against the previous release's artifact layout. Reverting keyless signing to key-based cosign is a workflow change plus published-key documentation, kept possible because the Updater verifies via configurable trust policy rather than a hardcoded mechanism.

#### Review conditions

- PENDING VALIDATION: the Apple Developer account decision is made — amend this ADR with the Developer ID signing/notarization workflow (or an explicit, documented decision not to notarize) and resolve the open-questions entry.
- goreleaser v3 ships, changes its licensing model, or moves required features to the paid tier.
- cosign publishes a v4 major or Sigstore deprecates an infrastructure component the pipeline depends on.
- Windows-phase entry: extend the matrix and decide Authenticode signing (new ADR expected).
- The project migrates CI off GitHub Actions, invalidating the OIDC identity assumptions in rules 2–3.
