### ADR-013 — Release Tooling: goreleaser with GitHub Actions, cosign, syft, and SLSA Provenance

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Updater, Package Manager
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation

#### Context

Releases are the trust boundary with every user: Volume 14 requires versioned, reproducible, verifiable artifacts for macOS and Linux (amd64/arm64), Windows later; the Updater verifies what it downloads; supply-chain integrity ranks under precedence items 1–3. Development and CI live on GitHub (Volume 11), releases are tag-driven (ADR-004), and builds are CGO-free cross-compiled Go binaries (ADR-001, ADR-007).

Web-verified facts (2026-07-11): goreleaser v2.17.0, MIT (no license change; "Pro" is a separate paid tier), natively integrating cosign signing and syft SBOM generation; cosign v3.1.1, Apache-2.0 (a v2→v3 major transition occurred — pin the major in CI); syft v1.46.0, Apache-2.0. All three are active.

#### Problem

Define the release pipeline: how binaries for the platform matrix are built, packaged, checksummed, signed, attested, published, and distributed to package managers — repeatably from CI, with no laptop in the loop.

#### Forces and constraints

- One tag MUST deterministically produce the complete artifact set: per-OS/arch binaries, archives, checksums, signatures, SBOMs, provenance, release notes.
- Signing MUST NOT depend on long-lived private keys in CI secrets when a keyless alternative exists (cosign's OIDC keyless flow against the GitHub Actions identity).
- macOS Gatekeeper treatment of downloads depends on Developer ID signing and notarization, which requires an Apple Developer account — an unresolved organizational decision.

#### Alternatives considered

| Criterion | goreleaser (selected) | Manual release scripts | ko | nfpm-only |
|---|---|---|---|---|
| Scope | Full pipeline: build matrix, archives, checksums, sign, SBOM, publish, brews | Whatever we write and maintain | Container images for Go apps | Linux packages (deb/rpm/apk) only |
| Fit for CLI binaries | Purpose-built | n/a | Wrong artifact type — Andromeda ships binaries, not images | Partial (no macOS, no archives/tap) |
| cosign/syft integration | Native (verified) | Hand-wired | Image signing focus | None |
| Homebrew tap publishing | Built-in | Hand-written | No | No |
| Maintenance burden | Config file in-repo | Highest; bespoke bash is where release bugs live | n/a | Medium, incomplete |
| License/status (verified) | MIT, v2.17.0, active | n/a | Not applicable to our artifact type | Subsumed: goreleaser uses nfpm internally for Linux packages |

- **Manual scripts** are release engineering's default failure mode: hand-maintained checksum, signing, and upload steps that drift and go untested until a release breaks. Rejected.
- **ko** publishes container images for Go services; Andromeda ships host binaries — the wrong artifact class. It MAY be revisited if a containerized channel is ever added.
- **nfpm-only** covers Linux package formats and nothing else (no macOS archives, tap, or signing orchestration); goreleaser drives nfpm internally, making standalone use strictly inferior.

#### Decision

1. Releases MUST be produced by **goreleaser** (pinned v2 line) running in **GitHub Actions**, triggered exclusively by release tags per ADR-004/ADR-015; no artifacts are built or published from developer machines.
2. Every release MUST include: per-platform archives for macOS/Linux (amd64/arm64), a checksums file, **cosign** signatures — keyless (OIDC, GitHub Actions identity), with the cosign **v3 major pinned in CI** — and per-artifact SBOMs generated by **syft** (pinned v1 line).
3. The workflow MUST publish **SLSA provenance attestations** binding artifacts to the source revision and the CI workflow identity.
4. A **Homebrew tap** (`homebrew-andromeda`) — the expected macOS/Linux install path — MUST be updated by the same pipeline; Linux deb/rpm/apk packages are produced via goreleaser's nfpm integration.
5. The Updater (Volume 14) MUST verify checksum and signature before applying any update; verification commands for humans are documented per release.
6. macOS Developer ID signing and notarization is PENDING VALIDATION: it requires an Apple Developer account decision. Until resolved, macOS artifacts ship with cosign verification plus documented Gatekeeper guidance; the open-questions register tracks the decision.

#### Rationale

goreleaser is the only candidate whose scope matches the whole problem — matrix builds, packaging, checksums, native cosign/syft orchestration, tap publishing — as verified, maintained, MIT-licensed configuration rather than bespoke scripting. Keyless signing removes the worst secret-management liability (a long-lived key in CI) by rooting trust in the repository's OIDC identity — the same identity SLSA provenance attests. Per-artifact SBOMs operationalize the ADR-002 license policy at the point of distribution. The one deferred item — Apple notarization — is isolated as PENDING VALIDATION so an organizational dependency cannot block the pipeline.

#### Positive consequences

- One-tag releases: reproducible, auditable, laptop-free; release bugs become reviewable config diffs.
- End-to-end verifiability: checksum → signature → provenance → SBOM, checkable by the Updater and downstream packagers.
- Homebrew and Linux package channels maintained at zero marginal effort per release.
- No signing keys to rotate, leak, or lose.

#### Negative consequences

- Deep coupling to GitHub Actions' OIDC identity: moving CI elsewhere re-establishes the signing trust model.
- Keyless verification requires trusting the Sigstore infrastructure chain — conceptually heavier than a static public key.
- Until notarization resolves, macOS users face Gatekeeper friction on direct downloads (Homebrew installs mitigate this in practice).
- goreleaser config accretes; it is production code, reviewed and staged (snapshot builds on PRs).

#### Risks

- **cosign v4 or Sigstore infrastructure changes** breaking verification: mitigated by the pinned major, CI verification tests, and release-candidate dry runs.
- **Compromise of the CI workflow** would sign malicious artifacts with a valid identity: mitigated by branch protection (ADR-004), environment-protected release workflows, provenance transparency, and SBOM-based third-party rebuild verification.
- **PENDING VALIDATION — macOS Developer ID notarization**: unresolved Gatekeeper posture for direct downloads pending the Apple Developer account decision; tracked in the open-questions register and below.

#### Reversal plan

Artifact contracts, not the tool, are the stable surface: names, checksums, signature and SBOM formats are specified in Volume 14. Replacing goreleaser reimplements that contract in CI steps (matrix `go build`, archive, cosign, syft, upload) — tedious but mechanical, verifiable against the previous release's layout. Reverting to key-based cosign is a workflow change plus published-key documentation, possible because the Updater verifies via configurable trust policy, not a hardcoded mechanism.

#### Review conditions

- PENDING VALIDATION: the Apple Developer account decision is made — amend this ADR with the Developer ID signing/notarization workflow (or an explicit, documented decision not to notarize) and resolve the open-questions entry.
- goreleaser v3 ships, changes its licensing model, or moves required features to the paid tier.
- cosign publishes a v4 major or Sigstore deprecates an infrastructure component the pipeline depends on.
- Windows-phase entry: extend the matrix and decide Authenticode signing (new ADR expected).
- CI migrates off GitHub Actions, invalidating the OIDC identity assumptions in rules 2–3.
