### ADR-014 — Credential storage via OS keychains (zalando/go-keyring) with an age-encrypted file fallback

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Secret Store, Authentication Layer, Platform Abstraction Layer (PAL), Configuration Manager
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation.

#### Context

Andromeda authenticates against model providers using API keys, OAuth tokens, device
authorization flows, service accounts, temporary credentials, and multiple named profiles.
Credentials are the most sensitive data the product handles: the Volume 0 precedence order
places "safety of the user, the system, and credentials" first and "legality and exclusive use
of official mechanisms" second. The glossary defines the Secret Store as "the credential-storage
abstraction over OS keychains and the encrypted-file fallback"; the single-home matrix assigns
the credential storage model to Volume 9 with provider auth flows in Volume 5. Platform targets
are macOS and Linux first, Windows in a future phase; ADR-001 fixes Go. This ADR selects the
storage mechanism and libraries behind the Secret Store.

#### Problem

Where and how does Andromeda persist credentials so that they are protected by the strongest
mechanism each platform offers, remain available where that mechanism is absent (for example
headless Linux without a D-Bus session), and are never written in plaintext — while keeping the
dependency surface small and maintained?

#### Forces and constraints

- Credentials MUST NOT be stored in plaintext configuration files, and `andromeda.toml` MUST
  NOT be a credential carrier.
- Platform mandate: macOS Keychain, Linux Secret Service, encrypted files only as an explicit
  fallback, Windows Credential Manager in the future phase.
- A provider subscription does NOT imply programmatic access; Andromeda MUST use only official,
  documented, authorized authentication mechanisms. It MUST NOT use reverse engineering,
  captured cookies, tokens extracted from other applications, private APIs, undocumented
  endpoints, or web-UI automation to obtain or exercise credentials. The Secret Store therefore
  only ever holds material issued through official flows.
- Headless and containerized Linux environments frequently lack a Secret Service implementation;
  the product still needs a storage path there.
- ADR-001 (Go) and the cgo-free stance of ADR-007 favor pure-Go dependencies.
- Dependencies MUST be actively maintained; abandonment of a credential-handling library is a
  direct product risk.

#### Alternatives considered

1. **`github.com/99designs/keyring`** — multi-backend design including a file fallback.
   Rejected: verified DORMANT, last release and commit 2022-12-19, roughly 3.5 years
   unmaintained as of 2026-07-11, no maintained successor found. An unmaintained library in the
   credential path is unacceptable under the precedence order.
2. **Hand-rolled per-OS keychain integration** — rejected: duplicates well-tested integration
   work and enlarges the audited surface this project owns, without functional gain.
3. **Encrypted file as the primary store on all platforms** — rejected: discards the
   OS-provided protections (OS-user authentication, access control, secret isolation) that the
   platform mandate requires as the first choice.
4. **Plaintext file with restrictive permissions** — rejected outright; violates the "no
   plaintext credentials" constraint.

#### Decision

Andromeda implements the Secret Store on **`github.com/zalando/go-keyring`** (MIT, verified
active, v0.2.8 released 2026-03-23) for OS-native backends: macOS Keychain (the library shells
out to `/usr/bin/security`), Linux Secret Service over D-Bus, and Windows Credential Manager
when the Windows phase arrives. Because zalando/go-keyring has no encrypted-file fallback
(verified in source), Andromeda implements its **own encrypted-file fallback using
`filippo.io/age`** (BSD-3-Clause, v1.3.1 released 2025-12-28), which exposes a public Go library
API (Encrypt/Decrypt), not only a CLI. The fallback is opt-in and explicit: it is never selected
silently, requires a deliberate user action, and the CLI/TUI MUST mark it as the lower-security
option relative to the OS keychain. The Secret Store sits behind the Platform Abstraction Layer
(PAL); no other component touches credential bytes directly.

#### Rationale

zalando/go-keyring is the maintained, minimal option that covers exactly the three mandated
OS backends under a permissive license. age is a maintained encryption library whose small
public Go API fits a single-purpose fallback file. Building the fallback in-house keeps the
mandated "explicit fallback" semantics under Andromeda's control (consent, UX warnings, file
location per ADR-022) instead of inheriting a dormant library's behavior. The split matches the
single-home matrix: Volume 9 specifies the model this ADR implements; Volume 5 consumes it for
provider auth flows.

#### Positive consequences

- Strongest available protection on each platform by default; one abstraction (Secret Store)
  for all consumers.
- Headless Linux and CI-like environments retain a working, documented credential path.
- Windows support in the future phase requires no library change.
- Both dependencies are permissively licensed (MIT, BSD-3-Clause) and verified active in 2026.

#### Negative consequences

- Andromeda owns the security-sensitive fallback code (format choices, passphrase handling,
  file permissions) and its audits.
- Two credential code paths must be tested and documented instead of one.
- On macOS, shelling out to `/usr/bin/security` adds a subprocess dependency to every keychain
  operation.

#### Risks

- Secret Service variability across Linux desktops may produce inconsistent prompts or failures;
  mitigation: detection plus guided fallback selection.
- The `/usr/bin/security` interface could change behavior across macOS releases; mitigation:
  integration tests per supported macOS version.
- Users may over-adopt the fallback out of convenience; mitigation: explicit opt-in, persistent
  lower-security marking in UX, and documentation that favors the keychain path.

#### Reversal plan

All access goes through the Secret Store interface. Replacing zalando/go-keyring (for example,
if it becomes unmaintained) or the fallback implementation means swapping backends behind that
interface plus a one-time migration command that re-reads and re-stores entries; no consumer
component changes.

#### Review conditions

- Before the Windows phase: re-verify zalando/go-keyring Credential Manager support and release
  cadence.
- If zalando/go-keyring goes 12 months without maintenance activity, re-evaluate backends.
- If the Secret Service landscape changes materially (for example, a successor standard), revisit
  the Linux backend.
