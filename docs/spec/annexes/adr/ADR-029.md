### ADR-029 — Schema evolution: forward-only migrations, no downgrade support

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Persistence Layer, Updater, Configuration Manager
- Related requirements: Volume 1 product objectives (PRD-010); Volume 2 chapter 10; formal requirement links are added at consolidation

#### Context

Andromeda's two authoritative databases (ADR-028) will evolve across releases for years, on
end-user machines, with no operator on hand. A binary of version N may open workspace
databases last touched by many different older versions — and, after a user downgrades or a
workspace directory syncs between machines, a binary may encounter a database *newer* than
itself. Data integrity and recoverability rank third in the corpus-wide precedence order
(Volume 0, chapter 01), above functional requirements. The exit-code scheme reserves code 9
for integrity errors, including failed migrations (Volume 0, chapter 03; ADR-016).

#### Problem

Define the schema evolution regime: are migrations bidirectional (upgrade and downgrade
scripts), forward-only, or replaced by versioned parallel schemas — and what happens when a
binary meets a database from its future, or a migration fails halfway on an unattended
machine?

#### Forces and constraints

- Downgrade scripts double the authored and tested surface and are exercised orders of
  magnitude less than upgrades — they rot precisely because they are rarely used.
- Lossy-by-nature: many migrations (dropped columns, merged tables, normalized values) have
  no faithful inverse; "downgrade support" is therefore partial by construction and dishonest
  if promised generally.
- Unattended reliability: migration runs at open time on user machines; the failure path must
  be self-contained (no operator), which favors backup/restore over inverse scripts.
- SQLite specifics: some DDL is non-transactional or restricted; large migrations use
  copy-and-swap table patterns; file-level backup before migration is cheap because
  authoritative files exclude caches (ADR-028 rule 5).
- Cross-machine sync makes future-schema encounters a certainty, not an edge case.
- The Updater (Volume 14) already maintains rollback of the *binary*; pairing binary rollback
  with database restore-from-backup gives a complete revert story without inverse migrations.

#### Alternatives considered

| Criterion | Forward-only + backup (selected) | Bidirectional up/down scripts | Versioned parallel schemas | No migrations (rebuild state) |
|---|---|---|---|---|
| Authoring/testing surface | One direction, always exercised | Doubled, down path under-tested | Multiplied readers/writers | None |
| Faithful reversal | Via file restore (exact) | Only for lossless changes | N/A | N/A |
| Unattended failure story | Restore backup, exit 9 | Attempt inverse of a half-applied script | Complex | Data loss |
| Honesty of contract | Explicit: no downgrade | Implies a guarantee it cannot keep | Explicit but heavy | Unacceptable for PRD-010 |
| Old binary, new DB | Clean refusal | Clean-ish, still incompatible reads | Reader per version | N/A |

- **Bidirectional migrations** promise a downgrade that lossy changes cannot deliver, and the
  inverse of a *failed, half-applied* upgrade is exactly the case inverse scripts handle
  worst. A file-level backup restores the pre-migration state exactly, always.
- **Versioned parallel schemas** (binary reads N-1 and N) buy gradual rollout at the cost of
  dual read/write paths in every Persistence Layer query — disproportionate for a local
  single-writer product.
- **No migrations** discards user history on upgrade; PRD-010 forbids it.

#### Decision

Schema evolution for both authoritative databases is **forward-only**, with recovery via
backup. Binding rules (operationalized in Volume 2 chapter 10; procedures in Volume 10):

1. Migrations are numbered, immutable scripts embedded in the binary; each database records
   applied migrations in `schema_migrations` (version, applied_at, script checksum) and
   mirrors the current version in `PRAGMA user_version`; disagreement between the two is an
   integrity error.
2. Before migrating, the Persistence Layer takes and verifies a backup copy of the database
   file, and runs `PRAGMA integrity_check` plus `PRAGMA foreign_key_check`; a database that
   fails pre-checks is not migrated (exit code 9).
3. Migrations apply strictly in order, each transactionally (copy-and-swap for
   non-transactional DDL); post-migration, both checks re-run and `user_version` updates
   last.
4. On any failure: stop, preserve both the current file and the backup, report the stable
   error naming the failing migration, exit with code 9. Recovery is restore-from-backup;
   no downgrade script exists or will be authored.
5. An older binary encountering a newer database refuses cleanly with an integrity/
   compatibility error (exit code 9), naming both versions; partial reads of future schemas
   are forbidden.
6. Product downgrade path: restore the paired binary + database backup (Updater rollback per
   Volume 14, database restore per Volume 10) — the *pair* reverts, never the schema alone.
7. Cache databases are exempt: they never migrate; layout changes drop and rebuild them.

#### Rationale

Forward-only migration is the only regime whose failure and reversal stories are exact:
a byte-identical pre-migration file beats any inverse script, including for lossy changes and
half-applied failures. It halves the authored surface and concentrates testing on the path
every user actually takes. The clean-refusal rule for future schemas turns an inevitable
sync/downgrade scenario from silent corruption risk into a defined, recoverable error — in
line with the precedence of data integrity over convenience.

#### Positive consequences

- Every migration path shipped is a path exercised; no bit-rotted down scripts.
- Exact recovery semantics for unattended failures: restore file, rerun.
- Half of the migration code and tests that a bidirectional regime would carry.
- Defined behavior — not corruption — when binaries and databases skew.

#### Negative consequences

- Downgrading Andromeda without a database backup means the newer-schema workspace is
  unreadable by the older binary (by design; the error names the remedy).
- Backups momentarily double a database's disk footprint during migration (bounded because
  caches are excluded from authoritative files).
- Users who sync one workspace between machines with different Andromeda versions must keep
  versions aligned or accept read refusal on the older machine.

#### Risks

- **Backup step failure** (disk full) blocking upgrades. Detection: pre-flight space check.
  Mitigation: refuse migration with a precise error before touching the database; user frees
  space and retries.
- **Checksum mismatch** from a tampered/edited shipped migration. Detection: rule 1 checksum
  verification. Mitigation: refuse with integrity error; reinstall binary.
- **Skew frustration** in multi-machine sync workflows. Detection: support channels.
  Mitigation: clear error naming both versions and the upgrade path; Volume 14 documents the
  alignment requirement.

#### Reversal plan

Adopting downgrade scripts later is additive: the forward chain, `schema_migrations` ledger,
and checksums remain valid; inverse scripts would be introduced from a chosen version forward
(never retroactively), by a superseding ADR. Adopting parallel-version readers is likewise
additive behind the Persistence Layer. Nothing in this decision writes data that a future
regime could not read.

#### Review conditions

- Telemetry/feedback (consent-based, Volume 10) shows migration failures or version-skew
  refusals exceeding the reliability targets Volume 12 sets for upgrades.
- A team-sync feature makes cross-version workspace sharing a first-class scenario rather
  than an edge case.
- SQLite gains transactional DDL guarantees that materially simplify rule 3's copy-and-swap
  discipline.
