### ADR-102 — Interactivity resolution order and the scope of `--yes`

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: CLI, Permission Manager
- Related requirements: FR-CLI-009, FR-CLI-010, NFR-CLI-003, PRD-005, PRD-009

#### Context

The CLI runs in three interaction contexts: a person at a TTY, a script on a developer
machine, and CI. Volume 1 settles the governing tension (tension rule 2): unattended runs
never bypass permissions — they resolve them from pre-approved policy, and anything
unresolved is denied and recorded; a hung prompt in CI is a defect, and so is a silent
grant. Volume 3 (chapter 06) states the consequence for the CLI: non-interactive mode never
prompts, and policy-unresolved permissions deny with exit code 5. What remains for Volume 8
is the mechanics: how the CLI decides it is non-interactive, and what the consent flags
(`--yes`, `--no-input`) do and — critically — do not cover.

#### Problem

Define a deterministic, testable rule for entering non-interactive mode, and bound the
authority of `--yes` so that a convenience flag for confirmations can never become a
permission-granting mechanism.

#### Forces and constraints

- Wrong-direction failures are asymmetric: prompting in CI hangs a pipeline (availability
  defect); not prompting a human skips a designed checkpoint (safety defect). The detection
  rule must fail toward *not prompting and denying*, never toward hanging.
- Two distinct consent classes exist and must not be conflated: **permission approvals**
  (Approval entities evaluated by the Permission Manager, Volume 9) and **command-level
  destructive confirmations** (CLI-local "are you sure" checkpoints for operations like
  `index remove` or overwriting files during `init --force`).
- TTY probes are heuristics: stdin can be a pipe while the user is present, and CI systems
  allocate pseudo-TTYs. Explicit signals must always win over probes.
- The `CI` environment variable is a de-facto convention across major CI systems, not a
  standard; reliance on it is an assumption recorded in the Volume 8 register.

#### Alternatives considered

1. **Ordered resolution (explicit flag > environment > `CI` > TTY probe) with `--yes`
   scoped to confirmations only (selected).**
2. **TTY probe only** — rejected: undetectable-CI cases (pseudo-TTY runners) would prompt
   and hang; the user has no override lever when the probe misreads.
3. **`--yes` also pre-grants permission approvals** — rejected outright: it would create a
   one-flag bypass of the permission model, violating PRD-005 and Volume 1 tension rule 2.
   Pre-granting is the Policy Engine's job through recorded, scoped, revocable policy
   (Volume 9), not a CLI flag's.
4. **Interactive prompts with a default-answer timeout** — rejected: a timeout that
   defaults to "no" still stalls pipelines for its duration, and one that defaults to "yes"
   is an unattended destructive action; both are worse than deterministic refusal.

#### Decision

1. **Interactivity resolution order**, first match wins, evaluated once at command start:
   1. `--no-input` present → non-interactive.
   2. `ANDROMEDA_NO_INPUT` set to a truthy value → non-interactive.
   3. `CI` environment variable set to a truthy value → non-interactive (CI mode,
      FR-CLI-009).
   4. stdin **or** stderr is not a terminal → non-interactive.
   5. Otherwise → interactive.
   There is no flag that forces interactivity against a failed TTY probe; prompts require a
   real terminal.
2. **`--yes` (`-y`) auto-confirms command-level destructive confirmations only.** It has no
   effect on permission approvals: in interactive mode approvals still prompt through the
   Permission Manager's presenter; in non-interactive mode approvals resolve from policy
   and unresolved ones deny with exit code 5 (E-SEC family, Volume 9).
3. In non-interactive mode, a required confirmation without `--yes` fails immediately with
   E-CLI-003 (exit code 2), naming the confirmation subject and the flag that would consent
   to it.
4. Every confirmation and its resolution (prompted answer, `--yes`, or non-interactive
   refusal) emits `cli.confirmation.resolved` and is auditable.

#### Rationale

The ordered rule makes the mode a pure function of documented inputs — reproducible in
tests and diagnosable in the field (`--debug` prints the resolution and its deciding rule).
Placing explicit signals above probes gives users and CI authors a deterministic lever;
placing the TTY probe last makes the default safe when nothing was declared. Scoping
`--yes` narrowly preserves the two-tier consent architecture: CLI confirmations guard
against fat fingers; the permission model guards against unauthorized side effects — only
the first is a CLI-local concern a flag may waive.

#### Positive consequences

- Zero interactive hangs by construction in CI (NFR-CLI-003 verifies mechanically).
- `--yes` in a script can never widen an agent's authority; security review of automation
  reduces to reviewing policy, not flag soup.
- Mode resolution is loggable and testable as a decision table.

#### Negative consequences

- Users piping stdin while present lose prompts and must re-run with explicit consent
  flags; the error message carries the exact re-invocation.
- Truthiness parsing of `CI`/`ANDROMEDA_NO_INPUT` values must be specified and maintained
  (FR-CLI-009 defines the accepted set).
- Vendor-specific CI detection beyond `CI` is deliberately out, so some CI systems without
  `CI` set fall through to the TTY probe (which still resolves correctly in practice —
  runner stdio is piped).

#### Risks

- The `CI` convention erodes or a major vendor diverges; mitigation: register assumption
  with PENDING VALIDATION on the extended detection list; the explicit tiers above it
  always work.
- Users cargo-cult `--yes` into every script, eroding its checkpoint value; mitigation:
  confirmations remain rare and destructive-only (FR-CLI-010 enumerates the class), so the
  flag's blast radius stays bounded.

#### Reversal plan

The resolution order is one function consulted by all commands; reordering or extending it
(e.g., vendor CI detection) is a localized change with a documented behavior note. The
`--yes` scope boundary is contract: widening it would require superseding this ADR and
amending Volume 9's approval model — by design, an expensive path.

#### Review conditions

- Field reports of prompts in CI or of hangs (either direction of misdetection).
- The PENDING VALIDATION on extended CI detection resolves with evidence that `CI` alone
  is insufficient in practice.
- Volume 9 introduces policy syntax that makes per-invocation approval pre-resolution
  expressible — re-examine whether E-CLI-003's guidance should point to it.
