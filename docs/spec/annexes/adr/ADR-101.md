### ADR-101 — CLI structured output: human format by default, `--json` everywhere, NDJSON for streams, versioned schemas

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: CLI, Observability
- Related requirements: FR-CLI-006, FR-CLI-007, NFR-CLI-001, PRD-009; SM-20 stability regime

#### Context

PRD-009 requires every capability to be available non-interactively with structured output,
and the CLI's structured-output schema is explicitly listed among the public contracts SM-20
measures. Volume 3 (chapter 06) fixes that the CLI renders human and JSON output; it leaves
the format mechanics — flag shape, envelope, streaming representation, schema identity and
versioning — to Volume 8. Two consumers with different needs read command output: humans at
terminals, and programs (scripts, CI, editor integrations, other agents) that must parse
results and stream progress without scraping prose.

#### Problem

Define one output contract that (a) serves both consumers from every command, (b) survives
message rewording and feature growth without breaking parsers, (c) represents long-running
streaming commands, and (d) has an identity that contract-diff tooling can version-check per
SM-20.

#### Forces and constraints

- Exit codes carry outcome class (ADR-016); structured output must carry everything finer.
- A schema per command is unavoidable (results differ), but parsers also need one stable
  outer shape to handle any command generically.
- Streaming runs emit events for minutes; a single JSON document cannot be parsed until
  complete, which defeats live consumption.
- Error output must reuse the ADR-016 envelope — a second error shape would fork the
  contract.
- Redaction rules (Volumes 9/10) apply to structured output exactly as to logs.

#### Alternatives considered

1. **Boolean `--json` + fixed result envelope + NDJSON for streams (selected).**
2. **`--format <table|json|yaml|template>`** — rejected for MVP: every additional format is
   a compatibility surface under SM-20; YAML adds a parser dependency and ambiguity
   (multi-document, implicit typing) for no automation capability JSON lacks. `--format` can
   be introduced later additively with `json` as an alias, so choosing the boolean now loses
   nothing.
3. **JSON always, human rendering as the option** — rejected: the default consumer at a TTY
   is a person (PRD-008); requiring a flag for readability inverts the common case.
4. **Server-sent-event-style framing for streams** — rejected: NDJSON (one JSON document per
   line) is parseable with a line reader in every language, matches the JSON-RPC-adjacent
   grammar economy of ADR-009/ADR-010/ADR-012, and needs no framing parser.

#### Decision

1. Every command supports the global boolean flag **`--json`**. Without it, output is the
   human format; with it, stdout carries JSON documents only.
2. Single-result commands emit exactly one **result envelope** document on stdout, with the
   fixed outer fields defined in FR-CLI-006 (`schema`, `command`, `ok`, `exit_code`, `data`,
   `error`, `warnings`, `meta`). Command-specific content lives entirely under `data`.
3. Streaming commands under `--json` emit **NDJSON**: one JSON document per line — zero or
   more stream documents (`andromeda.cli.stream.v1`, wrapping the Volume 10 event envelope)
   followed by exactly one terminal result envelope as the final line.
4. The `error` field, when non-null, is the ADR-016 error envelope's machine-consumable
   subset (code, category, severity, message, detail, recoverability, retry policy,
   recommended action, correlation ID).
5. **Schema identity**: every result document names its schema as
   `andromeda.cli.<command-path>.v<major>` in the `schema` field. Schemas are published as
   JSON Schema documents (ADR-024) in the repository and versioned under the SM-20 regime:
   additive changes keep the major; removals or type changes require a major bump plus a
   deprecation window.

#### Rationale

The boolean flag plus fixed envelope gives generic parsers one contract and specific parsers
one `data` schema per command — the same two-level design as ADR-016's exit codes plus error
codes, applied to output. NDJSON is the smallest streaming representation that is
line-buffered, language-neutral, and consistent with the JSON-everywhere grammar the product
already speaks. Naming the schema inside the document makes every captured output
self-describing, which is what allows contract-diff tooling and replay fixtures (Volume 13)
to check compatibility mechanically.

#### Positive consequences

- One flag teaches the whole surface; `--json` composes with every command including future
  ones (FR-CLI-006 makes it a definition-of-done item).
- Captured outputs are self-identifying artifacts usable as test fixtures and audit
  evidence.
- Live consumers (editors, CI annotators) read progress with a line loop, no SDK.

#### Negative consequences

- Two renderers per command must be maintained and tested (golden tests for both).
- NDJSON's one-document-per-line rule prohibits pretty-printed streaming output; debugging
  raw streams requires tooling (`jq`) rather than eyes.
- Deferring `--format` means table/CSV/YAML consumers must post-process JSON for now.

#### Risks

- Envelope bloat: fields accreting into `meta` without governance; mitigation: `meta` fields
  are enumerated in FR-CLI-006 and additions follow the change procedure.
- Schema drift between implementation and published JSON Schema; mitigation: Volume 13 CI
  check validates every command's emitted output against its published schema on every PR.
- Accidental prose on stdout in `--json` mode breaking parsers; mitigation: FR-CLI-007's
  stdout discipline plus a CI test that pipes every command's `--json` output through a
  strict parser.

#### Reversal plan

The envelope is produced by one renderer behind the command layer. Adding `--format` or
replacing NDJSON with a framed protocol would be additive (new flag values, new schema
majors) under SM-20; withdrawing `--json` itself is a major-version contract break executed
with a deprecation window and is not anticipated.

#### Review conditions

- Demand materializes for a second machine format (YAML/CSV) from real consumers — introduce
  `--format` additively.
- Stream consumers need server-push flow control beyond line-buffered NDJSON (aligns with
  ADR-012's review condition on JSON-RPC streaming ergonomics).
- SM-20 contract-diff reporting shows schema churn above one additive change per command per
  minor release — revisit schema granularity.
