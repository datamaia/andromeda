### ADR-107 — TUI shell shape: single-window screen-per-concern model with a fixed panel system

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: TUI
- Related requirements: FR-TUI-001, FR-TUI-002, FR-TUI-003, FR-TUI-009

#### Context

The TUI must present many concerns — session transcript, plans, execution, tool calls,
diffs, git, files, context, memory, logs, costs, plus platform screens — inside one
terminal. ADR-006 fixes the framework (Bubble Tea v2, one program owning the terminal);
Volume 3 fixes the TUI as an L4 driver over the Runtime API. What remains open is the
macro-structure: how concerns map to what the user sees and navigates, at sizes from
40×10 to full-screen 4K terminals, with the 80×24 floor as the reference experience.

#### Problem

Choose the TUI's structural model — how concerns are arranged, addressed, and navigated —
such that every concern is reachable at 80×24, wide terminals are not wasted, navigation
is deterministic and testable, and the model does not degenerate as screens are added.

#### Forces and constraints

- 80×24 must carry full functionality (NFR-TUI-001): any structure that *requires*
  side-by-side composition fails the floor.
- Streaming content (transcript, tool output) needs maximal vertical space; dashboards
  splitting many tiny panes starve it.
- Bubble Tea renders one view; internal "windows" are simulated, so every structural
  fantasy (floating windows, tabs-in-tabs) is bespoke code to maintain and test.
- Users of terminal tools bring strong conventions: full-screen views with fast switching
  (vim buffers, k9s resources, lazygit panels) are learned behavior.
- Golden-frame testing favors few, deterministic layouts over freeform arrangements.

#### Alternatives considered

1. **Screen-per-concern with a fixed panel system (selected).** One active screen at a
   time; each screen composes 1–4 fixed panels via three layout classes (`wide`,
   `standard`, `compact`); a ring plus go-to chords plus palette for switching.
2. **Tiling workspace (user-arranged split panes, tmux-style)** — maximal flexibility.
   Rejected: arbitrary arrangements explode the test matrix and defeat golden frames;
   at 80×24 splits produce unusable slivers; layout management UI (resize, move, zoom)
   is a large bespoke subsystem orthogonal to the product's value.
3. **Single mega-dashboard (everything visible, k9s-header-style)** — rejected: the
   transcript and diff views need the full frame to be useful; a dashboard at 80×24
   shows nothing usable; adding the eleventh concern breaks any fixed dashboard.
4. **Modal drill-down hierarchy (menus → detail, no lateral movement)** — rejected:
   engineering workflows hop laterally (session → diff → git → session); forcing
   traversal through menus adds keystrokes to the product's hottest paths.

#### Decision

The TUI is a **single-window, screen-per-concern** application: exactly one active screen
occupies the frame; each screen declares 1–4 panels laid out by the shared layout manager
through three layout classes (`wide` ≥ 120 columns with sidebar, `standard` 80–119 as the
reference, `compact` below 80×24, with a 40×10 operable floor and a size notice below
it). Navigation is the ring (`[`/`]`), go-to chords (`g` + letter), and the command
palette; overlays (modals, prompts, palette) stack above the screen and trap focus.
Wireframes (chapters 09–10) are structure-normative per screen; the panel inventory is
change-controlled via FR-TUI-009. Wide terminals enhance (sidebar, wider panels) but
never gate functionality.

#### Rationale

Screen-per-concern is the only structure that makes the 80×24 contract trivially true —
the reference layout *is* the floor layout. It matches the learned conventions of the
tool population Andromeda joins, keeps the golden-frame matrix linear in screens (not
exponential in arrangements), and gives streaming content the full frame. The fixed panel
system inside screens recovers the useful part of dashboards (summary + detail
side-by-side when space allows) without user-managed layout. The layout-class mechanism
concentrates all size adaptivity in one tested component instead of per-screen media
queries.

#### Positive consequences

- 80×24 completeness is structural, not aspirational; compact behavior is one shared code
  path.
- Golden frames: screens × classes × tiers × modes — a bounded, enumerable matrix.
- New concerns are additive: a new screen slots into the ring/chords/palette without
  disturbing existing layouts.
- Deterministic navigation (FR-TUI-003) falls out of the structure.

#### Negative consequences

- No simultaneous side-by-side of two arbitrary concerns (e.g., logs beside transcript)
  in standard class; users needing that run two TUI instances in their own multiplexer —
  a supported pattern, explicitly delegated to tmux/terminal splits rather than
  reimplemented.
- The go-to chord letter space is finite; screen additions must fit the mnemonic scheme
  or live palette-only.

#### Risks

- Screen proliferation could bloat the ring and chord table; mitigated by keeping
  platform screens out of the ring (palette-reachable) and by FR-TUI-009 change control.
- A future concern might genuinely need composition (e.g., watch logs while reviewing a
  diff); mitigated by the sidebar in wide class and, if demand proves out, a deliberate
  ADR for a second visible region rather than ad-hoc splits.

#### Reversal plan

The layout manager is the single authority for geometry; screens declare panels
abstractly. Moving to a tiling model would replace the layout manager and navigation
bindings while screen sub-models and their panels survive as the tiles. Golden frames
would be regenerated; the Runtime-facing code is untouched.

#### Review conditions

- Recurring user demand for simultaneous views that two-instance multiplexing does not
  satisfy (tracked via issues) — trigger the composition ADR.
- Screen count approaching the chord/mnemonic limit.
- Golden-matrix runtime becoming a CI bottleneck (would indicate the class/tier matrix
  needs pruning, a structural signal).
