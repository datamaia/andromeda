### ADR-056 — Capability enum extension (`token_counting`) and provenance-tracked capability resolution

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Provider Layer, Agent Engine, Context Manager, CLI/TUI
- Related requirements: FR-PROV-010, FR-PROV-011, FR-PROV-012

#### Context

The capability enum is a closed vocabulary owned by Volume 5 (Volume 2, INV-CAP-01; Volume 0
chapter 03: additions require an ADR). The authoring spine seeds fourteen values. The frozen
`ProviderPort.CountTokens` method (Volume 3, chapter 02) is contractually served "exactly
when the provider offers an official counting mechanism" — a declarable fact no seed value
expresses (`token_usage_reporting` covers response-side accounting, not request-side
counting). Separately, the generic OpenAI-compatible adapter cannot statically know an
arbitrary endpoint's capabilities, so declaration alone cannot be the whole model.

#### Problem

(a) How does an adapter declare an official token-counting mechanism? (b) How are effective
capability sets resolved when declarations, discovery metadata, user configuration, and
verification evidence can all speak — without permitting silent simulation or name-based
inference?

#### Forces and constraints

- The enum is closed and additions are rare, deliberate events (INV-CAP-01).
- Principle 2: no silent simulation; absence is reported; behavior never keys off names.
- The OpenAI-compatible adapter serves unknown endpoints; user knowledge must be admissible.
- Verification evidence must beat declarations when they conflict (honesty).

#### Alternatives considered

1. **Overload `token_usage_reporting` to imply counting** — rejected: conflates two
   different official mechanisms; a provider may report usage yet offer no counting
   endpoint, forcing adapters to lie in one direction.
2. **Open, adapter-extensible capability strings** — rejected: destroys the closed
   vocabulary that makes negotiation and honesty checks possible (INV-CAP-01/02).
3. **Declaration-only resolution (no config overrides, no probes)** — rejected: makes the
   generic adapter either uselessly conservative or dishonestly optimistic.
4. **Add `token_counting`; resolve sets from four provenance classes** — chosen.

#### Decision

Add one value, `token_counting` (level `both`), to the capability enum. Resolve effective
capability sets by the fixed procedure of Volume 5, chapter 02: `declared` baseline, then
`discovered` metadata, then `configured` overrides (additive only where the declaration
marks a capability configurable), then masking by `refuted` verification evidence — each
value carrying its provenance class, with capability changes evented (INV-MDL-03).

#### Rationale

The single addition keeps `CountTokens` honest without overloading semantics; provenance
classes admit user knowledge and empirical evidence into resolution while keeping every
claim attributable and every mask observable.

#### Positive consequences

- `CountTokens` availability is declarable, negotiable, and testable.
- The generic adapter serves unknown endpoints honestly via configurable capabilities.
- Refutation evidence wins over optimism; masks are visible events.

#### Negative consequences

- Resolution is more complex than a static set; determinism rules and tests must hold it.
- One more enum value for every capability matrix surface to display.

#### Risks

- Users over-granting configurable capabilities; mitigation: provenance display, probe
  level `probe`, honest E-PROV-006/E-PROV-018 failures (RISK-PROV-002).

#### Reversal plan

Removing the value or the provenance classes is a Volume 0 change-procedure amendment;
persisted sets remain readable (unknown values surface as validation errors per INV-CAP-02,
which is the designed downgrade behavior).

#### Review conditions

- Review if providers standardize documented capability-discovery metadata (provenance
  class `discovered` may then dominate and simplify).
- Review at each enum-addition proposal: this ADR's bar (a frozen contract surface needing
  a declarable fact) is the precedent.
