### ADR-031 — Go module and package layout

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: all components; Extension SDK; composition root
- Related requirements: FR-ARCH-001, FR-ARCH-002, PRD-007, PRD-011

#### Context

ADR-003 fixed a monorepo with the SDK as a separate Go module under `sdk/`; ADR-030 fixed the
layered hexagonal style. What remains is the concrete Go layout: how many modules, where port
interfaces live, how the 37 glossary components map to packages, and how the public API
surface is bounded. Go's `internal/` visibility rule and multi-module repositories are the
available mechanisms; the Volume 0 glossary names the components whose boundaries the layout
must express.

#### Problem

Define the module and package structure such that the ADR-030 layers are visible in paths,
the only importable public Go API is the SDK, one component maps to one package subtree, and
the layout survives growth without renegotiation.

#### Forces and constraints

- The public API surface must be minimal and deliberate: SM-20 measures contract stability,
  and everything importable is a de facto contract.
- Port interfaces are consumed by every layer; their package must be import-cycle-free and
  dependency-free.
- The SDK must not drag the product's dependency graph into extension builds (ADR-003 already
  requires a lean `sdk/` module).
- depguard-style enforcement (ADR-033) works on package path prefixes; the layout must make
  layer membership derivable from paths.
- AI-agent contributors navigate by naming: glossary-component-to-package mapping should be
  one-to-one and predictable.

#### Alternatives considered

1. **Single module, `internal/` component packages, ports in one package, SDK as second
   module (selected).** Layer and component visible in the path; `internal/` closes the API
   by default.
2. **Ports defined next to their consumers** (Go's "accept interfaces where you use them"
   idiom, one interface per consuming engine). Idiomatic for small programs, but here the
   same port has many consumers (chapter 02 ownership map), the SDK must mirror the
   contracts, and eleven volumes need one authoritative location to cite. Scattering the
   contract definition would fork it. Rejected for ports; the idiom remains fine for
   package-private helper interfaces.
3. **Multi-module workspace (module per component or per layer).** Maximal isolation, but
   Go multi-module repos multiply versioning, tagging, and dependency-update overhead per
   module, break atomic refactors across components (ADR-003's core argument), and buy
   nothing `internal/` does not already provide inside one binary. Rejected.
4. **Exported top-level packages (`pkg/` convention) instead of `internal/`.** Makes every
   package importable by the world, converting internal refactors into potential breaking
   changes; contradicts the bounded-public-API force. Rejected.

#### Decision

- **Two Go modules** in the monorepo: the main module (product) and `sdk/` (public Extension
  SDK), per ADR-003.
- Main module layout: `cmd/andromeda/` (composition root); `internal/core/` (L0);
  `internal/ports/` (L1 — all 18 port interfaces plus shared contract types, no I/O, no
  third-party imports); one `internal/<component>/` subtree per glossary component (L2/L3/L4
  assignments per Volume 3 chapter 01), with provider adapters under
  `internal/provider/adapters/<name>/`.
- **The SDK mirrors, never imports.** `sdk/` MUST NOT import any `internal/` package;
  extension-facing contract types are mirrored, with a CI equivalence check keeping mirror
  and source identical (mechanism: generated code or structural-equality tests — chosen at
  implementation, both satisfy this ADR).
- One component, one package subtree; no cross-subtree imports of another component's
  internals; layer membership is derivable from the path via the ADR-033 manifest.

#### Rationale

`internal/` gives the bounded-API property for free; a single ports package gives the corpus
its one citable contract location and the SDK a single mirroring target; per-component
subtrees make the glossary, the spec chapters, CODEOWNERS paths (ADR-003), and the import
graph all use the same names. The two-module split is the minimum that isolates extension
builds from the product dependency graph — every additional module would add versioning
overhead without adding isolation the binary needs.

#### Positive consequences

- Public Go API = `sdk/` exactly; contract-diff tooling (SM-20) has one module to watch.
- Layer enforcement reduces to path-prefix rules; ADR-033 stays simple.
- Predictable navigation: component name → package path, one-to-one.
- Atomic cross-component refactors remain single-PR (ADR-003 preserved).

#### Negative consequences

- Mirror maintenance between `internal/ports` and `sdk/` is ongoing work with CI cost.
- One large ports package concentrates contract churn; its file organization needs internal
  discipline (per-port files).
- `internal/` prevents downstream Go programs from importing product packages even when that
  would be convenient — deliberate, but occasionally annoying for tooling authors (the IPC
  surface is their sanctioned path).

#### Risks

- Mirror drift between SDK and ports if the equivalence check is weak; mitigation: the check
  is a required CI gate (FR-ARCH-001 acceptance criteria include it).
- The ports package accreting non-contract helpers; mitigation: the L1 purity rules
  (no I/O, no third-party imports) are depguard-enforced.
- Component subtree boundaries eroding via "shared util" packages; mitigation: shared code
  lives in `internal/core` (domain) or a manifest-listed L2 utility package, reviewed via the
  change procedure.

#### Reversal plan

Layout changes are `git mv` plus manifest updates plus import rewrites — mechanical and
tool-assisted in Go. Extracting the SDK to its own repository stays cheap per ADR-003's
reversal plan. Moving ports next to consumers (alternative 2) would be a package split with
type aliases during transition. None of these require re-deciding ADR-030.

#### Review conditions

- The mirror-equivalence mechanism proves fragile in practice (recurring drift incidents) —
  re-evaluate toward full code generation.
- A second binary deliverable appears (beyond `andromeda`) — revisit `cmd/` and composition
  sharing.
- Go module tooling changes materially (workspace/versioning improvements) that would lower
  multi-module costs — re-evaluate alternative 3 for the SDK boundary only.
