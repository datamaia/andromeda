### ADR-055 — Provider adapters declare themselves via a static, versioned Adapter Declaration manifest

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Provider Layer, Extension SDK, Package Manager, Configuration Manager
- Related requirements: FR-PROV-001, FR-PROV-002, FR-PROV-010

#### Context

Volume 5 must make every provider fact — capabilities, endpoints, auth methods, streaming
and tool-calling behavior, usage reporting, rate-limit signals, error mapping, health probes,
deprecations — available to the router, the conformance suite, and users, across nineteen
named adapters plus third-party ones (Volume 1 integration list; brief declaration set).

#### Problem

Where do provider facts live: implicit in adapter code, in central configuration, or in a
validated per-adapter artifact — such that honesty is checkable, registration is refusable,
and facts version with the adapter?

#### Forces and constraints

- Capability honesty (Principle 2) needs facts as data, not code behavior.
- Facts must be validatable without executing network code (offline, at registration).
- Extension adapters (Volume 6) must carry the same facts as built-ins.
- Facts change with provider APIs; changes must be visible, not silent (SM-20 spirit).

#### Alternatives considered

1. **Facts implicit in adapter code** — rejected: unverifiable without execution; honesty
   becomes a code-review judgment; no registration-time refusal.
2. **Central fact registry maintained in the core** — rejected: re-creates provider-specific
   knowledge outside adapters (violates Principle 1's adapter confinement) and turns every
   provider change into a core change.
3. **Static, versioned per-adapter declaration manifest, validated at registration** —
   chosen.

#### Decision

Every adapter ships an Adapter Declaration (structure in Volume 5, chapter 01) containing
the full declaration set. The registry validates it at registration (ADR-024 schema,
capability vocabulary, error-map targets, contract-major compatibility) and refuses invalid
declarations with E-PROV-019. Declarations are immutable per `AdapterVersion`; every fact is
sourced from official documentation, and undocumented facts are absent, recorded as
PENDING VALIDATION in the adapter catalog (chapter 09).

#### Rationale

A validated manifest turns provider facts into checkable inputs for negotiation, routing,
conformance testing, and user inspection, while keeping all provider specificity inside the
adapter package — data included.

#### Positive consequences

- Registration-time refusal of malformed or incompatible adapters.
- Capability honesty and behavior claims are mechanically testable against declarations.
- Fact changes are version changes — diffable, reviewable, announceable.

#### Negative consequences

- Declaration authoring is real work per adapter (mitigated by the SDK template, SM-01).
- Facts can lag providers between adapter releases; the drift surfaces as E-PROV-008
  patterns rather than silent misbehavior.

#### Risks

- Declarations copied between adapters without verification against official docs;
  mitigation: per-fact-group source strings and conformance cross-checks (RISK-PROV-002).

#### Reversal plan

The manifest is additive metadata around adapters implementing the frozen ProviderPort;
abandoning it would fall back to code-implicit facts by deleting validation, with no port or
consumer changes — an information loss, not an architectural rework.

#### Review conditions

- If per-adapter SDK adoptions (ADR-019) supply equivalent typed metadata, evaluate
  generating declarations from them.
- Review when the third-party adapter count makes declaration audits the bottleneck.
