### ADR-147 — Git hosting integrations as built-in tools over official APIs

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Tool Runtime, Git Engine, Authentication Layer, CLI, TUI
- Related requirements: FR-GIT-009, FR-GIT-010, FR-TOOL-001; ADR-019, ADR-014

#### Context

Andromeda the product integrates with Git hosting services — GitHub first, GitLab later —
to create and manage pull requests / merge requests, link issues, and read review state.
This is distinct from GitHub as the platform on which Andromeda itself is developed (Volume
11 chapters 03–07). The Volume 6 built-in tools catalog already names `github` and `gitlab`
tools; the Volume 1 provided constraints categorically exclude unofficial mechanisms
(scraping, private APIs, session-cookie reuse). The glossary's component list has no
dedicated "hosting client" component, and the 18 ports are frozen (FR-ARCH-003).

#### Problem

Where in the architecture do product-side hosting integrations live, through which
mechanisms do they reach the services, and how are GitHub and GitLab phased — without
minting a new component or port, and without provider-specific logic leaking outside an
adapter-shaped boundary?

#### Forces and constraints

- Frozen ports and component list: no new port; no new glossary component without a Volume 0
  amendment.
- Hosting calls are side-effecting network operations: they must pass permission evaluation
  (`external_service_access`, `network`) and produce audit records — exactly the mediation
  the Tool Runtime already provides for every tool (Principle 4, FR-TOOL-001).
- Credentials are service tokens: acquisition and storage must go through AuthPort and
  SecretStorePort (ADR-014), never plaintext configuration.
- Vendor-agnosticism (PRD-002 analog for hosting): CLI/TUI flows should not fork per
  provider for the common change-request operations.

#### Alternatives considered

1. **Extend the Git Engine to speak hosting HTTP APIs** — rejected. The Git Engine's charter
   (ADR-025) is subprocess git with parsing discipline; grafting HTTP clients onto it mixes
   two failure domains and bypasses the Tool Runtime's permission mediation.
2. **A new HostingPort and Hosting Engine component** — rejected. Requires amending the
   frozen port set and glossary for a capability the Tool Runtime already mediates; adds a
   port whose only consumers would be two tools.
3. **Shell out to `gh`/`glab` CLIs** — rejected as the contract. It adds runtime
   prerequisites beyond git, parses human-oriented output, and hides which endpoints are
   called from the permission model. (Official CLIs remain fine as user-side tools outside
   Andromeda.)
4. **Built-in tools over official HTTP APIs** — selected, described below.

#### Decision

1. Product-side hosting integrations are delivered as the **built-in `github` and `gitlab`
   tools** (Volume 6 catalog); Volume 11 chapter 02 owns their behavioral contract. All
   hosting side effects therefore pass tool-level permission evaluation, sandbox policy, and
   observability.
2. Both tools speak only **official, documented HTTP APIs** (GitHub REST/GraphQL; GitLab
   REST) per the ADR-019 HTTP baseline, with explicit `api_base_url` configuration for
   self-hosted/enterprise instances. No web automation, no undocumented endpoints, no
   cookie reuse.
3. A provider-neutral **change-request vocabulary** (one set of operations covering pull
   requests and merge requests) defines the tools' shared surface; provider-specific
   capabilities beyond it are declared per tool, never silently emulated.
4. Authentication uses AuthPort flows with material behind SecretStorePort references;
   supported grant types per provider are those officially documented for third-party
   applications, each marked PENDING VALIDATION until verified against current official
   documentation at implementation time.
5. Phasing: `github` tool at **Beta**; `gitlab` tool at **v1**. Neither is MVP.

#### Rationale

The Tool Runtime is the one place that already gives hosting calls everything Volume 9 and
Volume 10 demand — permission mediation, audit, events, timeouts, cancellation — so
delivering the integrations as tools reuses mediation instead of duplicating it. Official
APIs are the only mechanisms consistent with the Volume 1 provided constraints. The
change-request vocabulary keeps CLI/TUI flows provider-neutral, mirroring how ProviderPort
keeps model access vendor-neutral.

#### Positive consequences

- Zero new ports or components; the frozen architecture is untouched.
- Agents reach hosting operations exactly like any tool — with schemas, permissions, and
  audit — so PR creation by an agent is as governed as a file write.
- Enterprise GitHub/GitLab installations work by configuration, not code.
- The provider-neutral vocabulary makes adding a third hosting provider an additive tool.

#### Negative consequences

- Two API surfaces to track for drift and deprecations (RISK-GIT-004).
- The lowest-common-denominator vocabulary excludes provider-unique features from the
  neutral surface; users needing them use provider-specific tool operations.
- Tool-shaped delivery means hosting operations are unavailable wherever the Tool Runtime is
  (deliberately) restricted.

#### Risks

- Hosting APIs change authentication or rate-limit policy for third-party clients
  (RISK-PRD-002 analog); mitigation: capabilities are declared and verified per release;
  degradation is explicit, never silent.
- Token over-scoping grants more than the tools need; mitigation: chapter 02 declares
  minimal scopes per operation class; documentation instructs fine-grained tokens.

#### Reversal plan

If a dedicated hosting component ever becomes justified (many providers, heavy shared
logic), the tools' behavioral contract transfers to it unchanged and the tools become thin
adapters over it; consumers see the same tool surface. Dropping a provider follows tool
deprecation rules (Volume 6), not a breaking change to the other provider.

#### Review conditions

- At Beta exit: review GitHub tool usage and error telemetry before committing GitLab v1
  scope.
- When a third provider (e.g., Bitbucket, Gitea/Forgejo) is proposed: review whether the
  change-request vocabulary generalizes or needs a version bump.
- If either provider's official API terms change for third-party clients, re-validate the
  PENDING VALIDATION items of chapter 02 immediately.
