### ADR-017 — Testing stack: stdlib testing, go-cmp, testify, rapid property tests, native fuzzing, teatest for the TUI

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Core Domain, Runtime, CLI, TUI, Tool Runtime, Extension SDK
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation.

#### Context

Andromeda is implemented in Go (ADR-001) with a TUI built on bubbletea v2 (ADR-006). The
specification's verification methods (Volume 13) depend on a concrete testing toolchain covering
unit tests, deep structural comparison, property-based testing for parsers and protocol codecs
(Andromeda Runtime Protocol per ADR-009, configuration per ADR-008, schemas per ADR-024),
fuzzing of untrusted inputs, and deterministic testing of terminal UI behavior. Library facts
below were web-verified on 2026-07-11.

#### Problem

Which testing libraries form the supported stack, such that tests remain plain Go, external
dependencies stay test-only, licensing stays within policy, and the TUI can be tested without
manual terminal interaction?

#### Forces and constraints

- Test dependencies MUST NOT leak into shipped binaries; test-only scope is a hard boundary.
- License policy: the product is Apache-2.0 (ADR-002); test-only dependencies with file-level
  copyleft need an explicit policy flag rather than silent adoption.
- Property-based testing and fuzzing are the practical way to exercise parser/codec edge cases
  the specification requires (invalid config, malformed protocol frames, schema violations).
- The TUI (bubbletea v2 per ADR-006) needs programmatic driving of model/update/view cycles;
  hand-rolled terminal emulation is out of proportion for this project.
- Contributor accessibility: idiomatic `go test` workflows, no bespoke runners.

#### Alternatives considered

1. **ginkgo/gomega** — BDD-style suite runner and matcher DSL. Rejected: the DSL layer adds a
   second vocabulary on top of Go, complicates debugging and tooling integration, and its
   suite lifecycle diverges from plain `go test`, raising the contributor entry cost without a
   capability the chosen stack lacks.
2. **gopter** for property-based testing. Rejected: verified as less maintained than rapid, with
   a heavier generator API; rapid integrates with native fuzzing and has current releases.
3. **Hand-rolled TUI harness** — driving bubbletea programs with custom fakes. Rejected as the
   default: duplicates what teatest provides; retained only as a contingency (see Reversal
   plan).

#### Decision

The supported testing stack is:

- **stdlib `testing`** as the only test runner; all tests run under `go test`.
- **`github.com/google/go-cmp`** for structural comparison and diff output in assertions on
  domain values.
- **testify** assertions (`assert`/`require`) for concise checks where go-cmp diffing is not
  needed.
- **`pgregory.net/rapid`** (verified active, v1.3.0 released 2026-04-30, MPL-2.0) for
  property-based tests of parsers, codecs, and invariants. MPL-2.0 has file-level copyleft:
  rapid is restricted to test-only scope and MUST be flagged in the license policy as a
  reviewed exception to the permissive-only default.
- **Native Go fuzzing** (`go test -fuzz`) for untrusted-input surfaces: configuration parsing,
  protocol frames, schema documents, provider responses.
- **`github.com/charmbracelet/x/exp/teatest/v2`** (verified published 2026-07-05; lives under
  the experimental `/exp/` tree with pseudo-versions only) for TUI testing against bubbletea v2.
  It MUST be pinned to an exact pseudo-version and used in test-only scope.

#### Rationale

The stack keeps tests idiomatic: one runner, plain functions, no lifecycle DSL. go-cmp gives
reviewable diffs on the domain model's nested structures; testify covers the simple-assertion
majority ergonomically. rapid is the maintained property-testing option in the Go ecosystem in
2026 and complements — rather than replaces — native fuzzing: rapid states invariants over
generated values, fuzzing explores crash/panic surfaces with coverage guidance. teatest/v2 is
the matching test harness for the already-decided bubbletea v2 (ADR-006), from the same
ecosystem, and testing the TUI at the model/update/view level is the only maintainable
alternative to manual terminal QA.

#### Positive consequences

- Uniform `go test` workflow across unit, property, fuzz, and TUI tests; CI needs no special
  runners.
- Parser and protocol edge cases mandated by the specification get systematic, generative
  coverage.
- TUI behavior becomes regression-testable, enabling the Volume 8 acceptance criteria to be
  automated.
- All dependencies are test-only; the shipped binary's dependency set is unaffected.

#### Negative consequences

- Two assertion styles (go-cmp, testify) coexist; contributor guidelines must state when to use
  which.
- rapid's MPL-2.0 requires a standing license-policy exception and periodic license review.
- teatest/v2 is experimental with pseudo-versions: upgrades are manual, and breaking changes in
  the harness are possible between pins.

#### Risks

- teatest/v2 API churn (experimental `/exp/` status) could break the TUI test suite on upgrade;
  mitigation: exact pseudo-version pin, upgrades only as deliberate PRs with the suite green.
- Property tests can be flaky if authors use unbounded generators; mitigation: rapid's
  deterministic replay plus generator guidelines in Volume 13.
- testify misuse for deep structural equality produces poor failure output; mitigation: lint
  guidance directing structural comparisons to go-cmp.

#### Reversal plan

All choices are test-only and per-file replaceable: testify assertions can be mechanically
rewritten to stdlib checks; rapid properties can be ported to table-driven tests (losing
generative power); teatest can be replaced by a hand-rolled bubbletea driver behind a small
helper package that the suite already funnels through. No shipped artifact changes on reversal.

#### Review conditions

- When teatest graduates out of `/exp/` (or bubbletea v3 appears), re-pin and re-evaluate.
- If rapid's maintenance stops for 12 months or its license terms change, re-evaluate against
  then-current property-testing options.
- Review the license-policy exception for rapid at each release audit (ADR-013 SBOM output makes
  test/runtime scope visible).
