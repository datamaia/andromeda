### ADR-065 — Adapter catalog phasing: generic-adapter-first, dedicated adapters by declared value

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Provider Layer, Configuration Manager
- Related requirements: FR-PROV-080; Volume 1 chapter 05 (MVP provider seed)

#### Context

The brief mandates an integration list of 19 providers/servers, states that list membership
does not imply MVP support, and requires per-phase classification. Volume 1 fixes the MVP
seed: generic OpenAI-compatible + Anthropic + Ollama. Most catalog services — cloud (Groq,
Together, DeepSeek, xAI, OpenRouter) and local (vLLM, LM Studio, llama.cpp, LocalAI,
FastChat, Text Generation WebUI, LiteLLM) — expose OpenAI-compatible API surfaces and are
therefore reachable through one adapter with a configured base URL.

#### Problem

Which services get dedicated adapters, in which phase, and what does a dedicated adapter owe
beyond what the generic adapter already provides — without committing implementation effort
to services whose delta over the generic path is marginal?

#### Forces and constraints

- SM-01 measures integration effort; the generic adapter maximizes reachable services per
  unit of work.
- Vendor agnosticism (Principle 1) is proven by structurally different API families, not by
  adapter count.
- Every dedicated adapter is a permanent maintenance liability against a moving external
  service (RISK-PROV-080).
- Capability facts for most services are unverified at authoring time and must not be
  invented (assumptions policy).

#### Alternatives considered

1. **Dedicated adapters for all 19 at fixed phases** — rejected: multiplies maintenance
   surface for services whose generic-path delta is near zero; forces invention of
   unverified per-service facts now.
2. **Generic adapter only, forever** — rejected: discards real value where APIs genuinely
   differ (Anthropic, Gemini, Azure deployment model) and where discovery/pricing metadata
   matters (OpenAI, OpenRouter).
3. **Phase by provider popularity alone** — rejected: popularity does not measure the
   dedicated-adapter delta; a popular OpenAI-compatible service may need no dedicated
   adapter at all.

#### Decision

(a) Any OpenAI-compatible service is served from MVP by the generic adapter. (b) A dedicated
adapter exists only for declared value — structurally different API (Anthropic, Gemini,
Azure OpenAI), local-first anchoring (Ollama), or discovery/capability/pricing metadata and
provider-specific error normalization (OpenAI, OpenRouter, and peers). (c) Phases as
cataloged in Volume 5 chapter 09: MVP = `openai_compatible`, `anthropic`, `ollama`; Beta =
`openai`, `gemini`, `openrouter`, `mistral`; v1 = `azure_openai`, `groq`, `together`,
`deepseek`, `xai`, `vllm`, `lm_studio`, `llama_cpp`; v2 = `litellm`, `localai`; Future =
`fastchat`, `text_generation_webui`. (d) Catalog slugs are reserved and permanent; changes
go through the change procedure.

#### Rationale

The classification spends dedicated-adapter effort where the delta is structural or
metadata-rich, keeps every mandated service reachable from MVP, and defers unverifiable
specifics to each adapter's implementation with PENDING VALIDATION markers instead of
invented facts.

#### Positive consequences

- All 19 mandated integrations have a committed, justified phase.
- MVP effort concentrates on three adapters while covering the widest service set.
- Maintenance surface grows only where value is declared.

#### Negative consequences

- Services on the generic path lack curated discovery/pricing metadata until (and unless)
  their dedicated adapter ships.
- The catalog table becomes a governance artifact requiring phase-gate review.

#### Risks

- A "Future" service's API surface may bit-rot before any adapter ships; mitigation: Future
  items carry no commitment and the generic path degrades per capability detection.

#### Reversal plan

Phases move through the Volume 0 change procedure; promoting or demoting an adapter changes
the catalog table and Volume 15 sequencing, no contracts.

#### Review conditions

- At each phase gate: re-verify auth families and API surface claims for that phase's
  adapters (V5B-OQ-1).
- If usage data shows heavy generic-adapter use against one service, evaluate promoting its
  dedicated adapter.
