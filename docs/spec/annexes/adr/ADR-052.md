### ADR-052 — Run-start skill snapshot: skill influence resolved once, recorded, and immutable per run

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team (Volume 4 authoring)
- Components: Agent Engine, Workflow Engine, Skill Engine, Prompt Engine
- Related requirements: FR-WF-008

#### Context

Skills are versioned prompt-and-requirement bundles (Volume 2; format owned by Volume 6)
that alter agent behavior through Prompt Engine registry slots. Runs must be reproducible
(SM-12) and explainable (PRD-006): the Run's `config_snapshot` is written before the first
Turn and never changes (INV-RUN-04). The open question was *when* skill resolution happens
and whether a run tracks live skill changes.

#### Problem

When is a run's active skill set determined, and how do installations, upgrades, and
deactivations interact with runs already executing, so that skill influence on any given run
is a recorded, reproducible fact?

#### Forces and constraints

- SM-12 replay requires knowing exactly which skill content shaped every turn.
- Live re-resolution mid-run would make consecutive turns diverge for reasons invisible in
  the run record.
- Long workflow runs span hours; skill updates during them are likely.
- Sub-agent delegation must not become a skill-widening side channel.

#### Alternatives considered

1. **Per-turn resolution** — always freshest content; rejected: turn-to-turn behavioral
   drift with no recorded cause, and hash bookkeeping per turn bloats the hot path.
2. **Live re-resolution on registry change events** — rejected: same drift problem,
   plus a mid-run behavior change triggered by an unrelated `andromeda skill install` is a
   startling and unauditable interaction.
3. **Session-start snapshot** — coarser than needed; rejected: sessions are long-lived
   containers, and pinning skills for a whole session prevents deliberate adoption of a fix
   in the *next run*, which is the natural update point.

#### Decision

The active skill set is resolved **once at run start** from the three sources (agent
profile, workflow step declaration, session activation), verified (tools, capabilities,
trust, content hash), and snapshotted — names, versions, hashes, application order — into
the Run's `config_snapshot` before the first Turn. The snapshot is immutable for the run;
sub-agents inherit it unwidened; registry changes affect only subsequently started runs.

#### Rationale

Run start is the exact point where reproducibility bookkeeping already happens
(INV-RUN-04); aligning skill resolution with it makes the skill contribution one more
recorded input, with zero mid-run drift and zero per-turn overhead.

#### Positive consequences

- SM-12 replay reproduces identical prompt structures from the snapshot.
- "Which skills shaped this run" has one authoritative, persisted answer.
- Skill updates have a crisp adoption semantics: next run.

#### Negative consequences

- A skill fix cannot help a run already in flight; the run must be completed or cancelled
  and re-run.
- Long workflow runs may execute many steps with content already superseded in the registry.

#### Risks

- Workflow-spawned runs across a long workflow may straddle a skill upgrade, making sibling
  steps differ; mitigation: each spawned Run's snapshot is recorded, so the difference is
  visible and attributable; workflow authors can pin skill versions in step selectors.
- Snapshot immutability tempts callers to bypass skills for "dynamic" behavior; mitigation:
  prompt contribution outside registry slots is prohibited (FR-WF-008 constraint).

#### Reversal plan

A future opt-in "refresh skills at step boundary" flag for workflow definitions would be an
additive, recorded re-snapshot at a persisted boundary — the immutable-within-a-run
principle generalizes to immutable-within-a-snapshot-interval without breaking the record
model.

#### Review conditions

- Revisit if Beta workflow telemetry shows runs regularly exceeding 24 h wall-clock, where
  next-run adoption is too slow.
- Revisit alongside any Volume 6 introduction of dynamically parameterized skills.
