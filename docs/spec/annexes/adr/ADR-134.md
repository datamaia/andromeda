### ADR-134 — Configuration secret hygiene: fail-closed definite detectors plus universal sink redaction

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Configuration Manager, Logging, Event Bus, CLI, TUI, Persistence Layer
- Related requirements: FR-CFG-011, FR-CFG-010; ADR-014

#### Context

Configuration is structurally secret-free by product rule (INV-CFGP-02; ADR-014 makes the
Secret Store the only credential home, with reference indirection in config). Reality
disagrees: users paste keys into config files, and workspace files are committable and
shareable by design. Volume 10 owns the configuration surfaces where such a mistake would
propagate — error messages, validation reports, logs, events, exports, listings, and run
snapshots.

#### Problem

Decide how configuration handles secret-shaped values: tolerate and redact, or refuse —
and how rendering surfaces are made leak-proof against whatever detection misses.

#### Forces and constraints

- False negatives are unrecoverable (a leaked secret is compromised); false positives are
  recoverable (an error message with a remedy). The asymmetry should shape the policy.
- Detection can never be complete: arbitrary high-entropy strings are undecidable as
  secrets; a heuristic that blocks would lock users out on false positives.
- Vendor token formats are external facts that change; hardcoding them as normative
  behavior without verification violates the assumptions policy.
- Redaction must be a definition-time property of surfaces (ADR-016 envelope carries
  safe-context fields), not per-call discipline.

#### Alternatives considered

1. **Redact-and-continue for all detections.** Rejected: it normalizes secrets living in
   plaintext files; the file remains a leak (VCS, backups, sharing) even if Andromeda's
   own output is clean.
2. **No detection, policy only.** Rejected: the highest-likelihood leak (pasted key in a
   committed workspace file) would be entirely unmitigated until humans notice.
3. **Encrypting secret values inside config files.** Rejected: it makes `andromeda.toml`
   a credential carrier, contradicting ADR-014 and INV-CFGP-02, and imports key-management
   complexity the Secret Store already solves.
4. **Blocking on heuristic (entropy) matches too.** Rejected: entropy heuristics have
   irreducible false positives (hashes, ULIDs, opaque IDs); blocking on them makes
   legitimate configurations unloadable with no clean override.

#### Decision

Two-class detector registry, versioned and compiled in. **Definite** detectors (PEM
private-key blocks, authorization-header token shapes, URL userinfo passwords, registered
vendor token prefixes — the concrete vendor pattern set PENDING VALIDATION against each
vendor's published token-format documentation) fail the configuration with E-CFG-012,
whose remedy instructs moving the secret to the Secret Store and rotating it. **Heuristic**
detectors (length + Shannon entropy) warn and force redacted rendering without blocking.
Independently, every configuration sink — errors, reports, logs, events, exports,
listings, snapshots — renders values only through the redaction step; `sensitive`-classed
keys always render as tokens, with a truncated-hash form in run snapshots for equality
checking. Full rules: FR-CFG-011; effectiveness bound: NFR-CFG-004 (0 leaked fragments).

#### Rationale

The asymmetry argument decides the policy shape: block where confidence is categorical
(definite), never block where confidence is statistical (heuristic), and make rendering
leak-proof regardless, so detection quality bounds *file* hygiene while redaction bounds
*output* hygiene. Marking vendor patterns PENDING VALIDATION keeps external facts out of
normative text while still shipping the mechanism.

#### Positive consequences

- The most likely credential leak is stopped at entry, with a rotation instruction.
- Output-side leak risk is a testable identity property (NFR-CFG-004 target 0).
- Snapshot hashing preserves run reproducibility checks without disclosure.

#### Negative consequences

- Definite false positives (a test fixture that genuinely is a dummy PEM block) block
  loading with no override; the documented remedy is restructuring the fixture, which is
  friction by design.
- Heuristic redaction cosmetically hides legitimate opaque values in listings.

#### Risks

- Unrecognized secret shapes pass silently (residual risk of RISK-CFG-003); mitigated by
  registry updates per release and the structural reference-indirection rule.
- Registry pattern drift versus vendor reality; mitigated by the PENDING VALIDATION
  review gate per release.

#### Reversal plan

Detector classes, the registry, and the sink redaction step are each independent
mechanisms behind the Configuration Manager and the shared redaction handler; softening
E-CFG-012 to a warning, or retiring a detector, is a one-point change — but weakening is a
security posture change requiring a superseding ADR per FR-CFG-011 compatibility rules.

#### Review conditions

- Any confirmed secret leak through a configuration surface (immediate review).
- Definite-detector false-positive reports exceeding a handful per release cycle.
- Vendor token-format verification (the PENDING VALIDATION item) changes a registered
  pattern.
