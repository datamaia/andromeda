### ADR-011 — Observability Stack: OpenTelemetry SDK with Local-First Sinks; Logging via log/slog

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Observability, Logging, Telemetry, Persistence Layer
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation

#### Context

Observability is high in Andromeda's precedence order (item 8) and structurally important below it: audit, cost accounting, and debugging of agent runs all depend on correlated logs, traces, metrics, and events (Volume 10). The product is local-first and privacy-preserving (precedence item 5): the default destination for every signal is the user's machine, with network export strictly opt-in. Agent runs are naturally tree-shaped (run → turns → tool invocations → provider requests), which is exactly the trace model.

#### Problem

Choose the instrumentation APIs, wire formats, and default sinks for the three signal classes — traces, metrics, logs — such that local-first defaults, ecosystem interoperability, and low dependency weight hold together.

#### Forces and constraints

- Local-first default: all signals MUST be fully usable with zero network egress; export is opt-in under the Telemetry consent policy (Volume 10).
- Traces MUST correlate across components: one run's trace spans the Agent Engine, Tool Runtime, Provider Layer, and plugin/MCP subprocess boundaries (ADR-009/ADR-010 protocols carry correlation metadata).
- Logs are user-facing diagnostics: structured, redacted (credentials never logged — Volume 9), and inspectable without special tooling.
- Users operating fleets or CI will want standard export paths (OTLP) to their existing observability infrastructure; inventing a proprietary format would undercut interoperability.
- ADR-001 counts the OpenTelemetry Go SDK's maturity for traces/metrics among Go's verified strengths; dependency weight beyond that should stay minimal.
- Volume 10 owns the event envelope; events flow on the Event Bus (ADR-012) and are a distinct signal class from logs.

#### Alternatives considered

| Criterion | OTel SDK + slog (selected) | zap or zerolog + OTel | Fully custom tracing/logging |
|---|---|---|---|
| Trace/metric semantics | OpenTelemetry standard | OpenTelemetry standard | Proprietary; no ecosystem tooling |
| Log API | stdlib `log/slog`, zero extra dependency | Third-party logger dependency | Bespoke |
| Structured JSON logs | Built-in JSON handler | Yes, marginally higher throughput | At full implementation cost |
| Export interoperability | OTLP, off by default | OTLP for traces/metrics; logger-specific bridging | None without converters |
| Dependency weight | OTel SDK only | OTel SDK + logging framework | None external, maximum internal |
| Correlation (trace IDs in logs) | slog handler enriches records from span context | Same, via logger integration | Bespoke |

- **zap/zerolog** earn their place in services logging at extreme throughput. Andromeda is an interactive local tool; its logging volume is human-scale, and `log/slog` (standard library since Go 1.21) provides structured, leveled, handler-extensible logging with a JSON handler at zero dependency cost. A third-party logging framework would add a dependency, a second idiom for contributors, and no capability the product needs. Rejected.
- **Custom tracing** would tailor span semantics to agent runs but forfeit the entire OTel ecosystem (viewers, collectors, vendor backends) and put schema design, propagation, and instrumentation conventions permanently on us. The run-tree maps onto standard spans with attributes; nothing about the domain requires proprietary semantics. Rejected.

#### Decision

1. **Traces and metrics** MUST be instrumented with the OpenTelemetry Go SDK using standard semantic conventions plus Andromeda-specific attributes (run, session, provider, model, cost linkage) defined in Volume 10.
2. **Default sinks are local**: exporters write to local storage — structured files and/or SQLite tables in the Persistence Layer (ADR-007) — so traces and metrics are queryable offline by the CLI/TUI diagnostic commands.
3. **OTLP export** MAY be enabled explicitly by the user (config per Volume 10, consent policy enforced by Telemetry); it is off by default and MUST NOT be enabled by any implicit mechanism.
4. **Logging** MUST use stdlib `log/slog` with a JSON handler as the canonical structured format (a human-readable console handler MAY front it interactively). Log records within an active span MUST carry the trace/span IDs via a correlating handler. Redaction rules from Volume 9 apply at the handler layer.
5. Third-party logging frameworks MUST NOT be introduced; additional exporters require an amending ADR.

#### Rationale

OpenTelemetry is the only instrumentation choice that satisfies both ends of the tension: signals stay local by default, yet the moment a user opts in, they connect to their existing tooling with zero translation because the semantics and wire format are the industry standard. Pairing it with stdlib slog keeps the dependency graph flat where OTel adds unavoidable weight — the standard library gives structured JSON logging, handler composition (redaction, correlation) and contributor familiarity for free. The local sink through the Persistence Layer turns observability data into a queryable product feature (run inspection, cost reports) rather than write-only exhaust.

#### Positive consequences

- One correlation model end-to-end: trace IDs flow from CLI invocation through provider calls and subprocess boundaries into logs and cost records.
- Zero-egress default posture is verifiable (no exporter configured ⇒ no network path exists).
- Opt-in OTLP gives fleet/CI users standard integration without Andromeda-specific collectors.
- slog handlers give a single enforcement point for redaction and correlation.

#### Negative consequences

- The OTel Go SDK is a heavyweight dependency subtree — accepted as the price of standard semantics, and confined behind Volume 3 port interfaces.
- Local trace/metric storage consumes disk; retention policies (Volume 10) become mandatory bookkeeping.
- slog's throughput ceiling is below zap's; if a future high-frequency path (e.g., indexing at scale) becomes log-hot, sampling or level discipline is required rather than a framework swap.

#### Risks

- **Dependency bloat and API churn** in the OTel SDK: mitigated by pinning, wrapping instrumentation behind internal helpers, and instrumenting at component boundaries rather than pervasively.
- **Accidental telemetry egress** would be a privacy defect of the highest order: mitigated by making exporter construction impossible unless explicit configuration is present, and by a Volume 13 test asserting the no-config ⇒ no-exporter invariant.
- **Local signal volume** degrading performance on long agent sessions: mitigated by sampling knobs and retention limits owned by Volume 10, benchmark-checked in Volume 12.

#### Reversal plan

Instrumentation call sites are the sunk cost, and they are portable: OTel APIs are the industry-standard shape, so migrating to a hypothetical successor means re-binding wrappers, not re-finding instrumentation points. Sinks are independently reversible (file ⇄ SQLite ⇄ OTLP) since they are exporter configuration, not code structure. Logging reversal (slog → framework) is a handler swap at the single logger-construction site because all call sites use the stdlib API. The costliest realistic reversal — abandoning OTel for custom tracing — would keep span structure and attributes, changing only the recording backend.

#### Review conditions

- The OTel Go SDK's traces or metrics APIs undergo a breaking change or its maintenance cadence degrades materially (no release for more than 12 months).
- Volume 12 benchmarks attribute a missed latency/footprint target to instrumentation overhead at default settings.
- A native slog↔OTel logging bridge stabilizes upstream and would simplify rule 4's correlating handler — adopt via amendment.
- Telemetry consent policy changes in Volume 10 that alter what rule 3 must enforce.
