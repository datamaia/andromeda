### ADR-131 — Environment variable mapping: schema-matched compact names with double-underscore disambiguation

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Configuration Manager, CLI
- Related requirements: FR-CFG-004; FR-CLI-001 (Volume 8 CLI grammar and its environment-variable table)

#### Context

Volume 0 chapter 03 fixes the convention — `ANDROMEDA_` prefix, uppercased dotted path with
`_` separators, `__` for nesting "where ambiguous" — and delegates the exact algorithm to
Volume 10. Configuration keys themselves contain underscores (`default_profile`,
`max_output_bytes`), so a naive split on `_` cannot reconstruct dotted paths, and
dynamic-name tables (`providers.<slug>`, `mcp.servers.<name>`) make segmentation genuinely
ambiguous.

#### Problem

Define a total, deterministic env-name→key mapping that keeps the ergonomic compact form
(`ANDROMEDA_TUI_THEME_MODE`) working for the common case while never guessing when more
than one interpretation exists.

#### Forces and constraints

- POSIX-portable variable names cannot contain dots; underscores must encode both path
  separators and literal name characters.
- Guessing among multiple candidate keys would make behavior depend on schema evolution —
  adding a key could silently re-route an existing variable.
- The environment layer resolves after file layers, so declared dynamic-table entries are
  known at mapping time.
- Unknown `ANDROMEDA_*` variables must be surfaced, not ignored (stale or mistyped
  variables are a top misconfiguration class; Volume 8 already promises Volume 10 reports
  them).

#### Alternatives considered

1. **Double-underscore separators only** (`ANDROMEDA_AGENT__LOOP__MAX_ITERATIONS`).
   Rejected as the only form: verbose, hostile to the documented common case, and
   inconsistent with the Volume 0 example `ANDROMEDA_TUI_THEME_MODE`.
2. **A single JSON/TOML blob variable** (`ANDROMEDA_CONFIG_JSON`). Rejected: one opaque
   value defeats per-key attribution, diffing in CI logs, and per-key redaction.
3. **First-match/longest-prefix guessing** for compact names. Rejected: resolution would
   change when keys are added — a silent behavior change class.

#### Decision

Both forms are valid. Compact form: match the lowercased name against the full schema key
set by replacing dots with underscores; exactly one candidate resolves, zero or multiple
raise E-CFG-009 (with the explicit forms listed). Explicit form: `__` splits path
segments; single `_` is literal within a segment. Dynamic-table matching considers only
entries declared by lower layers; the environment overrides entries, never creates them.
Table-typed keys are not settable. Reserved control variables (`ANDROMEDA_CONFIG`,
`ANDROMEDA_WORKSPACE`, `ANDROMEDA_PROFILE`, `ANDROMEDA_NO_INPUT`, `ANDROMEDA_NO_COLOR`)
are excluded from mapping. Full algorithm: FR-CFG-004.

#### Rationale

The compact form stays ergonomic exactly as far as it is unambiguous, and the schema — not
heuristics — decides ambiguity. Refusing with the candidate list and their `__` forms makes
the failure self-remedying. Barring entry creation from the environment keeps dynamic
tables reviewable in files where they can be validated as a set.

#### Positive consequences

- The Volume 0 example works verbatim; CI recipes stay short.
- Every mapping outcome is deterministic under schema evolution: a new key can turn a
  working compact name ambiguous (loud error), never silently re-route it.
- Per-variable attribution and redaction fall out naturally.

#### Negative consequences

- Users must occasionally learn the `__` form when a collision arises.
- Mapping needs the schema and lower layers, so environment errors surface at resolution
  time rather than instantly at process start.

#### Risks

- Compact-name breakage when a colliding key ships (loud but annoying); mitigated by
  documenting the `__` form as the stable one for scripts.
- Dynamic-name entries containing uppercase or exotic characters are unaddressable from
  the environment; mitigated by the dynamic-name pattern restriction in the schema.

#### Reversal plan

The mapping is one pure function behind the Configuration Manager; switching to
explicit-only would be a breaking change shipped in a major release with E-CFG-009
messages teaching the migration for one deprecation cycle.

#### Review conditions

- E-CFG-009 ambiguity findings recur in the field (indicates the compact form is failing
  its ergonomic purpose).
- The Windows phase introduces case-insensitive environment semantics needing a PAL rule.
