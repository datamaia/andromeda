### ADR-016 — Structured error codes (`E-<AREA>-NNN`), a fixed error envelope, and a closed exit-code scheme

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Core Domain, CLI, TUI, Logging, Observability, Permission Manager
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation.

#### Context

Andromeda is a CLI and TUI product that is also scripted and embedded in automation: exit codes
are part of its public contract, and error content is consumed by humans, scripts, logs,
telemetry, and other agents. Volume 0 chapter 03 establishes area ownership for identifiers and
gives error-envelope-and-exit-codes a single home: Volume 0 owns the scheme, area owners mint
individual errors, and the consolidated catalog lives in `annexes/catalog-errors.md`. This ADR
records the design decision behind that scheme.

#### Problem

How are errors identified, described, and mapped to process exit codes so that every error has
one stable identity across releases, scripts can branch on outcomes without parsing prose, logs
and telemetry correlate on the same identity, and ownership of each error is unambiguous across
sixteen volumes and dozens of components?

#### Forces and constraints

- Exit codes are a scarce, flat namespace (practically 0–255, with shells reserving ranges);
  they cannot carry per-error identity and MUST stay coarse and stable.
- Error identity must survive message rewording and localization; user-facing text cannot be the
  key.
- Area ownership (chapter 03) requires that each error code be mintable by exactly one volume
  without cross-author collisions.
- Errors surface in different channels (terminal, JSON output, logs, events, telemetry) that
  need consistent, redactable content — security-relevant details MUST be separable from
  user-facing text.
- Recoverability and retry behavior are contract-level facts for agents and workflows, not
  presentation details.

#### Alternatives considered

1. **errno-style numbers only** — a flat integer catalog. Rejected: numbers carry no ownership
   or area information, collide across concurrent authors, and are hostile to grep-based
   traceability across a large specification and codebase.
2. **String-only errors** — free-text messages as the error identity. Rejected: rewording
   breaks scripts and dashboards; no stable key for telemetry; no machine-checkable catalog.
3. **Per-command ad-hoc codes** — each CLI command defines its own exit and error codes.
   Rejected: the same underlying failure (for example, a revoked credential) would surface with
   different codes per command, breaking cross-command scripting and observability correlation.

#### Decision

Error identity uses **`E-<AREA>-NNN`**, where `<AREA>` follows the ownership table in Volume 0
chapter 03 (the same area ownership as requirements) and `NNN` is a per-area sequence. Every
defined error declares a fixed **error envelope**: stable code, category, severity, user
message, technical message, cause, safe context data (safe-to-log), recoverability, retry
policy, recommended action, exit-code mapping, HTTP mapping where applicable, telemetry event,
and security implications.

Process **exit codes** are exactly:

| Code | Meaning |
|---|---|
| 0 | Success |
| 1 | General error |
| 2 | Usage error (invalid arguments or flags) |
| 3 | Configuration error |
| 4 | Authentication error |
| 5 | Permission denied (by Andromeda's permission model) |
| 6 | Tool execution failure |
| 7 | Provider failure |
| 8 | Timeout or cancellation |
| 9 | Integrity error (corrupted state, failed migration) |

Every error's envelope maps to exactly one exit code; every CLI command definition in Volume 8
states which exit codes it can produce. New exit codes require a superseding ADR.

#### Rationale

The two-level design matches the two consumers: scripts and CI need a tiny, stable exit
vocabulary (ten values covering the product's failure classes), while humans, logs, and
telemetry need precise per-error identity and rich envelope fields. Reusing chapter 03 area
ownership means error minting inherits the collision-free model already proven for
requirements, and the mandatory envelope forces every author to answer the operational
questions (retryable? recoverable? safe to log? which telemetry event?) at definition time
instead of at incident time.

#### Positive consequences

- Scripts branch on exit codes; dashboards and alerts key on `E-<AREA>-NNN`; both survive
  message changes.
- The safe-context-data and security-implications fields make redaction a definition-time
  property, supporting the Logging component's guarantees.
- The consolidated error catalog is mechanically checkable against the corpus.
- Recoverability and retry policy fields give the Agent Engine and Workflow Engine uniform
  failure semantics.

#### Negative consequences

- Defining an error is heavier: fourteen envelope fields per error instead of a message string.
- The coarse exit-code set means some distinct failures share an exit code; callers needing
  finer granularity must consume structured output instead.
- Area assignment for borderline errors (for example, a provider timeout during authentication)
  requires judgment and occasional register arbitration.

#### Risks

- Envelope fields could be filled with boilerplate, eroding their value; mitigation: review
  checklist and linter checks on the catalog.
- Exit-code overloading pressure as the product grows; mitigation: structured JSON output is the
  extension point, and the closed set is guarded by ADR review.
- Divergence between in-code error definitions and the specification catalog; mitigation: the
  catalog is generated or verified from source in CI (Volume 13 defines the check).

#### Reversal plan

Error codes and the envelope are declarative metadata around Go error values; remapping to a
different scheme is a mechanical transformation of the catalog plus a major-version note, since
exit codes and error codes are public contract surfaces under ADR-015 SemVer rules.

#### Review conditions

- If a failure class emerges that genuinely fits none of codes 1–9, propose a superseding ADR
  rather than overloading code 1.
- Review the envelope field list when Volume 10's telemetry and event envelope definitions are
  consolidated, to keep field names aligned.
- Revisit HTTP mappings if Andromeda ships a long-lived service surface beyond the current CLI
  and TUI scope.
