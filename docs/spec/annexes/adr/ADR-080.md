### ADR-080 — Extension distribution: declared package sources now, curated marketplace Future

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Package Manager, Skill Engine, Plugin Runtime, MCP Runtime, CLI
- Related requirements: FR-PLUG-005, FR-PLUG-006, FR-SKILL-005; INV-PKG-01..04 (Volume 2)

#### Context

Skills, plugins, and packaged MCP servers must travel from authors to installations. The
brief asks for installation, discovery, updating, uninstallation, and a "future marketplace
or registry". Volume 2 fixes the Package entity with `source` kinds `registry`, `archive`,
`path`, and `git`. A marketplace — hosted, curated, with accounts, moderation, and search —
is organizational infrastructure this project does not have and cannot assert into existence
(assumptions policy: no invented services). The distribution design must work from day one
with infrastructure that verifiably exists (git hosting, HTTPS file hosting, local paths)
while leaving a marketplace a compatible later addition rather than a redesign.

#### Problem

Decide how installations discover and acquire extension packages: a single central registry
Andromeda operates, a fully decentralized model, or configured sources — and how a future
curated marketplace relates to that choice.

#### Forces and constraints

- No invented external facts: a hosted Andromeda registry's existence, funding, and operation
  are unresolved organizational questions.
- Local-first: discovery and installation from local paths and pre-fetched archives must work
  offline; nothing may require a hosted service for core operation.
- Supply-chain posture (chapter 09 of Volume 6): acquisition must feed the verification
  pipeline (checksums unconditional, signatures per ADR-081) regardless of where a package
  came from.
- Enterprise reality: organizations run internal mirrors and want to pin exactly which
  sources their installations may consult.
- A later marketplace must not break the installed base: whatever ships at Beta is the
  compatibility floor.

#### Alternatives considered

1. **Central registry only** (a single hosted index Andromeda operates, npm-style) —
   rejected now: presumes infrastructure whose existence is PENDING VALIDATION; a single
   point of failure and moderation burden on day one; offline and air-gapped installs would
   still need a bypass path, which then becomes the real mechanism.
2. **Fully decentralized only** (git URLs and archives, no index concept) — rejected as the
   end state: no search/discovery story, no place for curation metadata, and every install is
   a copy-paste of a locator; acceptable as a floor, not as the ceiling.
3. **In-product marketplace at Beta** — rejected: curation, publisher identity, and hosting
   are organizational capabilities, not code; committing product surface to them now would
   couple the Beta to unresolved external decisions.

#### Decision

1. Extension acquisition works against **declared package sources**, configured per scope in
   `plugins.sources` and `skills.sources` (source schema owned by Volume 6 chapter 09). Four
   source kinds, matching the Volume 2 vocabulary:
   - `registry` — an HTTPS or file URL to a **registry index**: a JSON document (schema owned
     by chapter 09) listing packages with versions, kinds, archive locators, checksums,
     signature references, and publisher metadata. Anyone can host one; an index is data, not
     a service contract.
   - `git` — a git repository containing packages or an index, fetched through the Git Engine.
   - `archive` — a direct URL or path to one package archive.
   - `path` — a local directory (development, air-gapped transport).
2. Discovery (search/list) operates over the enabled sources in priority order and only on
   explicit user command — no implicit network at startup.
3. Every acquisition path feeds the same verification pipeline (FR-PLUG-007); source kind
   never weakens verification.
4. An **official Andromeda registry index** (hosting, operation, URL) is PENDING VALIDATION —
   an organizational decision tracked in the Volume 6 register. Until resolved, no default
   `registry` source ships enabled; documentation shows how to add sources.
5. A **curated marketplace** (hosted browsing, publisher accounts, moderation, ratings) is
   classified **Future**. Its committed compatibility floor: it presents as one more
   `registry`-kind source using the same index schema and the same verification pipeline, so
   marketplace adoption never requires a client redesign.

#### Rationale

Configured sources are the smallest mechanism that satisfies every hard constraint today:
they work offline (`path`, local `archive`), they work with infrastructure that exists (git
hosting, static file hosting), they give enterprises explicit control over provenance, and
they make the eventual marketplace an *instance* of the mechanism instead of a replacement
for it. Framing the registry index as a data format rather than a service keeps the client's
contract testable in CI with fixture indexes and keeps hosting decisions out of the product's
critical path.

#### Positive consequences

- Distribution works at Beta with zero new infrastructure; air-gapped installation is a
  first-class path, not a workaround.
- The marketplace question is decoupled from the client: index schema + verification pipeline
  are the stable interface either way.
- Source priority and enablement give organizations a policy lever (only the internal mirror,
  nothing else).

#### Negative consequences

- Without a default registry, first-run discovery is empty: users must add a source before
  `search` returns anything — documentation burden and onboarding friction.
- Ecosystem fragmentation risk: multiple independent indexes with no cross-search until a
  marketplace or aggregator exists.
- Index hosting quality (availability, staleness) is outside Andromeda's control and varies
  by source.

#### Risks

- A popular unofficial index could become a de-facto standard with weaker curation than the
  eventual official one; mitigated by signature policy (ADR-081) and trust classification
  being independent of source reputation.
- Index-schema evolution could strand old clients; the index carries a schema version and
  follows the SM-20 additive-evolution regime.

#### Reversal plan

Sources are additive configuration; removing a kind is a deprecation exercise, not a
redesign. If a central registry becomes the dominant path, the sources mechanism remains as
the enterprise/air-gapped path — the decision inverts defaults (ship the official source
enabled), which is a configuration change plus documentation, not a contract change.

#### Review conditions

- PENDING VALIDATION resolves: the official registry hosting decision is made — amend this
  ADR with the default-source policy and resolve the register entry.
- Marketplace commitment: when v2 scope is confirmed (Volume 1 phase definitions), the
  marketplace either enters scope with its own ADR (identity, curation, moderation) or is
  re-deferred explicitly.
- Field data shows source misconfiguration is a leading installation failure class: invest in
  source diagnostics and possibly a bundled default source.
