### ADR-089 — Semantic indexing is explicit opt-in; lexical indexing is the always-on baseline

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Indexing Engine, Context Manager, Configuration Manager, Provider Layer
- Related requirements: FR-IDX-001, FR-IDX-003, FR-IDX-005; NFR-MEM-001, NFR-IDX-001

#### Context

Semantic indexing embeds workspace content and memory records through an embedding provider
(ProviderPort `Embed`, ADR-020 storage). Embedding providers may be remote: every embedded
chunk's content then leaves the machine, and every re-embedding costs money. Volume 1 fixes
privacy and local operation above performance and convenience in the precedence order, and
the offline guarantee list requires indexing to work without Internet. Lexical indexing has
none of these externalities.

#### Problem

Should semantic indexing be enabled by default (best retrieval out of the box) or require
explicit configuration — given that a default embedding provider would either silently
transmit workspace content to a cloud service or silently depend on a local model the user
may not run?

#### Forces and constraints

- Requirement precedence: user/credential safety and privacy outrank retrieval quality.
- No silent egress: transmitting workspace content anywhere must be an explicit user choice
  (PRD-003; NFR-MEM-001's zero-egress default measurement).
- No invented defaults: presuming a specific local embedding model exists on the user's
  machine would violate the assumptions policy.
- Principle 2: absent capability is reported, never silently simulated.
- The offline guarantee needs a retrieval path with zero provider dependence.

#### Alternatives considered

1. **Semantic on by default with a bundled/assumed local embedding model** — rejected:
   assumes external facts (which local models exist, their licenses and quality) and adds a
   model-distribution burden out of scope for the product.
2. **Semantic on by default routed to a cloud embedding provider when credentials exist** —
   rejected outright: silent content egress by default inverts the precedence order and
   NFR-MEM-001.
3. **Lexical always-on; semantic strictly opt-in via explicit provider+model configuration**
   (selected).

#### Decision

Lexical indexing is the unconditional baseline: built for every workspace when indexing is
enabled, self-contained, offline-guaranteed. Semantic indexes exist only when
`index.semantic.provider` and `index.semantic.model` are explicitly set; the configured
provider must declare the `embeddings` capability. The pair fixes the index's immutable
embedding space (INV-IDX-03). Embedding requests are exempt from provider routing fallback:
they go to the configured provider or fail explicitly (E-IDX-002) — content never follows a
fallback to a provider the user did not name for this purpose. Absence of semantic
capability is reported (`Status`, degradation markers), and all consumers degrade to
lexical.

#### Rationale

The user's first search working offline on a fresh install, with zero egress and zero spend,
is the correct default for a local-first product; one explicit configuration line buys
semantic retrieval for users who choose it, with the choice itself serving as informed
consent about where content flows.

#### Positive consequences

- Zero-egress, zero-cost, offline-correct default posture; NFR-MEM-001/NFR-IDX-001 are
  measurable identities rather than configuration-dependent hopes.
- The embedding-space declaration doubles as the consent record and the INV-IDX-03 pin.
- No fallback surprises: embedding traffic is exactly where the user pointed it.

#### Negative consequences

- Out-of-the-box retrieval quality is lexical-only; semantic value requires a configuration
  step (documentation and `andromeda doctor` guidance mitigate, Volume 8).
- Two retrieval modes must both be first-class in consumers (FR-CTX-001 handles the
  degradation path).

#### Risks

- Users may conflate "search works" with "semantic search works" — mitigated by explicit
  degradation markers and `Status` reporting (FR-IDX-005).

#### Reversal plan

Enabling semantic-by-default later is a configuration-default change plus a consent flow —
additive and reversible; no schema or contract changes. The opt-in posture can also tighten
(e.g., per-workspace confirmation for remote spaces) purely in policy.

#### Review conditions

- A vetted, license-clean local embedding path becomes distributable within scope, making a
  privacy-preserving default viable (alternative 1's blocker resolving).
- Beta feedback shows the configuration step is a dominant adoption obstacle, and a consent
  flow can carry the privacy weight instead.
