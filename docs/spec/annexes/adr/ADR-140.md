### ADR-140 — Remote telemetry is opt-in and disabled by default, gated by an interactive, versioned, revocable consent record

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Telemetry, Policy Engine, Configuration Manager, CLI/TUI
- Related requirements: FR-OBS-010, FR-OBS-011, FR-OBS-012; NFR-OBS-006; ADR-011

#### Context

ADR-011 made OTLP export possible but "off by default and MUST NOT be enabled by any
implicit mechanism", deferring the consent policy to Volume 10. The product brief requires
an explicit decision on opt-in versus opt-out, the consent mechanics, and enterprise
policy. Andromeda is local-first and privacy-preserving by precedence (Volume 0 precedence
item 5); there is no hosted Andromeda backend (Out of Scope, Volume 1) — any export goes to
a user-chosen endpoint.

#### Problem

Decide whether remote telemetry is opt-in or opt-out, what constitutes valid consent, how
consent interacts with configuration (which is copyable) and with fleets under
administrative policy, and what invalidates consent.

#### Forces and constraints

- Privacy precedence: a default that ships data is indefensible for a tool that reads
  users' repositories and prompts, regardless of how well the payload is filtered.
- Opt-in telemetry yields biased, sparse data — the real cost of this decision, borne
  knowingly.
- Configuration files travel (dotfiles repos, machine images, CI); a config flag alone
  cannot represent a human decision.
- Non-interactive contexts (CI, headless) must never grant consent silently (PRD-009: no
  silent grant).
- Enterprises need to pin telemetry off fleet-wide.
- Informed consent requires knowing *what* ships: the collected-data catalog must be part
  of the consent act, and a broadened catalog must not inherit old consent.

#### Alternatives considered

1. **Opt-out (enabled by default with a disable flag)** — maximizes data, standard in some
   commercial tooling; irreconcilable with the privacy precedence and with ADR-011's
   explicit prohibition of implicit enablement. Rejected.
2. **Opt-in via configuration flag alone** — one copied dotfile opts a new machine in;
   consent becomes an artifact of file provenance, not human decision. Rejected.
3. **First-run consent prompt (ask everyone once)** — interactive consent, but a modal
   prompt at first run pressures the choice, breaks non-interactive first runs, and trains
   users to dismiss dialogs. Rejected in favor of a one-line informational notice with no
   prompt.
4. **Opt-in with interactive confirmation + recorded, versioned consent (selected).**

#### Decision

1. Remote telemetry is **opt-in and disabled by default**. No implicit mechanism —
   installation, update, CI variable, copied configuration — can enable it.
2. Valid consent = `telemetry.enabled = true` **and** an interactive confirmation that
   displays the versioned collected-data catalog and the endpoint, writing a consent record
   (consent version, product version, timestamp, endpoint, scope) to the global database.
3. Consent is **version-bound**: any broadening of the catalog bumps the consent version
   and invalidates prior consent until re-confirmation; endpoint changes require
   re-confirmation.
4. Consent is **revocable at any time**, with immediate export stop, queue purge, and
   installation-identifier destruction.
5. **Enterprise lock**: `telemetry.locked = true` at the system configuration layer pins
   telemetry disabled against user action.
6. The export pipeline is gated **constructively**: the exporter is not instantiated
   without valid consent (no code path exists to misfire).

#### Rationale

The only consent that survives scrutiny is specific (catalog + endpoint), informed
(displayed verbatim), affirmative (interactive), and revocable — so that is the mechanism.
Binding consent to the catalog version makes "we now collect more" impossible to slip past
users. Accepting biased opt-in data is the honest price; the alternative purchases data
with trust.

#### Positive consequences

- The zero-egress default is testable (NFR-OBS-006) and marketable as a property, not a
  promise.
- Fleet operators get a hard lock; CI can never opt in.
- Catalog changes get automatic user review via re-consent.

#### Negative consequences

- Telemetry volume will be low and self-selected; product decisions cannot lean on it
  uncritically.
- Re-consent on catalog broadening adds friction to telemetry evolution — intended, but
  real.
- The consent machinery (record, versioning, lock) is more code than a flag.

#### Risks

- Consent fatigue if the catalog churns: mitigated by treating broadenings as rare,
  reviewed contract changes (SM-20-style discipline).
- Users conflating local observability with telemetry: mitigated by strict naming
  separation in documentation and UI (local records never called "telemetry").

#### Reversal plan

Moving to opt-out would be a MAJOR product-posture change requiring a superseding ADR and
Volume 0 change procedure; the mechanics (record, catalog, lock) would survive either
polarity. Softening only the confirmation (e.g., config-only opt-in) is a one-component
change to the gate evaluation, also requiring a superseding ADR.

#### Review conditions

- Legal/regulatory requirements in target jurisdictions impose different consent mechanics.
- Evidence that the opt-in population is too small to validate reliability targets —
  re-evaluate incentives (never the default).
- A hosted ingestion offering is ever proposed (currently Out of Scope) — full re-decision.
