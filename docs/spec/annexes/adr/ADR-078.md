### ADR-078 — Skill package format: content-addressed directory with a TOML manifest, data only

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Skill Engine, Package Manager, Extension SDK, Prompt Engine
- Related requirements: FR-SKILL-001, FR-SKILL-004, FR-SKILL-005; INV-SKL-01..04 (Volume 2)

#### Context

Skills are Andromeda's packaged procedural knowledge: prompts plus declared requirements,
loadable by agents (Volume 2, Skill entity). Volume 2 fixes the boundary invariant INV-SKL-04
— skills carry no executable code — and requires per-version content immutability verified by
`content_hash` (INV-SKL-02). The format decision determines how skills are authored, reviewed,
tested, diffed, and distributed, and how the trust story ("a skill is reviewable text") holds
in practice. Andromeda already standardizes TOML for human-authored configuration (ADR-008)
and JSON Schema for machine-validated payloads (ADR-024).

#### Problem

Choose the on-disk format for a skill: single file or directory, manifest language, how
prompts and auxiliary content are referenced, how integrity is computed, and how the
no-executable-content rule is made mechanically checkable rather than aspirational.

#### Forces and constraints

- Skills must be reviewable as plain text in a code review (trust model, Volume 6 chapter 07)
  and diffable version-to-version.
- Prompt documents are Markdown-shaped prose; embedding them in a structured file (escaped
  strings) destroys reviewability and line-based diffs.
- The manifest carries typed declarations (requirements, selectors, semver ranges) that need
  schema validation with field-level findings (ADR-016 envelope quality).
- Integrity must bind the whole content set, not just the manifest (INV-SKL-02).
- Test fixtures (FR-SKILL-004) belong with the skill so its behavior is regression-checked at
  the source.
- The format is a public contract under SM-20 once Beta ships.

#### Alternatives considered

1. **Single Markdown file with front matter** (one `SKILL.md` with YAML/TOML front matter) —
   rejected: multi-document skills (checklists, templates, fixtures) would need embedding or
   ad-hoc sidecar conventions; front-matter parsing splits the manifest across two grammars;
   test fixtures have no home.
2. **JSON manifest** — rejected: hostile to hand authoring (no comments, trailing-comma
   traps); inconsistent with ADR-008's choice of TOML for every human-edited Andromeda file.
3. **Archive-only format** (skills exist only as packaged archives, no directory form) —
   rejected: kills the local development loop (edit, test, activate from a path) that
   FR-SKILL-004 and `skills.paths` depend on; packaging is distribution's concern
   (chapter 09), not the format's.
4. **Executable skills** (hooks or scripts allowed inside skills) — rejected outright:
   violates INV-SKL-04 and collapses the trust distinction between skills (data, reviewable
   as text) and plugins (code, sandboxed processes) that the whole extension trust model
   rests on.

#### Decision

A skill is a **directory** with a **`skill.toml` manifest** at its root:

1. The manifest (TOML per ADR-008) declares identity (`name`, `version`, `description`,
   `authors`, `license`), `format_version`, prompt document references, optional input JSON
   Schema and output artifact declarations, requirement declarations (tools, capabilities,
   skill dependencies as semver ranges), provider selectors, workflow associations,
   single-parent inheritance, override declarations, and deprecation metadata — validated
   against a JSON Schema (ADR-024) shipped with the binary and mirrored in `sdk/`.
2. Prompt documents, templates, and test fixtures are separate UTF-8 text files referenced by
   relative paths that MUST resolve inside the skill directory (no absolute paths, no `..`
   traversal, symlinks not followed).
3. Integrity is content-addressed: `content_hash` is SHA-256 over the sorted file manifest
   and per-file digests, computed at registration/installation and verified at every load.
4. The format is **data only**: no hook keys, no scripts, no binaries. Validation rejects
   executable content fail-closed (E-SKILL-006), with `additionalProperties: false` at the
   manifest top level so unknown declaration classes cannot smuggle behavior in.
5. `format_version` versions the format contract itself; the engine accepts the current and
   previous minor within a major (SM-20 regime).

#### Rationale

The directory-plus-manifest shape puts each content kind in its native medium: TOML for typed
declarations, Markdown files for prose that must diff cleanly, a fixtures tree for tests. The
content hash over the whole tree makes INV-SKL-02 enforceable mechanically, and the closed
manifest schema turns INV-SKL-04 from a review guideline into a validation error. Every
element reuses an already-decided mechanism (ADR-008, ADR-015, ADR-024), so the format adds
no new parsing or validation technology.

#### Positive consequences

- Skills are reviewable and diffable with standard text tooling; the trust claim "you can
  read what a skill does" is literally true.
- The local development loop needs no packaging step; the same directory is what an archive
  unpacks to.
- Field-level validation findings make authoring errors precise (E-SKILL-001 lists every
  finding, not the first).

#### Negative consequences

- A directory format is heavier to pass around ad hoc than one file; distribution relies on
  chapter 09 packaging for that.
- Path-containment and symlink rules add validation code that a single-file format would not
  need.
- Two schema technologies meet in one file (TOML carrying an embedded JSON Schema string for
  inputs), which authors must learn.

#### Risks

- Prompt-document conventions (slot names, section structure) could drift between skills and
  the Prompt Engine's expectations; mitigated by the composition report and golden tests
  (FR-SKILL-004) making rendered output inspectable.
- `format_version` discipline could erode if minors add semantics casually; guarded by SM-20
  contract-diff tooling once Beta ships.

#### Reversal plan

The manifest schema is the stable surface; the directory layout and hashing are internal to
the Skill Engine's loader. Migrating to a different container (for example, a single-file
bundle for transport) is a loader change plus a converter, with `format_version` gating the
transition. Reversing the data-only rule is not a format change but a trust-model change and
would require superseding this ADR together with a Volume 2 amendment to INV-SKL-04.

#### Review conditions

- A cross-vendor skill/prompt packaging convention emerges in the ecosystem: evaluate
  convergence or an import path.
- `format_version` needs its first major bump: review the support window against real skill
  population data.
- FR-SKILL-004 golden testing proves insufficient to catch behavior-relevant format drift:
  strengthen the format contract tests.
