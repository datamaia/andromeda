### ADR-146 — Destructive Git operation policy: dual gate plus safety refs

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Git Engine, Permission Manager, CLI, TUI
- Related requirements: FR-GIT-001, FR-GIT-007, FR-SEC-100; ADR-025, ADR-016

#### Context

The Git Engine executes mutations on the user's repository, including operations that can
discard commits, staged content, or working-tree changes: reset (mixed/hard), force push,
branch deletion, history rewriting (rebase, amend), checkout/restore over dirty files, and
clean. The brief's provided constraint is absolute: no silent destructive operations. The
permission model (Volume 9) supplies the `git_mutation` permission and the Approval flow;
data-integrity ranks third in the Volume 0 requirement precedence, above functional
requirements. Agents — not only humans — request these operations, and non-interactive modes
(CI, headless per ADR-032) cannot prompt.

#### Problem

Define when a Git mutation may proceed, what protection exists against unintended data loss
when it does, and how the policy behaves in non-interactive contexts — without making
routine, non-destructive operations (commit, branch creation) unusably heavy.

#### Forces and constraints

- Permission grants can be session- or workspace-scoped (Volume 9); a standing `git_mutation`
  grant must not silently authorize a force push hours later.
- git's own recovery surfaces are uneven: the reflog does not survive branch deletion for the
  deleted branch's log, and expires by time; stash covers only tracked changes it was asked
  to hold.
- Non-interactive paths treat anything but `allow` as denial (PRD-009); prompting is not
  available there.
- Repositories can be very large; any safety mechanism must cost O(refs written), not O(tree
  copied).

#### Alternatives considered

1. **Rely on git reflog alone** — rejected. Reflog entries for a deleted branch are removed
   with the branch; reflog expiry defaults prune older entries; untracked files removed by
   clean are never in the reflog. "Recoverable via reflog" is false precisely for several of
   the most destructive operations.
2. **Full repository backup before destructive operations** — rejected. Copying object
   stores or working trees on reference-scale repositories (Volume 1 chapter 06 reference
   conditions) costs seconds to minutes and gigabytes; users would disable it, defeating the
   policy.
3. **Prohibit destructive operations entirely** — rejected. Rebase, reset, and force push
   are legitimate engineering actions; prohibition drives users to raw git outside the audit
   trail, which is strictly worse for traceability (PRD-006).
4. **Dual gate plus safety refs** — selected, described below.

#### Decision

1. Every Git mutation requires a `git_mutation` permission decision through PermissionPort
   before the git subprocess is spawned (no exception, including agents and workflows).
2. Operations classified **destructive** in the Volume 11 chapter 01 catalog additionally
   require an **explicit, operation-specific confirmation**: an interactive Approval naming
   the operation, the refs it moves, and the content classes it can discard — or, in
   non-interactive contexts, a per-invocation explicit flag (Volume 8 defines the flag
   grammar). A standing permission grant never substitutes for this confirmation.
3. Before executing a destructive operation, the Git Engine records **safety refs** under
   `refs/andromeda/safety/<ulid>` pointing at every ref the operation will move or delete,
   plus a stash-like snapshot object for working-tree-discarding operations where git
   provides one. Safety refs are local-only, never pushed, retained for a configured period
   (default 30 days), listed and restorable through the Git Engine, and pruned only after
   retention expiry.
4. Force push additionally requires `git.allow_force_push = true` in configuration AND the
   confirmation of rule 2, and is refused against protected branches regardless.
5. Refusals are structured errors (E-GIT-007, E-GIT-008, E-GIT-012), never silent no-ops;
   every decision — grant or refusal — is audit-recorded (PRD-006).

#### Rationale

The dual gate separates two different questions: "may this actor mutate this repository?"
(permission, grantable at scope) and "does someone consciously accept this specific loss
risk?" (confirmation, never standing). Safety refs make the residual risk after a conscious
yes small and bounded: writing refs is O(1) per ref regardless of repository size, survives
branch deletion, and is independent of reflog expiry. The policy therefore satisfies "no
silent destructive operations" without taxing routine mutations, which pass only the
permission gate.

#### Positive consequences

- No destructive action can occur from a standing grant alone; agent autonomy stays inside
  PRD-005 bounds.
- Recovery after a confirmed-but-regretted operation is a local ref restore, executable
  offline.
- The classification lives in one catalog; new operations get a class at specification time,
  not at incident time.
- Audit records answer who confirmed what, when, for every discard risk.

#### Negative consequences

- Two prompts (permission + confirmation) can occur for a first destructive action in a
  session; UX must merge them into one coherent dialog (Volume 8).
- Safety refs occupy repository storage until retention expiry; `git gc` cannot prune
  objects they anchor.
- Non-interactive destructive operations need explicit flags, making some CI scripts more
  verbose — deliberately.

#### Risks

- Users disable confirmations wholesale if prompts feel gratuitous; mitigation: the
  destructive class is narrow and enumerated, and routine mutations never trigger it.
- Safety-ref namespace collision with user tooling; mitigation: the `refs/andromeda/`
  namespace is declared in this specification and checked for pre-existing use at first
  write (E-GIT-011 on conflict).
- A crash between safety-ref write and operation start leaves orphan refs; mitigation:
  retention pruning removes them; orphans are harmless.

#### Reversal plan

The classification table, the confirmation surface, and the safety-ref mechanism are each
independently reversible: classes can be re-assigned by document change; the safety-ref
mechanism can be replaced by a different snapshot mechanism behind the same "restorable
point recorded before every destructive operation" requirement (FR-GIT-007) without touching
the gate logic.

#### Review conditions

- Post-MVP, review confirmation friction telemetry (consent-gated, Volume 10) and incident
  reports: zero data-loss incidents with high prompt fatigue may justify narrowing the
  destructive class; any data-loss incident requires widening review.
- Review the 30-day default retention against observed repository storage impact.
- Review the classification whenever an amendment batch (ADR-145) adds operations.
