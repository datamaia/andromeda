### ADR-062 — Named authentication profiles bind credentials to providers

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Authentication Layer, Configuration Manager, Provider Layer, CLI, TUI
- Related requirements: FR-AUTH-008, FR-AUTH-009; Volume 2 Credential/Authentication Session entities

#### Context

The brief mandates multiple authentication profiles. Users hold several credentials for the
same provider (personal/work billing, region variants, key-per-project policies) and several
providers per workspace. Volume 2 gives Credentials stable labels and ULIDs; configuration
tables `[auth]` and `[providers.*]` belong to Volume 5; precedence mechanics belong to
Volume 10. Something must connect "which credential" to "which provider" per run without
placing material in configuration.

#### Problem

How are credential-to-provider bindings named, stored, selected, and recorded so that multiple
identities per provider coexist, selection is deterministic, ambiguity is an error rather than
a guess, and run records stay reproducible (SM-12)?

#### Forces and constraints

- No secret material in configuration (ADR-014); bindings may carry only references.
- Selection must work identically in interactive and non-interactive modes (PRD-009).
- Run reproducibility requires recording which identity was used (SM-12).
- Ambiguous silent picks would make cost and audit attribution wrong (PRD-006).

#### Alternatives considered

1. **Per-provider single credential** (one implicit credential per provider slug) — rejected:
   contradicts the mandated multiple-profiles capability; forces slug duplication as a
   workaround.
2. **Ad-hoc per-command flags naming credential labels directly** — rejected: no reusable
   named unit for Agent Profiles and workspaces to reference; selection logic scattered.
3. **Profiles as first-class database entities** — rejected: profiles are configuration
   (declarative, diffable, versionable with the workspace); the referenced Credential is the
   database entity.

#### Decision

Authentication Profiles are named configuration units: `[auth.profiles.<name>]` with
`provider` (slug) and `credential` (label) keys plus options. Selection precedence: explicit
CLI flag > Agent Profile binding > workspace configuration > `auth.default_profile` >
single-candidate inference. Zero or multiple candidates without a selector is the error
E-AUTH-010. Every run records the profile name used. `AuthPort.ListProfiles` exposes
profiles with fingerprints and status only.

#### Rationale

Named configuration units give one referenceable concept to CLI flags, Agent Profiles, and
workspace defaults; the strict precedence with ambiguity-as-error keeps selection
deterministic and auditable; keeping profiles out of the database keeps them portable with
project configuration while Credentials stay machine-local.

#### Positive consequences

- Work/personal and multi-region identities coexist cleanly per provider.
- Runs are attributable and reproducible with respect to identity used.
- Profiles travel with configuration; secrets never do.

#### Negative consequences

- One more indirection layer for the simple one-key case (mitigated by single-candidate
  inference).
- Profile/credential dangling references become a new error class to diagnose (E-AUTH-001
  names both ends).

#### Risks

- Users may proliferate near-duplicate profiles; mitigation: `ListProfiles` shows binding and
  status, and documentation prescribes naming conventions.

#### Reversal plan

Profiles are a resolution layer above AuthPort; collapsing to per-provider single credentials
would be a configuration migration (profile → provider default) with no port or entity
changes.

#### Review conditions

- If Agent Profile-level identity requirements (Volume 4 evolution) demand richer binding
  options, extend profile keys additively.
- Revisit single-candidate inference if support data shows it surprises users.
