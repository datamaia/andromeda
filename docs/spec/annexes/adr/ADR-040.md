### ADR-040 — One agent loop for interactive and autonomous operation

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Agent Engine, Runtime, Planner, Execution Engine
- Related requirements: FR-AGT-001, FR-AGT-002; PRD-001, PRD-009

#### Context

Andromeda serves interactive TUI sessions, one-shot CLI runs, and IPC-driven automation
(ADR-012, ADR-032). Comparable products often grow a "chat mode" engine and a separate
"autonomous mode" engine, which then diverge in permissions, records, and recovery. Volume 1
mandates parity between human and automated use (PRD-009) and a single runtime (PRD-001).

#### Problem

Should interactive conversation and autonomous execution be driven by one loop
implementation or by mode-specialized engines?

#### Forces and constraints

- PRD-009 parity: identical records, permissions, and observability in every mode.
- Recovery (PRD-010) and replay (SM-12) must hold for every run regardless of origin.
- Trivial conversational exchanges must not carry heavyweight ceremony.
- Two engines double the test surface of the most safety-critical component.

#### Alternatives considered

1. **Separate chat and autonomous engines** — rejected: guarantees divergence in exactly the
   properties (mediation, records, recovery) the product sells; doubles conformance surface.
2. **One engine with mode branches throughout** — rejected: mode conditionals scattered
   through the loop are the same divergence in one file; harder to audit than either option.
3. **One loop, mode-invariant, with mode expressed only at the edges** (permission
   resolution and presentation) — chosen.

#### Decision

The Agent Engine implements exactly one plan–act–observe loop (FR-AGT-001) used by every
driver and mode. Interactive versus non-interactive operation differs only in permission
resolution (prompting versus policy-only, Volume 9) and presentation. Every run — including
trivial conversational goals — executes through a plan (direct-execution plans, ADR-041).

#### Rationale

The loop is the product's safety-critical core; one implementation makes parity a structural
property instead of a test obligation, and concentrates the correctness investment (property
tests, crash injection, replay) where it pays across all modes at once.

#### Positive consequences

- PRD-009 parity checkable by record comparison, not argued by review.
- One recovery and one replay semantics; SM-11/SM-12 suites cover every mode.
- Drivers stay thin; new drivers (IPC clients) inherit full loop behavior.

#### Negative consequences

- Trivial exchanges pay the plan/record overhead (mitigated by direct-execution plans).
- The single loop is a critical component whose defects affect every mode.

#### Risks

- Latency ceremony creeping into conversational feel — watched via Volume 12 interaction
  budgets.
- Mode-specific pressure reintroducing branches — guarded by parity record tests.

#### Reversal plan

Splitting modes later would fork the engine behind the same Runtime API; records and events
are mode-agnostic, so persisted data survives such a fork unchanged.

#### Review conditions

Review if Volume 12 shows the loop overhead breaching interaction budgets for conversational
turns, or if a mode requires semantics that cannot be expressed at the edges.
