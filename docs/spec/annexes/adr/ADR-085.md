### ADR-085 — Two-axis memory taxonomy: storage layer × memory kind

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Memory Manager, Context Manager, Persistence Layer, CLI, TUI
- Related requirements: FR-MEM-001, FR-MEM-004, FR-MEM-007; Volume 2 chapter 07 (Memory Record)

#### Context

The product brief mandates eight memory concepts: Session Memory, Workspace Memory,
Long-Term Memory, Semantic Memory, Episodic Memory, User Preferences, Project Knowledge, and
Decision Memory. Volume 2 froze the Memory Record shape with a `layer` attribute
(`session` | `workspace` | `long_term`) and delegated the closed `kind` vocabulary to
Volume 7. A flat list of eight memory types would force every lifecycle rule (retention,
visibility, conflict resolution, projection schema) to be written eight times and would blur
which properties come from *where a record lives* versus *what a record is*.

#### Problem

How should the eight mandated memory concepts be modeled so that visibility/retention rules
and content/projection rules compose without duplication or ambiguity?

#### Forces and constraints

- Volume 2's `layer` attribute and ADR-028's database split already fix the storage axis.
- Retention and privacy properties track lifetime and visibility (layer), while projection
  schemas and conflict semantics track content class (kind).
- The glossary demands one name per concept; eight overlapping "memories" invite drift.
- The kind vocabulary must be closed (linter-enforceable, portable across exports).

#### Alternatives considered

1. **Flat eight-type enum** — rejected: "Workspace Memory" and "Project Knowledge" differ on
   different axes (location vs content); a flat enum forces artificial exclusivity (a
   preference stored per-workspace would need a ninth type).
2. **Free-form tags instead of kinds** — rejected: no closed vocabulary means no per-kind
   projection schemas, no per-kind retention defaults, and unlintable exports.
3. **Two orthogonal axes** (selected) — layer (session/workspace/long_term, frozen by
   Volume 2) × kind (`episodic`, `semantic`, `procedural`, `preference`, `decision`).

#### Decision

Memory is classified on two orthogonal axes. The kind vocabulary is closed with exactly five
values: `episodic`, `semantic`, `procedural`, `preference`, `decision`. The mandated
concepts map as: Session/Workspace/Long-Term Memory = the three layers; Episodic/Semantic
Memory = kinds; User Preferences = kind `preference`; Project Knowledge = kind `semantic` at
the `workspace` layer; Decision Memory = kind `decision`. Kind-specific structured
projections are: `preference` requires `{subject, value}`; `decision` requires
`{subject, decision, rationale}`; others MAY carry `{subject}`.

#### Rationale

The two axes factor the rule space cleanly: layer rules (visibility, storage home, retention
defaults) are written three times; kind rules (projection, conflict subjects, consolidation
targets) five times; every combination is well-defined without enumeration. All eight
mandated concepts exist with exactly one grid address each.

#### Positive consequences

- Lifecycle requirements (FR-MEM-004..008) parameterize over two small closed sets.
- New product concepts (e.g., a per-workspace preference) need no schema change.
- Exports carry two machine-checkable attributes instead of prose classifications.

#### Negative consequences

- Users see two classification dimensions instead of one friendly label; CLI/TUI views must
  present the grid comprehensibly (Volume 8's concern).
- Five kinds may prove too coarse; extending the vocabulary requires a change-procedure
  amendment (deliberate friction).

#### Risks

- Kind misassignment by agents degrades retrieval grouping — mitigated by kind validation at
  ingestion and user-visible provenance (FR-MEM-002); tracked under RISK-MEM-001.

#### Reversal plan

Kinds are data: a re-mapping migration (forward-only per ADR-029) can split or merge kind
values, and the closed vocabulary lives in one place (FR-MEM-001). Layer semantics cannot be
reversed without a Volume 2 amendment — they are load-bearing for ADR-028.

#### Review conditions

- Recurring user or agent need for a content class that maps poorly onto the five kinds.
- Volume 15 sequencing of team/shared-memory features (would add visibility scopes beyond
  the three layers and force revisiting the layer axis).
