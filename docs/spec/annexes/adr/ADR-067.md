### ADR-067 — Enterprise proxies via standard environment variables and configured trust anchors, with no TLS verification bypass

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Authentication Layer, Provider Layer, Configuration Manager
- Related requirements: FR-AUTH-006; ADR-019 (HTTP baseline)

#### Context

The brief mandates enterprise proxy support in the authentication scope. Enterprise networks
interpose authenticating proxies and TLS-inspection appliances whose interception roots are
not in system trust stores. The Go standard HTTP stack honors `HTTPS_PROXY`/`HTTP_PROXY`/
`NO_PROXY` conventions; ADR-019 fixes stdlib HTTP as the adapter baseline with uniform
behavior across adapters. Credential handling rules (ADR-014, FR-AUTH-009) apply to proxy
credentials like any other.

#### Problem

How do all provider and authentication requests traverse enterprise proxies — including
authenticating ones and TLS-inspecting ones — without creating a verification-bypass
mechanism that would silently void transport integrity for credential-bearing traffic?

#### Forces and constraints

- Proxy settings must apply uniformly to every adapter (ADR-019 uniformity) — per-adapter
  proxy code would drift.
- Proxy credentials are secrets; they follow the full storage model.
- A "skip TLS verification" escape hatch, once present, becomes the workaround of first
  resort and defeats credential-safety guarantees invisibly.
- Loopback endpoints (FR-PROV-084) must bypass proxies or local providers break in proxied
  environments.

#### Alternatives considered

1. **Environment variables only** — rejected as insufficient: no way to attach a stored
   proxy credential or a custom CA declaratively, and headless services often need explicit
   configuration.
2. **A `tls_skip_verify` option** for interception environments — rejected outright: it is a
   permanent downgrade path for all traffic including token exchanges; the sanctioned
   mechanism for interception proxies is trusting their root explicitly.
3. **System-store-only trust** (require admins to install roots system-wide) — rejected as
   the only path: correct where possible, but per-application bundles are the documented
   reality in many enterprises; supporting `ca_bundle` costs little and is explicit.

#### Decision

(a) Standard proxy environment variables are honored by default for all adapter and
authentication traffic. (b) The `[auth.proxy]` table (`url`, `no_proxy`, `credential`,
`ca_bundle`) overrides the environment explicitly; the `credential` key references a stored
Credential label — never inline material. (c) `ca_bundle` appends PEM roots to system trust
for provider connections. (d) No configuration, flag, or environment variable disables TLS
server verification for non-loopback endpoints; loopback traffic bypasses proxies by the
ADR-066 locality rule.

#### Rationale

Environment conventions serve the common case with zero configuration; the explicit table
serves managed deployments with attribution through configuration resolution; refusing a
verification bypass keeps the transport guarantee unconditional — the one setting whose
absence cannot be worked around is the one that protects credentials in flight.

#### Positive consequences

- Works in authenticated-proxy and TLS-inspection enterprises through explicit, auditable
  configuration.
- Uniform behavior across all adapters; one implementation, one test surface.
- No silent transport downgrade exists to misconfigure or socially engineer.

#### Negative consequences

- Environments with broken interception appliances have no quick bypass; they must obtain
  the appliance root (by design, but it will generate support friction).
- Proxy misconfiguration surfaces as connection failures that users may misattribute to
  providers (mitigated by E-AUTH-011 and doctor diagnostics).

#### Risks

- `NO_PROXY` semantics vary subtly across ecosystems; mitigation: Volume 10 documents the
  exact matching rules Andromeda applies, and doctor prints the effective proxy decision per
  endpoint.

#### Reversal plan

Proxy handling lives in the shared HTTP baseline beneath all adapters; changing mechanisms
(for example, adding PAC support) is additive there. The no-bypass rule is reversible only
through a superseding ADR — deliberately high friction.

#### Review conditions

- If validated enterprise deployments demonstrate a class of proxies unusable under these
  mechanisms, evaluate additive support (for example, PAC files) — never a verification
  bypass.
- Review when the Windows phase (v2 candidate) introduces platform proxy conventions.
