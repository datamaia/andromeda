### ADR-028 — Database topology: one workspace-local database plus one global database

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Persistence Layer, Workspace Engine, Configuration Manager, Memory Manager, Audit Log, Updater
- Related requirements: Volume 1 product objectives (PRD-003, PRD-006, PRD-010); Volume 2 chapter 10; formal requirement links are added at consolidation

#### Context

ADR-007 fixes SQLite as the storage engine; ADR-022 fixes directory resolution (XDG-style
paths via the PAL, project-local `.andromeda/`). What neither fixes is **topology**: how many
databases exist and which entities live where. Volume 2's model contains both
workspace-scoped data (sessions, runs, audit of workspace actions, workspace memory) and
machine-scoped data (providers, credentials, packages, releases, long-term memory, the
registry of known workspaces). Some entities exist at both scopes (configuration profiles,
agent profiles, plugins, skills, MCP servers, permissions, events, audit records).

#### Problem

Decide the database topology — one big store, per-scope stores, or something finer — such
that workspaces are portable and self-contained, machine-level secrets metadata never leaks
into workspace exports, concurrent multi-workspace use works, and rebuildable caches never
masquerade as authoritative state.

#### Forces and constraints

- **Workspace portability**: copying or archiving a workspace directory should carry its full
  history (PRD-010); a workspace must be analyzable on another machine.
- **Secret hygiene**: credential metadata must be structurally separated from anything a user
  might share (Volume 9; INV-CRED-01 stores only references, but even references and labels
  do not belong in a shared workspace file).
- **Concurrency**: several workspaces may be open at once; SQLite locking is per-file, so
  per-workspace files isolate write contention (WAL per ADR-007 handles readers/writers
  within one).
- **Blast radius**: corruption of one file should lose at most one scope (one workspace, or
  the machine-level registry — each independently backed up and migrated, ADR-029).
- **Simplicity**: every additional database multiplies migration, backup, and integrity
  surface; the count must stay small and principled.
- **Caches**: index/embedding data is large and rebuildable; it must not bloat authoritative
  files or their backups.

#### Alternatives considered

| Criterion | Workspace + global split (selected) | Single global database | Per-entity/per-area databases | Per-session databases |
|---|---|---|---|---|
| Workspace portability | Full — copy `.andromeda/` | None — history trapped in one machine file | Partial, complex assembly | Fragmented history |
| Secret separation | Structural (different files) | Policy-only (same file) | Structural but sprawling | Same problem as split, more files |
| Locking isolation | Per workspace | All workspaces contend | Fine-grained but cross-DB transactions impossible | Excessive file count |
| Migration/backup surface | 2 schemas | 1 schema | Many schemas | Unbounded instances |
| Blast radius | One workspace or global registry | Everything | Small per file, huge coordination cost | Small but incoherent |
| Cross-entity transactions within a scope | Yes (single file per scope) | Yes | No (SQLite cannot atomically span files without ATTACH caveats) | No |

- **Single global database** fails portability and secret separation at once, and makes every
  workspace contend on one write lock.
- **Per-area databases** (sessions.db, memory.db, audit.db, …) break the aggregate-per-
  transaction rule of Volume 2 chapter 01: Run aggregate writes would span files.
- **Per-session databases** fragment the workspace history that queries (audit, accounting,
  memory) need to traverse.

#### Decision

Andromeda persists authoritative state in exactly two database schemas, instantiated as:

1. **Workspace database** — `.andromeda/state.db`, one per Workspace. Holds everything
   workspace-scoped: the Workspace identity row, projects, sessions, the entire Run aggregate,
   workspace-scoped catalog entities, workspace events, traces, cost records, audit records
   (workspace chain), and session/workspace memory.
2. **Global database** — `<data_dir>/andromeda/global.db` (data directory per ADR-022), one
   per user per machine. Holds machine-level state: the workspace registry, providers and
   models, credentials and authentication sessions (metadata + Secret Store references only),
   global-scoped catalog entities, releases and update history, long-term memory, global
   events and audit records (global chain).

Supplementary rules:

3. **Scoped catalogs** (Configuration Profile, Agent Profile, Workflow, Skill, Plugin, MCP
   Server, Package, Extension, Permission) persist in the database matching their `scope`
   attribute, same table name in both schemas.
4. **Credentials and authentication sessions live only in the global database** — never in a
   workspace file, so workspace exports are structurally unable to carry them.
5. **Rebuildable caches** (index data, embeddings) live in separate cache databases
   (`.andromeda/index.db`); they carry no authoritative state, do not migrate (drop and
   rebuild), and their loss is never an integrity error.
6. Cross-database references are by ULID only (ADR-027); no ATTACH-based cross-file
   transactions in the authoritative path.

#### Rationale

The split matches the two real ownership domains — "this workspace" and "this machine/user" —
and nothing else earns a file. It gives structural (not policy) separation for credential
metadata, per-workspace lock isolation, a portable workspace artifact, exactly two schemas to
migrate and back up, and a clean rule for every future entity: follow the scope attribute.
The cache exclusion keeps authoritative files small enough for the backup-before-migration
discipline of ADR-029 to stay cheap.

#### Positive consequences

- Copying `.andromeda/` moves a workspace's complete history; deleting it cleanly severs one
  workspace.
- Credential metadata cannot appear in workspace exports by construction.
- Concurrent workspaces do not share write locks; WAL behavior stays per-workspace.
- Two schemas, two migration chains, two backup targets — bounded operational surface.

#### Negative consequences

- Queries spanning scopes (e.g., "cost across all workspaces") require reading multiple
  files; Volume 10 provides the aggregation path rather than a JOIN.
- The workspace registry in the global database can drift from on-disk reality (workspaces
  moved/deleted outside Andromeda); it is defined as a rebuildable cache row set, reconciled
  on open.
- Two audit hash chains (workspace and global) instead of one; verification tooling must
  handle both (Volume 9).

#### Risks

- **Registry drift** misleading workspace discovery. Detection: open-time reconciliation.
  Mitigation: registry defined as cache; workspace database is authoritative.
- **User confusion** about where data lives. Detection: support/feedback channels.
  Mitigation: `andromeda` diagnostics command surfaces both paths (Volume 8); documentation
  fixed by chapter 10's table.
- **Cross-scope invariant gaps** (e.g., global Permission referencing a workspace entity).
  Detection: spec review at consolidation; runtime validation. Mitigation: chapter 10 rule 1
  forbids cross-writes; cross-references are IDs with tombstone semantics.

#### Reversal plan

Merging to a single database (or splitting further) is a data migration plus a path change
behind the Persistence Layer: both schemas already share conventions (chapter 10), so tables
move verbatim with a scope column where needed. Because all access goes through the
Persistence Layer (ADR-007 rule 3), no component outside it observes the topology. The
workspace-portability property would be lost by a merge, so any such reversal requires
revisiting PRD-010-adjacent expectations explicitly.

#### Review conditions

- A future multi-user or team-sync capability (currently absent from scope) demands shared
  state with different ownership boundaries.
- Volume 12 benchmarks show the global database becoming a contention point (e.g., event
  writes from many concurrent workspaces).
- The workspace registry cache produces recurring user-visible drift defects despite
  reconciliation.
