### ADR-121 — Permission evaluation: deny-overrides with an ask-forcing tier

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Permission Manager, Policy Engine
- Related requirements: FR-SEC-100, FR-SEC-103, FR-SEC-105

#### Context

The permission model (Volume 9 chapter 05) evaluates overlapping standing grants, configured
policy rules, and built-in defaults for every side-effecting action. Overlap is legal by
design (Volume 2 Permission entity: no natural key, overlapping grants allowed), so the
evaluation order — not row uniqueness — must produce one deterministic outcome.

#### Problem

When multiple grants and rules match one query with conflicting effects (allow, deny, and
the need to force a prompt), which wins — and how do users predict it?

#### Forces and constraints

- Safety first (Volume 0 precedence order): a written prohibition must never be outranked.
- PRD-009: non-interactive evaluation must be resolvable without prompting, deterministically.
- Users author rules incrementally; the model must not require understanding rule age,
  insertion order, or scope-width arithmetic to predict outcomes.
- Volume 10 owns configuration-layer precedence; the evaluation-tier order must compose with
  it without duplicating it.
- Evaluation sits on the tool hot path (NFR-SEC-006 latency budget): one pass, no fixpoints.

#### Alternatives considered

1. **Most-specific-wins** (narrower selector or scope outranks broader, any effect) —
   rejected: specificity comparison across heterogeneous qualifiers (is a `path` glob more
   specific than a `command` pattern?) has no defensible total order; users cannot predict
   outcomes, and a narrowly-scoped allow could quietly outrank a broad organizational deny.
2. **Most-recent-wins** (later decision outranks earlier) — rejected: makes security posture
   depend on decision history order, invisible in configuration review; a stray
   `allow_once`-era grant could outrank a policy deny added earlier.
3. **First-match in declaration order** (firewall-style) — rejected for grants (rows have no
   meaningful order) and undesirable for rules: silent shadowing of later rules is a classic
   firewall foot-gun requiring tooling to detect.
4. **Deny-overrides with an ask-forcing tier** (chosen) — flat tier order: deny > ask >
   allow > default-ask.

#### Decision

Permission evaluation resolves in exactly four tiers: (1) any matching deny wins; (2) else
any matching `ask`-effect rule forces prompting (or denial where interaction is not
permitted); (3) else any matching allow wins; (4) else the default is ask. Within a tier,
outcomes are identical by construction and all matching rows of the winning tier are
recorded. Configuration-layer precedence (Volume 10) selects *which rules exist*; tier
precedence resolves *among applicable candidates*.

#### Rationale

Deny-overrides is the only order under which a written prohibition is unconditionally
effective — matching the Volume 0 precedence order and user intuition ("I denied it, so it
is denied"). The ask-forcing tier covers the real need that specificity tried to serve:
pinning interactive review for sensitive patterns (e.g., `git push *`) without revoking
broader conveniences. One-pass evaluation keeps the latency budget trivially satisfiable.

#### Positive consequences

- Outcomes are predictable from the rule set alone; no history or order reasoning.
- Security review of a configuration is local: finding any matching deny answers the
  question.
- The audit record's "winning tier plus matched rows" form is simple and complete.

#### Negative consequences

- No way to express "allow, overriding a broader deny" — by design; users must narrow the
  deny instead, which is more verbose.
- `ask`-forcing rules can surprise users who expected a standing allow to suppress prompts;
  surfaces must show the forcing rule in the prompt (chapter 05 requires it in the payload).

#### Risks

- Rule sprawl (many denies accumulating) degrades comprehension; mitigated by CLI/TUI
  grant/rule listing surfaces (Volume 8) and doctor checks for unmatched rules.
- Users emulating precedence via convoluted selectors; mitigated by documentation patterns.

#### Reversal plan

Tier resolution is one function inside the Permission Manager with a golden decision-table
suite; replacing the order is a contained change plus a MAJOR document change (Volume 0
chapter 10) because recorded decisions' meanings shift. Persisted grants need no migration —
only the resolver changes.

#### Review conditions

- If field usage shows recurring need for narrow allows under broad denies, revisit with an
  explicit exception construct (never implicit specificity).
- Review when organization-level policy distribution (Future phase) introduces centrally
  managed rules.
