### ADR-141 — Telemetry export transport: OTLP only, allowlist-built payloads, no log export

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Telemetry, Secret Store, Platform Abstraction Layer (PAL)
- Related requirements: FR-OBS-012, FR-OBS-013; ADR-011, ADR-140

#### Context

With consent decided (ADR-140), the remaining decisions are transport and payload
discipline: which protocol carries consented telemetry, which signal classes may leave the
machine at all, and how payloads are constructed so that the prohibited-data list (prompts,
paths, identities, free text) is enforced structurally rather than by scrubbing.

#### Problem

Choose the export protocol(s) and the payload construction model such that exported data is
exactly the documented catalog, transport is an official, widely supported standard, and no
second export path can accrete.

#### Forces and constraints

- ADR-011 already commits to OpenTelemetry semantics; a non-OTLP export format would forfeit
  the interoperability that justified OTel in the first place.
- There is no Andromeda-hosted endpoint (Out of Scope): the destination is user
  infrastructure, so a standard protocol is the only universal interface.
- Denylist scrubbing fails open (new fields leak by default); the prohibited list must be
  unrepresentable, not filtered.
- Log records are the most leak-prone signal class (free text by nature) and the least
  useful in aggregate.
- Endpoint authentication material is a secret (Volume 9): it must live behind Secret Store
  references, never inline in configuration.

#### Alternatives considered

1. **Custom JSON ingestion API** — requires an Andromeda-defined server contract nobody
   runs; violates the no-hosted-service scope and the interoperability rationale. Rejected.
2. **Multiple exporter plugins (StatsD, Prometheus remote-write, vendor SDKs)** — each
   additional path multiplies the egress audit surface and the redaction proof burden;
   users with such backends can bridge from an OTel collector they run. Rejected: one path,
   auditable once.
3. **Export logs as an opt-in signal class** — even redacted, log `msg` fields are free
   text with unbounded leak modes; aggregate error-code counters carry the diagnostic value
   at none of the risk. Rejected.
4. **Denylist scrubbing at the exporter** — fails open on every new field; contradicts the
   allowlist force above. Rejected.

#### Decision

1. Consented telemetry exports over **OTLP only** — `http/protobuf` by default, `grpc`
   selectable (`telemetry.protocol`) — to the user-configured endpoint. No other export
   transport, format, or destination exists in the product.
2. Exportable signal classes are **metrics and traces** (per consent scope). **Log records
   are never exportable** under any configuration.
3. Payloads are **allowlist-built** from the versioned collected-data catalog
   (FR-OBS-012): fields enter by name; prohibited sources are unreachable from the builder
   package under the ADR-033 dependency rules. Trace exports strip `andromeda.*` entity IDs
   (structural hashes preserve shape); instants coarsen to hour resolution.
4. Endpoint authentication uses Secret Store references (`telemetry.auth_secret_ref`);
   TLS verification is always on, with no insecure override key.

#### Rationale

OTLP is the one protocol every telemetry backend either speaks or has a collector for, and
it is the protocol the already-chosen SDK emits natively — one path, zero translation.
Restricting classes to metrics and traces plus allowlist construction moves the privacy
guarantee from runtime diligence to build-time structure: a leak requires changing the
catalog, which changes the consent version, which re-asks the user (ADR-140).

#### Positive consequences

- One egress path to audit, test (NFR-OBS-006), and document.
- Users integrate with existing observability stacks without Andromeda-specific tooling.
- The catalog document, the consent screen, and the builder are provably the same list.

#### Negative consequences

- Backends without OTLP support require the user to run a collector.
- No exported logs means some remote debugging workflows (fleet log aggregation) are
  unsupported by design.
- Structural hashing of trace identities reduces cross-install trace analytics — accepted
  for unlinkability.

#### Risks

- OTLP protocol evolution outpacing the pinned SDK: mitigated by the SDK pin and the
  register's open question on the semantic-conventions/protocol pin (PENDING VALIDATION at
  implementation).
- Allowlist drift from the rendered catalog: mitigated by the CI divergence check
  (FR-OBS-012 verification).

#### Reversal plan

Adding a transport is additive (a second exporter behind the same gate and builder) but
requires an amending ADR because it widens the audited egress surface. Removing OTLP would
be a breaking product change under SM-20. The allowlist builder is transport-independent
and survives any transport decision.

#### Review conditions

- OTLP is superseded or the pinned OTel Go SDK's exporter support changes materially.
- A legitimate fleet-operations need for exported logs is demonstrated — re-decide class
  exclusion with its own consent category, never silently.
- Users demonstrably cannot run collectors and demand a second protocol.
