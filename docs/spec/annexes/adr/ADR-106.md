### ADR-106 — Terminal capability detection: conservative signal ladder with configuration override

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: TUI, CLI
- Related requirements: FR-TUI-008, FR-UX-041, FR-UX-042; ADR-103 (CLI on/off styling decision), ADR-026 (tier structure)

#### Context

ADR-026 mandates four normatively defined rendering tiers (truecolor → 256 → 16 → none)
plus a light fallback theme, and requires that tiers be "normatively defined by Volume 8,
not left to runtime approximation heuristics alone." Terminals communicate their
capabilities poorly: `TERM` describes a terminfo entry that may under- or over-state
reality, `COLORTERM=truecolor|24bit` is a widespread but unstandardized convention,
multiplexers (tmux, screen) interpose their own `TERM`, and background color (needed for
dark/light mode selection) is only queryable via the OSC 11 control sequence, whose
support varies by emulator.

#### Problem

Define how the TUI selects a tier and a theme mode at startup: which signals are
consulted, in what order, what the defaults are when signals are absent or contradictory,
and how users override every automatic decision.

#### Forces and constraints

- Over-detection (claiming truecolor on a 16-color terminal) renders garbage; under-
  detection merely loses fidelity. The cost function is asymmetric, so the ladder must be
  conservative.
- Detection must be bounded in time: SM-06(b) gives TUI startup 500 ms p95 end-to-end; no
  probe may block rendering on a terminal that never answers.
- The user must always win: any detection can be wrong somewhere, so both tier and mode
  need closed-enum configuration overrides (`[tui.theme]`).
- The CLI needs only an on/off styling decision (ADR-103); the full ladder is a TUI
  concern — one shared token mapping, two consumption depths.
- Never invent: no undocumented per-emulator sniffing (e.g., parsing version strings of
  specific terminal apps) — only documented, widely deployed signals.

#### Alternatives considered

1. **Conservative signal ladder with config override (selected).** Configured tier, else
   ADR-103 off-switches, else `COLORTERM`, else `TERM` `256color` substring, else
   terminfo color capability, else none; mode from config, else bounded OSC 11 query,
   else dark.
2. **Terminfo-only detection** — trust the terminfo database exclusively. Rejected:
   terminfo entries routinely lag emulator reality (notably truecolor, which classic
   terminfo cannot express portably), and multiplexed sessions misreport; the result is
   systematic under-detection with no escape hatch except `TERM` surgery.
3. **Active probing (query-and-measure: emit sequences, read responses, time them)** —
   rejected as the primary mechanism: adds startup latency on every launch, breaks on
   terminals that swallow queries, and reintroduces the "runtime approximation
   heuristics" ADR-026 prohibits as the sole definition. OSC 11 for background color is
   retained as the one bounded, optional probe because no passive signal carries that
   information at all.
4. **Persist per-terminal learned capabilities** — a cache keyed by terminal identity.
   Rejected for MVP: terminal identity is itself unreliable over SSH/multiplexers, and a
   wrong cached entry is worse than a wrong fresh detection; the config override plus
   deterministic ladder covers the need without state.

#### Decision

Tier selection, first match wins: (1) `tui.theme.tier` explicit value; (2) styling
disabled per the ADR-103 resolution → `none`; (3) `COLORTERM` equals `truecolor` or
`24bit` → `truecolor`; (4) `TERM` contains `256color` → `ansi256`; (5) terminfo reports
color capability → `ansi16`; (6) otherwise `none`. Mode selection: (1) `tui.theme.mode`
explicit `dark`/`light`; (2) `auto` → OSC 11 background query with a 50 ms budget,
selecting by background luminance threshold 0.5; (3) no answer → `dark`. Resolution runs
once at shell start and on watched `[tui.theme]` changes — never mid-session re-probing.
Inside multiplexers no special-casing occurs: the ladder reads what the multiplexer
presents, landing conservatively (typically `ansi256`) unless `COLORTERM` passes through.
The `COLORTERM` convention and the OSC 11 support matrix are recorded as a register
assumption and an open question (PENDING VALIDATION) respectively in the Volume 8
register.

#### Rationale

The ladder encodes the asymmetric cost function: every step that could over-claim
requires an explicit signal (`COLORTERM`, `256color`), and silence degrades safely.
Putting configuration first makes every wrong answer a one-line fix, which is the honest
posture given that no detection method is universally correct. The single bounded probe
(OSC 11) is justified because mode selection has no passive alternative and a wrong mode
is highly visible; its 50 ms budget and dark default keep it harmless where unsupported.

#### Positive consequences

- Deterministic, fully documented resolution — `doctor` can print the decision and the
  deciding rule, and golden tests enumerate the ladder.
- Zero startup penalty beyond one bounded query, only in `auto` mode.
- Multiplexer behavior is predictable and explainable rather than special-cased.

#### Negative consequences

- Truecolor terminals that do not set `COLORTERM` render at `ansi256` until the user sets
  the config key — deliberate under-claiming.
- The 50 ms OSC 11 budget misses slow remote terminals (SSH with high RTT), yielding dark
  mode on some light terminals; the config override is the remedy.

#### Risks

- The `COLORTERM` convention could decay or fork in the ecosystem; the check is one
  ladder rung, trivially amended (register assumption tracks it).
- OSC 11 responses arriving *after* the budget could interleave with input parsing;
  mitigated by the input layer discarding late terminal responses by sequence class.

#### Reversal plan

The ladder is one pure resolution function consumed by FR-TUI-008. Replacing it (e.g.,
with learned capabilities at Beta) changes that function and its tests only; the tier
tables, theme engine, and configuration keys are untouched. A superseding ADR records any
replacement.

#### Review conditions

- Chapter 12 terminal-matrix testing shows the ladder misclassifying a Tier 1-relevant
  emulator population.
- A standardized capability negotiation gains real deployment (e.g., a terminfo extension
  for truecolor becoming ubiquitous) — add it as a ladder rung above `COLORTERM`.
- The OSC 11 open question resolves with a support matrix; revisit the 50 ms budget and
  the dark default.
