### ADR-133 — Configuration schema versioning: integer `config_version`, forward-only transforms, in-memory upgrade with explicit backed-up rewrite

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Configuration Manager, CLI, Updater
- Related requirements: FR-CFG-008; ADR-029 (regime pattern)

#### Context

Configuration files outlive binaries in both directions: upgrades meet old files, and
synced dotfiles or downgrades meet files written by newer versions. ADR-029 already chose
forward-only migration with backup-based recovery for the databases, on the argument that
a byte-identical pre-change artifact beats any inverse script. The configuration schema
needs its own versioning and migration regime, plus a deprecation lifecycle for key
renames that should not force immediate edits.

#### Problem

Choose the version marker, the migration direction and trigger, and the file-rewrite
policy such that old files keep working, future files fail cleanly and recoverably, and
hand-edited files are never modified behind the user's back.

#### Forces and constraints

- Configuration files are user-authored documents: silently rewriting them (reordering,
  dropping comments) is hostile, but old syntax must keep resolving.
- A single scalar version must order all schema states; migrations must be pure functions
  of the document (no environment- or machine-dependent transforms).
- Future-version files must be refused, not partially read — a future key could carry
  security-relevant semantics the old binary would silently drop.
- Deprecation must allow one key rename without breaking every existing file (SM-20
  deprecation window).

#### Alternatives considered

1. **SemVer string for the schema.** Rejected: configuration documents need only total
   ordering; major/minor/patch semantics duplicate what the transform chain already
   encodes and invite misclassification arguments.
2. **Silent automatic file rewrite on load.** Rejected: mutating user files as a side
   effect of running violates least surprise, races concurrent editors, and destroys
   comments without consent.
3. **Strict-only (no old-version tolerance: outdated files simply fail).** Rejected: it
   converts every upgrade into a mandatory config-editing session, punishing exactly the
   users who configured the most.
4. **Bidirectional migrations.** Rejected for the ADR-029 reasons: down-transforms are
   lossy by construction and under-exercised; the backup file is the exact inverse.

#### Decision

The schema version is a single monotonically increasing integer, declared per document
via the reserved `config_version` key (absent ⇒ current, validated strictly). Each version
increment ships an immutable ordered transform list. Old documents are upgraded
**in-memory on every load** (files untouched); persistent migration happens only through
the explicit migration surface, which writes and verifies a `<name>.toml.bak.v<old>`
backup before rewriting, and preserves the original on any failure (E-CFG-011). Documents
declaring a future version are refused with E-CFG-010 naming both versions and the backup
remedy. Deprecated keys resolve via aliases with warning findings (E-CFG-013) until their
removal major. Full rules: FR-CFG-008.

#### Rationale

In-memory upgrading gives strict-schema internals with tolerant file compatibility — the
resolver only ever sees the current schema, while user files change only on request, with
an exact recovery artifact. The regime deliberately mirrors ADR-029 so the product has one
migration story: forward-only, backup first, clean refusal of the future, restore as
recovery.

#### Positive consequences

- Upgrades never demand config edits; downgrades fail with a named remedy instead of
  misreading files.
- One transform chain, always exercised (every load of an old file runs it).
- Validation, suggestions, and documentation all speak the current schema only.

#### Negative consequences

- Load-time transform cost on every resolution of old files (bounded, linear; users are
  nudged toward persistent migration by the diagnostic).
- Comment preservation across rewrites is not guaranteed; the backup carries the original.

#### Risks

- Transform defects corrupt in-memory documents identically everywhere; mitigated by
  version-by-version and end-to-end (1→current) golden tests.
- Users ignore deprecation warnings until a removal major; mitigated by the migration
  surface rewriting renames mechanically.

#### Reversal plan

Adopting down-transforms later would be additive (forward chain and backups remain valid),
introduced from a chosen version forward by a superseding ADR — the same posture as
ADR-029's reversal plan.

#### Review conditions

- Transform-chain length or load-time cost becomes measurable against NFR-CFG-001.
- A schema change appears that cannot be expressed as a pure document transform.
- Field reports show E-CFG-010 refusals without usable backups (would motivate writing a
  forward-compatibility escrow copy at migration time).
