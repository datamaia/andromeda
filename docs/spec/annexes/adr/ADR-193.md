### ADR-193 — Support windows, release branches, and backport policy

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Updater, Package Manager, CLI
- Related requirements: FR-REL-012, FR-REL-013, FR-REL-014

#### Context

ADR-015 fixes SemVer and Conventional Commits; ADR-004 fixes trunk-based development with
release branches "for stabilization only". Volume 1 states that v1 enters maintenance under
a support policy, and the brief mandates definitions for support windows, backports, and
release branches. An open-source project with a small maintainer pool must promise support
it can actually deliver: every promised window is recurring engineering cost (backport
authoring, testing on the full Tier 1 matrix, releasing, advisories).

#### Problem

Define which released versions receive which classes of fixes for how long, on which
branches that work happens, and what qualifies for backporting — precisely enough that a
user can compute today whether their installed version is supported.

#### Forces and constraints

- Pre-1.0, contracts may change freely until MVP exit (Volume 1 Core discipline);
  promising long windows for 0.x versions would freeze what is meant to move.
- From v1, SemVer guarantees attach (Volume 1 phase definitions); users on major N−1 need
  a defined runway after N ships, or every major release strands them without security
  fixes.
- Each supported line multiplies CI cost: the full Tier 1 acceptance matrix gates every
  release, including patch releases (SM-17).
- ADR-004 allows release branches only for stabilization; a support policy must not
  reintroduce long-lived divergent development branches through the back door.
- Backports that include features quietly turn maintenance lines into forks; scope must be
  a closed list.

#### Alternatives considered

1. **Latest-release-only support forever** — rejected from v1: it makes every major
   release an immediate forced migration, hostile to the enterprises PRD-013 targets;
   acceptable (and adopted) for 0.x only.
2. **Fixed-duration windows per minor (e.g., every minor supported 6 months)** — rejected:
   with a monthly-ish minor cadence this yields many concurrently supported lines and
   quadratic CI cost; users get equivalent safety from "latest minor of the major".
3. **LTS designated minors** — rejected for now: LTS is a promise sized for a larger
   maintainer base; the review conditions name the signal that would justify it.
4. **Time-unlimited previous-major support** — rejected: an open-ended obligation no
   volunteer-scaled project can honor honestly.

#### Decision

1. **Pre-v1 (0.x):** only the latest published stable release is supported. Fixes ship
   forward in the next release; there are no 0.x backports. Release branches exist only to
   stabilize an imminent release (ADR-004).
2. **From v1:** at any moment the supported set is: (a) the latest minor of the current
   major — full support: bug fixes, security fixes, data-integrity fixes, delivered as
   patch releases; (b) the last minor of the previous major — security and data-integrity
   fixes only, for **12 months** from the first stable release of the current major.
   Everything else is end-of-life.
3. **Release branches** are named `release/vX.Y`, cut at the first `rc` of `X.Y`, and
   accept only cherry-picks from trunk (no direct feature work). Patch and maintenance
   releases tag from these branches; the branch for the previous major's last minor stays
   open exactly for its 12-month window.
4. **Backport qualification** is a closed list: security vulnerabilities, data-integrity
   or data-loss defects, regressions introduced by a supported release with no
   configuration workaround, and broken update/rollback paths. Features, contract changes,
   dependency upgrades without a security driver, and anything carrying a `BREAKING
   CHANGE` marker never backport.
5. Backports flow trunk-first: the fix lands on trunk (when applicable), then cherry-picks
   to the release branch via a pull request labeled `backport`, releases as a patch
   version, and is noted in the changelog of both lines.
6. Support-status is computable: release notes and the documentation state each line's
   end-of-support date; an unsupported installed version is reported by the update check
   (FR-REL-005) as guidance, never as blocking.

#### Rationale

"Latest minor of each supported major" is the narrowest set that keeps both populations
safe: upgraders within a major get everything by staying current on minors (SemVer makes
minor upgrades non-breaking), and previous-major users get a fixed, honest 12-month runway
to plan a major migration. The closed backport list keeps maintenance lines
security-shaped, and trunk-first flow guarantees no fix exists only on a dying branch.

#### Positive consequences

- Support status is a pure function of version and date — computable by tooling and users.
- CI cost is bounded: at most two concurrently supported lines.
- ADR-004's trunk discipline survives: branches are release infrastructure, not
  development venues.
- Enterprises get a plannable major-migration runway.

#### Negative consequences

- Users on old minors of the current major must first upgrade to its latest minor to
  receive fixes — deliberate, since minor upgrades are non-breaking by contract.
- 12 months may prove short for slow-moving organizations; extending it later is easy,
  shortening it is not.
- Maintaining the previous-major branch still costs a full Tier 1 matrix per patch.

#### Risks

- Backport-scope pressure ("just this one feature") eroding rule 4; mitigated by the
  closed list being lint-enforceable on PR labels and release audit (Volume 11
  automation).
- A security fix that cannot be cherry-picked cleanly onto the old line; mitigated by
  allowing a minimal divergent patch on the release branch when trunk equivalence is
  documented in the PR.

#### Reversal plan

Windows can be lengthened, or an LTS tier added, by a superseding ADR without breaking any
existing promise (both are strictly additive). Shortening a published window applies only
to majors released after the change, never retroactively — published end-of-support dates
are commitments.

#### Review conditions

- Maintainer capacity or funding changes materially (either direction).
- Enterprise adoption data (Volume 15 channels) shows the 12-month runway or the
  latest-minor rule causing measurable migration failure.
- A second concurrently supported major becomes normal (fast major cadence), stressing the
  two-line CI budget.
