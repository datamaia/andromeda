### ADR-175 — Mutation Testing: Scoped Packages, Phase-Gated Cadence, Tooling Pending Validation

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Core Domain, Extension SDK, CLI (repository tooling), Observability
- Related requirements: NFR-TEST-003, NFR-TEST-004, RISK-TEST-003; SM-14 (Volume 1, chapter 06)

#### Context

SM-14 binds line-coverage thresholds as a merge gate, but coverage measures execution, not
verification: assertion-free tests satisfy it while detecting nothing. Volume 13 needs a
defect-detection-strength measure for the packages the product leans on — Core Domain,
`internal/ports`/`sdk/` contract types, and the parser/codec packages (ADR-008, ADR-009,
ADR-024). Mutation testing is the established measure, but Go has no stdlib or de-facto
standard mutation tool, and candidate maintenance status could not be verified at authoring
time.

#### Problem

Should Andromeda adopt mutation testing, over which scope, at which cadence, and how can the
gate be specified now without committing to an unverified external tool?

#### Forces and constraints

- Coverage gaming (RISK-TEST-003) is otherwise unmitigated by any mechanical check.
- Mutation runs are computationally heavy: whole-suite execution per mutant makes per-PR gating
  on the full codebase impractical within the NFR-TEST-001 merge-gate budget.
- No external tool may be named as a dependency without verification (Volume 0 assumptions
  policy); tool facts are external information.
- The gate definition (scope, score formula, thresholds, cadence) must be stable even if the
  tool is swapped.

#### Alternatives considered

1. **Coverage plus review only.** Rejected: leaves RISK-TEST-003 with only human mitigation on
   a metric known to be gameable.
2. **Per-PR mutation gating on changed packages.** Rejected for now: unpredictable run cost
   breaches the merge-gate budget and partial-scope scores are incomparable. Revisit per the
   review conditions.
3. **Assertion-density lint heuristics.** Rejected as primary: counts assertions, not their
   power; retained as an advisory linter signal.

#### Decision

Mutation testing is adopted with the following fixed, tool-independent definition:

- **Scope:** Core Domain packages, `internal/ports` and `sdk/` contract packages, and the
  parser/codec packages for configuration, Andromeda Runtime Protocol frames, and JSON Schema
  handling. Other packages are out of mutation scope.
- **Cadence:** a scheduled weekly lane plus mandatory execution at Beta and v1 phase gates.
  Mutation testing is not a merge gate.
- **Score:** mutation score = killed mutants ÷ (killed + surviving mutants), computed per
  package after excluding mutants triaged and recorded as semantically equivalent; the triage
  list is versioned in the repository and reviewed.
- **Thresholds:** per NFR-TEST-004 (minimum 60% at the Beta gate, target 75% at the v1 gate on
  scoped packages).
- **Report format:** a JSON report per run (package, mutants generated, killed, survived,
  equivalents excluded, score) stored as gate evidence (Volume 13, chapter 04).
- **Tooling:** the concrete mutation tool is PENDING VALIDATION. Selection criteria are fixed
  now: operates on supported Go versions, runs under `go test` without a bespoke runner
  (ADR-017 discipline), supports package scoping and machine-readable output mappable to the
  report format above, and has verifiable maintenance activity. Evaluation happens before Beta
  entry; adoption lands as an amendment to this ADR.

#### Rationale

Fixing scope, formula, thresholds, cadence, and report shape makes the gate specifiable and
auditable today while quarantining the only unverifiable element — the tool — behind a
selection procedure. High-blast-radius scoping keeps run cost proportionate to risk; the
scheduled-plus-phase-gate cadence keeps the merge gate within budget while making mutation
strength a hard phase condition.

#### Positive consequences

- RISK-TEST-003 gains a mechanical mitigation with recorded evidence.
- The gate definition survives tool replacement; scores remain comparable across runs.
- Weekly trend data exists before the first gating evaluation at Beta.

#### Negative consequences

- Until tooling is validated, mutation scores cannot be produced; the Beta gate depends on
  closing this PENDING VALIDATION first.
- Equivalent-mutant triage is manual effort and a potential source of dispute; the versioned
  triage list mitigates but does not remove it.

#### Risks

- No candidate tool satisfies the criteria at evaluation time; fallback: hand-seeded fault
  injection sets for the scoped packages (a curated, versioned set of known-defect patches that
  the suite MUST fail on), which preserves the gate's intent at higher maintenance cost.
- Score churn from tool upgrades; mitigation: pin the tool version per release and re-baseline
  explicitly.

#### Reversal plan

Mutation testing is test-infrastructure-only: removing it deletes the lane and NFR-TEST-004
gating (a MAJOR document change per Volume 0, chapter 10) without any effect on shipped
artifacts. The fallback fault-injection set can substitute during any tooling gap.

#### Review conditions

- When the tooling PENDING VALIDATION is resolved (target: before Beta entry), amend this ADR
  with the selected tool and pinned version.
- If scoped-package mutation runs complete in under 10 minutes on the reference CI class,
  re-evaluate per-PR gating on changed packages.
- If the tool is unmaintained for 12 months, re-run the selection procedure.
