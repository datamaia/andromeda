### ADR-083 — Extension dependency resolution: constraint intersection with highest-satisfying version, no backtracking search

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Package Manager, Skill Engine, CLI
- Related requirements: FR-PLUG-005, FR-PLUG-006, FR-SKILL-002; INV-PKG-04 (Volume 2)

#### Context

Extension packages can depend on other extension packages: skills declare skill dependencies
as semver ranges (`skill.toml`, ADR-078), and bundles enumerate members. Plugins are
self-contained executables and declare no package dependencies in the current contract.
Dependency resolution runs inside `PackagePort.Resolve`, which must produce a complete,
side-effect-free plan (Volume 3) that installation then executes through the frozen Package
states. General-purpose package managers solve NP-hard constraint problems with backtracking
or SAT solvers; Andromeda's extension graphs are small (skills referencing a handful of
skills), and the product values predictable, explainable outcomes over maximal solvability.

#### Problem

Choose the resolution algorithm and its guarantees: how version constraints combine, which
version is selected, how deep graphs may go, what happens on conflict, and how resolution
interacts with sources — such that identical inputs always produce identical plans and every
failure is explainable in one sentence.

#### Forces and constraints

- Determinism: resolution feeds recorded plans and reproducible installs; heuristic or
  search-order-dependent outcomes are unacceptable.
- Explainability: when resolution fails, the user must see *which* constraints collided —
  a solver's "unsatisfiable" is not a diagnosis.
- Bounded work: resolution runs on user command with interactive latency expectations
  (NFR-PLUG-003); worst cases must be bounded by construction.
- Skill dependency chains are already bounded at depth 5 for inheritance (ADR-082); package
  graphs mirror that scale.
- Sources have priority order (ADR-080); the same package name may appear in several sources.
- One installed version per `(scope, name)` for plugins (Volume 2 natural key); skills may
  register multiple versions but activation selects one (FR-SKILL-002).

#### Alternatives considered

1. **Backtracking/SAT resolution** (pub/dart-, cargo-style) — rejected: solves a scale of
   problem the extension ecosystem does not have, at the cost of outcomes that are correct
   but hard to explain and implementation complexity that becomes a defect surface.
2. **Lockfiles as the reproducibility mechanism** — rejected as the primary mechanism:
   installations are imperative user actions, not project builds; the recorded resolution
   plan on the Package row already captures what was decided and why. A lockfile artifact MAY
   be revisited if declarative workspace manifests ever exist.
3. **Latest-wins without constraint intersection** (ignore ranges, always take newest) —
   rejected: violates the declared-requirements honesty that skills rely on (a skill tested
   against parent 1.x must not silently receive 2.0).
4. **Vendoring all dependencies into each package** — rejected: multiplies content, hides
   provenance, and breaks the one-version activation rules for shared dependencies.

#### Decision

`PackagePort.Resolve` implements **constraint-intersection resolution**:

1. Collect the requested package plus the transitive dependency closure, bounded at
   **depth 5** and **64 total nodes**; exceeding either bound fails resolution with the
   offending chain named (E-PLUG-009).
2. For each package name in the closure, intersect all version ranges that reference it. An
   empty intersection is a **conflict**: resolution fails, naming every constraint and its
   declaring package (E-PLUG-009). There is no backtracking to older roots.
3. For each name, select the **highest version satisfying the intersection** that is
   available across enabled sources, consulting sources in priority order; the first source
   (by priority) offering that version wins and is recorded in the plan. Deprecated versions
   are excluded from selection unless they are the only satisfier, in which case the plan
   carries a deprecation warning.
4. Already-installed packages satisfy constraints in place (no reinstall) when their version
   lies in the intersection; the plan marks them `satisfied`. Coexistence beyond that follows
   INV-PKG-04.
5. The output is a **resolution plan** — ordered actions with exact versions, source
   locators, expected checksums, and expected signature references — that is recorded with
   the installation and is the *only* input `Install` accepts (no resolution during
   installation).
6. Resolution never touches the network beyond reading enabled sources' indexes; with cached
   or local indexes it works offline.

#### Rationale

Intersection-plus-highest is the strongest guarantee expressible without search: it is a pure
function of (request, indexes, installed set), every failure carries its own explanation (the
constraints that emptied the intersection), and its worst case is linear in the bounded
closure size. The recorded plan separates deciding from doing, which is what makes the
Package state machine auditable — `Install` executes a reviewed artifact rather than making
choices mid-flight.

#### Positive consequences

- Identical requests against identical sources yield byte-identical plans — testable as a
  property (determinism suites).
- Conflict diagnostics name the colliding declarations directly; no solver-trace archaeology.
- Interactive latency is bounded by construction (NFR-PLUG-003 targets are enforceable).

#### Negative consequences

- Solvable-by-backtracking configurations (an older root version would satisfy everything)
  fail here; users must pin an older root explicitly — a deliberate trade of solvability for
  explainability.
- The 64-node/depth-5 bounds are policy constants that may need tuning as the ecosystem
  grows.
- First-source-wins for equal versions makes source priority order security-relevant
  configuration (documented with ADR-080).

#### Risks

- Dependency confusion: a public source offering a higher version of an internal package
  name could win selection; mitigated by source priority, per-source package-name policy
  (chapter 09 supply-chain rules), and signature policy (ADR-081) making publisher identity
  checkable.
- Ecosystem growth could outgrow the no-backtracking stance; the review conditions bound how
  long the assumption stands unexamined.

#### Reversal plan

The resolution plan schema, not the algorithm, is the stable surface (`Install` consumes
plans). Replacing the resolver with a search-based one preserves plans, records, and the
state machine; only `Resolve` internals and failure diagnostics change. Bounds are
configuration-adjacent constants revisable without contract change.

#### Review conditions

- Register/issue data shows real conflicts that backtracking would have solved occurring more
  than rarely (heuristic: a recurring support theme by v1): re-evaluate the algorithm.
- The 64-node or depth-5 bounds are hit by legitimate graphs: raise via change procedure with
  performance re-measurement.
- Declarative workspace extension manifests are introduced (a future volume decision): add
  lockfile semantics at that boundary.
