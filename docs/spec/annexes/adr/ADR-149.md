### ADR-149 — CI workflow security posture: least privilege, SHA pinning, untrusted forks

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: None (development-process decision; applies to the `andromeda` repository CI)
- Related requirements: FR-GH-012; ADR-013, ADR-004

#### Context

The `andromeda` repository's CI signs releases with the repository's OIDC identity
(ADR-013): a compromised workflow can publish validly signed malicious artifacts. The
repository is public and accepts fork pull requests from unknown contributors, including
AI-agent-driven ones. GitHub Actions workflows execute third-party actions resolved by git
reference, receive an installation token whose default permissions are configurable, and can
be triggered by events (`pull_request_target`, `workflow_run`) that combine untrusted input
with privileged context. Supply-chain attacks through mutable action tags and through
privileged-workflow injection are documented, recurring attack classes.

#### Problem

Fix the standing security rules for all repository workflows: token permissions, action
resolution, fork handling, secret exposure, and runner selection — such that the release
signing identity and repository secrets are unreachable from untrusted code.

#### Forces and constraints

- Fork PRs must still get full functional feedback (build, tests) or contribution quality
  collapses.
- Release workflows need elevated permissions (contents write, OIDC id-token) that no
  PR-triggered workflow may ever hold.
- Mutable references (`@v4`, `@main`) let an upstream compromise change executed code
  without any diff in this repository.
- Ergonomics pressure: SHA pinning and permission blocks are boilerplate that contributors
  will copy — the pattern must be uniform and tooling-updated.

#### Alternatives considered

1. **Trust-based defaults** (GitHub's default token permissions, tag-referenced actions) —
   rejected: leaves both documented attack classes open; incompatible with the precedence
   order (safety and supply-chain integrity above convenience).
2. **Org-level allowlist of verified actions with tag pins** — rejected as sufficient:
   allowlisting bounds *which* actions run but tags remain mutable; the allowlist is kept as
   a defense-in-depth layer on top of SHA pinning, not instead of it.
3. **Self-hosted runners for speed and cost** — rejected for this public repository:
   persistent runners retain state across jobs and are a lateral-movement target for fork
   PR code; ephemeral GitHub-hosted runners bound the blast radius.
4. **Least privilege + SHA pinning + fork isolation** — selected, described below.

#### Decision

1. **Token permissions:** every workflow declares a top-level `permissions:` block; the
   default is `contents: read` and nothing else. Jobs needing more declare it at job level
   with a comment justifying each grant. The absence of a permissions block fails the
   policy check.
2. **Action pinning:** every third-party action is referenced by **full commit SHA** with a
   trailing version comment; Dependabot maintains pin updates. Mutable references fail the
   policy check. First-party `actions/*` follow the same rule.
3. **Fork PRs are untrusted code:** they run only in the unprivileged `pull_request`
   context — no secrets, read-only token. Workflows combining privileged context with
   checkout of PR head code (`pull_request_target` + checkout, `workflow_run` consuming PR
   artifacts without validation) are prohibited; `pull_request_target` is permitted only
   for metadata automation (labels, comments) that never checks out or executes PR code.
   First-time contributors require maintainer approval before workflows run.
4. **Secrets and environments:** release, publish, and tap-update jobs run in GitHub
   **environments** with required reviewers and tag-pattern restrictions; OIDC `id-token:
   write` exists only in those jobs. No secret is available to any PR-triggered workflow.
5. **Runners:** GitHub-hosted, ephemeral runners only. Self-hosted runners are prohibited
   for this repository while it is public.
6. A **workflow policy check** (itself a required check) mechanically verifies rules 1–3
   for every workflow file change.

#### Rationale

The rules map one-to-one onto the real attack classes: token minimization bounds what any
compromised job can do; SHA pinning removes the mutable-reference vector; fork isolation
keeps untrusted code and privileged context disjoint by construction; environment gating
puts a human between any workflow and the signing identity; ephemeral runners remove
cross-job persistence. Each rule is mechanically checkable, so posture is a CI property,
not a review-time hope.

#### Positive consequences

- The release signing identity is unreachable from fork PRs by construction, not by review
  vigilance.
- Upstream action compromises cannot silently change executed code; every pin bump is a
  reviewable diff.
- The policy check makes the posture self-maintaining as workflows are added.

#### Negative consequences

- Pin-update PRs add review load (grouped weekly via Dependabot).
- Fork PRs cannot exercise secret-dependent integration paths; those run post-merge on
  `main` instead, delaying some feedback.
- First-contribution approval adds latency for new contributors.

#### Risks

- Reviewer rubber-stamping of pin bumps; mitigation: grouped updates with release-note
  links, and the org allowlist bounding which actions can appear at all.
- Policy-check gaps (new bypass primitives in Actions); mitigation: the check's rule set is
  versioned in-repo and reviewed at each phase gate against current published guidance.
- Post-merge-only integration paths letting a fork PR merge that breaks `main`; mitigation:
  revert-first discipline (ADR-004) and required non-secret integration coverage on PRs.

#### Reversal plan

Every rule is repository configuration or a required check; relaxing one is a reviewed
configuration change plus an amendment here. Moving CI off GitHub Actions re-opens ADR-013's
identity assumptions and requires a successor to this ADR for the new platform's
primitives.

#### Review conditions

- Any published bypass technique affecting a rule here triggers immediate review.
- If fork-PR feedback gaps measurably degrade contribution quality (rising post-merge
  failure rate), review the post-merge path split.
- At the v1 phase gate, review whether merge queues or deployment protection rules should
  join the posture.
