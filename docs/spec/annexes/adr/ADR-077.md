### ADR-077 — MCP surfaces bridge into existing runtimes under the reserved `mcp:` scheme

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: MCP Runtime, Tool Runtime, Context Manager, Prompt Engine, CLI, TUI
- Related requirements: FR-MCP-001, FR-MCP-003, FR-MCP-006, FR-TOOL-001; INV-MCPS-03 (Volume 2)

#### Context

MCP servers offer three surface kinds — tools, resources, and prompts — whose inventories are
declared by the server, change at the server's discretion (`listChanged` notifications), and
arrive from parties Andromeda does not control. Volume 2 fixes INV-MCPS-03: tools offered by
an MCP server enter the registry as ordinary Tool rows with `origin = mcp`. ADR-070 fixes the
tool identity grammar (`namespace.action`) for built-in and plugin origins, reserves the
built-in namespaces, and rejects cross-origin collisions — but explicitly leaves the MCP
naming form to this decision, having rejected origin prefixes as the *universal* primary form.

#### Problem

Decide how MCP-origin surfaces are named and where they land: do MCP tools, resources, and
prompts get their own execution and selection paths, or do they bridge into the Tool Runtime,
Context Manager, and prompt-selection surfaces — and under what identity, given that server
name sets are volatile, uncontrolled, and a typosquatting/poisoning surface?

#### Forces and constraints

- Principle 4 (uniform tool contract): a second execution path for MCP tools would bypass the
  validate → permission → sandbox → execute → record pipeline and fork observability.
- MCP server tool names are untrusted, unstable input; two servers may legitimately offer the
  same tool name, and a hostile server may deliberately mimic `fs.read` or another reserved
  name (ADR-070 threat analysis).
- ADR-070's collision-rejection rule assumes registration events are rare and author-fixable;
  MCP inventories change at runtime and rejection storms would make servers unusable.
- Users must be able to tell at a glance — in prompts, records, and the TUI — that a surface
  is MCP-provided and from which server (untrusted-content labeling, chapter 06 of Volume 6).
- Resources and prompts have no execution semantics; they are content and templates.

#### Alternatives considered

1. **A dedicated MCP execution path** (MCP tools invoked directly by the Agent Engine through
   the MCP Runtime) — rejected: violates Principle 4 and INV-MCPS-03; permission mediation,
   sandbox placement, and invocation records would need a parallel implementation that drifts.
2. **Flat `namespace.action` names for MCP tools under ADR-070 collision rejection** —
   rejected: cross-server collisions are routine, not exceptional; rejection would arbitrarily
   privilege whichever server connected first, and reserved-namespace impersonation would have
   to be policed per discovery instead of being impossible.
3. **User-assigned local aliases as the only MCP tool identity** — rejected: makes first use
   of every server a manual naming session and breaks skill/workflow portability, which needs
   a predictable name shape to declare requirements against.

#### Decision

MCP-origin surfaces bridge into the existing runtimes — no second execution or selection
system exists:

1. **Tools** register with the Tool Runtime as Tool rows (`origin = mcp`,
   `origin_ref = <MCP Server id>`) named **`mcp:<server>/<tool>`**, where `<server>` is the
   registration name and `<tool>` is the server-declared name. The `mcp:` scheme prefix is
   **reserved** for MCP-origin surfaces: no other origin may register a name beginning
   `mcp:`, and MCP-origin tools never register bare `namespace.action` names. Collisions
   across servers are impossible by construction; a single server offering two identical
   names is a protocol violation (E-MCP-006).
2. **Resources** are offered to the Context Manager as candidate Context Items with
   provenance `mcp:<server>`; content is fetched on selection under the connection's request
   budget and Volume 7 size policy.
3. **Prompts** are listed in CLI/TUI prompt selection under the `mcp:<server>/` namespace with
   argument validation against the server's declared argument list.

This complements ADR-070: the dotted grammar with collision rejection governs `builtin` and
`plugin` origins, whose name sets are stable and author-controlled; the origin-qualified
scheme governs MCP, whose name sets are volatile and server-controlled.

#### Rationale

Bridging keeps one contract, one permission model, and one observability regime for every
tool, which is the property SM-16(b) measures and the reason INV-MCPS-03 exists. The scheme
prefix solves the three MCP-specific naming problems simultaneously — collision-freedom
without arbitration, impossibility of reserved-namespace impersonation, and always-visible
provenance in every prompt, record, and screen — at the cost of verbosity that is acceptable
precisely because it is confined to the origin that needs it.

#### Positive consequences

- MCP tools inherit the full invocation pipeline, sandbox tiers, and audit chain with zero
  MCP-specific enforcement code above the bridge.
- Provenance is syntactic: any name beginning `mcp:` is third-party by inspection, in model
  context and UI alike.
- Skills and workflows can declare MCP tool requirements portably (`mcp:<server>/<tool>`),
  making their dependence on a server registration explicit rather than hidden.

#### Negative consequences

- MCP tool names are longer in model-facing declarations, spending prompt tokens.
- Renaming a server registration renames every bridged surface, invalidating skill selectors
  that referenced the old name (documented; rename is an explicit user action).
- Two registrations of the same server under different names yield distinct tool identities.

#### Risks

- Prompt-portability pressure could tempt future flat aliasing of MCP tools by default,
  reintroducing shadowing; guarded by ADR-070's rule that aliasing is always user-initiated.
- Very long server-declared tool names could exceed model tool-name limits when prefixed;
  the bridge truncation rule (reject with E-MCP-006 rather than truncate silently) keeps this
  visible.

#### Reversal plan

The bridge boundary is the reversal seam: identity is computed in one registration path in
the MCP Runtime. Moving to a different naming form (for example, per-server configurable
prefixes) changes that mapping plus recorded-name migration notes; the Tool Runtime,
pipeline, and records are name-agnostic. A reversal to a dedicated MCP execution path is
architecturally prohibited by INV-MCPS-03 and would require a Volume 2 amendment first.

#### Review conditions

- The MCP ecosystem standardizes a server-scoped tool identity or namespacing convention:
  align the scheme with it.
- Model providers impose tool-name length or character-set limits that the prefixed form
  breaches for a material fraction of reference servers (SM-15 interop set): revisit the
  identity mapping.
- User feedback shows the server-qualified names are a routine friction point in skills:
  evaluate portable indirection (declared requirements resolving through registration
  metadata) via the change procedure.
