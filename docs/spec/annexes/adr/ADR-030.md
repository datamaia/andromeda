### ADR-030 — Architectural style: layered hexagonal (ports and adapters)

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: all 37 components; composition root; Extension SDK
- Related requirements: FR-ARCH-001, FR-ARCH-002, FR-ARCH-003, PRD-002, PRD-007, PRD-011

#### Context

Andromeda is a single Go binary (ADR-001, ADR-003) whose product principles are structural
claims: no provider-specific logic outside adapters (Principle 1), core function without
network (Principle 3), one execution model for everything (PRD-001), OS specifics
encapsulated (PRD-011), and a platform whose extension surfaces are public contracts
(PRD-007). Eleven specification volumes are authored in parallel against Volume 3's
boundaries, and a large share of implementation is performed by AI agents that need
mechanically checkable rules rather than tacit conventions.

#### Problem

Choose the codebase-wide architectural style that determines how components depend on each
other, where infrastructure meets domain logic, and what "boundary" means concretely enough
to be enforced by tooling and referenced by ten later volumes.

#### Forces and constraints

- Principles 1–3 and PRD-011 are dependency-direction claims; the style must make them
  build-time properties, not review-time hopes.
- Parallel volume authoring needs stable, nameable seams (port interfaces) that later volumes
  elaborate without renegotiating structure.
- Go has no compiler-enforced architecture; enforcement must come from package layout plus
  lint (ADR-018) — the style must be expressible in those terms.
- Testing targets (SM-14's stricter core bar; contract suites in Volume 13) require engines
  testable without infrastructure.
- The team-scale reality: one codebase, many contributors (human and AI), no room for a style
  that depends on architectural folklore.

#### Alternatives considered

1. **Layered hexagonal — ports and adapters with explicit layers (selected).** Pure domain
   core; interface ports; application engines; infrastructure adapters; drivers; one
   composition root. Dependency rule enforceable with import checks.
2. **Conventional layered architecture (UI → service → data).** Familiar, but its
   dependency direction points *at* infrastructure: services import data-access concretions,
   which is precisely how provider- and platform-specific code metastasizes. It answers
   "where does code go" but not "what may know about what".
3. **Microkernel/plugin-first (everything an extension around a minimal kernel).** Matches
   the extensibility story superficially, but pushes first-party engines behind the same
   heavyweight indirection as third-party code, taxing the hot path (per-call process/RPC
   boundaries per ADR-009 economics) and exploding the protocol surface that must be
   versioned. Rejected as the *global* style; retained exactly where it belongs — the
   extension boundary (plugins, MCP).
4. **Free-form idiomatic Go ("package by feature, no formal layers").** Lowest ceremony and
   fine for small tools; at 37 components with parallel authors it provides no checkable
   answer to the dependency questions the principles pose. Rejected.

#### Decision

Andromeda adopts a **layered hexagonal architecture**: L0 Core Domain (pure), L1 Ports (18
frozen interfaces, Volume 3 chapter 02), L2 Engines (application logic; import L0/L1 only),
L3 Adapters (implement ports; the only code touching external systems; PAL among them), L4
Drivers (CLI/TUI/IPC; import L2 downward; imported by nothing), L5 composition root (the
only place engines and adapters meet). The dependency matrix in Volume 3 chapter 01 is the
normative statement; FR-ARCH-001 binds it; ADR-033 enforces it mechanically. The Extension
SDK mirrors L1 contracts without importing them (ADR-031).

#### Rationale

This is the only considered style whose core claims coincide with the product's: "vendor
agnostic" *is* "engines cannot import adapters"; "local-first" *is* "no engine depends on a
network adapter"; "portable" *is* "OS code only in one adapter family". Ports give the
parallel volumes their stable seams and give contract tests their natural fixtures. The
layering adds to classic hexagonal exactly one thing Go needs: an explicit, lintable layer
assignment per package, so the style survives contributors who never read the book.

#### Positive consequences

- Product principles become CI-checkable structure (FR-ARCH-001, NFR-ARCH-001).
- Engines test without infrastructure; adapters test against contract suites — the Volume 13
  pyramid falls out of the structure.
- Later volumes reference ports by name and never renegotiate boundaries.
- Adapter substitution (new providers, go-git fast paths per ADR-025, future platform
  backends) is localized by construction.

#### Negative consequences

- Indirection tax: every infrastructure interaction crosses an interface, and small features
  touch more files (port, adapter, wiring) than in free-form Go.
- Contract types in L1 must be designed early and carefully; sloppy port types propagate
  corpus-wide.
- The composition root grows large and must itself be kept organized.

#### Risks

- Layer erosion under delivery pressure — RISK-ARCH-001; mitigated by ADR-033 enforcement
  from the first commit.
- Anemic-core drift (logic leaking from L2 into adapters where it escapes the stricter test
  bar); mitigated by review conventions and the Volume 13 coverage split.
- Port-shape mistakes discovered late force change-procedure amendments (RISK-ARCH-002).

#### Reversal plan

Relaxations are cheap (delete rules); the expensive direction is the one this ADR prevents.
A partial reversal — say, collapsing L1 into L2 for internal-only ports — is a mechanical
refactor (interface moves) plus a matrix amendment, feasible per component. Full reversal to
an unlayered style would amount to deleting enforcement and letting entropy proceed; it needs
no plan beyond a superseding ADR.

#### Review conditions

- Two consecutive phase gates report the indirection tax materially slowing delivery
  (measured by Volume 13 review-cycle metrics) — revisit L1 granularity.
- Go ships language-level architectural enforcement that subsumes the manifest approach.
- The port set proves wrong-shaped during Volume 4–14 authoring (multiple change-procedure
  amendments to chapter 02) — re-examine the port decomposition before implementation scales.
