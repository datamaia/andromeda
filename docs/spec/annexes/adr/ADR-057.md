### ADR-057 — Unified stream event model normalized inside adapters

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Provider Layer, Agent Engine, TUI, CLI
- Related requirements: FR-PROV-020, FR-PROV-021, FR-PROV-022

#### Context

Providers stream over divergent wire protocols and event grammars (server-sent events,
chunked JSON lines, provider-specific event taxonomies with different delta granularities,
usage placement, and termination signals). `ChatStream` is frozen as `Stream[ChatEvent]`
with a tagged union (content delta, tool-call delta, usage, terminal) ending in exactly one
terminal event (Volume 3, chapter 02). Consumers — the agent loop, TUI rendering, CLI
structured output — need one grammar.

#### Problem

At which layer are wire streams normalized, and how rich is the shared event union — raw
passthrough with consumer-side interpretation, a rich provider-flavored union, or a minimal
normalized union produced inside each adapter?

#### Forces and constraints

- Principle 1: no provider-specific logic outside adapters; consumers must not parse wire
  formats.
- SM-08 bounds Andromeda-added streaming overhead (≤ 50 ms p95 per chunk).
- Tool-call argument deltas must assemble into complete JSON documents reliably.
- Reasoning artifacts may only surface when officially provided (Principle 7).
- The union is public contract surface (SM-20): every member added is a compatibility
  commitment.

#### Alternatives considered

1. **Raw passthrough with consumer-side adapters** — rejected: moves wire knowledge into
   every consumer, violating Principle 1 and multiplying parsing bugs.
2. **Rich union mirroring the most expressive provider** — rejected: encodes one vendor's
   taxonomy as the product contract (vendor shape leakage) and forces other adapters to
   fake events they do not have.
3. **Minimal normalized union produced inside adapters** — chosen.

#### Decision

Adapters translate every wire stream into the four-member `ChatEvent` union with the
ordering rules of Volume 5, chapter 03: index-ordered `content_delta` (part kinds `text`,
`reasoning_summary`), index/ID-keyed `tool_call_delta`, optional monotonic `usage_update`,
and exactly one `terminal` carrying a closed finish-reason set (`stop`, `length`,
`tool_calls`, `content_filter`, `cancelled`, `error`) with the raw reason preserved in safe
metadata. Translation is per-chunk with no whole-response buffering; transport failures
after first delivery terminate with E-PROV-009.

#### Rationale

The minimal union covers everything consumers act on; raw-reason preservation keeps
diagnostics lossless; per-chunk translation keeps overhead inside SM-08; confining wire
grammar to adapters is Principle 1 applied to streaming.

#### Positive consequences

- One rendering and one agent-loop path for all providers.
- Conformance can verify ordering, single-terminal, and incrementality mechanically.
- New adapters implement a translation, not a product surface.

#### Negative consequences

- Provider-specific stream events with no union member are reduced to safe metadata —
  intentionally invisible to consumers.
- Every future union member is a public-contract commitment (SM-20 discipline applies).

#### Risks

- Under-modeling: a capability class (e.g., richer official reasoning surfaces) may need a
  new member; mitigation: additive union evolution within a major line.

#### Reversal plan

The union is the frozen port's event type; enriching it is additive. A move away from
adapter-side normalization would be a major contract change under FR-ARCH-003's change
procedure — the reversal is deliberately expensive because the boundary is the product.

#### Review conditions

- Review when a second official reasoning-artifact shape or a new modality (streamed audio
  output) cannot be expressed as `content_delta` part kinds.
- Review against SM-08 measurements at each phase gate.
