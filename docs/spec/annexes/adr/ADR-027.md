### ADR-027 — Entity identifiers: ULIDs as primary IDs

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Core Domain, Persistence Layer, Event Bus, Observability, Extension SDK
- Related requirements: Volume 1 product objectives (PRD-006, PRD-010); Volume 2 identifier strategy (chapter 01); formal requirement links are added at consolidation

#### Context

Volume 2 defines 43 domain entities, most of them persisted as rows in the SQLite databases of
ADR-007/ADR-028. Every entity needs a primary identifier that works in four places at once:
database keys, cross-aggregate references, correlation fields in events/traces/audit records,
and human-facing surfaces (CLI output, log lines, bug reports). Identifiers are minted by
multiple uncoordinated processes: the Runtime, concurrent CLI invocations, and potentially
several machines whose exports are later compared or merged. Workspace databases are created
offline (Local First, PRD-003), so no central allocation service can exist. Write-heavy
record tables (messages, events, audit records) benefit from insertion-ordered keys in
SQLite's B-tree storage.

#### Problem

Choose one identifier scheme for all persisted domain entities that is collision-free without
coordination, sortable enough to keep writes append-mostly and support coarse time filtering,
compact and unambiguous in terminals, and implementable in Go (ADR-001) without CGO.

#### Forces and constraints

- No coordination: IDs must be mintable offline, concurrently, in multiple processes.
- Ordering: keys correlated with creation time keep B-tree inserts localized and make ID
  ranges meaningful; but wall clocks skew, so ordering cannot be trusted as semantics.
- Ergonomics: identifiers appear in CLI output, logs, and file names; length, case rules, and
  ambiguous characters matter.
- Privacy/security: identifiers leak whatever they encode; they must not act as capability
  tokens (Volume 9 owns authorization).
- Uniformity: one scheme for all entities keeps the Extension SDK, event envelope, and
  tooling simple (no per-entity key shapes).
- Integer autoincrement keys conflict with merge/export scenarios and leak row counts.

#### Alternatives considered

| Criterion | ULID (selected) | UUIDv4 | UUIDv7 (RFC 9562) | Integer autoincrement | KSUID | Snowflake-style |
|---|---|---|---|---|---|---|
| Coordination-free | Yes | Yes | Yes | No (per-DB sequence; collides on merge) | Yes | Needs node IDs |
| Time-ordered | Yes (ms) | No | Yes (ms) | Insertion-ordered only locally | Yes (s precision) | Yes |
| Canonical text length | 26 | 36 | 36 | variable | 27 | variable |
| Terminal ergonomics | Crockford base32, no ambiguous chars | Hex + dashes | Hex + dashes | Short but meaningless across DBs | Base62, mixed case | Decimal |
| Randomness per unit time | 80 bits/ms | 122 bits | 74 bits/ms | None | 128 bits/s | Sequence bits |
| Monotonic option within ms | Yes (monotonic source) | n/a | Optional per RFC | n/a | No (1 s granularity) | Yes |

- **UUIDv4** is coordination-free but unordered: random keys fragment B-tree inserts on
  write-heavy tables and carry zero time information for range filtering.
- **UUIDv7** is the closest contender — standardized, time-ordered, and widely implemented.
  It loses on text form: 36 characters of hex-and-dash versus 26 characters of
  case-insensitive Crockford base32 that excludes the ambiguous characters I, L, O, U.
  For a product whose primary surface is a terminal, the shorter, cleaner form is worth the
  divergence from the IETF format; the binary layouts are equivalent in capability.
- **Integer autoincrement** fails immediately: IDs collide across databases, leak counts,
  and cannot be minted before the row insert.
- **KSUID** has second-granularity timestamps (too coarse for intra-run ordering hints) and a
  27-char base62 form with case sensitivity.
- **Snowflake-style** schemes need coordinated node identifiers — a non-starter offline.

#### Decision

Every persisted domain entity row uses a **ULID** primary key. Binding rules (normative
restatement in Volume 2, chapter 01):

1. Canonical form: 26-character uppercase Crockford base32; stored, serialized, and displayed
   in exactly this form; parsers accept lowercase and normalize.
2. One monotonic ULID source per process: IDs minted within the same millisecond by one
   process are strictly increasing.
3. Explicit `sequence_no` (or rowid) fields — never ULID sort order — are the ordering
   authority wherever ordering is semantically load-bearing.
4. ULIDs are not secrets and not authorization tokens; they encode only creation time plus
   randomness, and no component may attach further semantics (no type prefixes).
5. Natural keys (names, slugs, versions) are unique constraints, never primary keys; external
   identifiers (commit hashes, provider model names) are never primary keys.
6. The generator implementation is chosen at implementation time against the ULID
   specification; it is not fixed by this ADR beyond being pure Go (ADR-001) and satisfying
   rules 1–2.

#### Rationale

ULID is the only candidate that simultaneously satisfies coordination-free minting, useful
time-ordering, and terminal-grade ergonomics. The 80 bits of per-millisecond randomness make
collision probability negligible for Andromeda's write rates even across merged exports. The
scheme is trivially representable as fixed-width `TEXT` in SQLite with a length check, keeps
hot tables append-mostly, and gives every correlation field in events, traces, and audit
records one uniform shape — which is what makes the observability chain (PRD-006) cheap to
implement and to read.

#### Positive consequences

- One identifier shape everywhere: model, storage, events, CLI, SDK.
- Offline, concurrent, multi-process minting with no allocation infrastructure.
- Append-mostly B-tree behavior on write-heavy tables; ID ranges support coarse time filters.
- Fixed-width keys simplify schema checks (`length(id) = 26`) and log parsing.

#### Negative consequences

- IDs reveal creation time (accepted: creation time is already recorded in `created_at` and
  is not sensitive under Volume 9's classification).
- Divergence from the IETF UUID format family; interoperability with UUID-expecting external
  systems requires representation at the boundary (exports remain strings, so this is
  cosmetic).
- Clock regressions can produce IDs that sort before earlier ones across process restarts —
  which is why rule 3 forbids semantic reliance on ID order.

#### Risks

- **Clock skew misleading humans**: users may assume ID order equals event order across
  machines. Detection: documentation and tooling always sort by `sequence_no`/timestamps.
  Mitigation: rule 3; CLI sorts by authoritative fields.
- **Randomness source weakness** on exotic platforms could raise collision odds. Detection:
  generator conformance tests (Volume 13). Mitigation: require crypto-grade randomness in the
  generator contract.

#### Reversal plan

Identifiers are opaque strings to every consumer, so a future scheme change (e.g., to UUIDv7)
is a data migration that rewrites key columns and foreign keys plus a compatibility window
accepting both forms in inputs. Because IDs never encode semantics (rule 4), no logic changes
— only formats. The cost is a heavy but mechanical forward-only migration (ADR-029), which is
why this reversal is documented and not expected.

#### Review conditions

- SQLite key-size or insert-locality profiling (Volume 12 benchmarks) attributes a missed
  target to TEXT ULID keys — revisit binary (BLOB) storage of the same ULIDs.
- An adopted external interoperability requirement mandates IETF UUID formats at a boundary
  that string conversion cannot satisfy.
- The ULID specification is superseded by an IETF standard with equivalent text ergonomics.
