### ADR-130 — Configuration precedence: invocation over environment over runtime overrides over scope layers, with scope-bound profiles

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Configuration Manager, CLI, TUI, Workspace Engine
- Related requirements: FR-CFG-001, FR-CFG-003, FR-CFG-005

#### Context

Volume 10 owns configuration precedence (single-home matrix). Andromeda has three scope
chains (global, workspace, project), each with a file and an optional named Configuration
Profile (Volume 2 entity, scopes `global`/`workspace`/`project`), plus three invocation-time
surfaces: CLI flags, `ANDROMEDA_*` environment variables, and session-scoped runtime
overrides (TUI/IPC). Every key-minting volume defers to one exact order.

#### Problem

Fix the exact, total precedence order across ten layers, including two contested placements:
where profiles sit relative to files, and where runtime overrides sit relative to the
invocation environment.

#### Forces and constraints

- Nearer scope must beat farther scope (project > workspace > global) — the universal
  expectation of layered configuration.
- A profile is a *deliberate selection*; selecting one and having it change nothing where
  the scope file disagrees would make profiles useless.
- CI must be able to pin behavior for a whole process via environment and argv; a
  mid-session mechanism silently unpinning CI settings would break automation contracts
  (PRD-009).
- Profiles must not become an inheritance system (INV-CFGP-04).

#### Alternatives considered

1. **Single profile layer between workspace and global** (one selected profile total).
   Rejected: it cannot express "this workspace's profile refines this workspace" — a
   global profile would be overridden by every workspace file, while a workspace profile
   could not override its own workspace file; scope-bound slots express both.
2. **Profile below its scope's file.** Rejected: values set by the file would be
   unoverridable by any profile, reducing profiles to defaults-of-defaults.
3. **Runtime overrides above environment/flags.** Rejected: flags and environment are the
   invocation contract, fixed at process start and visible to whoever launched the
   process; letting a TUI action or IPC call outrank them makes CI and wrapper behavior
   unauditable from the invocation record.

#### Decision

Highest first: CLI flag > environment variable > runtime override > project profile >
project file > workspace profile > workspace file > global profile > global file >
built-in defaults. Merge granularity is the key; arrays replace wholesale; profiles bind
at their scope's slot only. The order is a public contract under SM-20.

#### Rationale

The order preserves the two invariants users can reason with: "nearer wins" across scopes,
and "what I selected or typed most explicitly wins" within a scope (flag > env > override;
profile > file). Each contested placement follows from a stated force rather than
convention: profiles above their files because selection is deliberate; invocation
surfaces above runtime overrides because the invocation record must stay authoritative.

#### Positive consequences

- Every volume and command surface documents precedence by one reference.
- Profiles are useful at every scope without inheritance semantics.
- CI/wrapper behavior is fully determined by argv plus environment.

#### Negative consequences

- Ten layers is a lot to hold in mind; source attribution (FR-CFG-001) is mandatory
  compensation.
- An environment variable can shadow a deliberate mid-session override; the override event
  carries `effective = false` to make the shadowing visible.

#### Risks

- Users mispredict layer interactions (RISK-CFG-002); mitigated by attribution surfaces
  and the layer-matrix acceptance tests.
- A future scope (e.g. organization policy) will need a slot; the order admits insertion
  only via a superseding ADR and a major release.

#### Reversal plan

The order is data to the resolver (an ordered layer list); reordering is a small code
change but a breaking contract change — permitted only in a major release with migration
notes, per SM-20.

#### Review conditions

- An enterprise/managed-policy layer is requested (would sit above user layers and below
  invocation surfaces — requires a superseding ADR).
- Attribution telemetry or support load shows systematic misprediction concentrated on one
  placement decided here.
