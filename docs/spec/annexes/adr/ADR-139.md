### ADR-139 — Local traces unsampled with bounded retention; metric snapshots persisted to SQLite

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Telemetry, Observability, Persistence Layer
- Related requirements: FR-OBS-007, FR-OBS-008; ADR-011, ADR-007, ADR-028; Volume 2 Trace/Metric entities

#### Context

ADR-011 fixed OpenTelemetry instrumentation with local-first sinks (files and/or SQLite)
and named sampling knobs as a mitigation held in reserve. Volume 2 fixed the persisted
shapes (`traces`/`trace_spans`, definitional Metric registry, `metric_points`). Still open:
whether local traces are sampled, when spans and metric samples are written, and how local
signal volume is bounded.

#### Problem

Decide the local recording policy for traces and metrics such that SM-12 (replay
completeness) and SM-13 (100% side-effect attribution) hold, offline cost/performance views
work, and long sessions do not degrade the host (RISK-OBS-001).

#### Forces and constraints

- SM-13 tolerates zero orphan side effects; a sampled-out trace would orphan every side
  effect in the unsampled run — sampling and the traceability target are incompatible at
  the run level.
- Agent-run span volume is human-scale (tens to thousands of spans per run), unlike
  service-mesh tracing where sampling is existential.
- Writes must not sit on the hot path (TelemetryPort is non-blocking by contract).
- Disk is finite; retention must bound the store (ADR-011 negative consequence).

#### Alternatives considered

1. **Head/tail sampling locally** — standard for high-volume services; here it would break
   SM-12/SM-13 and save little, since volume is driven by run count, not request fan-out.
   Rejected locally (export-side reduction under consent remains available).
2. **File-based span sink (OTLP-file/JSONL)** — simple, but unqueryable without tooling;
   the CLI/TUI inspection views (Principle 7) need indexed queries by run, and Volume 2
   already homes traces in the workspace database. Rejected.
3. **In-memory only, no persistence** — trace views die with the process, breaking
   post-hoc inspection and SM-11-adjacent recovery analysis. Rejected.
4. **Persist every metric sample point** — unbounded growth for negligible diagnostic
   gain over interval snapshots. Rejected in favor of 60 s aggregated snapshots.

#### Decision

1. **Local traces are unsampled**: every run records a complete trace into
   `traces`/`trace_spans` (workspace database), spans written at span end in batches, with
   `Flush` bounded by the shutdown budget.
2. **Metric samples aggregate in memory** (OTel views); when
   `storage.metrics.local_persistence = true` (default), 60-second snapshots persist to
   `metric_points` (workspace/global per scope).
3. **Retention bounds both**: `storage.traces.retention_days` = 30 /
   `storage.traces.max_size_mb` = 512, whole-trace pruning only;
   `storage.metrics.retention_days` = 30.
4. Span attributes cap at 4 KiB per attribute; sampling knobs remain in the design as a
   reserve mitigation, off by default, and their activation is a recorded configuration
   act.

#### Rationale

Completeness is what the product sells (Principles 7 and 9): "every run inspectable, every
side effect attributable" is only true unsampled. The volume argument that justifies
sampling elsewhere does not hold at Andromeda's scale, while the retention bound converts
the residual growth concern into configuration. Snapshots give offline historical views at
one row per definition per minute — three orders of magnitude below raw samples.

#### Positive consequences

- SM-12/SM-13 are structurally achievable; validators need no sampling exceptions.
- Cost and performance history is queryable offline for the retention window.
- One storage engine (ADR-007) for all observability queries.

#### Negative consequences

- Trace storage grows with usage until retention binds (bounded at ~512 MiB per workspace).
- Batched span writes add workspace-database write load (off the hot transaction, budgeted
  by Volume 12).
- 60 s snapshot resolution limits historical metric granularity.

#### Risks

- Pathological runs (millions of spans) exceeding practical bounds: mitigated by span-count
  guards surfaced as diagnostics and by whole-trace retention.
- Metric snapshot writes colliding with heavy run writes: mitigated by WAL mode (ADR-007)
  and off-peak batching.

#### Reversal plan

Sampling can be enabled per the reserve knobs without schema change. The sink reverses per
ADR-011's plan (file ⇄ SQLite ⇄ OTLP are exporter configurations); the persisted model is
Volume 2's local shape, independent of any exporter.

#### Review conditions

- Volume 12 benchmarks attribute a target miss to span persistence at default settings.
- Retention-bound churn (traces pruned before users inspect them) appears in real usage —
  revisit defaults.
- OTel Go SDK changes to view/aggregation APIs materially affect the snapshot design.
