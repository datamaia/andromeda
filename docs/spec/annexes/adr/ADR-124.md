### ADR-124 — Redaction strategy: registry exact-match plus structural and heuristic layers at sink boundaries

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Secret Store, Logging, Event Bus, Observability, Memory Manager, Tool Runtime, Sandbox Engine
- Related requirements: FR-SEC-109, FR-SEC-102, NFR-SEC-004

#### Context

Secret material transits Andromeda legitimately: resolved for provider requests (Volume 5),
injected into granted child environments (Volume 9 chapter 06), echoed by tools that read
files containing it. Meanwhile every subsystem persists or emits content — logs (ADR-011),
events (ADR-012), error envelopes (ADR-016 safe-context-data), Tool Results, memory records
(Volume 7), traces, exports. INV-CRED-01 and INV-AUD-05 demand that no sink ever carries
material.

#### Problem

How does a system with dozens of sinks guarantee zero secret leakage — without relying on
every call site's author remembering to redact, and without pattern-guessing every provider's
token format (which the assumptions policy forbids inventing)?

#### Forces and constraints

- Definition-time redaction (ADR-016) requires one testable mechanism, not per-callsite
  judgment.
- The product *knows* its secrets: everything it resolved in-process came through
  `SecretStorePort` — exact knowledge beats format guessing.
- Provider token formats are external facts that change; a format-database approach would
  either invent unverified patterns or rot.
- Streaming outputs split content across chunk boundaries.
- False positives that mangle legitimate content (short common strings) are a real cost.
- Fail-open redaction (emit raw on failure) is unacceptable under the precedence order.

#### Alternatives considered

1. **Pattern-matching only** (regex catalog of known token shapes) — rejected as primary:
   requires maintaining unverifiable external format facts; misses anything nonstandard —
   precisely the secrets users paste by hand.
2. **Per-callsite manual redaction** — rejected: unauditable at scale; one forgotten
   callsite defeats the guarantee; contradicts ADR-016's definition-time discipline.
3. **Taint tracking** (propagate sensitivity through every value) — rejected: no practical,
   maintained taint system for Go fits ADR-001/ADR-007's dependency posture; the runtime
   cost and invasiveness are disproportionate.
4. **Registry exact-match + structural + heuristics, applied at sink boundaries** (chosen).

#### Decision

Redaction composes three layers applied by every sink before content leaves the producing
component: (1) a **registry** of all secret material resolved in the current process (plus
base64 and URL encodings), scrubbing matches ≥ 8 characters to `[REDACTED:<fingerprint>]`;
(2) **structural** redaction of schema-declared sensitive fields and authorization-class
HTTP headers by name; (3) additive **heuristics** for generic high-risk shapes (bearer
values, `token|secret|password|passwd|api_key|apikey|private_key` key-value pairs, PEM
private-key blocks) plus user-configured patterns (`security.redaction_patterns`).
Streaming sinks scrub with a 256-byte overlap window. Redaction failure withholds or marks
the payload — never raw emission (fail-closed, ADR-125).

#### Rationale

The registry turns the leakage problem from "guess what secrets look like" into "remember
what we ourselves resolved" — exact, format-agnostic, and testable with planted markers
(NFR-SEC-004's method). Structural redaction covers material the process never resolved
(user-typed content in known-sensitive fields); heuristics are a safety net that can only
add redaction, so their false-negative rate never weakens the stronger layers. Sink-boundary
placement makes the mechanism enumerable and CI-enforceable: new sinks register with the
leak-hunt suite.

#### Positive consequences

- Zero-leak guarantee is testable end-to-end with planted fixtures across all sinks.
- No dependency on provider token format knowledge; hand-pasted and nonstandard secrets are
  covered the moment they are resolved.
- One replacement grammar (`[REDACTED:<fingerprint>]`) keeps redacted logs correlatable
  without disclosure.

#### Negative consequences

- Secrets never resolved in-process and not in a declared field or heuristic shape can pass
  (e.g., a token inside a random file a tool prints, never stored in Andromeda) — the
  residual risk is recorded in the threat model (log leakage).
- Registry scrubbing adds a multi-pattern search on every sink write; bounded but nonzero
  (Volume 10 logging budgets absorb it).
- The ≥ 8-character floor leaves hypothetical very short secrets unmatched by the registry
  layer; recorded as an assumption (official flows do not issue such material).

#### Risks

- Registry lifetime bugs (material scrubbed only while registered); mitigated by
  registering at resolution and retaining fingerprints for the process lifetime.
- Overlap-window gaps for pathological chunk sizes; mitigated by property tests over split
  positions.

#### Reversal plan

Layers are independent: any layer can be strengthened or replaced behind the single
redaction function each sink calls. Moving to taint tracking (if the ecosystem matures)
would wrap, not replace, the registry.

#### Review conditions

- If NFR-SEC-004 runs ever find a leak class the three layers miss, add the class as a
  structural rule and review the layer design.
- Review the ≥ 8-character floor if any supported official flow issues shorter material.
- Review heuristic patterns annually against observed false-positive reports.
