### ADR-022 — Directory layout: XDG-style paths via adrg/xdg, Apple-native mapping on macOS, project-local `.andromeda/`

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Platform Abstraction Layer (PAL), Configuration Manager, Workspace Engine, Persistence Layer
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation.

#### Context

Andromeda persists global configuration (`andromeda.toml`), per-workspace state (`.andromeda/`),
caches, logs, and SQLite databases (ADR-007). Placement conventions differ per platform: Linux
follows the XDG Base Directory specification; macOS convention places application data under
`~/Library/Application Support` and caches under `~/Library/Caches`, while many terminal-first
tools on macOS still honor XDG variables when users set them. Verified 2026-07-11:
`github.com/adrg/xdg` (MIT) is maintained (commits 2026-06, v0.5.3), and on macOS it maps to
the two `~/Library` locations above by default while respecting explicitly set `XDG_*`
overrides — making the macOS path policy a decision this ADR must state explicitly.

#### Problem

Where exactly do Andromeda's configuration, data, state, and cache files live on each supported
platform, which library resolves those locations, and what happens when resolution fails or
users override it — so that every component asks one authority instead of hardcoding paths?

#### Forces and constraints

- Users and backup/sync tooling expect platform conventions: XDG dirs on Linux, `~/Library` on
  macOS.
- Terminal-centric users who deliberately set `XDG_*` variables expect them honored on any OS.
- Project-local state must travel with the workspace (glossary: the Workspace includes its
  `.andromeda/` state) and be trivially ignorable by version control.
- Config, data, and cache have different lifecycles (user-edited vs machine-managed vs
  disposable) and MUST NOT share a directory.
- Every path decision must flow through the PAL so components never compute paths themselves.
- Resolution can fail (unusual environments, stripped-down containers without HOME); behavior
  in that case must be defined, not accidental.

#### Alternatives considered

1. **Hardcoded dotfiles: `~/.andromeda` for everything** — rejected as the primary layout: it
   mixes config, data, and cache lifecycles in one hidden directory, ignores both the XDG
   specification on Linux and Apple conventions on macOS, and defeats OS-level cache cleanup
   and backup exclusion. It is, however, retained as the **defined fallback** when XDG
   resolution fails (see Decision).
2. **Strict XDG paths on macOS too** (`~/.config`, `~/.local/share`) — rejected as the default:
   it violates the platform's native convention and surprises macOS tooling; users who want it
   can still get it by setting `XDG_*` explicitly.
3. **Hand-rolled path resolution in the PAL** — rejected: reimplements a well-tested, verified-
   maintained library including its platform quirks, with no behavioral difference desired.

#### Decision

- Andromeda resolves base directories with **`github.com/adrg/xdg`** (verified maintained,
  v0.5.3) through the PAL; no component outside the PAL computes platform paths.
- **macOS policy, stated explicitly**: Andromeda follows the Apple-native mapping by default —
  data under `~/Library/Application Support`, caches under `~/Library/Caches` — but honors
  explicitly set `XDG_*` environment variables, in both cases exactly as adrg/xdg resolves
  them.
- **Global configuration** resolves to the platform config directory plus `andromeda.toml`;
  data (including Persistence Layer databases), state, and caches resolve to the corresponding
  platform data/state/cache directories.
- **Project-local state lives in `.andromeda/`** at the workspace root (per the glossary
  definition of Workspace); it holds workspace-scoped state and settings, and Volume 10 defines
  its precedence relative to global configuration.
- **Fallback**: if XDG resolution fails (for example, no resolvable home), Andromeda falls back
  to `~/.andromeda` (or an explicitly configured root), emits a diagnostic naming the fallback,
  and continues; the fallback layout is documented, not accidental.

#### Rationale

adrg/xdg gives verified, maintained, cross-platform resolution with exactly the macOS behavior
this ADR wants: native placement for the default user, override capability for the XDG-minded
user — resolving the ambiguity the verification file flagged. Separating config, data, and
cache by lifecycle enables OS cache cleanup, sane backups, and clean uninstalls. Routing all
resolution through the PAL keeps the future Windows phase a PAL-plus-library concern (adrg/xdg
already abstracts known folders) rather than a codebase-wide audit.

#### Positive consequences

- Predictable, platform-conventional locations; one documented answer to "where are my files".
- `XDG_*` overrides work uniformly, including on macOS, without Andromeda-specific mechanisms.
- Cache/data separation enables safe cache deletion and correct backup behavior.
- The Windows phase inherits a working resolution strategy.

#### Negative consequences

- Paths differ across platforms and documentation must present per-platform tables.
- The fallback path (`~/.andromeda`) creates a second layout support burden, though a small one.
- A third-party library sits in the startup path; its resolution semantics become part of
  Andromeda's observable behavior.

#### Risks

- adrg/xdg behavior changes (mapping or override semantics) could silently move files between
  versions; mitigation: pin the version, cover resolution with golden tests per platform, and
  treat mapping changes as migrations.
- Users with partially set `XDG_*` variables may get mixed layouts; mitigation: `andromeda`
  diagnostics display every resolved path and its source.
- Fallback overuse in containers could hide misconfiguration; mitigation: the fallback always
  emits a diagnostic.

#### Reversal plan

All paths flow through one PAL interface. Replacing adrg/xdg or changing the mapping policy
means changing that interface's implementation plus a documented migration (move files, leave a
marker). Project-local `.andromeda/` and the `andromeda.toml` name are product identity (Volume
0) and are not affected by any such reversal.

#### Review conditions

- Before the Windows phase: validate adrg/xdg known-folder mapping against the Volume 3
  platform matrix.
- If adrg/xdg maintenance stops for 12 months, re-evaluate (fork or replace behind the PAL).
- Revisit the fallback policy if diagnostics show it triggering in mainstream environments.
