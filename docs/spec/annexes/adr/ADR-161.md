### ADR-161 — Benchmark tooling and regression gating policy

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: benchmark harness, CI pipelines (Volume 11), release process (Volume 14)
- Related requirements: Volume 12 FR-PERF-005; NFR-PERF-001 through NFR-PERF-028

#### Context

Volume 1's metric governance requires performance metrics to be measured from MVP and gated
from their binding phase. ADR-017 fixes the testing stack (stdlib testing, testify, rapid,
native fuzzing) but says nothing about benchmarks, baselines, or how a regression blocks a
release. Without a declared gate, performance erodes one "small" regression at a time.

#### Problem

Which tooling produces Andromeda's benchmark measurements, how are baselines maintained,
and what exactly fails a release?

#### Forces and constraints

- Go's native benchmark facility (`go test -bench`) is idiomatic and zero-dependency for
  micro-benchmarks (ADR-001, ADR-017), but end-to-end metrics (cold start, TUI startup)
  require driving the real binary from outside.
- Statistical comparison needs a recognized method, not ad-hoc percentage arithmetic on
  single runs.
- CI environments are noisy; a gate that flakes gets disabled socially even if it stays on
  paper.
- Gate policy must distinguish "worse than promised" (absolute) from "worse than we were"
  (relative drift within the promise).

#### Alternatives considered

1. **Third-party continuous-benchmarking services** — rejected for the gate: adds an
   external service dependency to release integrity; MAY be adopted later for trend
   dashboards without owning the gate.
2. **Absolute thresholds only** — rejected: headroom between target and threshold would be
   consumed silently over years; drift detection is the point of baselines.
3. **Relative gates only** — rejected: a slowly re-baselined suite can ratchet past the
   product promise; the NFR thresholds must remain the hard floor.
4. **Statistical significance testing as the sole gate criterion** — rejected as sole
   criterion: significance without magnitude blocks trivial-but-real 1% changes;
   magnitude bands with benchstat's comparison output are more legible to contributors.

#### Decision

- Three-layer suite (Volume 12, chapter 03): Go micro-benchmarks; a scripted operation
  harness driving the real binary; scenario replays (soak, load, crash-injection).
- Micro-benchmark comparison uses **benchstat** from the official Go performance tooling;
  operation and scenario layers use the harness's own percentile reports over recorded raw
  samples.
- **Rolling baseline** = median of the last 5 mainline nightly results per benchmark, per
  runner class, per reference-machine designation.
- **Gate policy:** at a release gate, fail on (a) any gated metric breaching its NFR
  minimum threshold, or (b) p95 regression > 10% versus the rolling baseline; warn and open
  a finding at > 5%. Two consecutive failing runs confirm; noisy benchmarks (CV > 10%) are
  quarantined from relative gating and tracked as defects.
- Gating runs execute only on calibrated environments per ADR-160; whether GitHub-hosted
  runners can hold calibration or a dedicated runner is required is PENDING VALIDATION
  (open question in the Volume 12 register).

#### Rationale

Native tooling keeps benchmarks as ordinary code contributors already know; benchstat is
the community-standard comparator with sound statistics. The dual absolute/relative gate
implements both halves of the promise: never ship below threshold, never drift silently
inside it. Quarantine acknowledges that a flaky gate is a disabled gate.

#### Positive consequences

- Regression detection is mechanical, reviewable, and reconstructible from artifacts.
- Contributors run the same suite locally with the same comparison tool.
- The 5% warning band surfaces drift while it is still cheap to fix.

#### Negative consequences

- Baseline storage and runner-class bookkeeping add CI complexity.
- The 10%/5% bands are policy numbers that will need evidence-based tuning.
- Two confirming runs lengthen release pipelines on failure paths.

#### Risks

- Hosted-runner variance defeats calibration (mitigation: the PENDING VALIDATION
  investigation; fallback is a dedicated bare-metal runner); benchmark population changes
  silently shifting baselines (mitigation: population changes require explicit baseline
  resets per FR-PERF-005).

#### Reversal plan

The gate evaluator is a standalone tool consuming result documents; swapping comparison
tooling or bands changes the evaluator and this ADR, not the benchmarks or the NFRs.
Baseline history is retained so a policy change can be replayed against past results.

#### Review conditions

- Gate flake rate exceeds 2% of release runs, or the quarantine list exceeds 10% of gated
  benchmarks.
- The runner-calibration open question resolves (either way).
- benchstat or the Go benchmark facility materially changes its statistics.
