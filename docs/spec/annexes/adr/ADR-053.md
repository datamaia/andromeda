### ADR-053 — Workflow orchestration as supervised scheduler tasks; waiting states hold no threads

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team (Volume 4 authoring)
- Components: Workflow Engine, Task Scheduler, Event Bus
- Related requirements: FR-WF-009; NFR-WF-003

#### Context

ADR-023 and FR-ARCH-006 prohibit naked goroutines and mandate supervised, pooled,
backpressured concurrency. Workflow runs add a shape the interactive path does not have:
potentially many runs parked for hours in `awaiting_approval` or `paused`, plus parallel
steps and timers. A naive one-goroutine-per-workflow-run design would park goroutines on
gates and melt under many concurrent runs.

#### Problem

How does the Workflow Engine structure its concurrent work on the Task Scheduler so that
active orchestration is supervised and cancellable, while waiting runs cost effectively
nothing?

#### Forces and constraints

- FR-ARCH-004/006: everything cancellable through groups; no free-running goroutines.
- Gates can wait for a full expiry window (default 24 h); runs deadline at 168 h.
- Step failure must be routing data (chapter 06), not errgroup first-error propagation
  killing sibling steps.
- Volume 12 budgets memory per waiting run.

#### Alternatives considered

1. **Blocking orchestrator goroutine per run** — simple control flow; rejected: pool slots
   and stacks held across waits, first-error semantics fighting failure routing, and
   scale bounded by parked goroutines.
2. **Dedicated workflow thread pool outside the scheduler** — rejected: violates
   FR-ARCH-006's single-scheduler rule and forks supervision/observability.
3. **Event-driven continuations on scheduler tasks (chosen)** — control logic runs as
   short submitted tasks; waits are persisted state + event subscription + durable timer.

#### Decision

Each Workflow Run owns **one scheduler group** under its session. Orchestration executes as
**short-lived control tasks** on the `interactive` pool; spawned Runs run as per-step child
groups joined individually (failure = routing data); artifact/maintenance work rides
`background` with deferral on rejection. **Waiting states hold no pool slots and no
goroutines**: a waiting run is persisted state, one Event Bus subscription for its wake-up
events, and one durable deadline (ADR-051). Wake-ups submit fresh control tasks that re-read
persisted state and continue idempotently.

#### Rationale

This is the scheduler contract used as designed: supervision and cancellation come from the
group tree; cheap waiting comes from making the database, not a goroutine, the carrier of
run state — which chapter 07 already requires for durability.

#### Positive consequences

- Hundreds of parked runs cost one subscription and one row each (NFR-WF-003 budget).
- Cancellation reaches everything through one group subtree (FR-ARCH-004).
- Control tasks re-reading persisted state make continuations idempotent — the same
  property recovery needs.

#### Negative consequences

- Control flow is event-driven rather than linear; engine code is harder to read than a
  blocking orchestrator.
- Every continuation pays a state re-read; transition overhead is bounded by NFR-WF-003
  rather than being near-zero.

#### Risks

- Lost wake-up events (bus overflow policies) could strand a waiting run; mitigation: the
  ADR-051 sweep doubles as a liveness net — deadlines fire regardless, and the sweep
  re-evaluates gate states of live runs.
- Subscription leaks across many runs; mitigation: subscriptions close with the run group;
  NFR-ARCH-004 leak gates cover them.

#### Reversal plan

The event-driven core is internal to the Workflow Engine; a coroutine-styled layer could be
reintroduced over the same persisted-state contract without changing SchedulerPort usage,
groups, or any observable semantics.

#### Review conditions

- Re-evaluate if NFR-WF-003 transition-overhead measurements exceed the minimum threshold
  on reference hardware.
- Re-evaluate the wake-up design if Volume 10 event-delivery semantics change for the
  approval/workflow families.
