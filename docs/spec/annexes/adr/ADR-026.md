### ADR-026 — Brand identity as a closed design-token vocabulary with dark-first TUI theming and degradation tiers

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: TUI, CLI
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation.

#### Context

Andromeda has a defined brand: the palette and typography board (`docs/brand/branding.png`) and
the banner sketch (`docs/brand/banner-sketch.png`) establish a violet-on-near-black identity
with an ASCII-art cat mascot. The glossary defines "Design token" as "a named brand value
(color, typography role) from the brand definition; the closed set is specified in Volume 8",
and the single-home matrix assigns design tokens and theming to Volume 8. Terminals are the
product's primary canvas and vary widely: truecolor, 256-color, 16-color, and monochrome
environments all exist among target users, emulators choose their own fonts, and users run dark
and light backgrounds.

#### Problem

Which brand values are normative for the product's visual identity — in documentation, web
presence, and especially the TUI — and how does that identity survive terminals that cannot
reproduce exact colors or fonts, or that invert the assumed background?

#### Forces and constraints

- Consistent identity across README, documentation site, release collateral, and TUI is a UX
  requirement; ad-hoc per-surface color choices fragment the product.
- Terminal reality: fonts are chosen by the user's emulator — no application can enforce a
  typeface there; color depth varies by terminal and multiplexer.
- Accessibility: foreground/background contrast for text roles must meet a WCAG-style
  criterion, which constrains the exact Danger value on the near-black background.
- The token set MUST be closed (per the glossary definition): additions or changes go through
  specification change, not ad-hoc styling.
- Volume 8 owns the token-to-ANSI mapping and the `[tui.theme]` keys (single-home matrix and
  chapter 03 table ownership); this ADR fixes the brand source and vocabulary, not the mapping
  tables.

#### Alternatives considered

1. **No fixed brand** — let each surface (TUI, docs, site) style itself. Rejected: produces
   inconsistent UX across surfaces and unstable visuals across releases, and makes theming
   untestable because no normative expected value exists.
2. **Enforcing brand fonts in the terminal** — rejected as impossible: emulators render text in
   the user-configured font; an application cannot install or select terminal fonts. The
   brand's monospace choice can only ever be a recommendation there. This impossibility is
   documented so it is never re-attempted as a requirement.
3. **Dark-only theme** — rejected: light-background terminals are common; being unreadable on
   them fails basic usability regardless of brand fidelity.

#### Decision

Andromeda adopts the brand definition from `docs/brand/branding.png` and
`docs/brand/banner-sketch.png` as a **closed design-token vocabulary**:

- **Color tokens**: Primary `#7C5CFF` (violet accent), Secondary `#8C7B6E` (warm taupe),
  Tertiary `#F5F2ED` (warm off-white), Neutral `#121417` (near-black background). **Danger** is
  an accessible soft red whose exact value Volume 8 fixes — **PENDING VALIDATION** against a
  WCAG-style contrast criterion on the token backgrounds.
- **Typography tokens**: Geist for headline and body roles in documentation and web surfaces;
  JetBrains Mono for label and code roles — in terminals **recommended, never required** (see
  rejected alternative 2).
- **Mascot and wordmark**: the ASCII-art cat with four-pointed star accents and the lowercase
  serif wordmark from the banner sketch are the official marks.
- **Tagline**: "Your terminal companion for shipping great software."
- **TUI theming**: dark-first, derived from the tokens, with **degradation tiers**
  truecolor → 256-color → 16-color → no-color/monochrome, plus a light-terminal fallback
  theme. Each tier is normatively defined by Volume 8, not left to runtime approximation
  heuristics alone.
- **Ownership**: Volume 8 specifies the token-to-ANSI mapping per tier and the `[tui.theme]`
  keys; this ADR fixes the source images, token names, values, marks, and tier structure as the
  closed vocabulary Volume 8 maps.

#### Rationale

A closed token vocabulary makes brand consistency specifiable and testable: surfaces reference
tokens by name, golden tests assert tier outputs, and change control is explicit. The tier
ladder acknowledges terminal diversity honestly — identity degrades predictably instead of
breaking. Naming the font constraint impossible (not merely deprioritized) prevents a recurring
requirements mistake. Deferring the exact Danger value to Volume 8 under a contrast criterion
puts accessibility above aesthetic preference, consistent with the precedence order.

#### Positive consequences

- One recognizable identity from the GitHub banner to the TUI, stable across releases.
- Theming is testable: token values and per-tier mappings are normative expected values.
- Limited terminals get a designed experience, not an accidental one; monochrome operation is a
  first-class tier.
- The lipgloss styling stack (ADR-006 ecosystem) consumes exactly this token structure.

#### Negative consequences

- Four normative artifacts to maintain under change control: token set, tier mappings, light
  fallback, and marks.
- The closed vocabulary means even small visual adjustments require specification changes.
- Brand assets are raster images; values not stated here (for example, the board's tint scales)
  require careful transcription by Volume 8.

#### Risks

- **PENDING VALIDATION**: the exact Danger red must be fixed by Volume 8 and validated against
  the WCAG-style contrast criterion on Neutral `#121417` and the light fallback; until then, no
  surface hardcodes a danger color.
- 16-color mapping may drift visually from the brand (violet approximation varies by user
  palette); mitigation: Volume 8 maps against standard palettes and documents variance.
- Dark-first bias could leave light terminals under-tested; mitigation: TUI tests (ADR-017
  teatest) cover both backgrounds per tier.

#### Reversal plan

Tokens are referenced by name everywhere, so a rebrand is a value change in one vocabulary plus
regenerated assets; tier structure, `[tui.theme]` keys, and consuming code survive unchanged.
The tagline and marks are content-level and replaceable without code impact.

#### Review conditions

- Resolve the Danger-color PENDING VALIDATION when Volume 8 fixes the token-to-ANSI mapping
  and contrast checks.
- Review tier definitions when terminal capability detection is implemented, against real
  terminal population data.
- If brand assets are redesigned, supersede this ADR rather than restyling silently.
