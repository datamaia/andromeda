### ADR-015 — Semantic Versioning for product and public contracts, Conventional Commits with a fixed scope list

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Extension SDK, Configuration Manager, Plugin Runtime, Tool Runtime, Provider Layer, Updater, Package Manager
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation.

#### Context

Andromeda is developed openly on GitHub with full traceability between requirements, code,
tests, and releases. The product exposes many versioned surfaces beyond the binary itself:
the public API, internal contracts, schemas, the configuration version, the Plugin API, the
Tool API, the Provider API, the skill format, the workflow format, MCP compatibility, and
database migrations. ADR-004 fixes trunk-based development and ADR-013 fixes goreleaser-based
release automation with cosign signing and syft SBOMs; both need a machine-readable convention
for deriving versions and changelogs from history.

#### Problem

Which versioning scheme governs the product and its public contracts, and which commit-message
convention encodes change intent precisely enough that release tooling can compute the next
version, generate changelogs, and flag breaking changes without human reinterpretation?

#### Forces and constraints

- Extension authors (tools, plugins, skills, provider adapters) need a contract-level
  compatibility signal, not just a product version.
- Release automation (ADR-013) MUST be able to derive changelog sections and version bumps
  mechanically from commit history.
- Trunk-based development (ADR-004) produces a linear mainline where commit metadata is the
  primary record of intent.
- The specification mandates definitions for breaking changes, deprecation policy, support
  windows, backports, and release branches; the chosen scheme has to express all of them.
- Contributors are external and varied; the convention MUST be checkable in CI, not enforced by
  custom tooling knowledge.

#### Alternatives considered

1. **Calendar versioning (CalVer)** — rejected: encodes release timing, not compatibility. A
   platform whose Plugin API, Tool API, and Provider API are consumed by third parties needs a
   version component that changes if and only if compatibility changes.
2. **Perpetual 0.x versioning** — rejected: it suspends compatibility guarantees indefinitely,
   which contradicts publishing stable extension contracts.
3. **Free-form commits with a hand-maintained changelog** — rejected: manual changelogs drift
   from history, cannot drive automation, and place breaking-change detection on reviewer
   memory.
4. **A custom commit convention** — rejected: bespoke grammars need bespoke parsers and
   contributor education; Conventional Commits has existing ecosystem tooling and contributor
   familiarity.

#### Decision

Andromeda adopts **Semantic Versioning (SemVer)** for the product and for every public
contract: the public API, internal contracts, schemas, config version, Plugin API, Tool API,
Provider API, skill format, workflow format, MCP compatibility statement, and database
migrations each carry an explicit SemVer version, defined by their owning volumes.

Commit messages follow **Conventional Commits** with a closed scope list: `core`, `runtime`,
`agent`, `provider`, `auth`, `tool`, `plugin`, `skill`, `mcp`, `memory`, `context`, `index`,
`cli`, `tui`, `config`, `git`, `sandbox`, `perms`, `obs`, `docs`, `spec`, `ci`, `release`,
`deps`. Scopes outside this list fail CI; extending the list requires a specification change.
Breaking changes MUST be marked with `!` after the type/scope and a `BREAKING CHANGE` footer.
Release automation (ADR-013) derives changelog sections and version-bump proposals from commit
types and breaking-change markers.

```text
feat(provider)!: remove legacy completion capability flag

BREAKING CHANGE: adapters must declare capabilities via the Volume 5 enum.
```

#### Rationale

SemVer is the only widely understood scheme whose MAJOR/MINOR/PATCH semantics map directly onto
the compatibility promises Andromeda must make for extension contracts. Conventional Commits is
the only widely adopted commit grammar with off-the-shelf parsers, and its type + scope + `!`
structure carries exactly the three facts automation needs: what kind of change, in which area,
and whether compatibility broke. The closed scope list mirrors the component and area structure
of this specification, so commits become traceable to owning volumes — reinforcing the
traceability chain from Volume 0 (objective through commit to release).

#### Positive consequences

- Version bumps and changelogs are computed, not negotiated; releases become reproducible from
  history.
- Third-party developers can pin against contract versions (Tool API, Plugin API) independent of
  the product version.
- The scope list gives reviewers and tooling a shared, finite vocabulary aligned with the
  specification's areas.
- Deprecation policy, support windows, and backports have a precise vocabulary (MAJOR/MINOR/
  PATCH, pre-release identifiers) to be defined against.

#### Negative consequences

- Contributors must learn and pass the commit convention; malformed messages block merges.
- Multiple independently versioned contracts add bookkeeping: each owning volume must state its
  contract version and change rules.
- Squash-merge discipline is required so that mainline history remains the single source the
  automation parses.

#### Risks

- Mislabeled commits (a breaking change without `!`) would produce a wrong version bump;
  mitigation: CI checks plus reviewer checklist, and contract tests that fail on undeclared
  breaking changes.
- Scope-list rigidity could lag product structure; mitigation: the list is amendable through the
  normal specification change procedure.
- Changelog quality depends on commit-message quality; mitigation: PR templates and examples in
  contributor documentation.

#### Reversal plan

Both conventions are process-level, not embedded in shipped binaries. Reverting means updating
CI checks, contributor documentation, and the release pipeline configuration; already published
version numbers remain valid because any successor scheme can start from the last released
SemVer triple.

#### Review conditions

- If commit-convention CI failures exceed a nuisance threshold agreed by maintainers, revisit
  tooling and contributor guidance.
- When the Windows phase or new top-level components land, review the scope list for missing
  areas.
- Revisit changelog automation choices whenever ADR-013 tooling changes majors.
