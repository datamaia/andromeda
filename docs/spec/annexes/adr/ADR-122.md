### ADR-122 — Sandbox tiers: five subject profiles over one policy interface

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Sandbox Engine, Tool Runtime, Workflow Engine, Plugin Runtime, MCP Runtime
- Related requirements: FR-SEC-101, FR-SEC-106, FR-SEC-107

#### Context

ADR-021 fixes the layered enforcement mechanism (process-level floor, OS-level isolation
from Beta/v1) and Volume 3 fixes `SandboxPort` as the single launch path. What remains open
is the policy *shape*: Andromeda executes five structurally different subject kinds —
engine subprocesses, tool invocations, workflow stages, plugin processes, and locally
spawned MCP servers — with different lifetimes, declaration sources, and trust postures.

#### Problem

Does every execution share one sandbox profile, or does each subject kind get its own — and
if so, how do per-subject declarations and permission grants combine with defaults without
creating five divergent enforcement paths?

#### Forces and constraints

- One enforcement engine and one serializable policy schema (ADR-021's single-interface
  rule); tiers must not fork the mechanism.
- Long-lived extension processes (plugins, MCP servers) are the supply-chain surface
  (Volume 9 threat model chapter 03) and must default tighter than interactive tools.
- Workflow stages need a ceiling over their member invocations (Volume 4 stage semantics).
- Declarations come from untrusted parties (extension manifests); grants come from users;
  defaults come from this specification — combination must never widen.

#### Alternatives considered

1. **One uniform profile for everything** — rejected: forces the workspace-write default of
   tools onto plugin processes (too wide) or plugin restrictions onto tools (unusable), and
   gives workflows no ceiling construct.
2. **Free-form per-subject policies with no tier layer** — rejected: every caller would
   assemble policies from scratch, reproducing defaults inconsistently; security review
   would have no stable base profiles to reason about.
3. **Trust-level-based profiles** (policy keyed on origin trust rather than subject kind) —
   rejected as the primary axis: trust levels (Volume 6) modulate *grants*, but lifetime and
   data-directory shape follow the subject kind, not its trust; mixing axes produced a
   matrix explosion.
4. **Five subject-kind tiers with intersection semantics** (chosen).

#### Decision

The Sandbox Engine defines five tiers — `process`, `tool`, `workflow`, `plugin`,
`mcp_server` — as base profiles in one policy schema (Volume 9 chapter 06 tier table).
Effective policy is always the intersection of tier default, subject declaration, and
granted permissions; workflow-tier policies act as ceilings over member tool sandboxes;
plugin and MCP-server tiers receive no secret material without an extension-named
`credential_access` grant. The tier set is closed; additions require an ADR.

#### Rationale

Subject kind is the axis along which the *structural* differences run (lifetime, default
roots, override source); intersection is the only composition rule that is monotonically
narrowing, so no party — extension author, workflow definition, or misconfigured caller —
can widen containment by declaration. One schema keeps ADR-021's single-interface promise:
tiers are data, not code paths.

#### Positive consequences

- Security review reasons about five documented base profiles instead of per-execution
  ad-hoc policies.
- Extension manifests can only narrow; a malicious declaration cannot self-widen.
- The workflow ceiling gives Volume 4 a containment primitive matching stage semantics.

#### Negative consequences

- Tier defaults will fit some tools poorly (e.g., a legitimate tool needing home-directory
  access must be granted it explicitly every time policy allows) — friction is the price of
  narrow defaults.
- Five profiles to keep consistent as the policy schema evolves.

#### Risks

- Callers requesting the wrong tier for a subject; mitigated by `Prepare` validation
  (subject kind determines the legal tier) and conformance tests.
- Intersection emptiness surprising extension authors; mitigated by the explicit E-SEC-005
  pre-launch refusal with both sides of the intersection in the record.

#### Reversal plan

Tiers are rows in a base-profile table consumed by one engine; collapsing or adding tiers
changes data and the chapter 06 table, not port contracts. Recorded executions name their
tier, so historical records remain interpretable after any change.

#### Review conditions

- When OS-level isolation lands per platform (ADR-021 review), revisit whether tier
  defaults should request `os` isolation by default per tier.
- If WASM plugins arrive (ADR-009 v2 candidate), decide whether they are a sixth tier or a
  `plugin`-tier variant.
