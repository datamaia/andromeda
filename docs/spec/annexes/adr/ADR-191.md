### ADR-191 — Update channels, subscription floors, and automation consent policy

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Updater, CLI, Permission Manager, Configuration Manager
- Related requirements: FR-REL-005, FR-REL-006, FR-REL-007

#### Context

The brief mandates update channels (stable, beta, nightly, release candidate) and both
automatic and manual updates. Volume 2 freezes the Update process states; Volume 8 fixes the
manual surface (`andromeda update`, `update check`, `update rollback`, `--channel`) and the
consent conventions (destructive confirmation, `--yes`, non-interactive failure with its
usage-error code). ADR-015 fixes SemVer with pre-release identifiers. What remains open:
what each channel *means* in version terms, which releases a subscriber to a channel is
offered, and how far unattended automation may go before it conflicts with the corpus
precedence of user safety and consent (PRD-005, PRD-009).

#### Problem

Define the channel vocabulary as a closed enum with precise version semantics, the offer
rule that decides which Release a given subscriber sees, and the automation ladder
(check → download → apply) with its consent boundaries.

#### Forces and constraints

- Channel names surface in configuration (`[update].channel`), CLI flags, JSON output, and
  Release rows — they are a public contract and must be closed.
- Pre-release versions must order correctly under SemVer so "newer applicable release" is
  computable without channel-specific comparators.
- Applying an update replaces the installed product — a side-effecting action that binds to
  the `system_modification` permission and, per Volume 8, a destructive confirmation.
- Unattended environments (CI, servers) legitimately need fully non-interactive updates, but
  a default that self-modifies the product without consent violates PRD-005 and the Safe by
  Default posture.
- Major versions can break workflows and contracts (ADR-015); automating across a major
  boundary converts a consent decision into a surprise.

#### Alternatives considered

1. **Two channels only (stable + prerelease)** — rejected: it collapses "gating a release
   candidate" and "riding trunk nightly" into one risk class, though their stability and
   support expectations differ by design.
2. **Per-channel artifact trains with independent version lines** — rejected: SemVer
   pre-release identifiers already express maturity inside one ordered line; parallel lines
   would need custom ordering and duplicate the pipeline.
3. **Auto-apply enabled by default with an opt-out** — rejected: violates the precedence
   order (user safety and consent above convenience) and PRD-005; an agent product that
   silently replaces itself is the wrong default posture.
4. **No automation at all (manual only)** — rejected: the brief mandates automatic updates,
   and unattended fleets need them; the resolution is explicit opt-in, not absence.

#### Decision

1. The channel enum is closed: `stable`, `rc`, `beta`, `nightly`. Version semantics:
   `stable` = `X.Y.Z`; `rc` = `X.Y.Z-rc.N` cut from release branches for gating; `beta` =
   `X.Y.Z-beta.N` feature previews; `nightly` = `X.Y.Z-nightly.YYYYMMDD` scheduled trunk
   snapshots, where `X.Y.Z` is the next targeted release version. All channels are produced
   by the same ADR-013 pipeline with the full ADR-190 artifact and integrity set.
2. Channel maturity is ordered `stable > rc > beta > nightly`. The **offer rule**: a
   subscriber to channel `C` is offered the highest SemVer-ordered, non-yanked Release whose
   channel maturity is greater than or equal to `C` and whose upgrade path admits the
   installed version. A `stable` subscriber therefore never sees pre-releases; a `nightly`
   subscriber sees everything.
3. Switching to a less mature channel never downgrades: the installation stays on its
   current version until the new channel offers something SemVer-newer.
4. The automation ladder is three independent switches with fixed defaults: `auto_check`
   (default on), `auto_download` (default off), `auto_apply` (default off). Checking is
   metadata-only and side-effect-free; downloading stages verified artifacts without
   applying; applying always requires consent — interactive confirmation, `--yes`, or a
   recorded `always_allow_policy` decision for `system_modification` in unattended fleets.
5. `auto_apply` never crosses a major-version boundary: major updates are always offered,
   never self-applied, regardless of configuration.
6. Nightly Release rows are prunable after 30 days (Volume 2 permits pre-release pruning);
   `stable` and `rc` rows are retained indefinitely.

#### Rationale

Four channels map one-to-one onto the delivery phases the project already has (trunk,
feature preview, gating, general availability), and SemVer pre-release ordering makes the
offer rule a pure version comparison — no bespoke comparator, no channel-specific update
logic. The maturity-floor rule gives testers strictly more, never different, releases. The
three-switch ladder separates the harmless (check), the preparatory (download), and the
consequential (apply), putting the consent boundary exactly where the side effect is, and
the major-version carve-out keeps the one update class that can break contracts under
explicit human control.

#### Positive consequences

- One ordered version line; channels are filters, not forks.
- Unattended fleets get sanctioned full automation with an auditable recorded policy
  decision, not a bypass.
- Gating (rc) and bleeding-edge (nightly) populations are distinguishable in feedback and
  crash data.
- The offer rule is property-testable in isolation.

#### Negative consequences

- Four channels mean four populated release tracks to keep honest; nightly requires a
  scheduled pipeline in addition to tag-triggered releases.
- `auto_apply` off by default means most installations update on prompt, not silently —
  slower fleet convergence, by design.
- The no-downgrade rule means leaving `nightly` can strand a user on a pre-release for a
  while; the remedy (manual reinstall or rollback) is documented in FR-REL-005.

#### Risks

- Channel misconfiguration in the pipeline could publish a trunk snapshot as `stable`;
  mitigated by release-qualification gates (Volume 13) binding channel to workflow
  provenance, audited per FR-REL-016.
- Stale nightly subscribers after pruning could see no offers; mitigated by the offer rule
  falling through to more mature channels.

#### Reversal plan

Channels are metadata over one version line: removing a channel means ceasing to publish it
and mapping its subscribers to the next more mature channel (a configuration migration,
Volume 10). Changing automation defaults is a configuration-default change with release-note
prominence — no artifact or protocol impact.

#### Review conditions

- Beta-phase feedback shows the four-channel split under- or over-serves real populations.
- A team/fleet management feature lands that needs centrally managed update policy beyond
  `always_allow_policy`.
- ADR-015 pre-release grammar changes (it should not) or the Windows phase adds
  platform-specific channel constraints.
