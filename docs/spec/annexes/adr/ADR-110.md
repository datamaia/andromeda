### ADR-110 — Single TUI action registry: palette, quick actions, keybindings, and help are views over one catalog

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: TUI
- Related requirements: FR-TUI-060, FR-TUI-061, FR-TUI-062, FR-TUI-063, NFR-TUI-069

#### Context

The TUI exposes dozens of screens and hundreds of operations (Volume 8, chapters 07–10). The
brief mandates a command palette, quick actions, rebindable shortcuts, a help surface, and
keyboard-only operability. Each of those surfaces enumerates "the actions available here" —
and the product also promises driver parity (Volume 3, chapter 06): everything the TUI does
maps to Runtime API or port calls that the CLI can also reach.

#### Problem

How do four discovery surfaces (palette, quick actions, keymap, help) stay complete and
mutually consistent as screens and extensions add operations, without hand-maintaining four
parallel lists that drift apart?

#### Forces and constraints

- Keyboard reachability is a conformance target (NFR-TUI-069: 100% of actions reachable
  without a pointer); an action that exists only as a hardcoded key handler is unverifiable.
- Extensions contribute commands under the reserved `x` namespace at the CLI (ADR-104); the
  TUI needs a symmetric, collision-free mounting point.
- Permission mediation (Volume 9) must attach to the action, not to the surface that
  triggered it, or the same operation would be gated differently from the palette and from a
  keybinding.
- Golden-frame tests (ADR-006 teatest) need a stable enumeration to traverse.

#### Alternatives considered

1. **Central action registry (selected).** Every operation is a named, declared action;
   every surface renders from the registry.
2. **Per-screen key handlers plus a curated palette list.** Rejected: the curated list
   drifts from reality; unreachable and undocumented actions are undetectable; reachability
   cannot be tested mechanically.
3. **Leader-key menu system as the primary surface** (which-key style). Rejected as the
   primary mechanism: menus encode hierarchy, not searchability, and still need an
   underlying catalog; the help overlay (FR-TUI-063) provides the discoverable-key view
   instead.

#### Decision

The TUI maintains one **action registry**. Each entry declares: stable dotted identifier
(`<screen-or-domain>.<verb>`, e.g. `provider.verify`), title, context predicate (in which
screens/states it is available), default keybinding (may be empty), required permission
names from the Volume 9 enum (may be empty), destructiveness class (drives FR-UX-072
confirmation tiers), phase, and the Runtime API or port call it performs. The command
palette (FR-TUI-061) lists the registry filtered by context; quick actions (FR-TUI-062) are
a ranked subset; the keymap (chapter 07) binds keys to registry identifiers; the help
overlay (FR-TUI-063) renders the registry grouped by screen. Extension-contributed actions
mount under the `x.` identifier prefix, mirroring ADR-104. Executing an action follows the
same permission and confirmation path regardless of the invoking surface.

#### Rationale

One catalog makes completeness a property instead of a hope: reachability, help coverage,
and palette coverage are all mechanical assertions over the same data. Attaching permission
names and destructiveness to the action definition guarantees uniform gating. The dotted
identifier grammar matches the CLI's noun-verb tree (ADR-100 family), keeping the two
drivers explainable as one vocabulary.

#### Positive consequences

- NFR-TUI-069 is testable by registry traversal rather than by manual audit.
- New screens get palette, help, and keymap coverage by declaring actions — no parallel
  bookkeeping.
- Extension actions are namespaced and cannot shadow built-ins.
- The registry doubles as the golden-test enumeration for TUI automation.

#### Negative consequences

- Indirection: trivial one-key behaviors (scrolling, focus movement) must still be declared
  to be rebindable and testable, which adds ceremony.
- The context-predicate model must stay simple or availability logic becomes its own
  program.

#### Risks

- Registry sprawl: hundreds of fine-grained actions degrade palette usability. Mitigation:
  quick-action ranking, palette grouping by screen, and review of new actions at PR time.
- Divergence between a registered declaration and what the handler actually does (wrong
  permission names). Mitigation: mediation is enforced at the Runtime/port layer (Volume 9);
  the declaration is presentation metadata, not the enforcement point.

#### Reversal plan

The registry is a driver-internal structure; removing it means re-embedding key handling
per screen and hand-maintaining palette/help lists. No port, contract, or persisted format
depends on it, so reversal is contained to the TUI package — at the cost of losing the
mechanical reachability guarantee.

#### Review conditions

- Palette usability testing shows action counts above what ranking can organize.
- The extension surface (Volume 6) adds TUI panel extensions (Principle 6) that need richer
  contribution points than namespaced actions.
