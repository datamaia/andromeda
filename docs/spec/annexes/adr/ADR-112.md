### ADR-112 — Glyph capability tiers: a Unicode set and a complete ASCII fallback with declared parity

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: TUI, CLI
- Related requirements: FR-TUI-067, FR-TUI-066, RISK-TUI-072

#### Context

The TUI draws borders, markers, spinners, and badges. Volume 3 sets the minimum supported
geometry at 80×24 with UTF-8, but real environments fall short of full glyph coverage: the
Linux console renders a restricted font, legacy or misconfigured locales (`LC_CTYPE` without
UTF-8) mangle multibyte output, some terminal fonts lack box-drawing or braille ranges, and
double-width/ambiguous-width characters (emoji, East Asian ranges) are measured
inconsistently across emulators, corrupting column layouts. ADR-026 already defines color
degradation tiers; glyph degradation needs the same discipline. The CLI's human output is
ASCII-safe by default (FR-UX-002) and defers extended-glyph policy to this decision.

#### Problem

Which character repertoires may Andromeda's interfaces draw with, how is the repertoire
selected per invocation, and how is layout integrity protected against width-ambiguous
characters?

#### Forces and constraints

- A glyph that renders as tofu, a mangled byte sequence, or a double-width surprise is
  worse than a plain ASCII equivalent.
- The brand's mascot is ASCII art by definition (ADR-026), so brand identity survives the
  lowest tier by construction.
- Meaning must never be carried by a glyph alone (accessibility parity, chapter 12), so
  every symbolic glyph needs a textual or ASCII counterpart anyway.
- Probing character rendering is impossible without terminal round-trips; selection must
  rely on locale/TERM signals plus explicit configuration.

#### Alternatives considered

1. **Two declared glyph sets with automatic selection (selected).**
2. **Full Unicode always** — rejected: guarantees corruption on the Linux console and
   non-UTF-8 locales; violates the 80×24-anywhere floor.
3. **Three or more tiers** (full / extended / ASCII) — rejected: no reliable signal
   distinguishes "some Unicode" environments; two sets keep the parity table maintainable
   and the test matrix bounded.
4. **Emoji-rich chrome** — rejected: emoji width measurement varies by emulator and font,
   breaking column layouts; emoji carry no information a word cannot.

#### Decision

Andromeda defines exactly two glyph sets:

- **`unicode`**: single-cell characters only, restricted to a declared inventory — box
  drawing (U+2500 block), braille spinner frames, and a small symbol list (`…`, `•`, `→`,
  `✓`, `✗`, `✦`) — each with a declared ASCII equivalent.
- **`ascii`**: the complete fallback — `+ - |` borders, `-\|/` spinner, `...`, `*`, `->`,
  `ok`, `x`, `*` — with full informational parity.

Selection: the `tui.glyphs` key (`auto` | `unicode` | `ascii`, default `auto`); under
`auto`, the `unicode` set is chosen only when the locale (`LC_ALL`/`LC_CTYPE`/`LANG`)
declares UTF-8 and `TERM` is not `linux` or `dumb`; otherwise `ascii`. The chrome (borders,
markers, badges, spinners) MUST draw only from the selected set. Emoji and characters
outside the inventory MUST NOT appear in chrome; user and model *content* passes through
unmodified but is width-measured before layout, with width-ambiguous content confined to
cells that truncate rather than shift columns. The parity table in chapter 12 is normative;
specification wireframes are drawn in the `ascii` set.

#### Rationale

Two sets with a declared inventory make rendering testable per tier (golden frames per
set), keep the parity obligation small, and match the only trustworthy signals available
(locale and TERM). Confining chrome to single-cell glyphs removes the width-measurement
problem from every layout except content cells, where truncation bounds the damage.

#### Positive consequences

- Layout integrity on every supported terminal, including the Linux console and hostile
  SSH locale configurations.
- The ASCII set doubles as the accessible/no-frills aesthetic and the documentation
  wireframe language — one look, specified once.
- Golden tests enumerate two sets, not an open-ended font matrix.

#### Negative consequences

- Richer visuals (rounded corners, shade blocks, icons) are forgone even on capable
  terminals.
- Content width measurement still depends on a Unicode width library that trails Unicode
  releases.

#### Risks

- Width tables disagree with the user's emulator for ambiguous-width and emoji content,
  shifting content cells (RISK-TUI-072). Mitigation: chrome is immune by construction;
  content cells truncate; compatibility suite includes CJK/emoji fixtures.
- The `auto` heuristic misclassifies an environment. Mitigation: explicit `tui.glyphs`
  override; `doctor` prints the resolved set and deciding signal.

#### Reversal plan

Adding a richer set later is additive: a third named inventory with its own parity column
and golden frames, selected only by explicit configuration until signals justify auto
selection. Removing the `unicode` set degrades everything to `ascii` with no information
loss, by construction.

#### Review conditions

- Compatibility-suite evidence that a third tier has a reliable selection signal.
- Terminal ecosystem convergence on a queryable glyph/width capability (e.g., a widely
  deployed width-negotiation sequence).
