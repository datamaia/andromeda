### ADR-063 — OAuth for native clients: authorization code with PKCE over a loopback redirect, plus the device authorization grant

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Authentication Layer, CLI, TUI, Platform Abstraction Layer (browser launch)
- Related requirements: FR-AUTH-003, FR-AUTH-004; FR-AUTH-001

#### Context

The brief mandates OAuth and the Device Authorization Flow among supported mechanisms.
Andromeda is a native terminal application on macOS and Linux, frequently used over SSH and
in containers where no browser can be launched. OAuth 2.0 for native applications has
well-known public standards: the authorization code grant with PKCE (RFC 7636), loopback
interface redirection for native apps (RFC 8252), and the device authorization grant
(RFC 8628). Which provider offers OAuth to third-party clients at all is per-provider
PENDING VALIDATION (FR-AUTH-001 gate); this ADR fixes the client-side flow shapes Andromeda
implements when a provider does.

#### Problem

Which OAuth grant types and redirect mechanics does a terminal-native client implement so
that browserful and browserless environments are both served, without embedded client
secrets, interception-prone redirects, or prohibited grant types?

#### Forces and constraints

- Native apps cannot keep a confidential client secret; flows must be public-client-safe.
- SSH/container sessions have no launchable browser (PRD-009 forbids hanging prompts).
- Custom URI-scheme redirects are fragile on Linux terminals and register OS-level handlers
  Andromeda otherwise does not need.
- FR-AUTH-001 prohibits anything undocumented; only standard, provider-documented grants
  qualify.

#### Alternatives considered

1. **Implicit grant** — rejected: deprecated practice for native apps; tokens in redirect
   URLs; no refresh tokens.
2. **Resource owner password grant** — rejected: handles raw user passwords, contrary to the
   product's credential-safety posture, and widely withdrawn by providers.
3. **Custom URI-scheme redirect** instead of loopback — rejected: OS handler registration is
   platform-fragile, and a terminal process already owns the loopback pattern naturally.
4. **Out-of-band manual code copy only** — rejected as the primary path: worse UX than
   loopback where a browser exists; retained implicitly via the device grant's user-code
   model.

#### Decision

Andromeda implements exactly two establishment flows for providers documenting OAuth for
third-party clients: (a) **authorization code with PKCE (S256)** using a loopback redirect
listener on `127.0.0.1` with an ephemeral port and single-use `state`, launching the system
browser through the PAL; and (b) the **device authorization grant** for browserless
environments, with provider-controlled polling (`interval`, `slow_down`, `expires_in`
honored). Implicit and password grants are prohibited. Refresh tokens are stored per
FR-AUTH-009; non-interactive invocations refuse to start interactive flows.

#### Rationale

These are the two standardized shapes for native/public clients — one optimal where a browser
exists, one designed for exactly Andromeda's SSH/container reality. PKCE plus single-use state
addresses code interception and CSRF without secrets in the binary.

#### Positive consequences

- Both interactive-desktop and headless environments have a documented, standard path.
- No client secret ships in the product; no OS URI-handler registration.
- Flow implementations are provider-independent; adapters supply only endpoints and scopes.

#### Negative consequences

- Two flow implementations to test and maintain.
- Loopback listeners can collide with aggressive local firewall policies (surfaced as flow
  errors, not hangs).

#### Risks

- Provider OAuth implementations deviate from the RFC shapes; mitigation: per-adapter
  contract tests against recorded fixtures; deviations recorded at adapter implementation.

#### Reversal plan

Flows sit behind `AuthPort.Authenticate`; adding or retiring a grant type changes the
Authentication Layer only. Retiring loopback in favor of a future standard would be an
additive flow with configuration-selected preference.

#### Review conditions

- When the first provider validates OAuth support (V5B-OQ-2), confirm the flow shapes against
  its documentation.
- Revisit if OAuth 2.1 consolidation or successor native-app guidance changes recommended
  practice.
