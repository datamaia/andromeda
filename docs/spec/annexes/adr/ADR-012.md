### ADR-012 — Event Bus and External IPC: In-Process Typed Bus; Unix Domain Socket with JSON-RPC 2.0

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Event Bus, Runtime, Plugin Runtime, Platform Abstraction Layer (PAL)
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation

#### Context

Two related but distinct communication needs shape this decision. Internally, Andromeda's components (Agent Engine, Tool Runtime, Workflow Engine, TUI, Observability) are decoupled through events — the glossary defines the Event Bus as an in-process typed publish/subscribe channel, with the envelope and delivery semantics owned by Volume 10. Externally, a running Andromeda instance must be controllable and inspectable by other local processes: companion tooling, editor integrations, scripts driving a headless session. Andromeda is a single-binary, local-first tool (ADR-001); it runs no daemons and requires no services.

#### Problem

Select the internal event-distribution mechanism and the external IPC transport/protocol, keeping both consistent with local-first constraints and with the protocol grammar already adopted for plugins (ADR-009) and MCP (ADR-010).

#### Forces and constraints

- Events fan out to consumers with different tempos: the TUI must not lag because a slow subscriber (e.g., a trace writer) is behind — per-subscriber buffering and backpressure policy are required (concurrency model per ADR-023).
- Event payloads are typed and versioned (Volume 10 envelope); stringly-typed topics are a defect class to design out.
- No external broker: requiring a daemon (Redis, NATS, D-Bus service) would break single-binary local-first operation outright.
- External IPC crosses a security boundary on a shared machine: the endpoint MUST be access-controlled by filesystem permissions and MUST enforce the same Permission Manager mediation as any other entry point (Volume 9).
- Protocol economy: ADR-009 (ARP) and ADR-010 (MCP) already commit Andromeda to JSON-RPC 2.0; a third grammar needs strong justification.
- Windows later: the transport abstraction must have a Windows analogue without redesign (PAL).

#### Alternatives considered

**Internal distribution:**

| Criterion | Typed channel-based bus (selected) | External broker (NATS/Redis) | Direct component coupling |
|---|---|---|---|
| Process model | In-process, zero deps | Separate daemon | In-process |
| Typing | Go types per event, compile-checked | Bytes/strings on the wire | Typed but point-to-point |
| Local-first fit | Native | Broken (external service) | Native |
| Decoupling | Publisher ignorant of subscribers | Yes, at operational cost | None — N×M dependencies |

**External IPC:**

| Criterion | UDS + JSON-RPC 2.0 (selected) | gRPC over UDS/TCP | D-Bus | TCP + REST |
|---|---|---|---|---|
| Grammar reuse (ADR-009/ADR-010) | Same JSON-RPC 2.0 | New protobuf stack | New (D-Bus wire format) | New |
| Client accessibility | Any language; scriptable with a socket and JSON | Codegen toolchain per client | Linux-centric libraries | Universal but needs port management |
| Access control | Filesystem permissions on socket (0600) | TLS/authn machinery or UDS perms | Bus policy | Port exposure risk; loopback binding discipline |
| macOS support | Native | Native | Not native | Native |
| Windows path | Named pipes via PAL | Native | None | Native |
| Ops footprint | A socket file in the runtime directory (ADR-022) | Heavier server surface | System bus dependency | Open port on the host |

- **gRPC** brings streaming and codegen, but at the cost of a protobuf toolchain for every client author and a second RPC grammar in the product; nothing in the local-control use case needs its capabilities.
- **D-Bus** is idiomatic on Linux desktops only; it is not native on macOS and has no Windows story — it fails the platform matrix by itself.
- **Message brokers** and **TCP/REST** both violate the operational premise: no daemons, no open ports on developer machines by default.

#### Decision

1. **Event Bus**: Andromeda MUST implement an in-process typed event bus as its own package, built on Go channels — typed topics, per-subscriber bounded buffers, and an explicit per-topic backpressure/overflow policy (drop-oldest, block, or coalesce, declared per event family in Volume 10). The envelope, ordering, and delivery-semantics contract is Volume 10's; concurrency discipline follows ADR-023. External brokers MUST NOT be used for internal eventing.
2. **External IPC**: a running instance MUST expose its control surface over a **Unix domain socket** using **JSON-RPC 2.0**, sharing the protocol grammar (message shapes, error object conventions, versioned handshake) with the Andromeda Runtime Protocol (ADR-009). The socket lives in the per-instance runtime directory (ADR-022) with `0600` permissions; every request passes the Permission Manager like any other entry point. On Windows (future), the PAL maps the same protocol onto named pipes.
3. Events MAY be subscribed to over the IPC surface (bridged from the Event Bus through the Volume 10 envelope), making the bus observable without linking against Andromeda.

#### Rationale

Internally, Go channels already provide the queueing, select-based multiplexing, and shutdown semantics an event bus needs; wrapping them in a small typed package yields compile-time-checked events with zero dependencies — an external broker would add an operational tier to solve a problem the language runtime solves in-process. Externally, UDS + JSON-RPC 2.0 is the smallest mechanism that is simultaneously access-controlled (filesystem), portable (PAL analogue on Windows), scriptable from any language, and grammatically identical to the protocols Andromeda already speaks — one grammar across ARP, MCP, and IPC minimizes SDK, docs, and test surface.

#### Positive consequences

- Zero-dependency, compile-time-typed internal eventing; misuse surfaces at build time.
- Slow consumers cannot stall the Runtime; overflow behavior is explicit policy per event family, not accident.
- External tools integrate with a socket and JSON — no SDK, no codegen; the same conformance fixtures as ARP apply.
- No daemons, no open TCP ports, no broker to install: the local-first posture is preserved verbatim.

#### Negative consequences

- The bus is single-process by construction; any future multi-process architecture must bridge events over IPC rather than getting distribution for free.
- Bounded buffers mean lossy topics exist by design; consumers of drop-eligible families must tolerate gaps (the envelope's ordering metadata makes gaps detectable).
- JSON-RPC over UDS lacks gRPC's streaming ergonomics; high-volume subscriptions need explicit chunking/flow-control design in Volume 10.

#### Risks

- **Backpressure misconfiguration** (blocking policy on a hot path) could deadlock or lag the TUI. Mitigation: policy is declared per event family and race-/load-tested per ADR-017 and Volume 12.
- **Socket hygiene**: stale sockets after crashes, or predictable paths on shared machines. Mitigation: per-instance runtime directory with liveness checks at bind time (ADR-022); permissions `0600` enforced at creation.
- **Grammar divergence** between ARP and IPC over time. Mitigation: both are specified against the same JSON-RPC conventions section in Volume 6/Volume 10; shared conformance tests.

#### Reversal plan

The Event Bus is an internal package behind an interface: replacing channels with another in-process mechanism (or bridging to a broker if a multi-process future demands it) re-implements one package while publishers/subscribers keep their typed API. The IPC surface reverses per transport and per protocol independently: the JSON-RPC handler layer is transport-agnostic, so moving to gRPC (or adding a second transport) means adding a listener and codegen layer while the command semantics — specified in Volume 8/Volume 10, not in the transport — stay fixed.

#### Review conditions

- A multi-process architecture change (e.g., detached daemon mode) is proposed — the in-process bus assumption must be re-decided.
- Volume 12 measurements show event throughput or IPC round-trip latency breaching targets under representative TUI + tracing load.
- Windows-phase entry: validate the named-pipe PAL mapping and revise this ADR's transport section with measured behavior.
- IPC clients demonstrably need server-push streaming beyond what JSON-RPC notifications provide.
