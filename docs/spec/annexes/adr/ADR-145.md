### ADR-145 — Git operation catalog with phased, additive GitPort exposure

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Git Engine, Tool Runtime, CLI, TUI
- Related requirements: FR-GIT-001, FR-ARCH-003; ADR-025

#### Context

ADR-025 fixed the Git Engine as an encapsulated adapter over system git ≥ 2.40, and Volume 3
chapter 02 froze GitPort at fifteen methods: version, status, diff, stage/unstage, commit,
log, show, branch listing/creation/switching, patch application, and three worktree methods
(FR-ARCH-003). The product brief requires a substantially wider Git surface: tags, remotes,
fetch/pull/push, rebase, merge, cherry-pick, revert, reset, blame, hooks, conflicts,
submodules, sparse checkout, large-file support, ignore rules, signing, and protected-branch
awareness. The Volume 1 MVP minimum (item 12) commits only to "basic Git": status, diff,
stage, commit, branch, log. Volume 4's SDD workflow needs restore points at MVP and
release-preparation operations from Beta onward.

#### Problem

How does Volume 11 specify the full mandated Git operation surface without violating the
GitPort freeze, and in which delivery phases do the operations beyond the frozen fifteen
methods become available to consumers?

#### Forces and constraints

- FR-ARCH-003 freezes port names, method names, and arities; its alternative flow permits
  additive amendments only through the Volume 0 change procedure.
- ADR-025 prohibits any component outside the Git Engine from invoking git or parsing git
  output, so no side channel may deliver the extended operations.
- Every mutating operation must be individually permissioned and confirmable (Volume 9
  binding); a generic passthrough would collapse distinct risk classes into one opaque call.
- The MVP minimum deliberately bounds Git scope; front-loading the full surface would inflate
  MVP risk (RISK-PRD-004).

#### Alternatives considered

1. **Generic escape-hatch method** (`Run(args)`) on GitPort — rejected. It is itself a
   signature change to a frozen port, and a raw-argument passthrough defeats operation-level
   permission classification, typed results, and machine-readable parsing: callers would
   reimplement exactly what ADR-025 centralizes.
2. **Route extended operations through TerminalPort** as `git` command lines — rejected.
   ADR-025 forbids git invocation outside the Git Engine; the terminal path loses typed
   errors, structured results, and the destructive-operation gate of ADR-146.
3. **Defer everything beyond the frozen fifteen methods to v2** — rejected. Remote
   operations, tags, and history-modification operations are needed by Beta for the SDD
   release-preparation stage and for everyday engineering flows; deferral would push users
   back to raw git outside Andromeda's audit trail.
4. **Catalog now, additive port batches at phase boundaries** — selected, described below.

#### Decision

Volume 11 defines a single **Git operation catalog** (Volume 11, chapter 01) covering the
entire mandated surface. Each operation declares: operation class (read-only, additive
mutation, history mutation, destructive, remote), permission binding, confirmation
requirement, GitPort exposure, and delivery phase. Exposure follows three tiers:

1. **MVP tier** — exactly the fifteen frozen GitPort methods. No amendment needed.
2. **Beta tier** — tags, remotes (fetch, pull, push), merge, rebase, cherry-pick, revert,
   reset, blame, and conflict inspection are specified now at operation level; their port
   methods are proposed as **one additive GitPort amendment batch** through the Volume 0
   change procedure before Beta implementation begins.
3. **v1 tier** — submodule management, sparse-checkout management, and large-file-storage
   management follow as a second additive batch.

Amendment batches MUST be additive only (new methods, new optional contract-type fields);
no existing method may be renamed or re-signed. Until an operation's port method exists,
no consumer may reach it: the operation is absent, not emulated.

#### Rationale

The catalog gives the corpus one authoritative statement of the full surface — semantics,
permissions, and phases — while respecting the freeze mechanism FR-ARCH-003 itself provides
for genuine contract gaps. Batching amendments at phase boundaries keeps the number of
change-procedure events to two, aligns port growth with tested delivery increments, and
keeps the MVP surface identical to what Volume 1 committed. Operation-level methods (rather
than a generic call) preserve per-operation permission classes, typed results, and the
ADR-146 destructive-operation gate.

#### Positive consequences

- The frozen port and the mandated full surface are reconciled without exception or side
  channel.
- Permission and confirmation semantics attach to named operations, reviewable one by one.
- MVP scope stays exactly at the Volume 1 minimum; growth is planned, not improvised.
- Consumers (Tool Runtime, CLI, TUI) discover capability by phase, not by probing.

#### Negative consequences

- Two future change-procedure events are pre-committed, each requiring cross-volume review.
- Between phases, users needing an unexposed operation must use raw git themselves, outside
  Andromeda's audit trail.
- The catalog and the port drift risk exists between amendment batches and must be checked at
  consolidation.

#### Risks

- An amendment batch is rejected or delayed, stranding Beta features; mitigation: batches
  contain only operations already specified and reviewed in this volume, so the amendment is
  editorial, not a design negotiation.
- Scope pressure to smuggle operations through the terminal path before their tier arrives;
  mitigation: ADR-025's prohibition is CI-enforced via the ADR-033 prohibited-construct
  scanner (git invocation outside the Git Engine subtree fails CI).

#### Reversal plan

The catalog is specification metadata; removing an operation before its tier ships is a
document change. After exposure, methods are public contract surface under ADR-015 SemVer
rules: withdrawal follows the deprecation regime, not this ADR.

#### Review conditions

- At each phase gate (Beta, v1), review the pending amendment batch against actual workflow
  demand and drop operations with no consumer.
- If go-git read-only fast paths (ADR-025) are validated, review whether any catalog entry's
  implementation notes change (exposure and semantics do not).
- If a third amendment batch becomes necessary, review whether GitPort growth indicates a
  need to split the port in a major release.
