### ADR-024 — Schema validation: santhosh-tekuri/jsonschema/v6 for JSON Schema; typed schema layer for TOML configuration

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Tool Runtime, Configuration Manager, Extension SDK, MCP Runtime, Provider Layer
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation.

#### Context

Schemas are load-bearing contracts in Andromeda: every Tool declares schema-typed inputs and
outputs (Volume 6 owns the tool contract), structured outputs from providers are validated
against caller-supplied schemas (Volume 5), MCP tools (ADR-010) carry JSON Schema definitions,
and plugins (ADR-009) exchange schema-described payloads over the Andromeda Runtime Protocol.
Separately, `andromeda.toml` (parsed with go-toml v2 per ADR-008) needs validation beyond
parsing: types, ranges, closed enums, cross-key rules, and precedence-aware diagnostics (Volume
10 owns the configuration schema). Verified 2026-07-11: `github.com/santhosh-tekuri/jsonschema/v6`
is Apache-2.0, active (v6.0.2, commits 2026-06), `/v6` is the current major, it supports drafts
2020-12, 2019-09, 7, 6, and 4, and it passes the official JSON Schema test suite.

#### Problem

Which mechanism validates JSON Schema documents across tools, structured outputs, MCP, and
plugins — and how is TOML configuration validated — such that semantics are draft-correct,
consistent across every subsystem, and produce diagnostics precise enough for the error
envelope (ADR-016)?

#### Forces and constraints

- Draft correctness is non-negotiable: tool schemas arrive from third parties (MCP servers,
  plugins, skills) targeting different drafts; wrong draft semantics silently accept invalid
  data or reject valid data.
- One validator for the whole product: divergent validation behavior between the Tool Runtime
  and the MCP Runtime would make the same schema mean two things.
- License policy: runtime dependencies stay permissive (ADR-002 context); Apache-2.0 fits.
- Configuration is TOML, not JSON; forcing TOML through JSON Schema loses TOML-native types and
  produces diagnostics in the wrong vocabulary for users editing `andromeda.toml`.
- Validation failures surface through the ADR-016 envelope and must carry precise locations
  (instance path, failing keyword).

#### Alternatives considered

1. **`gojsonschema`** — rejected: verified as offering less draft support than
   santhosh-tekuri/jsonschema/v6; missing current-draft semantics disqualifies it for
   third-party schema interop where draft 2020-12 documents are the norm.
2. **Hand-rolled validation** — rejected: JSON Schema's applicator, reference, and dynamic
   keywords are notoriously subtle; reimplementing them would create a bug surface the official
   test suite exists to prevent, for zero product differentiation.
3. **JSON Schema for TOML configuration too** (convert TOML to JSON, validate, map errors
   back) — rejected as the primary mechanism: diagnostics would reference JSON pointers rather
   than TOML keys, TOML-native distinctions degrade in conversion, and configuration needs
   precedence- and migration-aware rules (Volume 10) a generic validator cannot express.

#### Decision

- **`github.com/santhosh-tekuri/jsonschema/v6`** (verified Apache-2.0, v6.0.2, active; note
  `/v6` is the current major — earlier majors are not to be adopted) is the single JSON Schema
  validator for: tool input and output schemas, structured-output validation, MCP-provided
  schemas, and Andromeda Runtime Protocol payload schemas. Supported drafts follow the library:
  2020-12, 2019-09, 7, 6, and 4.
- **TOML configuration is validated by a typed schema layer** built on top of ADR-008 parsing:
  Go types define structure, with declarative rules for ranges, closed enums, defaults,
  deprecations, and cross-key constraints, producing TOML-vocabulary diagnostics (table, key,
  line) that feed the ADR-016 envelope. Volume 10 owns this schema's content and versioning.
- Validation failures in both paths map to defined error codes with instance location and
  failing rule identified.

#### Rationale

For JSON Schema, correctness is measurable: passing the official test suite across five drafts
is exactly the guarantee third-party interop needs, and santhosh-tekuri/jsonschema/v6 provides
it under Apache-2.0, within license policy. Centralizing on one validator makes schema
semantics a product-wide constant. For TOML, the typed layer matches the medium: configuration
errors are a primary UX surface (exit code 3 per ADR-016), and users get diagnostics in the
language of the file they edited, with room for the precedence and migration semantics Volume
10 requires.

#### Positive consequences

- Identical, draft-correct schema semantics across Tool Runtime, MCP Runtime, Provider Layer,
  and plugin payloads.
- Third-party schemas written for any mainstream draft validate as their authors intended.
- Configuration diagnostics speak TOML (tables and keys), improving fixability of exit-code-3
  failures.
- The Extension SDK can document one validation behavior for tool and plugin authors.

#### Negative consequences

- Two validation mechanisms coexist (JSON Schema and the typed TOML layer); the boundary must be
  documented so authors do not use the wrong one.
- The typed TOML layer is code Andromeda owns, including its rule vocabulary and tests.
- Single-validator dependency concentrates risk in one library's maintenance.

#### Risks

- A future JSON Schema draft could appear before the library supports it; mitigation: declared
  supported drafts are part of the Volume 6 tool contract, and the review condition below
  triggers on new drafts.
- Version skew if any dependency pulls an older major of the same library; mitigation: module
  hygiene checks in CI ensuring only `/v6` is present.
- Divergence between the typed TOML schema and documented configuration reference; mitigation:
  the reference is generated from the schema definition (Volume 10).

#### Reversal plan

Both validation paths sit behind narrow internal interfaces (validate document against schema;
validate resolved config). Replacing the JSON Schema library means swapping the implementation
and re-running the official-suite-derived conformance tests; replacing the typed TOML layer
means regenerating diagnostics mappings. Tool and plugin authors are unaffected as long as
declared draft support is preserved.

#### Review conditions

- When a new JSON Schema draft is published or the library changes majors: re-evaluate and
  update declared draft support.
- If santhosh-tekuri/jsonschema maintenance stops for 12 months, evaluate alternatives against
  the official test suite.
- When Volume 10 fixes config versioning and migration rules, review the typed layer's rule
  vocabulary for completeness.
