### ADR-081 — Extension signature policy: checksums unconditional, cosign signatures policy-gated, invalid always blocks

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Package Manager, Policy Engine, Skill Engine, Plugin Runtime
- Related requirements: FR-PLUG-007, FR-SKILL-005; INV-PKG-01, INV-PKG-02 (Volume 2)

#### Context

Extension packages (skills, plugins, bundles) are third-party content crossing the
supply-chain boundary. Volume 2 fixes the Package attributes `checksum` (SHA-256, required)
and `signature_state` (`verified` | `unverified` | `invalid`) with INV-PKG-01 (checksum
verified before anything executes or registers) and INV-PKG-02 (`invalid` blocks installation
regardless of configuration). ADR-013 established the project's signing technology for
product releases: cosign (v3 major, Apache-2.0, verified 2026-07-11) with keyless OIDC
signing in CI. Third-party extension publishers are outside the project's control: some will
sign, many will not, and mandating signatures at Beta would empty the ecosystem.

#### Problem

Define what "signed" means for extension packages, how `signature_state` is computed, and
what each state permits — balancing supply-chain integrity against the reality that signature
adoption by independent publishers is gradual and unenforceable.

#### Forces and constraints

- INV-PKG-01/02 are fixed floors: checksums always verified; `invalid` always blocks.
- One signing technology: ADR-013 already puts cosign in the toolchain and in contributors'
  vocabulary; a second signature format doubles verification code and documentation.
- Keyless (identity-based) and key-based signing must both work: individual publishers may
  prefer keys; CI-published packages fit keyless identities.
- Verification must work offline for pre-fetched artifacts (air-gapped installs per ADR-080),
  which constrains how much verification may depend on online transparency-log lookups.
- Policy, not code, must decide what `unverified` means per installation (personal machine
  versus locked-down enterprise).

#### Alternatives considered

1. **Mandatory signatures for all packages** — rejected at Beta: would exclude the realistic
   majority of early publishers, pushing users toward `path`/`archive` side-loading that
   bypasses the mechanism entirely — a net security loss.
2. **Checksums only, no signature support** — rejected: checksums authenticate *content
   against the index entry*, not *publisher against content*; a compromised index rewrites
   both. Signatures are the only defense in that scenario, and deferring the mechanism would
   make later adoption a breaking ecosystem event.
3. **A bespoke Andromeda signature scheme** (project-managed keys, custom format) — rejected:
   duplicates what cosign already provides, adds key-management liability the keyless flow
   exists to avoid, and contradicts the one-technology force.
4. **GPG signatures** — rejected: no existing footprint in the project's toolchain, poor CI
   ergonomics, and the web-of-trust model does not map to how extension publishers actually
   operate; cosign is already the verified, adopted mechanism (ADR-013).

#### Decision

1. **Signature format**: extension package signatures are **cosign signatures over the
   package archive digest** — either keyless (certificate-based, bound to an OIDC identity)
   or key-based (a public key the user or a source's index metadata provides). The cosign
   **v3 major line is pinned** for verification, matching ADR-013.
2. **`signature_state` computation** at the `verifying` installation step:
   - `verified` — a signature is present and validates against the trust material the user or
     source configuration declares for that publisher (identity pattern or public key).
   - `unverified` — no signature is present, or a signature is present but no trust material
     is configured to evaluate it against.
   - `invalid` — a signature is present and fails validation against declared trust material,
     or the archive digest does not match the signed digest.
3. **Policy consequences**: `invalid` blocks installation unconditionally (INV-PKG-02).
   `unverified` proceeds only per the explicit trust policy in force (Volume 9 decision
   semantics; interactive default: prompt with provenance shown; non-interactive default:
   deny per PRD-009). `verified` proceeds and feeds the packaged-verified trust
   classification (Volume 9 vocabulary).
4. **Official packages** (`andromeda-` name prefix, Volume 2) MUST be signed by the project's
   release identity per ADR-013; an unsigned package claiming the official prefix is
   `invalid` by policy.
5. **Offline verification**: signature verification of pre-fetched artifacts MUST be possible
   without network access when the signature bundle and trust material are locally present;
   verification steps that would require online lookups are recorded as not evaluated rather
   than silently passed.

#### Rationale

The three-state model matches reality: it rewards publishers who sign (stronger trust
classification, smoother installs), it keeps unsigned content usable under explicit,
recorded consent instead of driving users around the mechanism, and it makes tampering
(`invalid`) a hard stop that no configuration weakens. Reusing cosign keeps one verification
implementation, one documentation story, and one set of CI patterns across product releases
(ADR-013) and extensions (this ADR).

#### Positive consequences

- Supply-chain verification exists from Beta with zero mandatory burden on early publishers.
- The trust classification pipeline (signature_state → trust level → activation/start policy)
  is uniform across skills, plugins, and packaged MCP servers.
- Enterprises can require `verified` for everything via policy without any product change.

#### Negative consequences

- Three states are harder to explain than signed/unsigned; UX must present provenance
  honestly without alarm fatigue on `unverified`.
- Keyless verification inherits trust in the Sigstore infrastructure chain (ADR-013 negative
  consequence), now on the extension path too.
- Publisher identity policy (which identities may sign which package names within a source)
  adds configuration surface to sources.

#### Risks

- Signature-stripping downgrade: an attacker who controls a source strips signatures to land
  in `unverified` instead of `invalid`; mitigated by per-source policy (a source or package
  pattern can be declared signature-required, making absence `invalid`) and by index entries
  carrying expected signature references.
- cosign v4 or Sigstore infrastructure changes breaking verification: same mitigation as
  ADR-013 (pinned major, verification tests, review condition).

#### Reversal plan

`signature_state` and its policy consequences are the stable contract; the signature
technology sits behind the Package Manager's verification step. Replacing cosign would swap
the verifier implementation and the trust-material configuration format, with `verified`
semantics preserved; packages already installed keep their recorded states. Tightening to
mandatory signatures later is a policy-default change, not a mechanism change.

#### Review conditions

- Signature adoption data at v1: if a supermajority of installed packages verify, evaluate
  flipping the default policy for `unverified` to deny-with-override.
- cosign publishes a v4 major or Sigstore deprecates infrastructure the verification path
  uses (shared with ADR-013).
- The official registry decision (ADR-080 review) lands: define the identity policy the
  official index enforces at publication time.
- A practical offline verification gap appears in air-gapped field use: strengthen the
  bundled-trust-material story.
