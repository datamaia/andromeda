### ADR-177 — Flaky-Test Quarantine: Build-Tag Exile with a Time Box, No Blind Retries

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: all components (test suites), Observability (CI metrics)
- Related requirements: FR-TEST-007, NFR-TEST-005, RISK-TEST-002

#### Context

A flaky test converts a gate from a verification into a dice roll: contributors re-run
instead of reading failures, and real defects ride through (RISK-TEST-002). Volume 13
requires deterministic suites (FR-TEST-007), but flakes will still occur; the response
mechanism must be fast, visible, auditable, and bounded in time.

#### Problem

Which mechanism removes a flaky test from gating suites quickly, keeps it visible and running,
forces resolution within a bounded period, and is mechanically enforceable in CI?

#### Forces and constraints

- The mechanism must be visible in code review (a quarantine is a change to the tree, not a CI
  console toggle).
- Quarantined tests must keep running somewhere, or the flake can never be diagnosed or proven
  fixed.
- Enforcement must be mechanical: a policy that depends on humans remembering dates will not
  hold (Volume 0 audit obligations).
- Retrying failures inside gating runs is indistinguishable from tolerating real intermittent
  defects.

#### Alternatives considered

1. **Automatic retry-on-failure in gating CI.** Rejected: masks both flakes and genuinely
   intermittent product defects; damages the meaning of a green gate.
2. **Central skip-list file.** Rejected: drifts from the code it governs, invisible at the
   test site, chronic merge-conflict surface, decaying ownership.
3. **Immediate deletion.** Rejected: silently loses coverage; a flake is often a real
   concurrency defect worth diagnosing.

#### Decision

- **Detection.** A test is declared flaky when (a) the nightly determinism lane (repeat and
  shuffled runs, FR-TEST-007) records divergent outcomes for it, or (b) a trunk gating run
  fails on it and the failure is triaged as non-reproducible with no product change implicated.
- **Quarantine mechanism.** The test file (or the isolated test) is moved behind the Go build
  tag `quarantine`, with a mandatory machine-readable comment on the tagged file:
  `// quarantined: <YYYY-MM-DD> <issue-URL>`. The change lands by pull request like any other
  change; the linked issue is labeled per Volume 11's taxonomy and carries P1 priority.
- **Quarantine lane.** A scheduled CI job runs `quarantine`-tagged tests non-gating and
  publishes their pass history; a flake is provably fixed only after 20 consecutive divergence-
  free runs in this lane plus the determinism lane.
- **Time box.** 14 calendar days from the quarantined date: the test is fixed and restored,
  rewritten, or deleted with a recorded justification in the issue. A repository check fails
  when any `quarantined:` date is older than 14 days, making expiry block the merge gate for
  everyone — the pressure is deliberate.
- **No blind retries.** Gating suites MUST NOT use automatic retry-on-failure. Bounded,
  explicit retry loops inside a test body (polling with an injected clock) are permitted; CI-
  level rerun-until-green is not.

#### Rationale

Build tags keep the quarantine in the tree — visible in review, greppable, attributable —
while excluding the test from gating compilation. The dated comment makes the time box
mechanically checkable; the non-gating lane preserves the diagnostic signal deletion
destroys.

#### Positive consequences

- Gating suites recover their signal within one pull request of detection.
- Every quarantine has an owner, an issue, a deadline, and a pass-history record.
- The 14-day check converts policy decay into an immediate, visible CI failure.

#### Negative consequences

- An expired quarantine blocks unrelated work until resolved; this is accepted as the forcing
  function, and deletion-with-justification is always available as the release valve.
- Tag-level quarantine of a file can exile neighboring healthy tests; mitigation: isolate the
  flaky test into its own file when quarantining.

#### Risks

- Quarantine becomes a graveyard (tests parked, deadline check circumvented by date edits);
  mitigation: date edits without issue-state change are flagged in review, and the quarantine
  lane's size is a published CI metric reviewed at phase gates (NFR-TEST-005).
- Deletion is overused under deadline pressure; mitigation: deletion requires a recorded
  justification and removes the test's traceability annotations, which the FR-TEST-002
  traceability check will surface as uncovered requirements.

#### Reversal plan

The mechanism is repository convention plus two small CI checks; replacing it (for example
with a CI-native quarantine feature) requires migrating tagged files and retiring the checks,
with no effect on shipped artifacts.

#### Review conditions

- If the quarantine lane holds more than 10 tests for two consecutive weeks, the determinism
  controls of FR-TEST-007 are re-audited — the intake rate, not the mechanism, is the problem.
- If Go's toolchain gains first-class test-quarantine support, re-evaluate the build-tag
  mechanism against it.
