### ADR-005 — CLI Framework: spf13/cobra with pflag

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: CLI, Configuration Manager
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation

#### Context

The `andromeda` executable exposes a substantial command surface (Volume 8): nested subcommands for sessions, agents, providers, tools, configuration, and diagnostics, plus global and per-command flags, shell completion, and generated help. The CLI is the primary non-interactive entry point and the launcher for the TUI (ADR-006). Go (ADR-001) offers three realistic approaches: the stdlib `flag` package, `urfave/cli`, or `spf13/cobra` with its companion flag library `spf13/pflag`. Separately, the cobra ecosystem is conventionally paired with `spf13/viper` for configuration — but Volume 10 specifies a custom Configuration Manager with strict TOML-only parsing (ADR-008), schema validation, deterministic precedence, and migration, which overlaps viper's territory.

#### Problem

Select the command-line parsing and command-tree framework for the `andromeda` executable, and explicitly bound its scope so that it does not absorb configuration responsibilities owned by the Configuration Manager.

#### Forces and constraints

- Volume 8 defines a deep command grammar: nested subcommands, aliases, global flags, per-command flags, and exit-code contracts per command.
- POSIX/GNU-style flags (`--flag`, `-f`, flag grouping) are the expected convention for a developer tool.
- Shell completion (bash, zsh, fish) and generated reference documentation are Volume 8 deliverables; hand-maintaining them invites drift from the implementation.
- Volume 10 owns configuration precedence (defaults < files < environment < flags); the CLI layer MUST feed resolved flag values into that pipeline rather than implement its own merging.
- Contribution accessibility: the framework should be one that Go contributors broadly already know.

#### Alternatives considered

| Criterion | cobra + pflag (selected) | urfave/cli | stdlib `flag` |
|---|---|---|---|
| Nested subcommand trees | Native, arbitrary depth | Supported | Hand-rolled dispatcher required |
| POSIX/GNU flag style | Yes (pflag) | Yes | No (single-dash only, no grouping) |
| Shell completions generated | bash/zsh/fish/powershell | Partial | None |
| Help/docs generation | Help, man pages, Markdown | Help | Minimal |
| Contributor familiarity | Very high in the Go CLI ecosystem | Moderate | Universal but low-level |
| Dependency surface | Two libraries | One library | Zero |
| Coupling risk | Conventionally paired with viper (avoidable) | Low | None |

- **stdlib `flag`** means writing and maintaining a bespoke subcommand dispatcher, completion generator, and help renderer — a framework of our own, which is the highest-cost option disguised as the lowest-dependency one. Its single-dash flag style also deviates from the conventions Volume 8 specifies.
- **`urfave/cli`** is a competent framework, but it offers weaker completion and documentation generation and materially lower contributor familiarity than cobra in the CLI-tool ecosystem Andromeda inhabits. Nothing it does better outweighs that.
- **cobra without pflag** is not meaningful; pflag is cobra's flag engine and supplies the POSIX/GNU behavior.

#### Decision

The CLI MUST be built on `github.com/spf13/cobra` with `github.com/spf13/pflag` for flag parsing. Scope is explicitly bounded:

1. cobra owns command-tree definition, argument/flag parsing, help text, completion generation, and documentation generation.
2. The Configuration Manager (Volume 10) owns all value resolution. Parsed flag values are handed to it as the highest-precedence layer; commands read resolved configuration, never raw flags, for any value that also exists in `andromeda.toml`.
3. `spf13/viper` MUST NOT be used. Configuration loading, merging, environment mapping (`ANDROMEDA_*`), validation, and migration are implemented by the custom Configuration Manager per Volume 10, using the ADR-008 TOML parser.
4. Exit codes follow the Volume 0 scheme; cobra's error handling is wrapped so that usage errors map to exit code 2.

Framework version and license are confirmed at dependency intake under the ADR-002 license policy.

#### Rationale

cobra + pflag is the only option that delivers the full Volume 8 surface — deep command trees, standard flag conventions, completions, and generated docs — without building a framework in-house. Its ubiquity among Go CLI tools maximizes contributor familiarity. The single genuine hazard of adopting cobra is drifting into viper by convention; this ADR closes that path explicitly because viper's implicit multi-source merging would compete with the deterministic precedence, strict validation, and migration semantics that Volume 10 requires the Configuration Manager to own.

#### Positive consequences

- The entire Volume 8 command grammar is expressible declaratively; completions and reference docs are generated from the same definitions, eliminating drift.
- Usage errors, help, and command discovery behave the way users of comparable developer tools already expect.
- Clean seam between parsing (cobra) and resolution (Configuration Manager) keeps configuration behavior testable in isolation.

#### Negative consequences

- Two third-party dependencies on the most user-visible code path.
- cobra's global-ish command objects and init-time registration style require project conventions to keep command definitions testable.
- Deliberately not using viper means implementing environment-variable mapping and layering ourselves — accepted, because Volume 10 requires custom semantics anyway.

#### Risks

- **Framework stagnation or breaking changes**: mitigated by pinning, and by confining cobra types to the CLI package boundary — command handlers receive plain resolved parameters, so the framework never leaks into the Runtime.
- **Convention pull toward viper** from contributors following cobra tutorials: mitigated by this ADR's explicit prohibition and a lint rule in CI rejecting the import.

#### Reversal plan

Because cobra types are confined to the CLI package and all behavior behind it consumes resolved configuration, replacing the framework (with urfave/cli or a bespoke dispatcher) is a re-mapping of command definitions, not a runtime change. The Volume 8 command grammar is the source of truth; a migration re-generates the same tree, help, and completions from it. Estimated blast radius: the CLI package and its golden-output tests only.

#### Review conditions

- cobra or pflag becomes unmaintained (no release or maintenance activity for more than 12 months) or adopts an incompatible license.
- Volume 8's command grammar evolves features the framework cannot express without forking.
- The prohibition on viper causes measurable duplicated effort that a scoped-down dependency could eliminate (re-evaluated only via a superseding ADR).
