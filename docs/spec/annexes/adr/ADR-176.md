### ADR-176 — Provider Test Doubles: Scripted Fakes First, Sanitized Recordings Second, Live Lanes Non-Gating

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Provider Layer, Agent Engine, Context Manager, Indexing Engine, Tool Runtime
- Related requirements: FR-TEST-006, RISK-TEST-004, RISK-TEST-005; SM-04, SM-10 (Volume 1, chapter 06)

#### Context

Much of Andromeda's behavior — agent loop, streaming, tool calling, accounting, error
normalization, routing — sits behind ProviderPort. Tests need providers that are
deterministic, offline, free, and able to exhibit rare conditions on demand; real provider
APIs are none of these. Meanwhile, adapter code parsing real HTTP responses must still be
exercised against realistic wire payloads, and genuine provider drift (SM-04, RISK-PRD-001)
must be detected somewhere.

#### Problem

How are provider-dependent behaviors tested so that gating suites are deterministic and
offline, adapter wire-parsing is exercised against realistic payloads, and real-world provider
drift is still detected — without real secrets in gating CI and without violating the offline
test discipline?

#### Forces and constraints

- Merge gates MUST be deterministic and offline (Volume 13, chapters 01 and 03); live APIs
  cannot participate in them.
- Fakes at the ProviderPort level cannot exercise adapter HTTP/parsing code at all.
- Recordings of real traffic can embed credentials and personal data; sanitization must be
  structural, not best-effort.
- Live testing requires credentials, spend, and rate-limit tolerance; it can never be a merge
  precondition without making third-party availability a merge dependency.

#### Alternatives considered

1. **Live-first testing.** Rejected: nondeterministic, secret-dependent, billed,
   offline-violating; a provider outage would block all merges.
2. **Recordings as the primary double.** Rejected as primary: cassettes ossify one historical
   response shape, cannot express scripted edge conditions, and every re-record is a
   sanitization risk. Retained as the secondary lane.
3. **ProviderPort-level fakes only.** Rejected as sole mechanism: leaves adapter HTTP,
   streaming, and error-mapping code untested until a live call happens.

#### Decision

Provider testing uses three lanes with fixed roles:

1. **Scripted fakes (primary, gating).** (a) An in-process fake implementing ProviderPort,
   driven by versioned scenario scripts (deterministic content, streaming chunk sequences with
   scripted pacing, tool-call sequences, usage figures, injected errors); used by all engine-
   and integration-level tests. (b) Local HTTP fake servers speaking the documented wire shapes
   of the MVP provider seed families (OpenAI-compatible, Anthropic, Ollama), bound to loopback,
   for adapter-level tests. Both kinds MUST themselves pass the ProviderPort contract kit.
2. **Sanitized recordings (secondary, gating for adapter parsing regression).** Recorded
   real interactions are captured with a structural sanitizer that replaces credentials,
   identifiers, and free-text content at record time; a scanner rejects any cassette containing
   secret-shaped material (E-TEST-002 family on divergence, secret scanning per Volume 11).
   Cassettes are re-recorded on a scheduled basis, never during merge-gate runs.
3. **Live lanes (scheduled, non-gating).** Scheduled CI jobs run the provider conformance
   suite against real providers using CI-held credentials, producing drift reports and issues —
   never blocking merges or releases by themselves. Findings feed adapter fixes and cassette
   re-recording.

#### Rationale

Each lane covers the gap the others leave: fakes give determinism and edge-condition
scripting; recordings keep parsers honest against real payloads without network; live lanes
detect drift where it happens. Kit-passing fakes bound fake/real divergence (RISK-TEST-004)
mechanically.

#### Positive consequences

- Merge gates are fully offline and deterministic while adapter code remains exercised.
- Edge and failure conditions (SM-10 reliability cases) are reproducible on demand.
- Provider drift surfaces as scheduled-lane findings with evidence, not as user bug reports.

#### Negative consequences

- Three lanes to maintain: scenario scripts, cassette re-recording, and live credentials
  rotation each carry upkeep cost.
- Non-gating live lanes mean a drift can land between scheduled runs; the exposure window is
  the lane's schedule interval.

#### Risks

- Sanitizer misses a novel secret shape; mitigation: deny-by-default structural sanitization
  (only allow-listed fields survive), plus repository-wide secret scanning over `testdata/`.
- Fake wire shapes drift from documented provider APIs; mitigation: cassette-based parsing
  tests and scheduled live conformance runs cross-check the fakes.

#### Reversal plan

All three lanes are test-only infrastructure. Any lane can be retired or replaced
independently; the ProviderPort contract kit and scenario-script format are the stable
interfaces, so replacing the fake implementation or the recording mechanism does not change
test call sites.

#### Review conditions

- If an official provider SDK is adopted for an adapter (ADR-019 PENDING VALIDATION path),
  re-evaluate whether that adapter's HTTP fake remains meaningful or should be replaced by
  SDK-level doubles.
- If live-lane drift findings exceed one confirmed drift per month per provider, shorten the
  lane's schedule interval for that provider.
