### ADR-108 — Keyboard model: non-modal fixed keymap with go-to chords at MVP; user remapping at Beta

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: TUI
- Related requirements: FR-TUI-003, FR-TUI-004; E-TUI-005

#### Context

The TUI is keyboard-first (mouse is enhancement-only per ADR-109). The keyboard model
determines learnability, speed, terminal compatibility, and testability. The candidate
population of conventions: modal editing (vim), chorded modifiers (emacs/IDE style),
single-key command sets with prefix chords (lazygit, k9s), and leader-key schemes. The
terminal constrains the space: key-release events and simultaneous non-modifier chords
are not delivered; function keys and exotic modifier combinations have unreliable escape
sequences across emulators and multiplexers (chapter 12 matrix); text inputs need
printable keys to insert text.

#### Problem

Choose the keyboard interaction model — modal or not, fixed or remappable, how screens
are addressed — balancing speed for daily users, zero-manual discoverability for new
users, and the terminal reality above.

#### Forces and constraints

- Approval prompts are safety-critical: the model must make accidental grants
  structurally impossible (no global key may double as a decision key).
- Discoverability: the status bar and help overlay can teach exactly one coherent scheme;
  two parallel schemes (e.g., full vim emulation plus defaults) double documentation and
  test surface.
- Golden tests script keystrokes; a fixed map keeps scripts stable across releases.
- Remapping is a real need (keyboard layouts, muscle memory, accessibility) but a
  validated remapping system (conflict detection, reserved keys) is meaningful scope.
- Chord timeouts must not swallow fast typists' input in text fields.

#### Alternatives considered

1. **Non-modal fixed map: global keys + contextual panel keys + `g`-prefix go-to chords;
   remapping deferred to Beta (selected).**
2. **Modal (vim-style normal/insert modes)** — rejected: modality's payoff is dense text
   *editing*, which the TUI does not do (no editor, Volume 1 out-of-scope); its cost —
   mode errors, the single most common vim-class user failure — lands on approval and
   interruption paths where errors are expensive; and every screen would need mode
   indicators to stay honest.
3. **Modifier-chord scheme (Ctrl/Alt combinations for everything)** — rejected: Alt is
   unreliable across terminals/macOS configurations, Ctrl collides with terminal and
   multiplexer bindings (Ctrl+B tmux prefix, Ctrl+S flow control), and dense Ctrl maps
   are undiscoverable without an IDE-style command surface.
4. **Remappable from MVP** — rejected: remapping without validation (conflicts, reserved
   decision keys, unreachable actions) is a support trap and a safety hole; validation is
   exactly the E-TUI-005 machinery scheduled for Beta. Shipping the fixed map first also
   gathers real friction data to inform the remapping design.

#### Decision

The keymap is **non-modal** with three resolution layers (overlay → focused panel →
global, FR-TUI-004): single printable keys for contextual actions outside text inputs,
`Ctrl`-modified keys for the small process-level set (interrupt, quit, suspend, palette),
and a `g`-prefix chord family for direct screen addressing with a 1.5 s timeout and a
hint toast. Inside text inputs, printable keys insert; only `Ctrl` globals, `Esc`, and
`Tab` bypass. Approval decision keys are digits local to the approval overlay, reserved
and never remappable. The global map and chord set are **fixed and closed at MVP**;
`[tui.keymap]` user remapping ships at **Beta** with validation (E-TUI-005) that rejects
conflicts, unreachable actions, and any rebinding of reserved decision keys. Vi-style
movement aliases (`j`/`k`) are included in the fixed map as aliases, not as a mode.

#### Rationale

The selected model matches the conventions of the adjacent tool population (lazygit, k9s
users transfer directly), needs no mode indicator, and keeps every safety-relevant key
either modified (`Ctrl+C`) or overlay-local (digits), so no navigation key can mutate
anything. The `g`-prefix chord gives O(1) screen access without burning the single-key
namespace or relying on fragile function keys. Deferring remapping converts a risky MVP
feature into a validated Beta feature with field data.

#### Positive consequences

- One scheme to document, teach in the status bar, and script in golden tests.
- Terminal-compatibility risk concentrated in a tiny `Ctrl` set chosen to avoid known
  collisions.
- Approval safety is structural (reserved, overlay-local decision keys).
- Text input never fights the keymap (insert-by-default in inputs).

#### Negative consequences

- Vim users do not get full modal navigation; only movement aliases.
- Chord timeout (1.5 s) is a felt latency for the hint path (mitigated: the chord fires
  immediately on the second key; the timeout only bounds abandonment).
- No personalization until Beta.

#### Risks

- Fixed-map friction for specific layouts (e.g., `[`/`]` on non-US keyboards); mitigated
  at MVP by palette and chords covering the same actions, and properly by Beta remapping.
- Chord prefix `g` colliding with a future contextual need on some screen; mitigated by
  the resolution order (panel layer sees keys first only for non-prefix keys; `g` is
  globally reserved).

#### Reversal plan

The keymap is a data table consumed by one resolver (FR-TUI-004). Introducing modality or
a different scheme replaces the resolver's state machine and the documentation/tests; no
screen logic changes (screens receive actions, not keys). The Beta remapping layer is
additive over the same table.

#### Review conditions

- Beta remapping ships: re-evaluate which parts of the fixed map become defaults versus
  reserved.
- Terminal-matrix testing (chapter 12) finds a chosen `Ctrl` binding colliding in a Tier
  1-relevant environment.
- Sustained user demand for a vim modal layer (tracked via issues) — would be a distinct
  opt-in ADR, never a silent addition.
