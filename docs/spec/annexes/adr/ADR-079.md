### ADR-079 — Andromeda Runtime Protocol 1.0: line-delimited JSON-RPC 2.0 with one-way authority

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Plugin Runtime, Extension SDK, Sandbox Engine, Tool Runtime
- Related requirements: FR-PLUG-001, FR-PLUG-002, FR-PLUG-003; INV-PLG-01..04 (Volume 2)

#### Context

ADR-009 fixed the plugin mechanism: subprocesses speaking JSON-RPC 2.0 over stdio, under the
name Andromeda Runtime Protocol (ARP), with message grammar, handshake, capability
declaration, lifecycle, and timeout/cancellation semantics delegated to Volume 6. This ADR
records the wire-level and authority-model decisions of ARP 1.0 as specified in Volume 6
chapter 08. Plugins are third-party code on a security boundary; the protocol shape
determines what a hostile or buggy plugin can even *express* toward the host.

#### Problem

Fix the concrete protocol contract: framing, frame limits, large-payload handling, version
negotiation shape, the method surface's direction of authority (may plugins call back into
the host?), and the reserved error codes — such that the protocol is implementable from its
documentation in any language, debuggable with text tools, and does not widen the plugin
attack surface beyond the subprocess boundary ADR-009 established.

#### Forces and constraints

- Language-agnosticism (ADR-009): framing must be implementable with a line reader and a JSON
  library — no length-prefix binary framing, no schema compilers.
- One mental model: MCP stdio servers (ADR-010) already use newline-delimited JSON-RPC on
  stdio; plugin authors frequently implement both.
- Security direction: every capability a plugin can request at runtime is a capability the
  host must gate at runtime; a callback surface (permissions, secrets, execution) would move
  policy enforcement into request handling on a hostile channel.
- JSON is a poor carrier for large binary payloads (ADR-009 negative consequence); tool
  results can be large.
- Streaming tool activity must reach the TUI live (PRD-008), so tool execution cannot be a
  single blocking request/response.

#### Alternatives considered

1. **Length-prefixed or HTTP-style framed JSON-RPC** — rejected: adds parser state machines
   to every SDK language for no capability gain; breaks the "debuggable with a pipe and a
   text editor" property; diverges from the MCP stdio convention plugin authors already know.
2. **Bidirectional authority** (plugin→host requests for permissions, secret resolution,
   nested tool calls) — rejected for 1.0: it converts the host's policy engine into an
   attack-reachable RPC service, makes every invocation's authority dynamic instead of
   recorded-at-grant, and complicates the audit chain (who asked for what, when, under which
   approval). Everything a 1.0 plugin legitimately needs travels *to* it: granted permissions
   ride in `arp.tool.execute` params, and manifest-declared credentials are delivered as
   sandbox-filtered environment entries at spawn.
3. **Inline binary payloads** (base64 blobs in frames) — rejected: memory amplification and
   frame-size blowups; out-of-band filesystem references inside sandbox-visible paths carry
   artifacts instead.
4. **Full semver protocol versions with capability flags** — rejected for 1.0: `MAJOR.MINOR`
   with a current+previous-minor support window is sufficient granularity for a protocol this
   small, and capability flags reintroduce the combinatorial testing burden the window rule
   avoids.

#### Decision

ARP 1.0 is fixed as specified in Volume 6 chapter 08:

1. **Framing** — JSON-RPC 2.0, UTF-8, one message per `\n`-terminated line, host→plugin on
   stdin, plugin→host on stdout; stderr is captured log output, never protocol. Maximum frame
   size 4 MiB; larger payloads pass out-of-band as filesystem references inside
   sandbox-visible paths.
2. **Versioning** — protocol versions are `MAJOR.MINOR` strings starting at `1.0`; the host
   supports the current and previous minor within a major; the handshake negotiates the
   highest mutual version, recorded per plugin (INV-PLG-02). Breaking changes require a major
   bump under SM-20/ADR-015 discipline.
3. **Handshake** — host-initiated `arp.initialize` (plugin silent on stdout until it
   arrives), manifest-consistency validation of the response, `arp.initialized` completion
   notification; one handshake per process lifetime.
4. **Authority is one-way** — plugin→host *requests* do not exist in 1.0; the plugin surface
   toward the host is limited to responses and the `arp.tool.event` / `arp.log`
   notifications. Any reverse-request surface is a future protocol minor with its own
   decision record.
5. **Reserved error codes** — `-32001` version unsupported, `-32002` unknown surface,
   `-32003` invocation unknown, `-32004` plugin busy, `-32005` shutting down; standard
   JSON-RPC codes keep their meanings. Tool-level failures are Tool Results (data), not
   protocol errors, mirroring ToolPort semantics.

#### Rationale

Every choice minimizes what a plugin author must build and what a hostile plugin can attempt.
Line-delimited framing is the simplest thing that works in every language and matches the MCP
convention; the 4 MiB cap plus out-of-band artifacts bounds memory without complicating the
common case; one-way authority keeps the permission model's enforcement point in the host's
pipeline (where it is tested and audited) instead of in a request handler facing untrusted
input. The tiny reserved-error vocabulary keeps failure semantics machine-checkable in the
conformance fixtures.

#### Positive consequences

- A conforming plugin is implementable from the specification with a line reader, a JSON
  library, and a loop — validated by the cross-language smoke plugin (FR-PLUG-001).
- Frames are human-readable and replayable; conformance fixtures double as documentation.
- The audit chain for plugin side effects is identical to built-in tools': grants are decided
  before dispatch and recorded on the invocation, never negotiated mid-flight.

#### Negative consequences

- No reverse channel means interactive plugin patterns (a plugin asking the user a question
  mid-execution) are inexpressible in 1.0; such flows must be modeled as tool results that
  the agent acts on.
- Per-line JSON parsing costs more CPU than binary framing for very chatty streams; bounded
  by NFR-PLUG-002 and the ADR-009 review condition on protocol overhead.
- Credentials delivered at spawn cannot be rotated into a running plugin; rotation takes
  effect at the next start.

#### Risks

- Frame-cap pressure: legitimate tools producing multi-megabyte structured results must
  spill to artifact references; SDK helpers make this the easy path, or authors will fight
  the cap.
- Embedded newlines in payload strings are legal JSON but must be escaped; a non-conforming
  writer corrupts framing — the conformance fixtures include this case (E-PLUG-003 class).

#### Reversal plan

ARP's method surface and lifecycle semantics, not its transport, are the stable contract
(ADR-009 reversal plan). Replacing line-delimited stdio with sockets or another framing
preserves method semantics and SDK-visible types; only the connection layer changes.
Introducing a reverse-request surface is additive (a protocol minor plus a superseding
authority-model decision), not a reversal.

#### Review conditions

- NFR-PLUG-002 breaches its minimum threshold in release benchmarks (shared trigger with
  ADR-009's protocol-performance review condition).
- Two or more credible plugin use cases require host callbacks that cannot be modeled as
  tool results: design the reverse surface as a protocol minor with its own ADR.
- MCP's stdio conventions change in a way that breaks the shared-mental-model rationale:
  re-evaluate alignment.
- The 4 MiB frame cap or the out-of-band artifact path proves wrong in field data (spillover
  rates, artifact-reference friction): retune via configuration and SDK helpers.
