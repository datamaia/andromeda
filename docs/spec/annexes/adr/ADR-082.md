### ADR-082 — Skill inheritance and composition: single parent, append-by-default, deterministic precedence, fail-closed conflicts

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Skill Engine, Prompt Engine, Agent Engine (consumer)
- Related requirements: FR-SKILL-002, FR-SKILL-003; INV-SKL-03 (Volume 2)

#### Context

Skills combine: a team skill refines an organization skill; several skills are active in one
session at once; a workspace wants to adjust a globally installed skill without forking it.
The brief mandates inheritance, composition, and overrides. Skills are prompt-bearing data
(ADR-078), so combination semantics directly shape what enters the model's context — an
auditability and reproducibility surface (SM-12), not merely a convenience feature. The Prompt
Engine (Volume 4) assembles prompts from named slots; skills contribute text into those slots.

#### Problem

Define how skills combine: how many parents a skill may extend, whether children replace or
extend parent content, in what order multiple active skills apply, and what happens when two
skills target the same thing — such that the combined result is deterministic, attributable
to specific skill versions, and never silently surprising.

#### Forces and constraints

- Reproducibility (SM-12): identical skill sets must produce identical prompt contributions;
  order-of-load or hash-map iteration order must never leak into behavior.
- Attribution: every line of skill-contributed prompt text must be traceable to one skill
  version (injection forensics, Volume 9 threat model).
- Requirement honesty (INV-SKL-03): combining skills must never hide a requirement — a
  composed set's requirements are at least the union of its members'.
- Authoring simplicity: skill authors are not programmers by assumption; multiple inheritance
  and resolution orders (MRO) are expert-level semantics.
- Overrides are necessary (adjusting a parent's checklist) but are also the mechanism by
  which one skill could silently neutralize another's guardrails.

#### Alternatives considered

1. **Multiple inheritance with linearization** (C3/MRO-style) — rejected: resolution-order
   subtleties are notorious even among programmers; diamond patterns would make prompt
   provenance genuinely hard to explain in a composition report.
2. **No inheritance, composition only** — rejected: the refine-a-base-skill case would force
   copy-forking, losing upstream updates and multiplying near-duplicate skills — the problem
   inheritance exists to solve.
3. **Implicit override-by-precedence** (higher-precedence skill silently replaces same-named
   contributions) — rejected: silent replacement is exactly the guardrail-neutralization
   attack shape; replacement must be declared by the author and visible in the report.
4. **Runtime-conditional composition** (skills activating or suppressing each other by
   runtime predicates) — rejected for the format's current major: it would make the composed
   result depend on runtime state, breaking the static determinism that activation records
   rely on.

#### Decision

As specified in Volume 6 chapter 07:

1. **Inheritance** is single-parent (`extends.parent`, semver range allowed), bounded at
   depth 5, cycle-checked. Children inherit prompt documents and requirement declarations;
   child declarations **append by default**. Replacing a parent prompt document requires
   `overrides.prompts = true` *and* explicit naming of each replaced document.
2. **Composition** of multiple active skills is deterministic: workspace scope over global
   over builtin; within a scope, explicit activation order; ties broken by name. Prompt
   contributions concatenate in precedence order into their Prompt Engine slots; requirement
   sets union.
3. **Conflicts fail closed**: two skills replacing the same prompt document or declaring the
   same named artifact output are a composition conflict (E-SKILL-005) that the engine never
   auto-resolves; the user chooses.
4. Every composition emits a **composition report** — every contribution, its source skill
   and version, every override applied, and the set hash — attached to the activation record
   and the run record.
5. Trust does not launder through composition: the composed set's trust classification is the
   minimum of its members' for approval purposes, and the union rule means composition can
   never *reduce* the requirement set a prompt references.

#### Rationale

Single-parent chains with append-by-default keep the mental model at "a child adds to its
parent unless it explicitly says otherwise" — explainable in one sentence and mechanically
reportable. Deterministic precedence makes the composed result a pure function of the
activation set, which is what the set hash certifies and what replay needs. Fail-closed
conflicts convert the most dangerous interaction (two skills fighting over the same target)
from a silent precedence outcome into a user decision with both parties named.

#### Positive consequences

- Composition reports give injection forensics and behavior debugging a complete, attributed
  picture of what skill text entered a run.
- Deterministic set hashes make skill configuration drift detectable across machines and
  sessions.
- The minimum-trust rule prevents a high-trust wrapper from smuggling a low-trust dependency
  past activation policy.

#### Negative consequences

- Depth-5 single-parent chains cannot express genuine multi-base combinations; authors must
  restructure as composition (activating both bases) instead.
- Fail-closed conflicts interrupt users where a lenient engine would proceed; conflict UX
  quality becomes load-bearing.
- Append-by-default can produce long concatenated prompts; token budgeting for skill slots is
  delegated to the Context Manager's budgeting (Volume 7).

#### Risks

- Authors may circumvent conflict detection by trivially renaming targets, composing
  semantically contradictory instructions the engine cannot see; mitigated by the report
  making all contributions reviewable, and by golden tests (FR-SKILL-004) at author time.
- Precedence surprise (workspace silently outranking global) if users forget scope order;
  mitigated by the report and by listings showing effective precedence.

#### Reversal plan

Composition semantics version with `format_version` (ADR-078). Loosening (e.g., permitting
declared multi-parent inheritance) is an additive format minor with resolver extension;
tightening is a major. The composition report schema is additive-evolution per Volume 2
serialization rules, so tooling built on reports survives semantic revisions.

#### Review conditions

- Real skill populations show depth-5 or single-parent as a routine limitation (register
  feedback, issue signals): design multi-parent or deeper chains with an explicit
  linearization ADR.
- Conflict-interruption frequency in field data is high enough to indicate the model is
  fighting authors rather than protecting users: revisit conflict classes.
- The Prompt Engine's slot model (Volume 4) changes shape: re-verify concatenation semantics
  against the new slots.
