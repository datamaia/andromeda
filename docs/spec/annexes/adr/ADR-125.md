### ADR-125 — Fail-closed security decision path: evaluation failure denies, audit failure blocks

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Permission Manager, Audit Log, Sandbox Engine, Secret Store
- Related requirements: FR-SEC-100, FR-SEC-105, FR-SEC-111, FR-SEC-109

#### Context

The security subsystems — permission evaluation (Volume 9 chapter 05), audit recording
(chapter 08), sandbox policy application (chapter 06), and redaction (chapter 07) — sit on
every side-effecting path. Each can itself fail: storage errors during grant reads,
unwritable audit databases, malformed stored selectors, redaction-layer faults. Something
must be specified for the failure of the enforcement machinery itself, or implementers will
decide ad hoc under deadline pressure.

#### Problem

When a security mechanism cannot complete — not "the answer is no" but "the question could
not be answered" — does the guarded action proceed?

#### Forces and constraints

- Volume 0 precedence order: safety of user, system, and credentials first; implementation
  convenience last.
- PRD-005: no side effects without decided permissions "under any configuration, in any
  mode" — an availability-shaped exception would hollow the guarantee.
- INV-AUD-03: every catalogued action produces exactly one Audit Record; an action that
  proceeds unrecorded violates the invariant permanently (the record cannot be
  reconstructed honestly later).
- Fail-closed converts security-subsystem outages into product outages — a real usability
  cost that must be bounded by recovery paths.
- SM-16(b)'s 100% mediation target admits no "except when the checker was down" asterisk.

#### Alternatives considered

1. **Fail-open with logging** (proceed on evaluation failure, log loudly) — rejected:
   creates an attack recipe (induce evaluation failure, act freely); the log entry may
   itself be lost if logging shares the failing resource.
2. **Fail-open for "low-risk" permissions only** (e.g., `read`) — rejected: requires a
   risk taxonomy orthogonal to the granted model, and `read` is precisely the exfiltration
   permission (threat model chapter 02).
3. **Queue-and-wait** (block until the subsystem recovers, indefinitely) — rejected as the
   general rule: unbounded blocking violates context-cancellation discipline (FR-ARCH-004);
   bounded retry followed by failure is the compatible shape.
4. **Fail-closed with bounded retry and explicit errors** (chosen).

#### Decision

The security decision path fails closed, uniformly: (1) permission evaluation that cannot
complete resolves as deny with E-SEC-002 after one bounded retry for transient store errors;
(2) an audited action whose Audit Record cannot be appended does not execute — E-SEC-014
after one writer-level retry; (3) sandbox policy that cannot be applied (including empty
policy intersections and unavailable required isolation under `refuse`) does not launch;
(4) redaction that cannot be applied withholds or marks the payload, never emitting raw.
Every fail-closed outcome is an explicit, coded error — never a silent no-op — and repeated
occurrences are incident-response triggers (chapter 08).

#### Rationale

Fail-closed is the only behavior consistent with the precedence order and with PRD-005's
"under any configuration" wording: guarantees that dissolve under partial failure are not
guarantees. The bounded-retry-then-error shape respects FR-ARCH-004 cancellation discipline
while giving transient faults (a locked database, a momentary D-Bus hiccup) one chance to
clear. Making each outcome a distinct coded error keeps the availability cost diagnosable
and keeps SM-16(b) honestly measurable.

#### Positive consequences

- The mediation guarantee holds under fault injection, not just on the happy path —
  directly testable (NFR-SEC-002 probes include induced subsystem failures).
- Attackers gain nothing by degrading the security subsystems; degradation is containment.
- Incident triggers on repetition convert chronic enforcement failures into visible events.

#### Negative consequences

- Security-subsystem outages (disk full stopping audit writes) halt side-effecting work
  until resolved — deliberate, but users experience it as breakage; `andromeda doctor`
  must diagnose all four failure classes (recovery-path duty).
- Read-only, non-catalogued operations continue during such outages, which users may find
  inconsistent; the audited-action catalog (chapter 08) is the explicit boundary.

#### Risks

- Overly broad interpretation could fail-close paths that are not security decisions
  (e.g., telemetry), degrading availability without safety gain; mitigated by scoping this
  ADR to the four named mechanisms — telemetry failure explicitly never fails operations
  (Volume 3 TelemetryPort rule).
- Retry storms against a struggling store; mitigated by the single-retry bound.

#### Reversal plan

The rule is enforced at four choke points (evaluation resolver, audit writer, sandbox
`Prepare`/`ApplyPolicy`, redaction function). Relaxing any of them is a MAJOR document
change (Volume 0 chapter 10) because it weakens a product guarantee; the mechanical change
itself is local to the choke point.

#### Review conditions

- If field data shows fail-closed audit blocking is a dominant failure mode, review storage
  hardening (reserved space for audit writes) before reviewing the rule itself.
- Review when headless/enterprise deployments (ADR-032 evolution) demand different
  availability trade-offs — any relaxation requires a superseding ADR.
