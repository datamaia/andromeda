### ADR-111 — Accessible output mode as a first-class linear rendering profile

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: TUI
- Related requirements: FR-TUI-065, FR-TUI-066, NFR-TUI-069

#### Context

The brief requires accessibility, including "screen readers where viable". Full-screen TUIs
are structurally hostile to terminal screen readers: they repaint arbitrary screen regions
via cursor addressing, so assistive technologies cannot compute a meaningful reading order
or detect what changed. Line-oriented terminal output, by contrast, is the case terminal
screen readers handle: appended lines are announced as they arrive. Terminals expose no
per-application accessibility tree for TUI content, and no standard environment variable
announces that assistive technology is active.

#### Problem

How does an interactive, panel-based TUI remain usable with screen readers and other
assistive setups without forking the product or pretending the panel layout is readable?

#### Forces and constraints

- Reading order and change announcement are the blocking problems; colors and glyphs are
  already handled by the no-color and ASCII tiers (ADR-026, ADR-112).
- CLI non-interactive mode exists (FR-CLI-009) but is not an interactivity substitute: the
  product promise includes interactive sessions.
- Assistive technology cannot be auto-detected from a terminal; activation must be explicit.
- The mode must not change capability: every action available in panel rendering must be
  available in linear rendering (registry-driven, ADR-110).

#### Alternatives considered

1. **A distinct linear rendering profile (selected).** Same session model and action
   registry, rendered as an append-only transcript with plain-text prompts.
2. **Point screen-reader users at the CLI.** Rejected: denies interactive parity and
   contradicts PRD-009's spirit — automation parity is not an accessibility answer.
3. **Platform accessibility API integration** (VoiceOver/AT-SPI trees). Rejected as not
   viable: terminal emulators do not expose application-level widget trees for the hosted
   program; Andromeda has no surface to attach one to.
4. **Retrofit announcements into the panel renderer** (status-line hints). Rejected: does
   not fix reading order, the structural problem.

#### Decision

The TUI implements an **accessible output mode**: a rendering profile that replaces panel
composition with linear, append-only output on the primary screen (no alternate screen, no
cursor-addressed repaints), renders prompts as numbered plain-text questions, tables as
`label: value` lines, and announces state transitions as text lines. It is activated
explicitly — `tui.accessible_output` configuration key, `--accessible` flag — never by
detection. Capability parity is registry-enforced: every action is reachable by typed
command through the palette's linear form. Whether this mode is *sufficient* for specific
screen readers (VoiceOver with Terminal.app, Orca with GNOME Terminal) is **PENDING
VALIDATION** through testing with real assistive technology; the mode's rendering contract
(FR-TUI-065) is normative regardless, because linear output also serves braille displays,
high-magnification use, and transcript-based workflows.

#### Rationale

Linear rendering attacks the two structural failures (reading order, change detection)
instead of decorating an unreadable layout. Making it a profile of the same model — not a
second application — keeps behavior, permissions, and observability identical, and the
ADR-110 registry makes the parity claim checkable.

#### Positive consequences

- A defined, testable accessibility answer instead of an aspiration.
- The linear renderer doubles as a debugging/transcript view and degrades gracefully over
  slow links.
- No capability fork: one model, two renderers.

#### Negative consequences

- A second renderer to maintain and test (golden tests double for shared screens).
- Dense visual affordances (side-by-side diffs) need designed linear equivalents, which is
  real specification and implementation work.

#### Risks

- The mode ships but real screen-reader sessions remain impractical (verbosity, emulator
  quirks) — RISK-TUI-071. Mitigation: the PENDING VALIDATION test pass with named AT
  configurations at Beta; findings feed rendering adjustments.
- Feature drift: new screens implemented only for panel rendering. Mitigation: registry
  parity check plus linear golden tests as a required suite.

#### Reversal plan

If validation shows a different mechanism serves users better, the linear renderer remains
useful as a transcript view; the accessibility commitment would be re-specified by
superseding this ADR rather than silently dropping the mode.

#### Review conditions

- Completion of the assistive-technology validation pass (Beta phase gate).
- Terminal ecosystem changes (emulators exposing accessibility trees or a standard AT
  signal) that enable a stronger integration.
