### ADR-114 — Terminal support tiers and capability probing: probe, degrade, never crash

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: TUI, CLI, Platform Abstraction Layer (PAL)
- Related requirements: FR-TUI-068, NFR-TUI-070, FR-TUI-067

#### Context

Volume 3 (chapter 07) names the reference terminal emulators — Terminal.app, iTerm2, GNOME
Terminal, Konsole, Alacritty, kitty, WezTerm — and makes tmux and GNU screen first-class
multiplexer targets, with capability tiers "probed, never assumed". Capabilities that vary
across this population include color depth, glyph rendering, mouse reporting, bracketed
paste, OSC 52, and focus events. terminfo describes much of this but has known gaps (no
standard truecolor capability entry; multiplexers re-advertise their own `TERM`), and the
per-emulator facts are external and drift with emulator releases.

#### Problem

Define what "supported terminal" means for release gating, and fix the runtime order in
which capability signals are consulted, so behavior on any terminal — including unknown
ones — is deterministic and degraded rather than broken.

#### Forces and constraints

- Release gates need a finite, testable terminal set; "works everywhere" is not a gate.
- Users run terminals outside any reference list; refusing to start there is unacceptable
  for a terminal-first product (PRD-008, PRD-011).
- Per-emulator capability claims cannot be asserted from documentation memory; they must
  be verified by the Volume 13 compatibility suite and are PENDING VALIDATION until then.
- ADR-103 already fixes the CLI's color on/off order; the TUI needs the fuller ladder
  (tier, glyphs, mouse, paste, clipboard) without contradicting it.

#### Alternatives considered

1. **Two support tiers plus a fixed probing order with conservative baseline (selected).**
2. **Hardcoded per-emulator capability database** — rejected: drifts with every emulator
   release, misidentifies forks and multiplexed sessions, and duplicates terminfo badly.
3. **Trust terminfo exclusively** — rejected: truecolor and OSC 52 are not reliably
   described; multiplexers mask the outer terminal.
4. **Interactive capability round-trip probing at startup** (query sequences) — rejected
   as the primary mechanism: adds startup latency and hangs on non-answering terminals;
   MAY be adopted later for specific capabilities under a timeout, via review.

#### Decision

1. **Support tiers.** *Tier A* — the seven Volume 3 reference emulators plus tmux and GNU
   screen, on Tier 1 platforms: the compatibility suite (NFR-TUI-070) and golden-frame
   tests gate releases on them. *Tier B* — any terminal with a resolvable `TERM`: the
   degradation ladder applies, defects are triaged but do not gate releases.
2. **Probing order** (per capability, first decisive signal wins): explicit configuration
   (`[tui]`/`[tui.theme]` keys) → Andromeda environment controls and ecosystem conventions
   (aligned with ADR-103's order for color) → terminfo for the advertised `TERM` →
   conservative baseline. The baseline is 16-color, `ascii` glyph set, no mouse, no OSC 52,
   no focus events — the profile every supported terminal can render.
3. **Unknown or hostile environments never crash the TUI**: an unresolvable or `dumb`
   `TERM` falls to the CLI path (E-CLI-005 semantics for interactive screens); everything
   else gets the baseline. Degradations are observable (`tui.render_profile.changed`,
   `doctor` output), never silent — matching the PAL's explicit-degradation rule.
4. **The compatibility matrix** (chapter 12) records the *expected* per-terminal profile;
   every cell is PENDING VALIDATION until the Volume 13 suite has verified it on that
   emulator, and the matrix is re-verified per release on Tier A.

#### Rationale

Tiers make the release gate finite while keeping the long tail functional through the
baseline. The probing order puts intent above convention above description above guesswork
— the same philosophy as ADR-103, extended to the TUI's larger capability set. Treating the
matrix as claims-to-verify rather than facts respects the assumptions policy: emulator
behavior is external information.

#### Positive consequences

- A finite, automatable release gate; a deterministic story for every other terminal.
- One documented ladder explains any rendering outcome; `doctor` can print the resolved
  profile and the deciding signal per capability.
- No startup latency spent on round-trip probing.

#### Negative consequences

- Conservative baseline means capable-but-unrecognized terminals start degraded until the
  user configures overrides.
- Maintaining Tier A across emulator releases is recurring CI cost (macOS and Linux
  runners with real emulators or their headless equivalents).

#### Risks

- terminfo entries misdescribe a Tier A terminal, shipping a wrong default profile.
  Mitigation: the compatibility suite compares probed profile to expected matrix cell and
  fails the gate on mismatch.
- Multiplexer indirection (outer terminal ≠ inner `TERM`) yields under-featured profiles.
  Mitigation: tmux/screen are Tier A themselves; documented passthrough settings; explicit
  overrides.

#### Reversal plan

Tier membership and the ladder are specification tables plus one resolution function.
Adding round-trip probing later inserts one rung between conventions and terminfo without
changing consumers. Tier changes are register-recorded specification amendments.

#### Review conditions

- Compatibility-suite evidence that the baseline is wrong for a material share of Tier B
  terminals.
- Emergence of a reliable, low-latency capability query protocol across the Tier A set.
- Windows Terminal/ConPTY entering scope in the v2 phase (new tier rows).
