### ADR-061 — Two-layer error normalization with raw-identity preservation

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Provider Layer adapters, Provider Router, Logging, Observability
- Related requirements: FR-PROV-050, FR-PROV-002, FR-PROV-040

#### Context

ADR-016 fixes error identity (`E-<AREA>-NNN`) and the fourteen-field envelope; Volume 3
port rule 2 forbids raw driver/HTTP errors from crossing ports. Provider failures originate
in heterogeneous wire conditions (statuses, documented error-body schemas, transport
failures) and locally (pacing, breakers, routing). Retries, breakers, and fallback all key
off failure classification, and misclassification amplifies failures.

#### Problem

Who classifies provider failures and on what evidence — a single flat mapping in each
adapter, classification in consumers, or a layered pipeline — and how is provider-specific
diagnostic detail kept without leaking vendor types or unredacted content?

#### Forces and constraints

- Retryability must be uniform per code across all adapters (ADR-059 keys on it).
- Locally generated failures (E-PROV-015, E-PROV-016, pacing) have no wire origin.
- Diagnostics need the provider's raw error identity; security needs bounded, redacted
  excerpts (Volume 9).
- Unmapped conditions must degrade to defined codes, never to undefined behavior
  (RISK-PROV-001 containment).

#### Alternatives considered

1. **Consumer-side classification** — rejected: moves wire knowledge above the port,
   violating Principle 1 and duplicating logic per consumer.
2. **Single adapter-layer mapping producing final errors** — rejected: adapters cannot know
   routing context (attempt numbers, candidates skipped, fallback disposition), so
   envelopes would be incomplete or router logic would leak downward.
3. **Two-layer pipeline: adapter maps, router enriches and decides** — chosen.

#### Decision

Adapters map wire conditions to normalized E-PROV codes via their declared `ErrorMapping`
rules (validated at registration, FR-PROV-002), preserving the provider's documented error
identity and a redacted excerpt bounded at 2048 bytes in safe technical detail; unmapped
conditions normalize to E-PROV-008 (undocumented shape) or E-PROV-012 (documented but
unclassified). The router mints locally generated codes, enriches envelopes with routing
context, and derives retryability, breaker eligibility, and fallback classes exclusively
from the chapter 06 matrix — never from raw wire conditions.

#### Rationale

Each layer classifies what only it can see: adapters know wire schemas, the router knows
policy context. Totality (every condition lands on a defined code) turns provider drift
into a measurable signal (E-PROV-008 rates) instead of undefined behavior.

#### Positive consequences

- Uniform retry/breaker/fallback semantics per code across all adapters.
- Lossless-enough diagnostics with definition-time redaction (ADR-016 spirit).
- Leak checks are mechanical: no raw type may cross the port.

#### Negative consequences

- Two mapping artifacts to maintain per adapter (wire map + matrix row references).
- Bounded excerpts occasionally truncate long provider diagnostics.

#### Risks

- Wrong mapping rules misclassify failures; mitigation: registration-time validation,
  fault-injection coverage of every mapping row (NFR-PROV-004).

#### Reversal plan

The pipeline is internal to the Provider Layer; collapsing to one layer or re-splitting
changes no consumer surface. Code meanings are public contract (ADR-016 reversal rules
apply to any remapping).

#### Review conditions

- Review when E-PROV-008 rates for any adapter exceed its fault-injection baseline in
  production telemetry (consented) — the signal that mapping rules lag the provider.
- Review the excerpt bound if real diagnostics repeatedly truncate below usefulness.
