### ADR-044 — Prompt template format: restricted Go text/template with a versioned slot registry

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Prompt Engine, Skill Engine, Configuration Manager
- Related requirements: FR-AGT-013, FR-AGT-014; NFR-AGT-003; PRD-006, PRD-007

#### Context

Prompts are versioned internal assets (Volume 1) that skills and users extend (Principle
6), and run records reference them for reproducibility (SM-12). The engine needs a template
format, a composition mechanism, and a registry model — in Go (ADR-001), with deterministic
rendering as a hard requirement.

#### Problem

Which template technology and composition model give versioned, deterministic,
contribution-friendly prompt rendering without opening an injection or exfiltration
surface?

#### Forces and constraints

- Determinism: byte-identical output for identical inputs (NFR-AGT-003) — no clock,
  environment, filesystem, or randomness access from templates.
- Composition: profiles, skills, and overrides must combine without string concatenation.
- Trust: workspace-provided templates are untrusted code-adjacent assets (Volume 9).
- No new runtime dependency for a core path if the standard library suffices (ADR-001
  spirit).

#### Alternatives considered

1. **A custom template language** — rejected: design and tooling cost with no capability
   gain over a restricted stdlib engine; contributors would learn a bespoke syntax.
2. **Third-party template engines** — rejected: adds a dependency at a security-sensitive
   core path; sandboxing guarantees would rest on external maintenance.
3. **Sprintf-style composition in code** — rejected: makes prompts code changes, killing
   the override/skill contribution surface (Principle 6).
4. **Go `text/template`, restricted, with declared slots and a versioned registry** —
   chosen.

#### Decision

Templates use Go `text/template` syntax with a closed function allowlist excluding all I/O,
time, environment, and randomness; each template declares identity
`(namespace, name, semver)`, typed slots, and a parameter schema; composition happens only
through declared slots; the registry resolves `workspace` > `global` > `builtin` with
trust-gated override loading and skill contributions confined to `skill/<name>` namespaces
(FR-AGT-013). Rendering is pure and provenance-recorded (FR-AGT-014).

#### Rationale

The stdlib engine is deterministic once its function set is closed, universally familiar to
Go contributors, and dependency-free; slots turn composition into a checked interface
instead of concatenation conventions; SemVer per template lets records pin behavior
exactly.

#### Positive consequences

- Determinism is enforceable by construction (allowlist), not by review.
- Skills and overrides extend prompts through a typed, auditable surface.
- Zero new dependencies on the core rendering path.

#### Negative consequences

- `text/template` logic is deliberately limited; complex prompt logic must live in engine
  code or slot preparation.
- The allowlist must be maintained as the stdlib evolves.

#### Risks

- Overrides as an injection vector — mitigated by trust gates, off-by-default workspace
  overrides, and provenance (RISK-AGT-004).

#### Reversal plan

Templates are declarative assets; migrating to another engine is a mechanical translation
plus re-issuing golden renders, versioned as template major bumps under SM-20 rules once
the format is public.

#### Review conditions

Review when the skill format (Volume 6) publishes templates as a public contract surface,
and if slot expressiveness proves insufficient for skill composition in practice.
