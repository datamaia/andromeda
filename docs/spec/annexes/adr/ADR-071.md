### ADR-071 — Tool output handling: bounded inline results with artifact spillover

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Tool Runtime, Terminal Engine, Context Manager, Persistence Layer
- Related requirements: FR-TOOL-001, FR-TOOL-006; INV-TRES-04 (Volume 2)

#### Context

Tool Results are persisted JSON payloads inside the run record stream (Volume 2, chapter 04),
are delivered to models as context, and — for plugin and MCP tools — cross a JSON-RPC process
boundary that handles large binary blobs poorly (ADR-009 negative consequence). Filesystem
reads, terminal captures, and HTTP responses routinely produce megabytes; model context
windows and the workspace database are both finite. Volume 2 already gives Tool Result a
`truncated` flag and a `spillover_artifact_id`, and gives Command Execution capture artifact
references, but the size policy and its mechanics are Volume 6's to fix.

#### Problem

Decide how tool output that exceeds practical inline size is represented so that no output is
silently lost, records stay bounded, models receive usable content, and the full data remains
retrievable and attributable.

#### Forces and constraints

- Precedence order: data integrity and recoverability rank above performance and convenience.
- SM-13 traceability: every side effect and its evidence must resolve to persisted records.
- Model context budgeting (Volume 7) needs predictable upper bounds on tool payloads.
- JSON-RPC framing (ADR-009/ADR-010) makes multi-megabyte inline payloads a latency and memory
  hazard.
- Truncation that looks like completion corrupts agent reasoning.

#### Alternatives considered

1. **Unbounded inline payloads** — rejected: unbounded database rows, unbounded IPC frames,
   and context assembly failures on large outputs.
2. **Hard truncation without spillover** — rejected: destroys data the user may need
   (integrity precedence); the agent cannot recover the lost remainder.
3. **Always-spill (every output an artifact)** — rejected: one artifact per invocation
   multiplies filesystem churn for the common small-output case and slows the hot path.

#### Decision

Tool output is delivered and persisted inline up to an **effective output cap**: the lesser of
the tool's declared `max_output_bytes` and the `tools.max_output_bytes` configuration key
(default 1 MiB). Output beyond the cap is written to an Artifact (`kind = tool_output`,
content-addressed per Volume 2), the inline payload carries the leading portion plus
`truncated = true`, `payload_size_bytes` (untruncated size), and `spillover_artifact_id`, and
the runtime emits `tool.output.truncated`. Truncation is always marked, never silent; the
terminal counterpart applies the same rule to stdout/stderr captures via Artifact references.
Binary content is never inlined into JSON payloads: tools return binary data as artifact
references regardless of size.

#### Rationale

The two-tier shape keeps the hot path allocation-light for the overwhelmingly common small
outputs while guaranteeing that nothing observable is destroyed: the full content survives as
a content-addressed, attributable Artifact, and the record explicitly says both that and how
much was withheld inline. Making binary-as-reference unconditional keeps every JSON surface
(records, ARP, MCP bridging, exports) text-safe.

#### Positive consequences

- Bounded rows, frames, and context items with a single knob per tool and per installation.
- Agents can detect truncation and choose to read the artifact selectively (e.g., ranged
  `fs.read` against the artifact path).
- One rule serves ToolPort results and TerminalPort captures.

#### Negative consequences

- Two retrieval paths (inline and artifact) that consumers must both handle.
- Spilled artifacts consume disk under retention policy (Volume 10) and add a cleanup surface.

#### Risks

- A tool could emit pathological output faster than the spill path absorbs it; mitigated by
  the resource limits of FR-TOOL-006 (output-rate accounting counts toward the cap and breach
  terminates the invocation with E-TOOL-009).
- Agents ignoring `truncated` reason over partial data; mitigated by the flag being part of
  the payload the model sees, with SDK fixtures asserting its presence.

#### Reversal plan

The cap and spillover are runtime policy around an unchanged Tool Result shape (Volume 2
already defines the fields). Raising caps, adding streaming retrieval, or moving spillover to
a different store are configuration or adapter changes that do not alter the contract; full
reversal to unbounded inline would only require ignoring the cap, which the review conditions
gate.

#### Review conditions

- Volume 12 benchmarks show the 1 MiB default materially truncating common workflows.
- Volume 7 context budgeting wants a lower or model-dependent inline bound.
- A binary-capable transport (post-v1) changes the binary-as-reference rule's cost basis.
