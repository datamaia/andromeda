### ADR-137 — Event envelope: persist-then-publish with a closed compiled registry and per-family overflow policy

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Event Bus, Persistence Layer, Observability, all producer components
- Related requirements: FR-OBS-001, FR-OBS-005, FR-OBS-006; Volume 2 INV-EVT-01..05; ADR-012

#### Context

ADR-012 fixed the transport (in-process typed channel bus, at-most-once, bounded buffers)
and deferred the envelope, ordering, delivery semantics, and per-family overflow policy to
Volume 10. Volume 2 fixed the persisted Event entity and its invariants (registered names,
immutability, run correlation, redaction, no suppression in quiet modes). Nine volumes mint
event names concurrently against whatever contract this decision sets.

#### Problem

Decide (a) how durability relates to bus delivery, (b) how event names and payload schemas
are governed, and (c) which overflow policy applies to which traffic — such that the
activity record is complete (PRD-006), the TUI never stalls (SM-07), and the event space
stays reviewable as the corpus and extensions grow.

#### Forces and constraints

- The bus is deliberately lossy under pressure (ADR-012); the product promise of a complete
  record (SM-12/SM-13) is absolute — the two must not depend on each other.
- Volume 2's write discipline already couples state transitions with record appends
  transactionally; events describing transitions can ride the same transaction for free.
- Extensions (plugins, MCP servers) must be observable without gaining the power to mint
  arbitrary names into a namespace users and tooling trust.
- Payload schemas need evolution without breaking consumers (SM-20 counts the event
  envelope among public contracts).

#### Alternatives considered

1. **Publish-then-persist (bus as source of truth, async writer)** — a slow writer or a
   crash between publish and persist loses record rows; the durable trail would inherit the
   bus's loss profile. Rejected: violates PRD-006/PRD-010 precedence of integrity over
   latency.
2. **Durable bus (persistent queue with acknowledgments)** — turns the in-process bus into
   a broker with delivery state, contradicting ADR-012's zero-dependency channel decision
   and adding redelivery semantics no consumer needs (durable consumers can read the
   database). Rejected.
3. **Runtime-extensible event registry (extensions register names dynamically)** —
   uncontrolled name growth, unreviewable payload schemas, and a supply-chain surface where
   a malicious extension pollutes the trusted activity record. Rejected in favor of a closed
   compiled registry with extension activity surfaced under Volume 6-minted names.
4. **Single overflow policy for all traffic** — either everything blocks (TUI stalls) or
   everything drops (progress coalescing impossible, security signals lost silently).
   Rejected in favor of per-family policy.

#### Decision

1. **Persist-then-publish**: `lifecycle`/`action` events append transactionally with the
   writes they describe; bus publication follows commit and is a latency optimization, not
   the record.
2. **Closed compiled registry**: every name is declared in code with family, payload JSON
   Schema, required correlation, privacy class, retention class, producer, and documented
   consumers; unregistered emission is rejected (E-OBS-002).
3. **Per-family overflow policy**: `drop_oldest` (lifecycle, action, telemetry), `coalesce`
   (progress), bounded-`block` 100 ms then drop (security) — fixed in Volume 10 chapter 04.
4. **Payload evolution**: additive within a `schema_version`; breaking changes bump the
   version; envelope-shape changes follow SM-20.

#### Rationale

Coupling durability to the existing transactional write discipline makes the record's
completeness exactly as strong as the database's — one guarantee instead of two. A lossy
bus on top is then honestly what it is: a live view. The compiled registry makes the event
space a reviewed artifact (like the metric registry and the error catalog), and per-family
policy spends loss where loss is cheap.

#### Positive consequences

- Record completeness is testable via crash injection at one seam (transaction boundary).
- Consumers have one recovery story: re-sync from persisted rows.
- The consolidated event catalog can be generated from the registry mechanically.

#### Negative consequences

- Producers of lifecycle events pay a database write inside the hot transaction (budgeted
  by Volume 12).
- New events require code changes and registry review — heavier than ad-hoc emission.
- Live consumers must tolerate gaps by design.

#### Risks

- Registry review becoming a bottleneck: mitigated by area-owner autonomy (each volume
  mints its own names) with mechanical grammar/schema checks.
- Transaction bloat from oversized payloads: mitigated by the 256 KiB payload cap and
  reference-not-content discipline.

#### Reversal plan

The registry is declarative metadata — moving to runtime registration is additive. The
persist-then-publish coupling is localized in producer helpers; a durable-bus future would
replace publication mechanics while envelope, registry, and persisted schema stay intact.

#### Review conditions

- A multi-process architecture (ADR-012 review condition) forces re-deciding delivery.
- Volume 12 measurements attribute a hot-path budget miss to transactional event appends.
- Extension ecosystems demonstrate a genuine need for extension-declared event names.
