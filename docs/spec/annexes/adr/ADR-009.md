### ADR-009 — Plugin Mechanism: Subprocess with JSON-RPC 2.0 over stdio (Andromeda Runtime Protocol)

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Plugin Runtime, Tool Runtime, Extension SDK, Sandbox Engine
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation

#### Context

Plugins are Andromeda's mechanism for third-party extensions beyond what skills and MCP servers provide: custom tools, provider adapters, exporters, and commands (Volume 6). The extension ecosystem is explicitly polyglot — choosing Go for the core (ADR-001) must not force Go on extension authors. Plugins sit on a security boundary: third-party code executing on the user's machine, subject to the permission model and Sandbox Engine (Volume 9, ADR-021). Volume 0 fixes the internal protocol name: **Andromeda Runtime Protocol**.

#### Problem

Choose the mechanism by which the Plugin Runtime loads, isolates, communicates with, and supervises plugins: in-process loading, managed subprocesses, or sandboxed bytecode — and the wire protocol between host and plugin.

#### Forces and constraints

- Language-agnosticism: a plugin author MUST be able to use any language that can read stdin and write stdout.
- Fault isolation: a crashing plugin MUST NOT crash Andromeda (precedence item 1, safety).
- Security: plugin capabilities are mediated by the Permission Manager and Sandbox Engine; the mechanism must give the host a hard process boundary to police.
- Andromeda already speaks JSON-RPC 2.0 over stdio as an MCP client (ADR-010); every additional protocol grammar increases SDK, documentation, and testing surface.
- No Go-version coupling: the mechanism must not require plugin and host to share a toolchain or ABI.
- Local-first: no network listener requirements for local plugins.

#### Alternatives considered

| Criterion | Subprocess + JSON-RPC 2.0/stdio (selected) | hashicorp/go-plugin (gRPC) | WASM via wazero | Go `plugin` package |
|---|---|---|---|---|
| Language-agnostic | Yes: stdio + JSON | In principle; gRPC/protobuf toolchain required per language | Yes, if the language compiles to WASM/WASI | No — Go only |
| Toolchain coupling | None | protobuf codegen for every SDK | WASM target maturity varies by language | Exact Go version + flags match required |
| Fault isolation | OS process boundary | OS process boundary | In-process but memory-sandboxed | None — shared address space |
| Sandboxing hooks | Full OS-level process controls (ADR-021) | Same | Strong capability model, but host-mediated I/O only | None |
| Debuggability | Human-readable JSON frames, replayable | Binary protobuf frames | Guest debugging still rough | n/a |
| Consistency with existing stack | Same grammar as MCP stdio (ADR-010) and IPC (ADR-012) | New protocol stack | New runtime + new ABI design | n/a |
| Platform coverage | Everywhere Andromeda runs | Everywhere | Everywhere wazero runs | No Windows support; fragile on macOS |

- **Go `plugin` package — rejected outright.** It requires host and plugin built with the identical Go toolchain version and build flags, offers no fault isolation, excludes every other language, and does not support Windows. It fails four constraints simultaneously.
- **hashicorp/go-plugin** is production-proven for exactly this problem shape, but its cost lands on plugin authors: a gRPC/protobuf toolchain in every SDK language, binary frames that resist ad-hoc debugging, and Go-centric idioms. It would also give Andromeda two RPC grammars with no needed capability in exchange.
- **WASM via wazero** offers attractive long-term properties: in-process speed, a capability-based sandbox, no process management. It is not chosen now because real plugins need network, subprocess, and PTY access, and designing that host-function surface is premature before the plugin API stabilizes. It is retained as the v2 in-process isolation candidate, PENDING VALIDATION.

#### Decision

Plugins run as **subprocesses** managed by the Plugin Runtime, communicating over **JSON-RPC 2.0 on stdio**. This protocol is the **Andromeda Runtime Protocol** (ARP), whose message grammar, handshake (with protocol version negotiation), capability declaration, lifecycle states, and timeout/cancellation semantics are specified in Volume 6. Framing follows the MCP stdio transport conventions, keeping one mental model for implementers. The Extension SDK ships the Go reference implementation; the protocol is the contract, so any language can implement it. Plugin processes execute under Sandbox Engine policy (ADR-021) with permissions mediated per Volume 9. WASM-based in-process plugins via wazero are a v2 candidate, PENDING VALIDATION (see Review conditions).

#### Rationale

The subprocess model is the only alternative satisfying every hard constraint — language-agnosticism, fault isolation, sandboxability, zero toolchain coupling — using only mechanisms Andromeda already needs: process supervision (also required for MCP stdio servers) and the JSON-RPC grammar shared with ADR-010 and ADR-012. Extension authors who implement both MCP servers and plugins keep one mental model, and human-readable frames make plugins debuggable with a text editor and a pipe.

#### Positive consequences

- Any language with stdio can implement a plugin; the SDK is a convenience, not a requirement.
- Crash, hang, and resource-exhaustion containment at the OS process boundary; the Plugin Runtime supervises restarts and timeouts.
- One protocol grammar (JSON-RPC 2.0) across plugins, MCP, and external IPC (ADR-012).

#### Negative consequences

- Per-call serialization and process-boundary latency penalize chatty APIs — the Volume 6 API is designed coarse-grained accordingly.
- Process lifecycle management (spawn, health, restart, orphan cleanup) is real engineering owned by the Plugin Runtime.
- JSON encoding suits large binary artifacts poorly; Volume 6 defines out-of-band artifact passing (filesystem handles) rather than inline blobs.

#### Risks

- **Protocol versioning debt**: ARP evolves with the product; without discipline, plugin breakage follows. Mitigation: handshake version negotiation from day one; versioning policy tied to ADR-015.
- **Sandbox gaps**: a subprocess inherits ambient authority if launched naively. Mitigation: launch path goes exclusively through the Sandbox Engine (ADR-021); direct `exec` of plugins outside it is a defect.
- **PENDING VALIDATION — wazero/WASM as v2 in-process plugin isolation**: feasibility for real plugin workloads (network, subprocess, PTY host functions) is unvalidated; tracked in the open questions register and in Review conditions below.

#### Reversal plan

The ARP contract, not the transport, is the stable surface. Migrating transports (sockets for performance, WASM host calls in v2) preserves the JSON-RPC grammar and Volume 6 lifecycle semantics: the Plugin Runtime transport layer and SDK connection code change; plugin business logic does not. A full reversal to hashicorp/go-plugin — plausible only after a demonstrated protocol-performance failure — would keep the subprocess supervision layer and swap the wire format: a contained change in the Plugin Runtime plus SDK regeneration.

#### Review conditions

- Volume 12 benchmarks show ARP round-trip overhead breaching tool-invocation latency targets for representative plugins.
- PENDING VALIDATION: before any v2 commitment, a spike must validate wazero-based WASM plugins (host-function surface for network/subprocess/PTY, guest-language coverage, performance) — outcome recorded in a superseding or amending ADR.
- The MCP ecosystem ships a standard plugin-style extension protocol that would let Andromeda converge ARP with it.
- More than two SDK languages accumulate hand-maintained protocol code, indicating a schema-driven codegen investment is due.
