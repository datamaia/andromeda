### ADR-007 — Persistence: SQLite via modernc.org/sqlite

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Persistence Layer, Memory Manager, Indexing Engine, Audit Log
- Related requirements: Volume 1 product objectives; formal requirement links are added at consolidation

#### Context

Andromeda persists sessions, runs, memory records, indexes, audit records, cost records, and assorted runtime state locally (Volume 10 owns storage; Volume 7 memory/indexing; Volume 9 audit). The store must be embedded (local-first, no server process), transactional (crash recovery is precedence item 3, data integrity), queryable (memory retrieval, index queries, audit inspection), and portable across macOS/Linux now and Windows later. ADR-001 makes CGO-free cross-compilation a load-bearing property of the release pipeline (ADR-013).

Web-verified facts (2026-07-11): `modernc.org/sqlite` is an active, CGO-free, pure-Go SQLite driver, BSD-3-Clause, v1.53.0 (2026-06-21), embedding SQLite 3.53.2. WAL is enabled via the DSN pragma `?_pragma=journal_mode(WAL)`. Its throughput is roughly 75% of the CGO driver `mattn/go-sqlite3`, workload-dependent. A known footgun: `modernc.org/libc` MUST be pinned to the exact version in the driver's `go.mod` (upstream issue #177).

#### Problem

Choose the storage engine and Go driver for the Persistence Layer such that durability, query capability, and the CGO-free portability guarantee hold simultaneously.

#### Forces and constraints

- Local-first: no external database process; state lives under the Andromeda directories (ADR-022).
- Crash safety and recoverability MUST be provided by the storage engine (transactions, journaling), not reinvented above it.
- The Memory Manager and Indexing Engine need relational queries and full-text search; the Audit Log needs append-heavy writes with integrity.
- The release matrix cross-compiles from one CI host (ADR-013); a CGO dependency would require per-target C toolchains and break that.
- Concurrency: TUI reads while the Runtime writes — WAL-mode concurrent readers are required.
- Performance targets are owned by Volume 12; the driver choice must be benchmark-gated, not assumed.

#### Alternatives considered

| Criterion | modernc.org/sqlite (selected) | mattn/go-sqlite3 | bbolt | Flat files (JSON/JSONL) |
|---|---|---|---|---|
| Engine | SQLite 3.53.2, pure Go translation | SQLite via CGO | Native Go B+tree KV | None |
| CGO-free cross-compilation | Yes | No — C toolchain per target | Yes | Yes |
| SQL, joins, FTS | Full SQLite feature set | Full SQLite feature set | None (KV only) | None |
| Transactions/crash recovery | SQLite WAL | SQLite WAL | Single-writer transactions | Manual, error-prone |
| Relative performance | ~75% of mattn (verified, workload-dependent) | Baseline | High for KV patterns | Poor at scale |
| External inspectability | Any SQLite tooling | Any SQLite tooling | Custom tooling | Text editors |
| Ecosystem status (verified) | Active, v1.53.0 (2026-06-21) | Not re-verified for this decision | Not re-verified | n/a |

- **mattn/go-sqlite3** is the performance baseline but reintroduces CGO: cross-compilation then needs a C cross-toolchain per OS/arch pair, complicates the goreleaser matrix, slows builds, and undermines the single-binary reproducibility story. That trade is only worth making if benchmarks prove the pure-Go driver inadequate — hence the gate below.
- **bbolt** is a solid embedded KV store, but Andromeda's access patterns (relational queries over sessions/runs, FTS over memory and indexes, ad-hoc audit queries) would force us to build query and index layers by hand on top of it.
- **Flat files** fail transactional integrity and query requirements immediately; they remain appropriate only for exports and artifacts, not for state.

#### Decision

The Persistence Layer MUST use SQLite as the storage engine, accessed through `modernc.org/sqlite`, pinned. Binding rules:

1. WAL mode MUST be enabled via the DSN pragma `?_pragma=journal_mode(WAL)`.
2. `modernc.org/libc` MUST be pinned in `go.mod` to exactly the version required by the driver's own `go.mod` (upstream issue #177); CI MUST fail on divergence.
3. All schema access goes through the Persistence Layer; no component embeds SQL against the store directly (single-home: Volume 10).
4. Driver-specific behavior (DSN construction, pragmas) is confined to one package so the driver is swappable.

Adoption is benchmark-gated: Volume 12 defines representative workloads (session write throughput, memory retrieval latency, index build time), and the ~75% relative-performance figure is accepted only while those targets are met.

#### Rationale

SQLite is the only candidate satisfying transactional durability, SQL/FTS query power, zero-process operation, and universal tooling at once; the real decision is the driver. `modernc.org/sqlite` preserves the CGO-free property that the whole distribution strategy (ADR-001, ADR-013) rests on, at a verified, bounded, workload-dependent performance cost that the I/O-bound product profile can absorb — and that we verify rather than assume, via the benchmark gate. Files remain plain SQLite databases, inspectable and recoverable with standard tooling regardless of driver.

#### Positive consequences

- Cross-compilation for the full release matrix from one CI host, no C toolchains.
- Crash recovery, atomic transactions, concurrent readers under WAL — inherited, not reimplemented.
- State files are standard SQLite: debuggable, backup-friendly, migration-friendly.
- Driver isolation keeps a future swap to the CGO driver (or successors) mechanical.

#### Negative consequences

- Verified throughput penalty (~75% of the CGO driver, workload-dependent) — accepted under the benchmark gate.
- The pure-Go translation layer is a large, unusual dependency; debugging driver-level issues is harder than with the canonical C library.
- The libc pin adds dependency-management friction on every driver upgrade.

#### Risks

- **Benchmark failure**: representative workloads miss Volume 12 targets due to the driver. Detection: phase-exit benchmark runs. Mitigation: the reversal plan's driver swap.
- **libc version drift** causing subtle runtime faults (issue #177 class). Detection: CI pin check (Decision rule 2). Mitigation: automated dependency-update tooling treats driver+libc as one atomic pair.
- **Upstream stall** of the translation project. Detection: release monitoring. Mitigation: driver isolation package; SQLite file format is engine-portable.

#### Reversal plan

Because the on-disk format is standard SQLite, reversal never migrates data — only the driver. Swap path: introduce `mattn/go-sqlite3` behind the same Persistence Layer package (optionally as a build-tag variant first), accept the CGO consequences in the release pipeline (per-target toolchains in ADR-013's workflow), and re-run the Volume 12 benchmark suite to confirm the gain justifies the portability cost. A partial reversal (CGO driver only on platforms where native builds are cheap) is possible for the same reason.

#### Review conditions

- Any Volume 12 storage benchmark misses its target and profiling attributes the miss to the driver — this is the standing benchmark-gated review condition.
- `modernc.org/sqlite` misses two consecutive upstream SQLite version updates or goes 12 months without a release.
- The libc pinning requirement (issue #177) is resolved upstream, allowing rule 2 to be relaxed by a superseding ADR.
- Windows-phase entry: re-validate driver behavior and WAL semantics on Windows before committing the platform matrix.
