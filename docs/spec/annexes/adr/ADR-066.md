### ADR-066 — Offline behavior: passive detection, fail-fast classification, resolved-address locality

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Provider Layer, Authentication Layer, Agent Engine
- Related requirements: FR-PROV-084, FR-PROV-085; PRD-003; SM-05

#### Context

Local First is a product identity property: the offline guarantee list (Volume 1) is
contractual and SM-05 additionally requires zero observed network-access attempts under the
offline condition. The provider layer is the component most tempted to probe connectivity
(health checks, reachability tests, retry loops). Volume 2 freezes the Provider connection
states; the loopback question (what counts as "local") decides which endpoints receive
offline guarantees and plain-HTTP allowance.

#### Problem

How does the provider layer know it is offline, what does it do about it, and what exactly is
a "local" endpoint — such that offline machines generate zero provider-layer traffic, cloud
timeouts never block local work, and loopback privileges cannot be acquired by naming tricks?

#### Forces and constraints

- SM-05: 0 network-access attempts observed offline — active probing is disqualified by the
  metric itself.
- Retry policies for transient errors must not turn connectivity loss into retry storms.
- Captive portals make "reachability" checks lie; only real requests fail honestly.
- Plain HTTP is acceptable only where the OS guarantees locality (loopback), not where a
  name claims it.

#### Alternatives considered

1. **Active connectivity probing** (periodic health pings, portal detection) — rejected:
   violates the zero-attempts metric, generates beaconing from otherwise-quiet machines, and
   portals defeat it anyway.
2. **OS network-status APIs** as an offline oracle — rejected: platform-divergent semantics,
   wrong answers for VPN/split-tunnel setups, and an answer that real requests must verify
   anyway.
3. **Hostname-based locality** (`localhost`, configured allowlist) — rejected: DNS can bind
   any name to any address; locality by name is spoofable configuration.
4. **Treating LAN endpoints as local** — rejected: LAN reachability without Internet is
   real, but Andromeda cannot verify network topology; guarantees limited to what the OS
   enforces (loopback).

#### Decision

(a) **No probing, ever**: connectivity state is inferred passively from real request
failures; nothing polls, pings, or beacons — including at startup. (b) **Fail fast,
classify**: connectivity-class failures surface within normal timeouts, are exempt from
transient-retry policy, and move the Provider to `unavailable`; recovery happens on next
real use or explicit re-verification only. (c) **Locality by resolved address**: an endpoint
is local exactly when it resolves to loopback (`127.0.0.0/8`, `::1`) or a Unix domain
socket; only local endpoints get offline guarantees, plain-HTTP allowance, and proxy bypass.

#### Rationale

Passive detection is the only strategy consistent with the zero-attempts metric and honest
under portals; fail-fast-without-retry keeps offline latency bounded and prevents storms;
resolved-address locality delegates the trust decision to the OS instead of to strings.

#### Positive consequences

- Offline machines are network-silent by construction; SM-05 is testable directly.
- Local sessions never wait on cloud timeouts or retries.
- Locality privileges are unspoofable via DNS or configuration names.

#### Negative consequences

- Providers stay `unavailable` until actually used again — status displays can lag reality.
- Self-hosted LAN servers get remote treatment (TLS required) even on trusted networks.

#### Risks

- Users may misread lagging `unavailable` status as a defect; mitigation: status output
  states "until next use or explicit verify" and offers the verify command.

#### Reversal plan

Detection strategy is internal to the Provider Layer router and Authentication Layer; adding
an opt-in active health check later would be an additive, consent-gated feature with its own
metric exemption — no contract or state changes.

#### Review conditions

- If Volume 12's reliability work defines a user-visible health dashboard needing fresher
  state, revisit opt-in verification scheduling.
- Revisit the LAN stance if a verifiable local-network attestation mechanism becomes part of
  the PAL.
