### ADR-049 — Specification-driven development as a fixed fourteen-stage pipeline with bounded feedback loops

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team (Volume 4 authoring)
- Components: Workflow Engine, Agent Engine, Permission Manager
- Related requirements: FR-WF-001; RISK-WF-001, RISK-WF-003

#### Context

The project brief mandates a specification-driven development workflow with fourteen named
stages (intake through release-preparation). PRD-012 commits to process-as-declared-workflow.
The open questions were structural: is the stage set an ordered pipeline or a free graph, are
stages skippable, and where do feedback loops and human gates sit?

#### Problem

How are the fourteen mandated stages arranged and tailored so that runs are comparable and
auditable across projects, while small changes are not crushed by ceremony?

#### Forces and constraints

- The stage list is a provided requirement; the arrangement is a design decision.
- Auditability wants a fixed, recognizable shape; practicality wants tailoring.
- Unbounded loops (review → implementation → review …) are a runaway-cost path for
  autonomous agents.
- Gate fatigue (RISK-WF-003) argues against gating every stage by default.

#### Alternatives considered

1. **Free-form stage graph per project** — maximum flexibility; rejected: runs stop being
   comparable, "SDD" stops meaning anything specific, and audit tooling cannot rely on any
   shape.
2. **Skippable stages via configuration flags** — rejected: silent stage skipping erodes
   exactly the guarantees the workflow exists to provide; absence of a stage becomes
   invisible in audits.
3. **Fixed pipeline, no loops** — rejected: real verification failures need a return path
   to implementation; without one, every test failure kills the run.

#### Decision

`spec-driven-dev` is a **fixed, ordered fourteen-stage pipeline**. Exactly four feedback
loops exist — validation, testing, review, and security-review, each routing back to
implementation — every loop bounded by a declared `max_loop_iterations`. Stages are never
skipped at run time; tailoring happens through **gate profiles** (`strict`, `standard`,
`minimal` — selecting which stage gates require human approval) and through new definition
versions for teams that need a genuinely different process.

#### Rationale

A fixed shape makes every SDD run navigable by the same mental model and the same tooling;
bounded loops make worst-case cost computable before a run starts; gate profiles tune
ceremony without touching auditability — an ungated stage still runs and still records its
artifacts.

#### Positive consequences

- Runs are comparable across projects and time; audit tooling relies on a stable shape.
- Loop budgets cap autonomous spend on failing verification cycles.
- Gate tailoring is explicit configuration, recorded per run.

#### Negative consequences

- Trivial changes pay the full pipeline's fixed overhead when run through SDD (RISK-WF-001);
  the answer is authoring lighter workflows, not bending this one.
- Processes that genuinely differ (e.g., research spikes) need their own definitions.

#### Risks

- Users fork the builtin definition and drift; mitigation: versioned definitions with
  provenance make drift visible; the format (ADR-048) keeps forks structurally valid.
- Loop budgets set too low frustrate legitimate iteration; mitigation: budgets are
  per-definition values, adjustable by version.

#### Reversal plan

The stage set lives in one builtin definition; restructuring ships as `spec-driven-dev`
version 2.x under NFR-WF-001's compatibility regime, with the old version still resolvable
for reproducibility (INV-WFD-04).

#### Review conditions

- Beta adoption metrics (completion/abandonment per stage) trigger a stage-set review if
  abandonment concentrates in specific stages.
- Revisit the four-loop topology if recorded runs show recurring need for a
  requirements-level loop.
