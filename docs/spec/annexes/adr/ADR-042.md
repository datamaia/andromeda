### ADR-042 — Task retry policy: bounded backoff with side-effect gating

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Execution Engine, Tool Runtime, Permission Manager
- Related requirements: FR-AGT-011; PRD-005, PRD-010

#### Context

Task attempts fail transiently (rate limits, network, busy locks) and deterministically
(bad inputs, denied permissions). ADR-016 makes retryability a per-error envelope fact.
Volume 2 gives Tasks an `attempt` counter and full side-effect attribution (File Change,
Command Execution, Patch records). Recovery semantics already forbid assuming interrupted
work complete (PRD-010).

#### Problem

When may the Execution Engine retry a failed task attempt automatically, and with what
schedule?

#### Forces and constraints

- Transient failures should not kill runs (reliability, SM-10 adjacency).
- Retrying past a landed side effect duplicates effects: double commits, re-applied
  patches — a data-integrity violation (precedence order item 3).
- Whether an effect landed is knowable from attribution records, which persist before
  classification runs (INV-TASK-05).
- Unbounded or unjittered retries amplify provider pressure.

#### Alternatives considered

1. **Retry everything retryable** — rejected: ignores landed side effects; the duplication
   failure mode is exactly what the attribution system exists to prevent.
2. **Never auto-retry** — rejected: pushes every transient blip to the user or the model,
   wasting budgets and attention; models handle retry decisions worse than a classifier.
3. **Idempotency-declared retries only** (tools declare idempotency; retry only idempotent
   tools) — insufficient alone: task attempts span multiple invocations; adopted as an
   input signal rather than the whole rule.
4. **Retryable-and-effect-free automatic retry with bounded jittered backoff; everything
   else gated** — chosen.

#### Decision

Automatic retry only when the attempt's terminal error is envelope-retryable **and** the
attempt recorded zero side effects. Schedule: exponential backoff — base delay × multiplier
^ (attempt − 1), capped, ±20% jitter, bounded attempts (`agent.execution.retry.*`).
Side-effect-bearing or non-retryable failures require plan revision or explicit user
confirmation; confirmations are recorded as Approvals. Unclassifiable errors are treated as
non-retryable (fail closed).

#### Rationale

The two conditions are independently checkable from persisted facts at classification time,
making retry decisions auditable and replayable; gating on recorded effects encodes the
integrity precedence directly instead of trusting tool idempotency claims.

#### Positive consequences

- Transient failures self-heal within bounds; duplication cannot happen silently.
- Retry decisions are reconstructable from the record stream (SM-12).

#### Negative consequences

- Conservative: an idempotent effectful attempt still gates, costing a confirmation.
- Classification waits for the attempt's batch commit (bounded, but on the failure path).

#### Risks

- Attribution gaps (an effect a tool failed to record) defeat the gate — mitigated by
  Volume 6 tool-contract attribution obligations and SM-13 orphan detection.

#### Reversal plan

The classifier is a pure function over envelope + attribution records; loosening or
tightening it is a configuration-shaped change plus this ADR's supersession.

#### Review conditions

Review when tool idempotency declarations (Volume 6) mature enough to justify auto-retrying
declared-idempotent effectful attempts, or if confirmation friction dominates field
feedback.
