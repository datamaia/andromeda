### ADR-088 — Chunking: deterministic structure-preferring windows with overlap

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Indexing Engine
- Related requirements: FR-IDX-002, FR-IDX-003, FR-IDX-004

#### Context

Semantic and lexical indexing (Volume 7 chapter 04) require splitting file content and
memory records into retrieval units. Chunk geometry drives retrieval precision, embedding
cost (ADR-020 stores one vector per chunk), incremental-update cost (only changed chunks
re-embed), and the usefulness of hit spans handed to the Context Manager. Language-aware
parsing (tree-sitter-class syntax chunking) exists in the ecosystem but adds a large
dependency surface with per-language grammars.

#### Problem

Define the chunking strategy: unit sizes, boundary selection, overlap, and determinism
guarantees — balancing retrieval quality against dependency weight, embedding spend, and the
incremental-update property that unchanged content keeps unchanged chunk hashes.

#### Forces and constraints

- Determinism is mandatory: chunk sets feed content hashes (INV-EMB-02 staleness) and
  cross-platform identical indexes (Volume 13 golden fixtures).
- Chunks bound embedding cost: geometry changes multiply spend across every workspace.
- Boundaries aligned to human/model reading structure (headings, blank lines, indentation)
  measurably beat arbitrary windows for code-and-docs corpora retrieval; full syntax parsing
  is heavier than its marginal gain justifies at MVP.
- A changed region should perturb only nearby chunks (incremental friendliness).

#### Alternatives considered

1. **Fixed-size windows (bytes or tokens) with no structure awareness** — rejected: splits
   mid-function and mid-sentence, degrading both embedding quality and the readability of
   hit spans; cheapest but visibly worst.
2. **Full syntax-aware chunking via language parsers** — deferred: a per-language grammar
   dependency tree (and its update treadmill) for a quality delta the MVP corpus does not
   need; revisit as an extension surface (indexers are extensible per Principle 6).
3. **Structure-preferring windowed chunking** (selected): token-budgeted windows that snap
   to a fixed boundary preference order.

#### Decision

Chunking is windowed with structural boundary preference, parameterized by `[index.chunk]`:
target 400 estimated tokens (ADR-087 heuristic), hard cap 512, trailing overlap 40.
Boundary preference order: Markdown/heading boundaries, blank lines, top-level indentation
decreases, line boundaries; never inside a line. Each chunk carries a stable locator (path,
byte offsets, line span) and a content hash. Memory records chunk as whole records (capped
at ingestion below the chunk maximum). Oversized single lines become one chunk, embedded
truncated at the cap with the truncation recorded. The algorithm is deterministic over
bytes and configuration, with no per-language dependencies.

#### Rationale

Structure preference captures most of the retrieval-quality benefit of syntax awareness at
none of the dependency cost; the hard cap bounds embedding input; overlap preserves
cross-boundary continuity for similarity search; line-integrity keeps hit spans directly
renderable in diffs and editors-of-the-terminal contexts (Volume 8 views).

#### Positive consequences

- Deterministic, dependency-free, platform-identical chunking.
- Localized re-chunking under edits keeps incremental embedding spend proportional to
  change size.
- Locators double as `source_ref` values for Embedding rows (Volume 2) without extra
  mapping.

#### Negative consequences

- Boundary heuristics are approximations: some code constructs still split awkwardly.
- The token budget uses estimated tokens for sizing (ADR-087), so chunk token counts vary
  slightly across true tokenizers.

#### Risks

- Pathological content (generated one-liners, huge tables) produces low-quality chunks —
  bounded by the cap-and-record rule and the per-file byte cap (FR-IDX-001 scope rules).

#### Reversal plan

Chunking parameters and algorithm version are part of the index layout: changing them bumps
`index_schema_version`, which triggers rebuild-not-migrate (Volume 2 rule). A syntax-aware
chunker can ship later as an alternative implementation behind the same chunk contract
(locator + hash), per-index selected, without disturbing existing indexes until rebuild.

#### Review conditions

- Retrieval-quality golden suites (Volume 13) attribute systematic misses to boundary
  quality.
- An indexer-extension implementation of syntax-aware chunking matures under Principle 6's
  extension surface and benchmarks favorably on the reference repository.
