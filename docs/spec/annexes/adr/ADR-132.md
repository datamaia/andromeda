### ADR-132 — Configuration includes: explicit, bounded, containment-checked, local-only

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Configuration Manager, PAL
- Related requirements: FR-CFG-006, FR-CFG-002

#### Context

The brief mandates configuration includes. Includes act on trust-boundary input: a
workspace file is often committed to version control and edited by multiple people, and an
include directive is a request to read another file at load time. Chapter 01 of Volume 10
fixes layer semantics; this ADR fixes the include design inside a layer.

#### Problem

Provide file composition (shared team policy, split provider fleets) without creating an
exfiltration primitive, an unbounded load graph, or a second precedence system.

#### Forces and constraints

- Attribution must survive composition: every key must trace to a concrete file and line.
- A committed workspace file must not be able to read arbitrary paths on a teammate's
  machine (`include = ["../../home/user/x.toml"]`).
- Cycles and deep graphs must fail loudly, not hang or explode memory.
- Includes must not smuggle in sub-layer semantics that compete with FR-CFG-001.

#### Alternatives considered

1. **No includes.** Rejected: the brief mandates them, and the observed workaround —
   generating config files — destroys attribution and auditability entirely.
2. **Glob includes** (`include = ["conf.d/*.toml"]`). Rejected for now: load order would
   depend on directory listing semantics; explicit ordered lists keep merge order
   reviewable. A globbing extension would be additive later.
3. **Remote includes (URLs).** Rejected outright: network at resolution time violates
   Principle 3 (local first) and turns configuration loading into a supply-chain surface.
4. **Include-as-layer** (each include its own precedence layer). Rejected: precedence must
   have exactly ten documented layers (ADR-130); composition happens inside a layer.

#### Decision

`include` is a reserved root key: an ordered array of relative paths, resolved against the
including file's directory. Includes merge in listed order beneath the including file's
own body (the physical layer file always wins within its layer). Nesting ≤ 8 deep, ≤ 64
files per layer, ancestor cycles are errors, and every resolved path must stay inside the
containment root — the global configuration directory for the global file, the workspace
root for workspace and project files — with symlinks resolved before the check. Included
files cannot set `config_version` or `default_profile`. No URLs. All violations are
E-CFG-007 sub-classes.

#### Rationale

Explicit ordered lists preserve reviewability; containment converts the exfiltration
primitive into a validation error; the bounds make the worst-case load graph small and
diagnosable; and keeping composition inside a layer preserves the ten-layer contract.

#### Positive consequences

- Team policy files compose without file generation.
- Attribution names the exact included file and line for every key.
- The security review surface of includes is four rules (order, bounds, containment,
  reserved keys).

#### Negative consequences

- No globbing means fleet-style `conf.d/` layouts need explicit lists.
- Containment blocks legitimate cross-tree sharing (e.g. one file shared by two
  workspaces); the supported pattern is duplication or a global-layer include.

#### Risks

- Users symlink files into the workspace to defeat containment — accepted: the symlink
  itself is a deliberate local action, and resolution-before-check still bounds what a
  *committed* file can reach on other machines.
- Include-heavy configurations approach the bounds; detection via E-CFG-007 frequency,
  remedy is raising bounds additively.

#### Reversal plan

Includes are resolved entirely inside the Configuration Manager's loader; removing or
extending the feature (globs, higher bounds) changes one loader stage. Removal would be a
breaking change (major release) with the migration being manual inlining, which the
migration surface could automate.

#### Review conditions

- Recurrent demand for glob includes or shared cross-workspace fragments.
- Any demonstrated bypass of the containment rule.
- The bounds (8 deep / 64 files) proving too small in real fleets.
