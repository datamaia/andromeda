### ADR-032 — Headless server mode: included, same binary over the IPC surface, Beta phase

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Runtime, CLI, Permission Manager, Policy Engine, Event Bus, PAL
- Related requirements: FR-ARCH-007, FR-ARCH-008, PRD-009; Volume 8 command grammar (naming)

#### Context

PRD-009 mandates human/automation parity: every interactive capability available
non-interactively, with structured output, deterministic exit codes, and policy-resolved
permissions. ADR-012 gives every running instance a Unix-socket JSON-RPC control surface.
Two automation shapes exist: one-shot non-interactive CLI runs (spawn, execute, exit — the
CI shape, UC-07), and a *persistent* instance that automation drives over time — editor
integrations, long-lived agent sessions addressed by scripts, remote operation over SSH.
The one-shot shape is uncontroversial and owned by Volume 8. The persistent shape — a
"headless server mode" — needs an explicit inclusion, shape, and phase decision, because it
smells like a daemon, and Andromeda has committed to running no daemons (ADR-012 context).

#### Problem

Decide whether Andromeda includes a persistent headless mode; if so, whether it is the same
binary and process model or a separate server build; how it is driven; and in which phase it
ships.

#### Forces and constraints

- PRD-009 parity: a persistent mode must add no capability and lose none except rendering,
  or parity forks.
- No daemons: nothing self-installs, self-starts, or outlives its invoker (local-first
  operational posture, ADR-012).
- The IPC surface already exists for *every* instance; a separate server build would create a
  second security surface and a second test matrix.
- Unattended permission semantics are already settled: policy-resolved, deny-on-unresolved,
  never a hanging prompt (Volume 1 tension rule 2).
- MVP scope pressure is real (RISK-PRD-004): the MVP automation commitments are the
  one-shot CLI and CI journeys, not persistent instances.

#### Alternatives considered

1. **Same binary, headless operating mode over the existing IPC surface, Beta (selected).**
2. **No persistent mode — one-shot CLI only.** Simplest, but editor integrations and
   long-lived remote sessions would each pay full startup and lose session continuity per
   call; automation would inevitably fake persistence with expect-style TUI scripting — the
   worst of all surfaces. Rejected.
3. **Separate server binary/build (an `andromeda-server`).** A second deliverable, second
   hardening target, second contract surface; violates the single-binary identity (PRD-011)
   for no capability gain — the runtime is identical either way. Rejected.
4. **Persistent mode at MVP.** Parity pressure argues for it, but the MVP exit criteria
   (Volume 1) do not require it, and shipping it early would freeze the IPC method
   vocabulary (Volumes 8/10) before those volumes stabilize it. Rejected on sequencing, not
   on substance.

#### Decision

Andromeda includes a **persistent headless operating mode** with these properties:

1. **Same binary, same process model**: a headless instance is the standard process with
   drivers limited to the IPC surface; no TTY required; started and owned by its invoker
   (shell, service manager, editor, CI step). Andromeda itself never installs or registers it
   as a system service.
2. **Driven exclusively through the ADR-012 IPC surface** (FR-ARCH-007 rules: peer
   verification, versioned handshake, full permission mediation, redacted event bridging).
3. **Permissions are policy-only** in this mode: no prompt path exists; unresolved
   permissions deny and record (PRD-009).
4. **Phase: Beta.** The one-shot non-interactive CLI remains the MVP automation surface
   (Volume 8); the persistent mode ships at Beta once the IPC method vocabulary (Volumes 8
   and 10) is specified. FR-ARCH-008 binds the behavioral requirement; command naming is
   Volume 8's.

#### Rationale

The persistent mode falls out of decisions already made — every instance has the socket, the
Runtime already serves multiple drivers, unattended permission semantics already exist. Making
it an *operating mode* rather than a product variant keeps one binary, one security surface,
one test matrix, and exact PRD-009 parity by construction. Beta sequencing respects both MVP
scope discipline and the dependency on Volumes 8/10 for the method vocabulary the mode is
driven through.

#### Positive consequences

- Editor and automation integrations get session continuity without TUI scraping.
- One hardening surface: everything learned securing the IPC socket serves both modes.
- Parity is structural: the mode cannot drift from interactive behavior because it *is* the
  same runtime behind the same API.

#### Negative consequences

- A long-lived instance changes the resource profile (idle footprint, log growth); Volume 12
  must budget it explicitly.
- Lifecycle responsibility shifts to invokers (service managers, editors); documentation must
  be explicit that Andromeda will not manage its own persistence.
- The IPC protocol becomes load-bearing for third parties earlier than it otherwise would,
  raising its compatibility stakes (chapter 09 regime applies).

#### Risks

- Scope creep toward daemon features (auto-start, multi-user serving); mitigation: this ADR
  explicitly excludes them — any such feature requires a superseding ADR.
- Idle-instance resource regressions going unnoticed; mitigation: Volume 12 budgets an idle
  headless profile and CI tracks it.
- Divergence between CLI semantics and IPC method semantics; mitigation: both drive the same
  Runtime API (FR-ARCH-002), and parity record-comparison tests gate releases (FR-ARCH-008
  acceptance criteria).

#### Reversal plan

The mode is a thin driver configuration over existing surfaces. Withdrawing it (or deferring
past Beta) removes the entry point and its documentation while the IPC surface — which
ADR-012 requires regardless — remains for one-shot tooling. No engine, adapter, or contract
changes would be required in either direction.

#### Review conditions

- Volume 8/10 IPC method vocabulary lands: confirm Beta sequencing still holds at that phase
  gate.
- Demand emerges for multi-client concurrent driving of one instance beyond what the session
  model supports — re-evaluate the concurrency contract of the mode.
- Volume 12 idle-footprint measurements breach budgets — revisit what stays resident in
  headless instances.
