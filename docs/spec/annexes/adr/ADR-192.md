### ADR-192 — Retained-version rollback and binary–database pairing

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team
- Components: Updater, Persistence Layer, Platform Abstraction Layer (PAL)
- Related requirements: FR-REL-006, FR-REL-008, FR-REL-016

#### Context

SM-19 commits to rollback in ≤ 30 s p95, executable offline, from locally retained
artifacts. The PAL Updater surface provides atomic replace-or-restore with a retained
previous version (Volume 3, chapter 07); ADR-029 makes database migrations forward-only with
pre-migration backups and clean refusal of future schemas, and states the product downgrade
path as "binary + database backup revert as a pair". Volume 8 ships `andromeda update
rollback` at MVP. The open decisions: how many versions are retained and where, what
rollback restores by itself, and how the binary–database pairing is surfaced so rollback
never silently strands data.

#### Problem

Fix the retention policy (count, location, lifecycle) and the rollback semantics —
specifically what happens when the version being rolled back migrated a database schema
after it was applied.

#### Forces and constraints

- Offline rollback (SM-19) forbids re-download: everything needed must already be on disk.
- Retained binaries cost tens of megabytes each; unbounded retention erodes the disk
  budgets Volume 12 sets.
- ADR-029 rule 5: an older binary refuses a newer database cleanly. A binary-only rollback
  across a migration boundary therefore produces a *working but refusing* installation
  unless the database question is answered at rollback time.
- Restoring a pre-migration database backup discards everything written after the
  migration — an irreversible, user-consequential act that must never be automatic.
- The Update process machine has exactly one rollback-shaped terminal state
  (`rolled_back`); manual rollback and automatic apply-failure restore should share
  mechanics, not diverge.

#### Alternatives considered

1. **Retain all previous versions** — rejected: unbounded disk growth for a capability
   (multi-step rollback) with no requirement behind it; air-gapped reinstall covers the
   long-tail case.
2. **Automatic paired restore (binary + database) on rollback** — rejected: silently
   discarding post-migration writes violates the data-integrity precedence; the *user*
   owns that trade-off.
3. **Block rollback across migration boundaries entirely** — rejected: it turns a
   recoverable situation into a dead end; ADR-029's backups exist precisely to make the
   paired revert possible.
4. **Retain two or more versions by default** — rejected as default (kept as
   configuration): the dominant rollback scenario is "the update I just applied is bad",
   which one retained version covers; `keep_versions` remains configurable for cautious
   fleets.

#### Decision

1. The Updater retains the previously installed binary (and its verification metadata) in
   the Updater area of the platform data directory (ADR-022 resolution). The default
   retention count is 1 (`[update].keep_versions`); retained versions beyond the count are
   deleted oldest-first after a successful apply.
2. `Rollback` (UpdaterPort) restores the retained binary via the PAL atomic
   replace-or-restore surface, offline, and re-verifies its recorded checksum before the
   swap. No network, no re-download, ever.
3. Rollback is **binary-only by default**. Before swapping, the Updater compares the
   schema versions the retained binary supports (recorded at retention time) against the
   current databases. If any database is newer, the rollback proceeds only after an
   explicit, separate confirmation that names the consequence: the restored binary will
   refuse that database (ADR-029 rule 5) until the user either restores the pre-migration
   backup (documented procedure, data written since the migration is lost) or re-applies
   the newer product version.
4. Automatic restore after a failed `Apply` uses the same retained artifact and the same
   PAL surface; it lands the Update process in `rolled_back` without the schema dialogue
   (a failed apply cannot have migrated anything: migrations run at first start of the new
   version, not during apply).
5. Every rollback — manual or automatic — appends to update history, emits
   `update.version.rolled_back`, and records the schema-pairing outcome.

#### Rationale

One retained version, one atomic PAL surface, and one recorded pairing decision give
rollback semantics that are exact and honest: the binary revert is always safe and offline;
the data revert is a distinct, named, consented act because it is the only part that can
destroy information. Recording supported schema versions at retention time makes the
boundary check a local table lookup — no version-knowledge heuristics at rollback time.

#### Positive consequences

- SM-19 is achievable by construction: local artifact, one rename-swap, one verification.
- No silent data loss path exists anywhere in update or rollback.
- Apply-failure recovery and manual rollback share one tested mechanism.
- Disk cost is bounded and configurable.

#### Negative consequences

- Rolling back across a migration boundary is a two-step experience (binary now, data
  decision explicitly) — deliberately less smooth than a silent paired restore.
- `keep_versions = 1` cannot skip back two versions; that path is air-gapped reinstall.
- Retention metadata (supported schema versions per retained binary) is one more record to
  keep consistent, covered by FR-REL-016's history invariants.

#### Risks

- Users may ignore the schema warning and file the ADR-029 refusal as a bug; mitigated by
  the refusal error naming the exact remedy and the rollback confirmation naming the exact
  consequence.
- Retained-artifact corruption discovered at rollback time; mitigated by checksum
  re-verification before swap and E-REL-006 guidance to air-gapped reinstall.

#### Reversal plan

Raising the default retention count or adding multi-step rollback is additive
(configuration default plus a selection argument on the existing port method is an additive
contract change). Adopting automatic paired restore would supersede decision 3 by a new ADR;
nothing in the retained layout precludes it, since ADR-029 backups are already taken before
every migration.

#### Review conditions

- SM-19 measurements miss target on reference hardware.
- Support data shows multi-version rollback demand or recurring schema-boundary confusion.
- The Windows phase, where locked running binaries change swap mechanics (PAL
  locked-file-aware strategy), lands.
