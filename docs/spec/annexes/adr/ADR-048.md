### ADR-048 — Workflow definition format: TOML-authored, compiled to canonical JSON

- Status: Accepted
- Date: 2026-07-11
- Deciders: Andromeda specification team (Volume 4 authoring)
- Components: Workflow Engine, Configuration Manager, Persistence Layer
- Related requirements: FR-WF-002; NFR-WF-001

#### Context

Workflow definitions are declared, versioned orchestrations (PRD-012) that humans author,
packages distribute, and the engine compiles into the Workflow entity's `definition` JSON
document (Volume 2 chapter 06). Volume 3 chapter 04 delegates the source-format decision to
Volume 4. The project already standardizes TOML for configuration (ADR-008), JSON Schema for
validation (ADR-024), and canonical JSON for persisted documents (Volume 2 chapter 10).

#### Problem

Which human-authored source format do workflow definitions use, and how does it relate to
the persisted canonical form, so that authoring is consistent with the rest of the product,
validation is mechanical, and the format can evolve as a public contract (SM-20)?

#### Forces and constraints

- One authoring language across the product lowers the learning cost; users already write
  `andromeda.toml`.
- The persisted form must be canonical JSON for hashing (`source_hash`), diffing, and
  schema validation (ADR-024).
- Package-distributed workflows need deterministic compilation: same source, same compiled
  document.
- The format is a public contract under SM-20 — its surface should be as small as possible.

#### Alternatives considered

1. **YAML sources** — familiar from CI systems, but adds a second configuration language,
   a large parser dependency, and YAML's well-known ambiguity traps (implicit typing,
   anchors); rejected.
2. **JSON sources** — canonical already, but hostile to hand-authoring (no comments);
   rejected as the authoring surface, retained as the compiled form.
3. **A custom DSL** — maximum expressiveness, maximum cost: parser, tooling, documentation,
   and a proprietary contract; rejected.

#### Decision

Workflow definitions are authored in **TOML** (parsed per ADR-008), compiled
deterministically into a versioned **canonical JSON document** (`schema_version` integer,
starting at 1) validated by JSON Schema (ADR-024). TOML is the only authored format at Beta.
The compiled schema is the SM-20 contract surface; the TOML surface maps 1:1 onto it.

#### Rationale

TOML keeps one authoring language product-wide; canonical JSON keeps one persisted/hashed
form; the 1:1 mapping means only one schema evolves. Determinism makes `source_hash`
meaningful for integrity and immutability (INV-WFD-02).

#### Positive consequences

- One parser stack, one validation stack, reused from ADR-008/ADR-024.
- Definitions diff, hash, and migrate like every other persisted document.
- Invalid-example conventions of Volume 0 apply directly (` ```toml invalid `).

#### Negative consequences

- Users arriving from YAML-based CI systems must translate; no YAML import exists at Beta.
- TOML's nesting verbosity makes deep step structures wordier than YAML equivalents.

#### Risks

- Format pressure to grow expressions/conditionals into a programming language; mitigation:
  machine-checkable criteria are a closed predicate set (FR-WF-002 validation), and
  computation belongs in `transform` steps.
- Divergence between TOML surface and compiled schema; mitigation: the compiler is
  generated/tested from one schema definition (golden round-trip tests).

#### Reversal plan

The compiled JSON document is the contract; adding a second authoring frontend (YAML import)
or replacing TOML is an additive frontend change that does not touch stored definitions,
validation, or execution.

#### Review conditions

- Revisit if Beta feedback shows definition authoring is a measured adoption blocker
  (RISK-WF-001 detection channels).
- Revisit the closed predicate set if ≥ 3 recorded workflow-author requests need dynamic
  criteria that predicates cannot express.
